# Propose Refactoring

Analyze the ish prototype codebase for Rust code smells and produce an RFP that documents the findings and proposes specific refactorings for human review.

**This skill makes no code changes.** It reads, analyzes, and writes one document: the RFP.

## When to Use

- The user wants a formal code quality review before starting a refactoring effort
- The user invokes `/propose-refactoring` (optionally with a crate or file path)

## Scope

Default scope: all `.rs` files under `proto/`, excluding `proto/target/`.

If the user provides a path argument, restrict analysis to that crate or file.

## Analysis Procedure

### Step 1 — Run the detection scripts

Run all four scripts from `.agents/skills/propose-refactoring/scripts/`. Capture their output for use in the RFP. Pass the scope directory as the first argument.

```bash
bash .agents/skills/propose-refactoring/scripts/long-functions.sh <scope>
bash .agents/skills/propose-refactoring/scripts/complexity-check.sh <scope>
bash .agents/skills/propose-refactoring/scripts/detect-duplicates.sh <scope>
bash .agents/skills/propose-refactoring/scripts/unwrap-check.sh <scope>
```

### Step 2 — Manual code review

Read the top offenders identified by the scripts. For each significant finding, read enough surrounding code to understand:

- What the function/module does
- Why it is long, complex, or duplicated (structural reason, not just size)
- What the natural refactoring would be

Rust-specific smells to look for beyond what the scripts detect:

| Smell | What to look for |
|-------|-----------------|
| God function | A single `fn` with multiple unrelated concerns: parsing, validation, side effects, and error mapping all together |
| Repeated `match` arms | Identical or near-identical `match` blocks on the same enum in different functions |
| Panic-on-error | `.unwrap()` / `.expect()` in non-test library code where `?` or `Result` propagation is possible |
| Clone chains | Sequences of `.clone()` calls that suggest ownership design could be improved |
| Leaky abstraction | `pub` items in a module that are only used by the module's own submodules |
| Implicit coupling | Two functions that always call each other, suggesting they should be one or should share a helper |
| Misplaced logic | Business logic in `Display` impls, or formatting logic in core interpreter functions |
| Large enum arms | `match` arms longer than ~20 lines, suggesting `Extract Method` (Rust: extract to a helper `fn`) |

### Step 3 — Prioritize findings

Group findings into three tiers:

- **High** — affects correctness risk, panic potential, or makes future work significantly harder
- **Medium** — reduces readability or makes the code harder to extend
- **Low** — style or minor structural improvements

Only include findings that are actionable in the prototype stage (no speculative "future-proofing" refactorings).

### Step 4 — Write the RFP

Save the RFP to `docs/project/rfp/refactoring-<scope-slug>.md` where `<scope-slug>` is derived from the scope (e.g., `refactoring-interpreter`, `refactoring-prototype`). If the RFP index at `docs/project/rfp/INDEX.md` exists, add a row for the new RFP.

Use this RFP structure:

```markdown
---
title: "RFP: Refactoring — <Scope>"
category: rfp
audience: [ai-dev, human-dev]
status: draft
last-verified: <date>
depends-on: []
---

## Refactoring Patterns Reference

The following Rust refactoring patterns apply to the findings above:

| Pattern | When to apply |
|---------|--------------|
| Extract function | Function body exceeds ~50 lines or has distinct phases |
| Consolidate match arms | Two or more `match` blocks on the same enum with identical arms |
| Replace unwrap with `?` | `.unwrap()` / `.expect()` in `fn` returning `Result` |
| Extract module | Group of tightly related items that currently live in a larger file |
| Introduce type alias | Repeated complex type signatures (e.g., `HashMap<String, Vec<Arc<Mutex<...>>>>`) |
| Newtype wrapper | Primitive obsession — bare `String` or `usize` used for distinct concepts |
```

### Step 5 — Report to the user

Tell the user:
- The path to the RFP file
- The count of findings by priority tier (High / Medium / Low)
- The single highest-priority finding, as a one-sentence summary
- That no code was changed

## Constraints

- **Do not modify any `.rs` files, `Cargo.toml`, or any file outside `docs/project/rfp/`.**
- Do not run `cargo clippy` or `cargo build` — read the source files directly.
- Do not propose speculative refactorings for hypothetical future requirements.
- Prototype-stage code: do not flag the absence of documentation comments as a smell unless the missing doc causes genuine confusion.
- If a large function is large because it is a large `match` on a large enum (and each arm is simple), note it but rate it Low unless the arms themselves are complex.

## Scripts Reference

| Script | Purpose |
|--------|---------|
| `long-functions.sh` | Lists `fn` bodies exceeding 50 lines, sorted by length |
| `complexity-check.sh` | Estimates cyclomatic complexity per file (if/match/loop/&&/\|\|) |
| `detect-duplicates.sh` | Finds non-trivial lines repeated across two or more files |
| `unwrap-check.sh` | Lists all `.unwrap()` and `.expect()` calls outside test modules |
