## Decision Register

All decisions made during design, consolidated here as the authoritative reference.

| # | Decision | Outcome |
|---|----------|---------|
| 1 | Scope of this audit | All 10 in-scope items plus generic types, type narrowing, literal types, intersection types, and custom type guards. Specifications of newly-in-scope type features must be refined first, with open questions resolved as decision points. |
| 2 | Should maturity.md and roadmap.md be corrected? | Yes. Also add maturity matrix to the authority order and ensure skills maintain it. |
| 3 | Error representation model | Errors are ordinary objects with an `Error` entry (requires `message: String`). `SystemError` extends `Error` and requires an error code. The language processor only throws `SystemError`s. `RuntimeError` struct is eliminated. `new_error` builtin is removed. |
| 4 | Throw restriction enforcement | Only objects with the `Error` entry can be thrown. Enforced via assurance ledger: `throw` audits its argument for the `Error` entry, auto-adding it if the object qualifies (has `message: String`), or raising a system error discrepancy if it doesn't. |
| 5 | Type annotation enforcement | Implement live-audit type checking for annotated variables and parameters, integrated with the assurance ledger (`types(live)` feature state). Build features in dependency order, interleaving as needed. |
| 6 | Naming conventions | Define ish naming conventions before renaming anything. Use a "no surprises" approach: research conventions in popular languages (especially Rust and TypeScript), then adopt a coherent set. Do not adopt piecemeal. |
| 7 | Assurance ledger runtime level | Foundation + queries (Alternative A). Clean separation between VM and ledger — VM instantiates ledger, feeds it states, calls it per statement. Ledger has stateless live-audit logic for easy unit testing. |
| 8 | `with` block tests | Yes, include in this work. |
| 9 | Error catalog approach | Audit all error paths (Alternative A). Structure documentation so agents keep the catalog up to date. |
| 10 | Implementation priority and phasing | Phased. Error-related items pushed to later phases since errors should use type and assurance ledger mechanisms. |
| 11 | Type features and the assurance ledger | Many features originally in types.md (e.g., mutability enforcement, null safety, overflow config) are actually assurance ledger features, not type system features. They should be relocated or cross-referenced appropriately. |
| 12 | Error spec visibility | Error handling needs its own entry in the spec index (currently absent). The error spec must be updated to incorporate assurance-ledger integration. |

### Feature 2: Naming Conventions for ish Code

#### Research: Conventions in Popular Languages

| Language | Variables/Functions | Types/Classes | Constants | Modules/Packages |
|----------|-------------------|---------------|-----------|-----------------|
| **Rust** | `snake_case` | `PascalCase` | `SCREAMING_SNAKE_CASE` | `snake_case` |
| **TypeScript** | `camelCase` | `PascalCase` | `SCREAMING_SNAKE_CASE` or `PascalCase` | `camelCase` (files vary) |
| **Python** | `snake_case` | `PascalCase` | `SCREAMING_SNAKE_CASE` | `snake_case` |
| **Go** | `camelCase` (unexported) / `PascalCase` (exported) | `PascalCase` | `PascalCase` or `camelCase` | `lowercase` |
| **C#** | `camelCase` (local) / `PascalCase` (public) | `PascalCase` | `PascalCase` | `PascalCase` |
| **Java** | `camelCase` | `PascalCase` | `SCREAMING_SNAKE_CASE` | `lower.dot.case` |
| **Kotlin** | `camelCase` | `PascalCase` | `SCREAMING_SNAKE_CASE` | `lowercase` |
| **Swift** | `camelCase` | `PascalCase` | `camelCase` (let) | `PascalCase` |

#### Analysis

There are two dominant schools:
- **snake_case functions** (Rust, Python): function/variable names use underscores
- **camelCase functions** (TypeScript, Java, Kotlin, Swift, Go, C#): function/variable names use camelCase

Both schools universally use **PascalCase for types**. Both universally use **SCREAMING_SNAKE_CASE for constants** (with minor exceptions).

The existing ish prototype uses `snake_case` for all builtins (`type_of`, `is_type`, `to_string`, `str_length`, etc.). Rust (the implementation language) uses `snake_case`. ish also has shell integration where `snake_case` is native.

However, TypeScript (a key inspiration) uses `camelCase`. Adopting `camelCase` would mean renaming all 45 existing builtins.

#### Issues to Watch Out For
- Choosing `camelCase` for functions means renaming all 45 builtins and updating 150+ acceptance tests.
- Choosing `snake_case` means the spec's `isType`/`validate` names must change.
- Mixed conventions (some `camelCase`, some `snake_case`) would violate the "no surprises" principle.
- Keywords (`for`, `while`, `if`, `fn`, `let`, `mut`) are already lowercase — consistent with both schools.

#### Critical Analysis

**Alternative A: snake_case for functions/variables (Rust/Python school).**
- Pros: Matches all 45 existing builtins. Matches shell conventions. Matches Rust (implementation language). No migration needed. Natural for ish's shell integration (shell commands are lowercase/underscored).
- Cons: Diverges from TypeScript convention. Less familiar to JavaScript/TypeScript developers.

**Alternative B: camelCase for functions/variables (TypeScript/Java school).**
- Pros: Familiar to the largest developer population (JS/TS/Java/C#/Kotlin/Swift). Matches the spec as written.
- Cons: Requires renaming all 45 builtins and updating all tests. Inconsistent with shell conventions. Inconsistent with Rust.

**Alternative C: camelCase for user code, snake_case for builtins/system functions.**
- Pros: Keeps existing builtins. User code looks TypeScript-like.
- Cons: Two different conventions in the same language — confusing. Violates "no surprises."

**Recommendation**: Alternative A (`snake_case`). The implementation, the shell environment, and all existing code already use it. The "no surprises" principle favors internal consistency over matching any one external language. The spec should be updated to match.

#### Proposed Conventions

| Category | Convention | Examples |
|----------|-----------|---------|
| Variables | `snake_case` | `my_value`, `user_name` |
| Functions | `snake_case` | `get_user`, `is_type`, `to_string` |
| Types | `PascalCase` | `String`, `List`, `MyObject`, `Error` |
| Constants | `SCREAMING_SNAKE_CASE` | `MAX_SIZE`, `DEFAULT_TIMEOUT` |
| Modules | `snake_case` | `file_utils`, `http_client` |
| Entry types | `PascalCase` | `Error`, `SystemError`, `Mutable` |
| Standards | `snake_case` | `streamlined`, `cautious`, `rigorous` |
| Keywords | `lowercase` | `fn`, `let`, `mut`, `if`, `for`, `while`, `throw`, `try`, `catch` |

#### Decisions

**Decision:** Which naming convention school — snake_case (A), camelCase (B), or mixed (C)?
--> snake_case

### Feature 3: Assurance Ledger Runtime

#### Architecture

The ledger runtime has two components with a clean separation:

**Ledger Engine** (new module, possibly new crate `ish-ledger` or module within `ish-vm`):
- Stateless functions for live-audit logic — given a statement + active feature states + entries → produces audit result (pass, auto-fix, discrepancy)
- Entry type registry — stores entry type definitions (Error, SystemError, Mutable, etc.)
- Standard registry — stores standard definitions with feature state maps
- No reference to the interpreter or VM state

**VM Integration** (in `ish-vm`):
- Standard scope stack — tracks which standard is active in the current scope
- Entry store — attaches entries to values/items and queries them
- Audit bridge — calls ledger engine on each statement, applies results

#### Proposed Implementation

1. **Define `FeatureState` enum**: `Optional`, `Live`, `Pre`, plus feature-specific states.

2. **Define `Standard` struct**: name, optional parent, feature map (`HashMap<String, FeatureState>`).

3. **Define `EntryType` struct**: name, optional parent, required properties with types.

4. **Define `AuditResult` enum**: `Pass`, `AutoFix(Vec<Action>)`, `Discrepancy(DiscrepancyReport)`.

5. **Implement `StandardRegistry`**: register/lookup standards, resolve inheritance chains.

6. **Implement `EntryTypeRegistry`**: register/lookup entry types, validate entries against types.

7. **Implement stateless `audit_statement(statement, active_features, entries) -> AuditResult`**: the core audit function, dispatchable per statement kind.

8. **Pre-register built-in standards** (`streamlined`, `cautious`, `rigorous`) and built-in entry types (`Error`, `SystemError`, `Mutable`, `Type`) at startup.

9. **Add standard scope stack to interpreter**: push on `@standard[name]`, pop on scope exit.

10. **Add entry tracking to values**: entries stored alongside values in the environment.

11. **Wire audit bridge**: after statement construction, before execution, call `audit_statement`. Apply auto-fixes. Report discrepancies as system errors.

12. **Add acceptance tests**:
    - Define and apply a custom standard
    - Query active feature state
    - Entry type definition and validation
    - Built-in standards are available
    - Standard inheritance (extends)
    - Standard scope (push/pop)

--> The proposed implementation assumes that the ledger related datatypes will be defined in the VM.  There are three places these datatypes could be defined: In the AST, in the VM, or in ish running on the VM.  Consider each of these alternatives, with pros and cons, and make a recommendation.  Note that the ish system will not support any mechanism for mutating standards, entry types, or entries.  It will also not support any mechanism for injecting arbitrary values at runtime.  They are all intended to be used in pre-audit.  As a result, they need to stable at module and function declaration time.  Every possible entry state needs to be inferrable from the code being declared in order to conduct a pre-audit.

### Feature 4: Type Annotation Live-Audit

#### Dependency

Requires Feature 3 (ledger runtime) — type checking is the `types(live)` feature state.

#### Proposed Implementation

1. **Extend audit logic for `types` feature**:
   - When `types` is `live`: check type annotations at assignment and function call sites.
   - When `types` is `optional`: no checking (current behavior).
   - When `types` is `pre`: future (requires static analysis pass).
--> This is confusing.  We may need to introduce alternative names for this. Propose alternative naming schemes, with pros and cons, and a recommendation.
--> The VM always checks for type compatibility discrepancies when every statement is executed.
--> Optional means that explicit type specification is optional. (Either `let x` or `let x:i32` is allowed)
--> Live means that explicit type specification is required, and it is checked during live audit (`let x` is not allowed, because no type is specified)
--> Pre means that explicit type specification is required, and it is checked during pre audit.
--> The live/pre modes make more sense when considering null checks.  Live null checking implies that a function like like `fn my_fn(a) { println(a.value)}` will throw a null dereference error at runtime, and so has a return type of `void|NullDereferenceError`, while pre checking means that the same function will throw a NullDereferenceError at build time, and so can never even be declared.
--> Live mode for explicit type specification provides little value, and maybe it shouldn't even be supported.  It is not completely meaningless, as a function like `fn another_fn(a){ if a > 0 {let x}}` has a path where the statement is never reached and so the function never throws.

2. **Implement type compatibility checking**:
   - Simple types: match by name (`int`, `float`, `string`, `bool`, etc.)
   - Union types: value matches if it matches any member
   - Optional types: value matches if it matches the inner type or is null
   - Object types: structural matching (all required properties present with correct types)
   - Function types: parameter count and return type matching
   - List types: element type checking
   - Tuple types: position-by-position matching

3. **Wire into interpreter**:
   - On `let x: T = expr` — after evaluating `expr`, audit type of result against `T`
   - On function call — audit each argument against parameter type annotation
   - On function return — audit return value against return type annotation

4. **Add acceptance tests** under `cautious` standard:
   - Variable with correct type annotation passes
   - Variable with wrong type annotation raises type error (E004)
   - Function parameter type checking
   - Union type matching
   - Optional type matching
   - Object structural matching

### Feature 5: Generic Types — Specification Refinement

#### Current Open Questions (from types.md)

The spec defines generic types but has unresolved questions:

1. **Variance** — Should generics be covariant, contravariant, invariant, or declaration-site specified?
2. **Generic function types** — Exact syntax for generic function signatures?
3. **Constraints** — Syntax for type constraints (`T extends Comparable`)?
4. **Defaults** — Syntax for default type parameters?
5. **Inference** — How are generic parameters inferred from usage?

#### Critical Analysis

**Variance:**
- **Alternative A: Invariant by default** (Rust approach). `List<Cat>` is NOT assignable to `List<Animal>`.
  - Pros: Safe. Prevents unsound mutations. Simple mental model.
  - Cons: Verbose — users must write explicit conversions.
- **Alternative B: Declaration-site variance** (Kotlin/C# approach). `List<out Animal>` for covariance.
  - Pros: Flexible. Expressive.
  - Cons: Complex. Users must understand variance.
- **Alternative C: Use-site variance** (Java `? extends T`).
  - Pros: Familiar to Java developers.
  - Cons: Complex wildcard syntax. Verbose.
- **Alternative D: Defer variance — implement generics as invariant initially, revisit later.**
  - Pros: Ship sooner. Variance is a refinement.
  - Cons: May paint into a corner.
- **Recommendation**: Alternative D. Invariant by default, with variance as a future refinement. This matches Rust's approach and avoids premature complexity.

**Assurance ledger interaction**: The `types(live)` feature should validate generic type arguments at instantiation sites. Under `streamlined`, generics are unchecked. Under `cautious`/`rigorous`, type arguments are validated.

#### Decisions

**Decision:** Variance strategy for generic types?
--> I believe that the answer to this question is meaningless and/or obvious in the ish type system.  But please check my reasoning. ish is structurally typed, so by definition a Cat, which has all of the structural properties of Animal, can always be used in place of an Animal, unless Animal is defined as closed, meaning it is not allowed to have any extra properties.  In which case, how could a type Cat, which extends Animal even exist?  Attempting to declare Cat would cause a discrepancy.  Now, you could have a situation where Animal is defined as open, and Cat extends Animal, and someone is attempting to assign a `List<Cat>` to a `List<Animal&@closed>` (I'm not sure about the syntax for this, and am not proposing it here.) In this situation, a discrepancy would occur. I should also note that the type spec says something about closed being the default, which is incorrect.  A type declaration statement like `type Cat = Animal & { sound: "Meow" }` is neither open or closed.  It may be inferred or coerced to either. To declare an open/closed type, it must be explicitly annotated so.  An object literal like `let x = { sound: "Meow" }` is closed by default.  To declare an open object literal, it must be explicitly annotated so.  A statement like `let x: Cat =  { sound: "Meow" }` will result in a closed cat.  The assurance ledger will combine the indeterminate open/closed state from the Cat type with the closed object literal, and the variable will have both sets of entries.

**Decision:** Should generic type checking be live-audit only, or also support pre-audit?
--> Generics should be checked by both live audit and pre audit.  However we are not implementing pre-audit yet.

### Feature 6: Literal Types — Specification Refinement

#### Current State

The spec mentions literal types (concrete values as types, e.g., `5`, `"hello"`, `true`) but does not detail syntax or semantics.

#### Critical Analysis

**Alternative A: Full literal types (TypeScript style).**
- `let x: 5 = 5` — the type of `x` is literally `5`, not `int`.
- Enables discriminated unions: `type Direction = "north" | "south" | "east" | "west"`.
- Pros: Powerful. Enables precise typing. Works well with unions for exhaustive matching.
- Cons: Complex type inference (when does `5` widen to `int`?). Requires a solid widening strategy.

**Alternative B: Literal types only in type positions (not inferred).**
- Users can write `let x: "hello" = "hello"` but if they write `let x = "hello"`, the type is `String`.
- Pros: Simpler inference. Explicit opt-in.
- Cons: Less useful for discriminated unions (must always annotate).

**Alternative C: Defer literal types entirely.**
- Pros: Simplifies scope.
- Cons: Discriminated unions become harder to type.

**Recommendation**: Alternative A with widening rule: unnannotated bindings widen to the base type. Annotated bindings with literal types stay narrow. This matches TypeScript's behavior and is the least surprising.

**Assurance ledger interaction**: Under `types(live)`, literal type annotations are checked at assignment. Under `streamlined`, they're ignored.

#### Decisions

**Decision:** Literal type approach — full (A), explicit-only (B), or defer (C)?
--> Please check my reasoning on this decision, and evaluate any possible inconsistencies or implementation issues it may generate.  The existing specification of literal types was written before the assurance ledger, and this is a new specification in the context of the assurance ledger.  Every value has both a (structural) type and (behavioral) entries.  So, the statement `let x = 5` does not have a type of `5`.  Rather it has a type of `int` (Some integer type, it can be inferred to any of the integer types later).  In a live audit, it has an actual value entry of `5`.  In a pre audit, it has a possible values entry of `5`.  (This is a key difference in the way that live and pre audit works.  Both audits check the code one statement at a time.  Live audit checks for discrepancies using the actual value, while pre audit tracks and audits against all possible values.)  Anyway, in our example, x does not have a type of `5`, it has an actual value entry of `5`.  ish should support adding allowed value entries to a value as well.  This supports the three use cases of literal types without actually being literal types.  The three use cases, as I see them, are: 1. Inferring statement reachability, using possible values; 2. Inferring type possibility (`never` type) using possible values; 3. Checking actual or possible values against allowed values.  Check whether languages that support type literals have other use cases that we need to consider.  Anyway, one consequence of this approach is that type declarations in ish are not really pure structural declarations.  In addition to defining stuctural types, a type declaration provides a hook on which to hang behavioral entries.  This may cause ambiguity between types and entries, and we should carefully consider our terminology.

**Decision:** Widening rule — should unannotated bindings always widen?
--> Actually, unannotated bindings should wait to be inferred later.  In the statement `let x = 5`, we can infer that x is of some integer type, but not exactly which one.  So, rather than converting the string literal `5` to a particular integer type, the interpreter should retain the string literal, and parse converting later when the actual type is inferred.  The compiler will infer the type as part of a pre-audit, and so can choose a type immediately.  If no type can be inferred, the compiler will widen.

### Feature 7: Intersection Types — Specification Refinement

#### Current State

The spec mentions intersection types (`A & B`) but does not detail them.

#### Critical Analysis

**Alternative A: Full intersection types.**
- `type AB = A & B` — a value must satisfy both `A` and `B`.
- For objects: all properties from both types.
- For primitives: `int & string` is `never` (no value can be both).
- Pros: Powerful for mixins and object composition.
- Cons: Complex. What does `A & B` mean when both have conflicting property types?

**Alternative B: Intersection types for objects only.**
- Restrict `&` to object types where it means "merge all properties."
- Pros: Simpler. Covers the primary use case.
- Cons: Inconsistent restriction.

**Alternative C: Defer intersection types.**
- Pros: Simplifies scope.
- Cons: Limits expressiveness for object composition.

**Recommendation**: Alternative A. Intersection types are a natural complement to union types. The rule for conflicts: conflicting property types are intersected recursively; if the intersection is `never`, the whole intersection is `never`.

**Assurance ledger interaction**: Under `types(live)`, intersection type annotations are validated at assignment.

#### Decisions

**Decision:** Intersection type approach — full (A), objects-only (B), or defer (C)?
--> Full

**Decision:** How to handle conflicting property types in intersections?
--> Conflicting property types produce a `never` type.

### Feature 8: Type Narrowing — Specification Refinement

#### Current State

The spec describes type narrowing through control flow but provides no implementation details.

#### Critical Analysis

**Alternative A: Control-flow narrowing (TypeScript style).**
- After `if is_type(x, "int")`, `x` is narrowed to `int` in the true branch.
- After `if x != null`, `x` is narrowed to exclude `null`.
- Pros: High value. Most common TypeScript feature used by developers.
- Cons: Requires data-flow analysis — more complex than simple type checking. Needs a narrowing engine that understands `is_type`, `!=`, truthiness, and custom guards.

**Alternative B: Narrowing only via explicit `is_type` checks.**
- Only `is_type(x, "T")` narrows. No narrowing from `!=`, truthiness, etc.
- Pros: Simpler to implement. Clear narrowing triggers.
- Cons: Misses common patterns (`if x != null`).

**Alternative C: Defer narrowing.**
- Pros: Simplifies scope.
- Cons: Types remain wide, requiring manual assertion.

**Recommendation**: Alternative B for initial implementation, with Alternative A as a future refinement. `is_type` narrowing covers the most important use case and has a clear implementation path.

**Assurance ledger interaction**: Narrowing is orthogonal to assurance level — it refines type knowledge regardless of the active standard. However, narrowed types should be used by the `types(live)` audit to avoid false positives.

#### Decisions

**Decision:** Type narrowing approach — full control-flow (A), is_type-only (B), or defer (C)?
--> Full control-flow.  Note that this is an assurance ledger behavior.  This is what the assurance ledger is for.  After every statement, it creates a revised set of entries for every value.  This **might** be implemented by mutating the entry set, but logically speaking the assurance ledger is creating a completely new entry set for after statement execution, which is coincidentally very similar to the before-execution entry set.  The ledger is not just checking the ledger.  It is also maintaining it.  There is a ledger entry for structural type, and this is just one of the entries that gets checked and maintained.

### Feature 9: Custom Type Guards — Specification Refinement

#### Current State

The spec mentions custom type guards: developer-defined functions that provide type narrowing beyond structural checks. No syntax is specified.

#### Critical Analysis

**Alternative A: TypeScript-style type predicates.**
```
fn is_fish(pet: Animal) -> pet is Fish {
    return obj_has(pet, "swim")
}
```
- After `if is_fish(pet)`, `pet` is narrowed to `Fish`.
- Pros: Familiar syntax. Powerful.
- Cons: Requires `is` keyword in return type position. Complex interaction with type system.

**Alternative B: Entry-based type guards.**
Use assurance ledger entries to mark functions as type guards:
```
@[type_guard(Fish)]
fn is_fish(pet: Animal) -> bool {
    return obj_has(pet, "swim")
}
```
- Pros: Uses existing assurance ledger mechanics. No new syntax needed.
- Cons: Less intuitive. Entry connects function to type indirectly.

**Alternative C: Defer custom type guards.**
- Built-in `is_type()` covers basic cases. Custom guards are a refinement.
- Pros: Simplifies scope.
- Cons: No user-defined narrowing.

**Recommendation**: Alternative B. Custom type guards fit naturally as an assurance ledger feature — a function annotated with a `TypeGuard` entry. This demonstrates the ledger's extensibility and avoids new syntax.

#### Decisions

**Decision:** Custom type guard approach — TypeScript-style (A), entry-based (B), or defer (C)?
--> Defer custom type guards.

### Feature 10: Error Representation Overhaul

#### Architecture (per Decisions 3 and 4)

Errors in ish are ordinary objects with specific assurance ledger entries:

**`Error` entry type:**
```
entry type Error {
    message: String
}
```
An object with the `Error` entry must have a `message` property of type `String`. Any object can be an error by having this entry.

**`SystemError` entry type:**
```
entry type SystemError extends Error {
    code: String
}
```
A `SystemError` is an `Error` that also has a `code` property (e.g., `"E001"`). The ish language processor only throws `SystemError`s.

**Throw semantics:**
- `throw expr` evaluates `expr` to a value.
- The assurance ledger audits the throw: the thrown value must have the `Error` entry.
- If the value is an object with a `message: String` property but no `Error` entry, the ledger auto-adds the entry (auto-fix).
- If the value lacks a `message: String` property, the ledger raises a discrepancy, and a `SystemError` is thrown about the type mismatch instead.

**Examples:**
```ish
// Works: object has message, Error entry auto-added
throw { message: "Something bad happened" }

// Works: explicit entry
@[Error]
throw { message: "File not found", path: "/tmp/foo" }

// Fails: no message property → system error about discrepancy
throw { errno: 73 }
// SystemError { code: "E007", message: "throw requires Error entry: missing 'message: String' property" }
```

#### Proposed Implementation

1. **Define `Error` and `SystemError` entry types** as built-in entry types in the ledger.
2. **Replace Rust `RuntimeError` struct** — the interpreter uses ish objects with `SystemError` entries internally. The Rust layer wraps/unwraps these as needed for control flow.
3. **Remove `new_error()` builtin** — errors are created as ordinary object literals.
4. **Update `is_error()` builtin** — checks for `Error` entry (not `__is_error__` property).
5. **Update `error_message()` builtin** — reads `message` property.
6. **Add `error_code()` builtin** — reads `code` property (returns null if not a SystemError).
7. **Update throw handler** — audit the throw argument; apply auto-fix or raise discrepancy.
8. **Update catch handler** — bound variable receives the thrown object.
9. **Update all interpreter error sites** — create SystemError objects with appropriate error codes.
10. **Update acceptance tests** — remove tests for throwing non-error values; add tests for entry-based error identity.

--> Note that to aid in error handling, there should be a hierarchy of error types extending SystemError.  For example, FileError extends SystemError, and FileDoesNotExist error extends FileError. There should be a error code for each leaf of the hierachy.
--> Add a `CodedError` layer to the error hierarchy.  Any Error with a code property of type string is a `CodedError`.  `SystemError` is reserved for errors with one of the well known ish error codes.
