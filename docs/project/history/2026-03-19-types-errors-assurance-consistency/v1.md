### 1.3 Types — Specification vs. Implementation Gaps

**Specification**: [docs/spec/types.md](../../spec/types.md) defines a comprehensive type system.

**What's implemented and tested (consistent)**:
- Primitive types (Int, Float, String, Bool, Null, Char) — runtime values, `type_of()`, `is_type()`, conversions
- Object types — literal construction, property access, mutation builtins
- List types — literal construction, index access, mutation builtins
- Function types — declarations, parameters, closures, lambdas, higher-order

**What's parsed but not enforced (gap)**:
- Generic types (`<>`) — AST `TypeAnnotation::Generic` exists, parser handles it, but no instantiation or checking
- Union types (`|`) — AST `TypeAnnotation::Union` exists, parser handles it, but no runtime representation or checking
- Tuple types — AST `TypeAnnotation::Tuple` exists, parser handles it, but tuples are just lists at runtime
- Optional types (`?`) — AST handles it via union with null, but no enforcement
- Type annotations on variables — parsed and stored on AST nodes, but interpreter ignores them
- Return type annotations — parsed and stored, but interpreter ignores them
- Function parameter type annotations — parsed and stored, but interpreter ignores them
- Type aliases — AST node exists for type definitions but interpreter ignores them

**What's not parsed or implemented (gap)**:
- Literal types (concrete values as types)
- Intersection types (`&`)
- Type narrowing through control flow
- Custom type guards
- `validate()` builtin
- Object open/closed distinction
- Property mutability annotations
- Numeric precision configuration
- Integer overflow configuration (wraps silently, not configurable)
- Implicit conversion rules
- Type widening/inference rules
- Higher-kinded types
- Type metatype (`Type` for runtime operations)

**What's not tested (gap)**:
- None of the "parsed but not enforced" features have acceptance tests
- `isType()` (spec function) is tested as `is_type()` (builtin) — naming inconsistency

### 1.4 Errors — Specification vs. Implementation Gaps

**Specification**: [docs/spec/types.md](../../spec/types.md) § Error Handling and [docs/errors/INDEX.md](../../errors/INDEX.md).

**What's implemented and tested (consistent)**:
- `throw` statement with any value
- `try`/`catch`/`finally` blocks
- `new_error(message)` builtin
- `is_error(value)` builtin
- `error_message(error)` builtin
- `defer` statement (LIFO order)
- Error propagation through function calls
- Re-throw from catch blocks
- Exit code 1 for unhandled errors

**What's specified but not implemented (gap)**:
- Error codes (E001–E006 defined in catalog, but `RuntimeError` has no code field)
- Error categories (all errors are generic `RuntimeError`)
- Type-based catch dispatch (catch clauses parse type annotations but ignore them)
- Multiple catch clauses (parsed and executed, but all clauses match any error)
- Error union types in function signatures (no syntax enforcement)
- `undeclared_errors` feature (no tracking of which functions throw)
- Stack traces / error context
- Custom error types (only structural `{ message, __is_error__ }`)
- `with` blocks (parsed and executed but no tests — untested feature)

**Naming inconsistency**:
- The error-handling proposal decided that only `Error` instances can be thrown, but the prototype allows throwing any value. This is an actual behavioral gap.

### 1.5 Assurance Ledger — Specification vs. Implementation Gaps

**Specification**: [docs/spec/assurance-ledger.md](../../spec/assurance-ledger.md) — comprehensive.

**What's implemented (parser/AST only)**:
- `@standard[name]` syntax — parsed, AST node created, interpreter ignores
- `@[entry(params)]` syntax — parsed, AST node created, interpreter ignores
- `standard name [features]` definition — parsed, AST node created, interpreter ignores
- `standard name extends base [features]` — parsed, AST node created, interpreter ignores
- `entry type name { fields }` — parsed, AST node created, interpreter ignores

**What's not implemented at all**:
- Standard registration (no runtime storage for defined standards)
- Standard application (no mechanism to activate a standard in scope)
- Entry tracking (no metadata recorded on items)
- Pre-audit checks (no build-time analysis)
- Live-audit checks (no runtime validation)
- Discrepancy reporting (no conflict detection)
- Built-in standards (streamlined, cautious, rigorous not pre-defined)
- Feature state logic (no mechanism to query active feature states)
- Entry inference from native syntax

**Testing**: Zero acceptance tests for any assurance ledger feature.

## Part 2: Scope Determination

The RFP asks to audit features that "have already been specified." This section determines which features are in and out of scope.

### In Scope (Specified and Expected to Work)

These features have specifications and accepted proposals. They should be consistent across spec, docs, prototype, and tests:

1. **Primitive types** — fully specified, should be fully working
2. **Object types** — specified in types.md
3. **List types** — specified in types.md
4. **Function types** — specified in types.md
5. **Error handling basics** — try/catch/finally/defer/throw specified and accepted
6. **Error creation** — `new_error()`, `is_error()`, `error_message()`
7. **Error catalog** — E001–E006 defined
8. **Assurance ledger syntax** — accepted proposal, full specification
9. **Standard definitions** — syntax specified and accepted
10. **Entry annotations** — syntax specified and accepted

### Out of Scope (Not Yet Fully Specified or Explicitly Deferred)

These features are mentioned in specs but have open questions, are placeholder-status, or are acknowledged as future work:

1. **Generic types** — specified but ~20 open questions remain
2. **Type narrowing** — specified but no implementation path defined
3. **Literal types** — mentioned but not detailed
4. **Intersection types** — mentioned but not detailed
5. **Custom type guards** — mentioned but not detailed
6. **Pre-audit** — specified but acknowledged as requiring static analysis (future)
7. **Memory model** — placeholder status
8. **Polymorphism strategy** — placeholder status
9. **Pattern matching** — not yet designed

### Decisions

**Decision:** Which of the "in scope" items should we actually address in this proposal, and should any "out of scope" items be pulled in?
--> All of the features listed as in scope should be addressed by this proposal, as well as the first five features listed as out of scope. We will need to refine the specifications of the "out of scope" type features first.  Pull the open questions for these features into the next version of the spec as decision points.  Propose alternatives for them, with pros and cons and a recommendation.  When proposing alternatives, consider that the type system was specified before the assurance ledger, and that now that we have the assurance ledger, the type system should rely on it.  Many features originally specified as part of the type system can now be implemented using the assurance ledger, and have nothing to do with typing at all.  Update the roadmap and maturity matrix to be correct for all of these features whether or not they are in scope.

### Feature 2: Error Code Integration

#### Issues to Watch Out For
- The error catalog defines E001–E006, but `RuntimeError` is a single struct with only `message` and `thrown_value`. Adding error codes changes the runtime error representation.
- Existing tests check error messages, not codes — adding codes should not break existing behavior.
- The error-handling proposal decided to use result types under the hood — this may affect how error codes are represented.

#### Critical Analysis

**Alternative A: Add error codes to RuntimeError, emit them in error messages.**
- Pros: Makes the error catalog real; structured errors enable better tooling; aligns spec with implementation.
- Cons: Requires touching every error creation site in the interpreter. Error messages change format.

**Alternative B: Add error codes to RuntimeError but only expose them programmatically, not in messages.**
- Pros: No breaking changes to error message format; codes available for tooling.
- Cons: Users can't see codes unless they use `error_code()` builtin.

**Alternative C: Leave error codes as documentation-only for now; implement when the type system can support custom error types.**
- Pros: No implementation effort; avoids premature decisions about error representation.
- Cons: Error catalog remains disconnected from implementation; spec/impl inconsistency persists.

#### Proposed Implementation (Alternative A)

1. Add `code: Option<String>` to `RuntimeError` in `ish-vm/src/error.rs`.
2. Define error code constants (E001–E006) in a new `error_codes` module.
3. Update interpreter error creation sites to attach appropriate codes.
4. Add `error_code(error)` builtin.
5. Format error messages as `[E001] Unhandled throw: <message>`.
6. Add acceptance tests for error codes.

#### Decisions

**Decision:** Which alternative for error code integration — wire them in (A/B) or leave as documentation (C)?
--> An object is designated as being an error by having the Error entry. The Error entry requires the message property, and requires it to be a string.  So, a user defined error can be any object with the annotation and a message property.  There should also be a SystemError entry, which also requires an error code.  The ish language processor should only throw SystemErrors, and the error catalog should enumerate all possible error codes for these.  There should not be anything that corresponds to a `RuntimeError`.  Errors are just ordinary objects that have the right entries and properties.

### Feature 3: Throw Restriction Enforcement

#### Issues to Watch Out For
- The accepted error-handling proposal says only `Error` instances can be thrown.
- The prototype currently allows throwing any value (and tests verify this behavior).
- Enforcing the restriction would break the existing "throwing non-error values" test.

#### Critical Analysis

**Alternative A: Enforce the restriction now — only Error instances can be thrown.**
- Pros: Aligns with accepted proposal; cleaner error handling semantics.
- Cons: Breaking change; removes flexibility; requires updating the existing test. Users who throw strings lose that ability.

**Alternative B: Enforce the restriction only under a standard (e.g., `cautious` or `rigorous`).**
- Pros: Backward compatible; demonstrates assurance ledger in action; gradual adoption.
- Cons: More complex implementation; requires assurance ledger runtime (Feature 6).

**Alternative C: Defer enforcement; document the intended restriction.**
- Pros: No breaking changes now.
- Cons: Spec/impl gap persists.

#### Proposed Implementation (Alternative A)

1. In the interpreter's `throw` handler, check if value is an Error (has `__is_error__` property).
2. If not, wrap in an Error: `new_error(to_string(value))`.
3. Update the "throwing non-error values" test to expect wrapped Error behavior.

#### Decisions

**Decision:** Enforce throw restriction now (A), tie to assurance level (B), or defer (C)?
--> The error spec was written before the assurance ledger spec, and so did not properly incorporate assurance ledger features.  It needs to be updated.  Also, it needs to be more prominent in the spec.  Currently, errors are not even referenced in the spec index.  Only Errors can be thrown.  This should be enforced.  But an Error is just a type of object, defined using the normal type and assurance ledger mechanics.  So, for example, the statement `throw { message: "Something bad happened" }` is supported: First, an object literal is constructed, with a message property.  Second, it is passed to the throw statement.  The assurance ledger audits the statement and sees that a throw statement requires an Error entry.  It adds the error entry to the object.  Then the object is thrown.  Another example is `throw { errno: 73 }`.  First, an object literal is constructed, with a message property.  Second, it is passed to the throw statement.  The assurance ledger audits that statement and sees that a throw statement requires an Error entry.  Adding an error entry to `{ errno: 73 }` creates a discrepancy.  Errors are required to have a message property.  So, instead of throwing the requested object, the system instead throws a system error about the Error type discrepancy.  Also, there is no need for a `new_error` function, and it should be removed.

### Feature 4: Type Annotation Awareness (Minimal)

#### Issues to Watch Out For
- Full type checking requires a type-checker pass, which is a huge undertaking.
- The spec describes both low-assurance (no checking) and high-assurance (full checking) modes.
- The prototype currently ignores all type annotations completely.

#### Critical Analysis

**Alternative A: Implement live-audit type checking for annotated variables and parameters.**
- Pros: Type annotations become meaningful; demonstrates the assurance continuum; catches bugs.
- Cons: Significant implementation effort; need to define checking semantics for all type annotation variants.

**Alternative B: Implement only basic `is_type()` enforcement on assignment when type annotations are present.**
- Pros: Simple to implement (just check the annotation matches `type_of()` at assignment time); small step toward type safety.
- Cons: Only covers simple types (not generics, unions, objects); limited value.

**Alternative C: Leave type annotations as decorative; focus on `validate()` builtin instead.**
- Pros: `validate()` is explicitly in the spec; gives users a manual way to check types; no implicit behavior changes.
- Cons: Annotations remain meaningless.

**Alternative D: Leave type annotations as decorative for now; implement after assurance ledger runtime exists.**
- Pros: Type checking is a feature of the assurance ledger (`types(live)` and `types(pre)` states); implementing it standalone would need to be re-done. Correct layering.
- Cons: Kicks the can further.

**Recommendation**: Alternative D or C. Type checking is architecturally part of the assurance ledger. Implementing it standalone creates technical debt. A `validate()` builtin gives users immediate value without architectural conflict.

#### Proposed Implementation (Alternative C — validate builtin)

1. Add `validate(type_string, value)` builtin that throws a type error (E004) if `type_of(value) != type_string`.
2. Add acceptance tests for `validate()`.

#### Decisions

**Decision:** Which approach to type annotations — live enforcement (A), basic checking (B), validate builtin only (C), or defer to assurance ledger (D)?
--> Implement live-audit type checking for annotated variables and parameters.  The reason this RFP addresses types, errors, and the assurance ledger all together is that these features have complex interdependencies and are all, to some extent, specified in terms of each other.  It is time to bring the whole complex system together, and make sure not only that all these features are working, but that they are consistent with each other and that they use each other properly instead of being siloed.  The implementation must make sure to build them in dependency order, and that may mean implementing implementing part of one, then part of another, then back to the first one, etc.

### Feature 5: isType / is_type Naming Reconciliation

#### Issues to Watch Out For
- Spec says `isType(t, i)` (camelCase, type-first argument order).
- Prototype implements `is_type(value, type_string)` (snake_case, value-first).
- The language-syntax proposal decided on specific naming conventions but didn't address builtin naming.

#### Critical Analysis

**Alternative A: Rename to match spec — `isType(type, instance)`.**
- Pros: Spec consistency; camelCase may be the intended convention.
- Cons: Breaking change for any existing ish programs; `is_type` has 15 existing tests.

**Alternative B: Update spec to match implementation — `is_type(value, type_string)`.**
- Pros: No breaking changes; snake_case is common in ish builtins (all current builtins use snake_case).
- Cons: Spec diverges from intended convention.

**Alternative C: Support both as aliases.**
- Pros: No breakage; flexible.
- Cons: Two names for one thing; confusing.

**Recommendation**: Alternative B. All existing builtins use snake_case (`type_of`, `to_string`, `str_length`, etc.). The spec should be updated to match the established convention, and the argument order should be value-first (consistent with `type_of(value)`).

#### Proposed Implementation (Alternative B)

1. Update `docs/spec/types.md` to use `is_type(value, type)` with snake_case and value-first order.
2. Keep existing implementation unchanged.

#### Decisions

**Decision:** Rename builtin to match spec (A), update spec to match builtin (B), or alias both (C)?
--> We have not yet decided on naming conventions for code written in ish.  This proposal should include those naming conventions, then this feature should follow them.  The naming conventions should take a "no surprises" approach, researching what naming conventions are used in other languages, prioritizing both the most popular languages and Rust and Typescript specifically.  We should be careful not to adopt conventions from other languages piecemeal, in a way that leads to inconsistencies within the ish conventions.

### Feature 6: Assurance Ledger Minimal Runtime

#### Issues to Watch Out For
- The assurance ledger specification is comprehensive but completely unimplemented beyond parsing.
- Full implementation (pre-audit, all 22 features) is a massive undertaking.
- Without *any* runtime, the assurance ledger is purely decorative — which contradicts the purpose.
- Several other features (type enforcement, throw restriction, error tracking) depend on assurance ledger feature states.

#### Critical Analysis

**Alternative A: Implement a minimal runtime — standard registration, scope tracking, feature state queries.**
- Pros: Foundation for all other assurance-dependent features; standards become meaningful; demonstrates the design.
- Cons: Moderate implementation effort; need to decide which features to actually enforce.

**Alternative B: Implement standard registration only (no enforcement).**
- Pros: Low effort; standards are stored and queryable; sets up for future enforcement.
- Cons: Standards still don't *do* anything visible.

**Alternative C: Implement one end-to-end feature — `types(live)` enforcement under `cautious` standard.**
- Pros: Demonstrates the full assurance pipeline from standard definition to runtime checking; concrete value.
- Cons: Higher effort; requires both ledger runtime AND type checking.

**Alternative D: Defer all ledger runtime; focus on non-ledger fixes first.**
- Pros: Lower scope; fixes immediate inconsistencies.
- Cons: Ledger remains decorative; features that depend on it cannot proceed.

**Recommendation**: Alternative A or B. The ledger needs a foundation before any feature enforcement can be layered on top.

#### Proposed Implementation (Alternative A)

1. Add `StandardRegistry` to interpreter state — stores defined standards with their feature maps.
2. Add `active_standard` stack to interpreter — tracks which standard is active in current scope.
3. When executing `StandardDef`, register the standard in the registry.
4. When executing `Annotated` with `@standard[name]`, push standard onto scope stack.
5. Add `query_feature(name) -> FeatureState` method that checks the active standard.
6. Pre-define `streamlined`, `cautious`, and `rigorous` in the registry at startup.
7. Add acceptance tests: define standard, apply standard, query feature state (via a new `assurance_feature()` debugging builtin).

#### Decisions

**Decision:** Which level of ledger runtime — foundation + queries (A), registration only (B), one end-to-end feature (C), or defer (D)?
--> Alternative A, foundation + queries.  Also note that the implementation should have a clean separation between the VM and the assurance ledger.  The VM instantiates a ledger, and it is responsible for feeding it the correct states (entry type, standard, and entries) and calling it with every statement.  The assurance ledger itself should have completely stateless logic for live auditing, for easy unit testing.
