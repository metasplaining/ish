## Decision Register

All decisions made during design, consolidated here as the authoritative reference.

| # | Decision | Outcome |
|---|----------|---------|
| 1 | FutureRef equality semantics | `Rc::ptr_eq` identity equality |
| 2 | Await-on-non-future behavior | E012 thrown in clear-cut cases: awaiting a non-function-call value that is not a Future, or awaiting a call to a function/builtin with an explicit unyielding entry. **Pass-through only for the ambiguous case:** calling a function with no Yielding entry that returns a non-Future. |
| 3 | Error code for await-on-non-future | E012 — await type mismatch. Active in prototype for clear-cut cases. |
| 4 | Spawn-on-non-function behavior | E013 — spawning anything but a function (or builtin) call is an error. Spawning a call to a function/builtin with an explicit unyielding entry is also E013. |
| 5 | Parallel builtin mechanism | Two-signature: `Simple` (sync) and `Parallel` (marshaled args + `spawn_blocking`) |
| 6 | Which builtins become parallel | `print`, `println`, `read_file`, `write_file` only |
| 7 | Parallel builtin return type | Return `Future<T>`. At low assurance, calling without `await`/`spawn` implies `await`. The only way to obtain a `Future` value is via `spawn`. |

| 10 | Builtin–function parity | Builtins appear to outside observers exactly like regular functions. All builtins carry parameter metadata (names, types), return type, and a yielding classification. ## Feature 1: FutureRef Identity Equality

### Summary

The `PartialEq` implementation for `Value::Future` currently returns `false` for all comparisons. This violates reflexivity (`f == f` is `false`) and is incorrect. Futures use identity-based equality: two future values are equal if and only if they reference the same underlying `Rc`.

### Proposed Implementation

Single change in `proto/ish-vm/src/value.rs`, in `impl PartialEq for Value`:

```rust
// Futures use identity equality (same Rc pointer)
(Value::Future(a), Value::Future(b)) => Rc::ptr_eq(&a.inner, &b.inner),
```

This replaces the current line:

```rust
// Futures are never equal (identity-based, not value-based)
(Value::Future(_), Value::Future(_)) => false,
```

### Tests

- `let f = spawn work(); f == f` → `true`
- `let f1 = spawn work(); let f2 = spawn work(); f1 == f2` → `false`
- `let f = spawn work(); let g = f; f == g` → `true` (same Rc via clone)

## Feature 2: Spawn Validation and Await Semantics

### Summary

Three rules govern `spawn` and `await`:

1. **Spawn validation (E013):** `spawn` applied to anything other than a function call (or builtin call) is E013. Spawning a call to a function or builtin with an explicit unyielding entry is also E013.

2. **Await validation (E012):** `await` applied to a value that is not a `Future` is E012 — **except** when the await sub-expression is a call to a function that has no Yielding entry (the ambiguous case). In that case, the non-Future result passes through.

3. **The ambiguous case:** A function with neither `async` keyword nor `@[unyielding]` annotation has no Yielding entry. The prototype cannot determine whether such a function yields internally (e.g., `fn my_fn() { println('I yield') }` calls a parallel builtin, making it yielding, but it has no Yielding entry). When `await` is applied to a call to such a function and the result is not a `Future`, the result passes through rather than throwing E012. This is a pragmatic compromise — breaking correct programs is worse than permitting a no-op `await`.

### E012 Decision Table

| Await sub-expression | Result type | Outcome |
|---------------------|-------------|---------|
| Function call, function has yielding entry | `Future` | Normal await |
| Function call, function has unyielding entry | non-`Future` | **E012** |
| Function call, function has no Yielding entry | non-`Future` | Pass-through (ambiguous) |
| Function call, function has no Yielding entry | `Future` | Normal await |
| Parallel builtin call | `Future` | Normal await |
| Simple builtin call (unyielding entry) | non-`Future` | **E012** |
| Variable or literal | `Future` | Normal await |
| Variable or literal | non-`Future` | **E012** |

### Proposed Implementation — Await

In `proto/ish-vm/src/interpreter.rs`, the `Expression::Await` handler changes to:

```rust
Expression::Await { expr } => {
    // 1. Audit async_annotation feature (existing)
    // 2. Evaluate expr
    let val = self.eval_expression(task, yield_ctx, expr, env).await?;

    if let Value::Future(ref future_ref) = val {
        // Normal await — take JoinHandle and .await (existing logic)
        // ...
    } else {
        // Non-future result. Check the expression form:
        // - If expr is a Call and the callee has no Yielding entry → pass-through
        // - Otherwise → E012
        if is_call_to_unentryed_function(expr, env) {
            // Ambiguous case: function has no Yielding entry.
            // Pass through to avoid breaking correct programs.
            Ok(val)
        } else {
            Err(RuntimeError::system_error(
                format!(
                    "cannot await a value of type '{}' — await requires a Future",
                    val.type_name()
                ),
                "E012",
            ))
        }
    }
}
```
--> This won't work.  The checking needs to be done before `let val = self.eval_expression(task, yield_ctx, expr, env).await?;`, since that is where the function is called and where the result is awaited.  We want to check whether the expression is awaitable before even attempting to await it.  Is this a more fundamental error, in the grammar itself?  Why does Expression::Await have an expression as a parameter, instead of a FunctionCall?

The helper `is_call_to_unentryed_function` inspects the expression AST:
- If the expression is `Expression::Call { function, .. }`, resolve `function` to a value
- If the value is `Value::Function(f)` and `f.is_async` is false and the function has no unyielding annotation → return true (ambiguous)
- If the value is `Value::BuiltinFunction(b)` and `b.is_yielding` is false → return false (unyielding, should be E012)
- Otherwise → return false

Note: this requires looking up the callee without re-evaluating side effects. Since function names are typically simple identifiers, this is a scope lookup. For complex call expressions, the conservative default is E012 (not ambiguous).

### Proposed Implementation — Spawn

In `proto/ish-vm/src/interpreter.rs`, the `Expression::Spawn` handler adds validation before spawning:

```rust
Expression::Spawn { expr } => {
    // 1. Check that expr is a Call expression
    if !is_call_expression(expr) {
        return Err(RuntimeError::system_error(
            "spawn requires a function call",
            "E013",
        ));
    }

    // 2. Resolve the callee and check for unyielding entry
    if let Some(callee) = resolve_callee(expr, env) {
        match callee {
            Value::Function(f) if has_unyielding_entry(&f) => {
                return Err(RuntimeError::system_error(
                    "cannot spawn an unyielding function",
                    "E013",
                ));
            }
            Value::BuiltinFunction(b) if !b.is_yielding => {
                return Err(RuntimeError::system_error(
                    format!("cannot spawn unyielding builtin '{}'", b.name),
                    "E013",
                ));
            }
            _ => {} // proceed
        }
    }

    // 3. Spawn the task (existing logic)
    let mut spawned_vm = self.spawn_clone();
    // ...
}
```
--> Same thing.  Why does Expression::Spawn expect an expression rather than a FunctionCall?

## Feature 3: Parallel Builtins with Function Parity

### Summary

Builtins that perform blocking I/O (`print`, `println`, `read_file`, `write_file`) become **parallel builtins**. They use a two-signature architecture: simple builtins remain synchronous closures; parallel builtins marshal arguments into `Send`-safe form, execute on `tokio::task::spawn_blocking`, and return a `Future<T>`.

All builtins — simple and parallel — carry the same metadata as regular functions: parameter names, parameter types, return type, and a yielding classification. To an outside observer, builtins are indistinguishable from regular functions:

- **Simple builtins** have an unyielding entry. Attempting to `await` or `spawn` a simple builtin produces E012 or E013 respectively.
- **Parallel builtins** have yielding and parallel entries. They can be `await`ed or `spawn`ed.
- The interpreter applies the same parameter-count checking and parameter/return-type auditing to builtin calls as it does to regular function calls.

Parallel builtins return `Value::Future` internally, but **the caller never sees a bare `Future`** in normal usage:

- **Without `await_required`** (low assurance): calling a parallel builtin without `await` or `spawn` implies `await`. So `println("hello")` is equivalent to `await println("hello")`. The future is created, immediately awaited by the VM, and the resolved value is returned. Output ordering is preserved.
- **With `await_required` set to `required`**: the VM throws an error if a yielding function is called without explicit `await` or `spawn`. The caller must write `await println("hello")` or `spawn println("hello")`. In the `await` case, the future is created and awaited — the caller receives the resolved value. In the `spawn` case, the caller receives a `Future`.
- **The only way to obtain a `Future` value is via `spawn`.** In all other cases, the future is consumed by `await` (explicit or implied) and the caller receives the resolved value.

Builtins are temporary scaffolding. They will be replaced by standard library functions when the standard library becomes available. An open question is created to track this.

### Proposed Implementation
--> Instead of changing the builtin implementation, let's add the implementation for compiled functions, then convert all of the builtins to use the compile function implementation.  Ish supports two kinds of functions: Compiled and interpreted.  The interpreted kind are already implemented.  The compiled kind are almost exactly like the intepreted kind, except for a few small differences.  We need a new FunctionImplementation type, which is an enumeration of `Interpreted(Statement)`, `Concurrent(Shim)`, and `Parallel(ParallelShim)`. IshFunction is currently defined as having a `body` property, of type Statement.  This should be replaced with an `implementation` property, of type FunctionImplementation.  A `Shim` is a function that has the approximately the same method signature as IshVM.exec_statement (something like: fn <'a>(vm: &'a mut IshVm, task: &'a mut TaskContext, yc: &'a mut YieldContext, env: &'a Environment) -> Pin<Box<dyn Future<Output = Result<ControlFlow, RuntimeError>> + 'a>>).  A parallel shim is similar, but returns a future.  So, in `call_function_inner`, the VM detects whether the function is interpreted, concurrent, or parallel.  If the function is interpreted, it calls exec_statement.  If the function is concurrent, it calls the shim.  If the function is parallel, it calls the parallel shim.  `call_function_inner` has already checked all of the function parameters and packed them into the env parameter in any event.  So, to implement the builtins as compile functions, each of them needs to be converted to have either the Shim or ParallelShim function signature, associated with some metadata with new_function, and registered into the environment using `env.define`.  No other special handling should be needed for them.

**Step 4: Update interpreter dispatch in `call_function_inner`**

The builtin dispatch path now mirrors the regular function path:

```rust

    // 1. Check arg count against b.params.len()
    if args.len() != b.params.len() {
        return Err(RuntimeError::system_error(
            format!("'{}' expects {} arguments, got {}",
                b.name, b.params.len(), args.len()),
            "E003",
        ));
    }

    // 2. Audit parameter types (same logic as regular functions)
    for (i, (val, expected_type)) in args.iter().zip(&b.param_types).enumerate() {
        if let Some(type_ann) = expected_type {
            // Type audit against annotation (existing audit logic)
        }
    }

    // 3. Dispatch by kind
    match &b.kind {
        BuiltinKind::Simple(func) => {
            let result = func(&args)?;
            // 4. Audit return type
            // ...
            Ok(result)
        }
        BuiltinKind::Parallel(func) => {
            let call = func(&args)?;
            // Execute parallel call (see Step 5)
            let future_val = execute_parallel_call(call, &config);

            // 5. Implied await at low assurance
            // If no await_required or not "required", immediately await
            // the future and return the resolved value.
            // Otherwise, return the Future for the caller to handle.
            // ...
        }
    }

    // Ledger builtins remain intercepted by name before this point
}
```

**Step 5: Execute function for `spawn_blocking`**

```rust
fn execute_parallel(call: ParallelCall) -> Result<ParallelResult, String> {
    match call {
        ParallelCall::Print { text, newline } => {
            if newline { println!("{}", text); } else { print!("{}", text); }
            Ok(ParallelResult::Null)
        }
        ParallelCall::ReadFile { path } => {
            std::fs::read_to_string(&path)
                .map(ParallelResult::Text)
                .map_err(|e| format!("{}: {}", path, e))
        }
        ParallelCall::WriteFile { path, content } => {
            std::fs::write(&path, &content)
                .map(|_| ParallelResult::Null)
                .map_err(|e| format!("{}: {}", path, e))
        }
    }
}
```

The interpreter wraps this in a `spawn_blocking` → `spawn_local` bridge:

```rust
let handle = tokio::task::spawn_blocking(move || execute_parallel(call));
// Bridge back to LocalSet thread to create Value (which contains Rc, so !Send)
let local_handle = tokio::task::spawn_local(async move {
    match handle.await {
        Ok(Ok(result)) => Ok(result.into_value()),
        Ok(Err(msg)) => Err(RuntimeError::system_error(msg, "E008")),
        Err(join_err) => Err(RuntimeError::system_error(
            format!("parallel builtin panicked: {}", join_err), "E010"
        )),
    }
});
let future_val = Value::Future(FutureRef::new(local_handle));

// Implied await if no await_required or at low assurance:
// The interpreter checks the call context and, if no explicit await/spawn
// is present and await_required is not "required", immediately awaits
// the future and returns the resolved value.
```

For `print`/`println` with an active `output_sender` (interactive shell mode):

1. Send the message and a `oneshot::Sender` through the crossbeam channel
2. The shell writes the output and sends the acknowledgment back through the `oneshot`
3. The `spawn_blocking` (or `spawn_local`) task waits for the acknowledgment before resolving the future
4. This ensures the future does not resolve until the output is actually written

### Test Impact

Because of implied await at low assurance, **existing tests do not need to change**. `println("hello")` continues to work — the VM internally creates a future, immediately awaits it, and returns the resolved value. Output ordering is preserved because the implied await blocks until the I/O completes.

Tests under `@standard[cautious]` or any standard where `await_required` is `required` will need explicit `await`:

```ish
@standard[cautious] {
    await println("hello")    // required — no implied await
    let content = await read_file("path.txt")   // required
}
```

New tests should cover:
- Parallel builtins with implied await (low assurance) — output ordering preserved
- Parallel builtins with explicit await (high assurance)
- Parallel builtins with `spawn` — returns `Future`, output may be deferred
- `read_file` and `write_file` in parallel via `spawn` — concurrent I/O
- Parameter count errors on builtins (same E003 as regular functions)
- `await len([1,2])` → E012 (simple builtin has unyielding entry)
- `spawn len([1,2])` → E013 (simple builtin has unyielding entry)
