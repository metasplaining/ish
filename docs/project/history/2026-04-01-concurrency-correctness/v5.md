# Proposal: Concurrency Correctness Fixes

*Generated from [concurrency-correctness.md](../rfp/concurrency-correctness.md) on 2026-04-01. Revised 2026-04-01 (v5) with inline decisions.*

## Decision Register

All decisions made during design, consolidated here as the authoritative reference.

| # | Decision | Outcome |
|---|----------|---------|
| 1 | FutureRef equality semantics | `Rc::ptr_eq` identity equality |
| 2 | Await-on-non-future behavior | E012 thrown in clear-cut cases: awaiting a call to a function/builtin with an explicit unyielding entry. **Pass-through only for the ambiguous case:** calling a function with no Yielding entry that returns a non-Future. |
| 3 | Error code for await-on-non-future | E012 — await type mismatch. Active in prototype for clear-cut cases. |
| 4 | Spawn-on-non-function behavior | E013 — spawning a call to a function/builtin with an explicit unyielding entry. |
| 5 | Parallel builtin mechanism | Builtins become compiled functions with `FunctionImplementation::Concurrent` or `FunctionImplementation::Parallel` bodies. No separate `BuiltinFn`/`Value::BuiltinFunction`. |
| 6 | Which builtins become parallel | `print`, `println`, `read_file`, `write_file` only |
| 7 | Parallel builtin return type | Return `Future<T>`. At low assurance, calling without `await`/`spawn` implies `await`. The only way to obtain a `Future` value is via `spawn`. |
| 8 | Parallel builtins and `await_required` | When `await_required` is `required`, VM throws if no explicit `await` or `spawn`. Caller never receives a bare `Future`. |
| 9 | Parallel builtin I/O completion | Future must not resolve until the I/O operation is actually complete. Shell communication requires an acknowledgment message back after output is written. |
| 10 | Builtin–function parity | Builtins are regular `IshFunction` values with compiled implementations. They carry parameter metadata (names, types), return type, and yielding classification. `Value::BuiltinFunction` is eliminated. Dispatch through `call_function_inner` is unified. |
| 11 | Grammar-level await/spawn restriction | `await` and `spawn` take a function call in the grammar, not an arbitrary expression. `await_op ~ call_expr` instead of `await_op ~ unary`. The AST `Await` and `Spawn` nodes contain a `FunctionCall` (callee + args). Invalid forms like `await 42` still parse (parser-matches-everything) but produce `Incomplete` nodes. |
| 12 | FunctionImplementation enum | `IshFunction.body: Statement` is replaced by `IshFunction.implementation: FunctionImplementation`. Three variants: `Interpreted(Statement)`, `Concurrent(Shim)`, `Parallel(ParallelShim)`. `call_function_inner` dispatches on the variant. |

## Feature 1: FutureRef Identity Equality

### Summary

The `PartialEq` implementation for `Value::Future` currently returns `false` for all comparisons. This violates reflexivity (`f == f` is `false`) and is incorrect. Futures use identity-based equality: two future values are equal if and only if they reference the same underlying `Rc`.

### Proposed Implementation

Single change in `proto/ish-vm/src/value.rs`, in `impl PartialEq for Value`:

```rust
// Futures use identity equality (same Rc pointer)
(Value::Future(a), Value::Future(b)) => Rc::ptr_eq(&a.inner, &b.inner),
```

This replaces the current line:

```rust
// Futures are never equal (identity-based, not value-based)
(Value::Future(_), Value::Future(_)) => false,
```

### Tests

- `let f = spawn work(); f == f` → `true`
- `let f1 = spawn work(); let f2 = spawn work(); f1 == f2` → `false`
- `let f = spawn work(); let g = f; f == g` → `true` (same Rc via clone)

## Feature 2: Grammar-Level Spawn/Await Restriction

### Summary

Currently, `await` and `spawn` are unary prefix operators in the grammar (`await_op ~ unary`), meaning they can be applied to any expression: `await 42`, `spawn "hello"`, etc. The AST `Await` and `Spawn` nodes wrap a generic `Box<Expression>`.

This is a grammar-level error. `await` and `spawn` should only accept function calls. The grammar is changed so these operators take a `call_expr` (function call expression) as their operand. The AST nodes are changed to contain a `FunctionCall` (callee + args) instead of a generic `Expression`.

Invalid forms like `await 42` still parse — parser-matches-everything — but produce an `Incomplete` AST node rather than a valid `Await` node. This moves the structural validation from the interpreter to the parser.

### Grammar Change

Current grammar:

```pest
unary = {
    not_op ~ unary |
    neg_op ~ unary |
    await_op ~ unary |
    spawn_op ~ unary |
    postfix
}
```

New grammar:

```pest
unary = {
    not_op ~ unary |
    neg_op ~ unary |
    await_op ~ call_expr |
    spawn_op ~ call_expr |
    postfix
}

// A call_expr is a primary followed by call_args
call_expr = { primary ~ call_args }
```

When the parser encounters `await` followed by a non-call expression (e.g., `await 42`), the `call_expr` rule does not match. The parser-matches-everything philosophy means this still produces an AST node — an `Incomplete { kind: IncompleteKind::AwaitNonCall }` or similar — rather than failing.

### AST Change

Current:

```rust
Await {
    expr: Box<Expression>,
},
Spawn {
    expr: Box<Expression>,
},
```

New:

```rust
Await {
    callee: Box<Expression>,
    args: Vec<Expression>,
},
Spawn {
    callee: Box<Expression>,
    args: Vec<Expression>,
},
```

This mirrors the existing `FunctionCall` structure. The `callee` is the function expression and `args` are the call arguments. Since `await` and `spawn` now grammatically require a call, the callee/args split happens at parse time.

### Interpreter Change — Await

With the grammar guaranteeing a function call, `Expression::Await` can resolve the callee *before* calling it:

```rust
Expression::Await { callee, args } => {
    // 1. Audit async_annotation feature (existing)

    // 2. Resolve the callee
    let callee_val = self.eval_expression(task, yc, callee, env).await?;

    // 3. Check yielding classification before calling
    match &callee_val {
        Value::Function(f) if has_unyielding_entry(f) => {
            return Err(RuntimeError::system_error(
                format!("cannot await unyielding function '{}'",
                    f.name.as_deref().unwrap_or("<anonymous>")),
                "E012",
            ));
        }
        _ => {} // proceed
    }

    // 4. Evaluate arguments
    let arg_vals = eval_args(task, yc, args, env).await?;

    // 5. Call the function
    let val = self.call_function_inner(task, yc, &callee_val, &arg_vals).await?;

    // 6. If the result is a Future, await it
    if let Value::Future(ref future_ref) = val {
        // Take JoinHandle and .await (existing logic)
        match future_ref.take() {
            Some(handle) => match handle.await {
                Ok(result) => result,
                Err(join_err) => {
                    if join_err.is_cancelled() {
                        Err(RuntimeError::system_error(
                            "awaited task was cancelled".to_string(), "E011"))
                    } else {
                        Err(RuntimeError::system_error(
                            format!("awaited task panicked: {}", join_err), "E011"))
                    }
                }
            },
            None => Err(RuntimeError::system_error(
                "future has already been awaited".to_string(), "E011")),
        }
    } else {
        // Non-future result from a function without an unyielding entry.
        // This is the ambiguous case — pass through.
        // The function has no Yielding entry, so we cannot determine
        // whether it is truly yielding or unyielding.
        Ok(val)
    }
}
```

Key difference from v4: the callee is resolved and checked *before* calling, not after. The grammar guarantees `await` wraps a call, so callee/args are always available. E012 is thrown before the function executes if it has an unyielding entry. The ambiguous case (function with no Yielding entry returns non-Future) is a pass-through after the call.

### Interpreter Change — Spawn

```rust
Expression::Spawn { callee, args } => {
    // 1. Resolve the callee
    let callee_val = self.eval_expression(task, yc, callee, env).await?;

    // 2. Check yielding classification before spawning
    match &callee_val {
        Value::Function(f) if has_unyielding_entry(f) => {
            return Err(RuntimeError::system_error(
                format!("cannot spawn unyielding function '{}'",
                    f.name.as_deref().unwrap_or("<anonymous>")),
                "E013",
            ));
        }
        _ => {} // proceed
    }

    // 3. Evaluate arguments in the current context
    let arg_vals = eval_args(task, yc, args, env).await?;

    // 4. Spawn a new task that calls the function
    let mut spawned_vm = self.spawn_clone();
    let callee_clone = callee_val.clone();
    let handle = tokio::task::spawn_local(async move {
        let mut task_ctx = TaskContext::new();
        let mut yield_ctx = YieldContext::new();
        task_ctx.push_defer_frame();
        let result = spawned_vm.call_function_inner(
            &mut task_ctx, &mut yield_ctx, &callee_clone, &arg_vals,
        ).await;
        spawned_vm.pop_and_run_defers(&mut task_ctx, &mut yield_ctx).await;
        result
    });
    Ok(Value::Future(FutureRef::new(handle)))
}
```

Key difference from v4: spawn evaluates args in the parent context, then spawns a task that calls the already-resolved function with already-evaluated arguments. E013 is checked before spawning. The grammar prevents `spawn 42` from reaching this code at all (it would be an `Incomplete` node).

### E012 Decision Table

Since `await` now grammatically requires a function call, the table simplifies:

| Callee classification | Result type | Outcome |
|----------------------|-------------|---------|
| Function with yielding entry | `Future` | Normal await |
| Function with unyielding entry | — | **E012** (before call) |
| Function with no Yielding entry | `Future` | Normal await |
| Function with no Yielding entry | non-`Future` | Pass-through (ambiguous) |

Cases like `await 42` or `await "hello"` no longer reach the interpreter — the parser produces an `Incomplete` node.

### Error Catalog

Add E011, E012, E013 to `docs/errors/INDEX.md` and `docs/spec/errors.md`:

| Code | Structural Type | Summary | Production Sites |
|------|----------------|---------|-----------------|
| E011 | `ConcurrencyError` | Concurrency error — cancelled task, panicked task, assurance discrepancy, already-awaited future | `interpreter.rs` (await, spawn, audits) |
| E012 | `TypeError` | Await type mismatch — `await` applied to a call to an explicitly unyielding function | `interpreter.rs` (Expression::Await) |
| E013 | `ConcurrencyError` | Spawn type mismatch — `spawn` applied to a call to an explicitly unyielding function | `interpreter.rs` (Expression::Spawn) |

## Feature 3: Compiled Function Implementation

### Summary

The `IshFunction` struct currently has a `body: Statement` field that holds the interpreted AST body. This is replaced by an `implementation: FunctionImplementation` enum with three variants:

- **`Interpreted(Statement)`** — existing tree-walking execution via `exec_statement`
- **`Concurrent(Shim)`** — a compiled function that runs on the current async context, with the same signature as `exec_statement` (takes `&mut IshVm`, `&mut TaskContext`, `&mut YieldContext`, `&Environment`)
- **`Parallel(ParallelShim)`** — a compiled function that marshals arguments into `Send`-safe form, executes on `spawn_blocking`, and returns a `Future`

All builtins become regular `IshFunction` values with compiled implementations. `BuiltinFn` and `Value::BuiltinFunction` are eliminated. The `Value` enum loses the `BuiltinFunction` variant — builtins are just `Value::Function` with a `Concurrent` or `Parallel` implementation. To an outside observer, builtins are indistinguishable from regular functions.

### `FunctionImplementation` Type

```rust
/// The execution strategy for a function.
pub enum FunctionImplementation {
    /// Tree-walking interpreter — existing behavior for user-defined functions.
    Interpreted(Statement),

    /// Compiled synchronous function — runs on the current async context.
    /// Has access to IshVm, TaskContext, YieldContext, Environment.
    Concurrent(ConcurrentShim),

    /// Compiled parallel function — marshals args, runs on spawn_blocking,
    /// returns a Future.
    Parallel(ParallelShim),
}

/// A compiled function that executes in the current async context.
/// Same capability as exec_statement — can call other functions,
/// access the environment, yield, etc.
pub type ConcurrentShim = Rc<
    dyn for<'a> Fn(
        &'a mut IshVm,
        &'a mut TaskContext,
        &'a mut YieldContext,
        &'a Environment,
    ) -> Pin<Box<dyn Future<Output = Result<ControlFlow, RuntimeError>> + 'a>>
>;

/// A compiled function that marshals its arguments into Send-safe form
/// for execution on a blocking thread.
pub type ParallelShim = Rc<
    dyn Fn(&Environment) -> Result<ParallelCall, RuntimeError>
>;
```
--> There should only be two kinds of FunctionImplementation: Interpreted and Compiled.  And there should only be one kind of shim: Shim.  Shims are not async.  If the shim is wrapping an unyielding function, it marshalls the arguments, calls the function, and marshalls the result into a Value, which it returns.  If the shim is wrapping a yielding function, it marshalls the arguments, spawns the function using LocalSet, and returns a Value of type Future. If the shim is wrapping a parallel function, it marshalls the arguments, spawns the function as a regular task (or calls a function that returns a future so it doesn't have to do any task spawning itself), and returns a Value of type Future.  I'm a little unclear about how/when the shim has a chance to marshall the result from the internal TokioFuture into the Value held by the Value::Future.  For all shims, marshalling the parameters and the result to/from VM Value objects is the shim's responsibility.

Note: `ConcurrentShim` takes `&Environment` (where the callee's arguments have already been bound by `call_function_inner`). The shim reads parameter values from the environment using the function's param names. `ParallelShim` also takes `&Environment` and reads arguments the same way, but produces a `ParallelCall` (Send-safe) instead of executing directly.

### `IshFunction` Change

```rust
// Current:
pub struct IshFunction {
    pub name: Option<String>,
    pub params: Vec<String>,
    pub param_types: Vec<Option<ish_ast::TypeAnnotation>>,
    pub return_type: Option<ish_ast::TypeAnnotation>,
    pub body: Statement,           // <-- interpreted body
    pub closure_env: Environment,
    pub is_async: bool,
}

// New:
pub struct IshFunction {
    pub name: Option<String>,
    pub params: Vec<String>,
    pub param_types: Vec<Option<ish_ast::TypeAnnotation>>,
    pub return_type: Option<ish_ast::TypeAnnotation>,
    pub implementation: FunctionImplementation,  // <-- replaces body
    pub closure_env: Environment,
    pub is_async: bool,
}
```

The `is_async` field serves as the yielding classification:
- `is_async: false` — unyielding (simple builtins, functions with `@[unyielding]`)
- `is_async: true` — yielding (parallel builtins, functions with `async` keyword)
- Functions with neither annotation leave `is_async: false` but have no explicit unyielding annotation — this is the ambiguous case for E012.

To distinguish the ambiguous case from explicitly-unyielding, a separate mechanism is needed: either checking for `@[unyielding]` in the function's annotations (for interpreted functions) or a flag on `IshFunction`. The simplest approach is a new field:

```rust
pub has_yielding_entry: Option<bool>,
// None = no Yielding entry (ambiguous)
// Some(true) = yielding (async, parallel)
// Some(false) = unyielding (explicit @[unyielding])
```

### Dispatch in `call_function_inner`

With `Value::BuiltinFunction` eliminated, `call_function_inner` only handles `Value::Function`:

```rust
fn call_function_inner<'a>(...) -> Pin<Box<dyn Future<...>>> {
    Box::pin(async move {
        match func {
            Value::Function(f) => {
                // 1. Arity check (same for all implementations)
                if args.len() != f.params.len() {
                    return Err(RuntimeError::system_error(
                        format!("'{}' expects {} arguments, got {}",
                            f.name.as_deref().unwrap_or("<fn>"),
                            f.params.len(), args.len()),
                        "E003",
                    ));
                }

                // 2. Audit parameter types (same for all implementations)
                // ...existing param type audit logic...

                // 3. Create call environment and bind parameters
                let call_env = f.closure_env.child();
                for (name, val) in f.params.iter().zip(args.iter()) {
                    call_env.define(name, val.clone());
                }

                // 4. Dispatch by implementation
                match &f.implementation {
                    FunctionImplementation::Interpreted(body) => {
                        // Existing logic: push defer frame, async_stack,
                        // yield budget, exec_statement, pop, audit return
                        // ...
                    }
                    FunctionImplementation::Concurrent(shim) => {
                        // Call the shim with VM access
                        let cf = shim(self, task, yc, &call_env).await?;
                        match cf {
                            ControlFlow::Return(v) => Ok(v),
                            ControlFlow::Normal => Ok(Value::Null),
                            _ => Ok(Value::Null),
                        }
                    }
                    FunctionImplementation::Parallel(shim) => {
                        // Marshal args and spawn
                        let call = shim(&call_env)?;
                        let handle = tokio::task::spawn_blocking(
                            move || execute_parallel(call)
                        );
                        let local_handle = tokio::task::spawn_local(async move {
                            match handle.await {
                                Ok(Ok(result)) => Ok(result.into_value()),
                                Ok(Err(msg)) => Err(RuntimeError::system_error(
                                    msg, "E008")),
                                Err(join_err) => Err(RuntimeError::system_error(
                                    format!("parallel function panicked: {}",
                                        join_err), "E010")),
                            }
                        });
                        Ok(Value::Future(FutureRef::new(local_handle)))
                    }
                }
            }
            _ => Err(RuntimeError::system_error(
                format!("cannot call {}", func.type_name()), "E006"
            )),
        }
    })
}
```
--> Shims don't need to be awaited.  They are not async.

### Builtin Registration

All builtins are registered as `IshFunction` values via `new_function` (or a new helper). The `BuiltinConfig` (output channel) is captured in closures.

**Simple builtins** use `FunctionImplementation::Concurrent`:

```rust
// Example: len(value: any) -> int
let len_shim: ConcurrentShim = Rc::new(|_vm, _task, _yc, env| {
    Box::pin(async move {
        let value = env.get("value").unwrap();
        let result = match &value {
            Value::String(s) => Value::Int(s.len() as i64),
            Value::List(l) => Value::Int(l.borrow().len() as i64),
            _ => return Err(RuntimeError::system_error(
                format!("len: unsupported type '{}'", value.type_name()), "E004"
            )),
        };
        Ok(ControlFlow::Return(result))
    })
});

env.define("len", new_function(
    Some("len".into()),
    vec!["value".into()],
    vec![None],    // param type: any
    Some(TypeAnnotation::Simple("int".into())),
    FunctionImplementation::Concurrent(len_shim),
    global_env.clone(),
    false,  // is_async = false (unyielding)
));
```

Simple builtins have `is_async: false` and `has_yielding_entry: Some(false)` — explicitly unyielding. `await len(...)` produces E012 and `spawn len(...)` produces E013.

**Parallel builtins** use `FunctionImplementation::Parallel`:

```rust
// Example: println(value: any) -> null
let println_shim: ParallelShim = Rc::new(|env| {
    let value = env.get("value").unwrap();
    let text = format_value(&value);
    Ok(ParallelCall::Print { text, newline: true })
});

env.define("println", new_function(
    Some("println".into()),
    vec!["value".into()],
    vec![None],    // param type: any
    Some(TypeAnnotation::Simple("null".into())),
    FunctionImplementation::Parallel(println_shim),
    global_env.clone(),
    true,  // is_async = true (yielding)
));
```

Parallel builtins have `is_async: true` and `has_yielding_entry: Some(true)` — explicitly yielding.

**Ledger builtins** use `FunctionImplementation::Concurrent` (they need `&mut IshVm` access for the ledger):

```rust
// Example: active_standard() -> string
let active_standard_shim: ConcurrentShim = Rc::new(|vm, _task, _yc, _env| {
    Box::pin(async move {
        // Access vm.ledger directly
        let name = vm.ledger.active_standard_name();
        Ok(ControlFlow::Return(Value::String(Rc::new(name))))
    })
});
```

### Implied Await for Parallel Functions

When `call_function_inner` returns a `Value::Future` from a `Parallel` implementation, the behavior depends on the call context:

- **Called via `Expression::Await`:** The `Await` handler (Feature 2) receives the `Future` and awaits it. The resolved value is returned.
- **Called via `Expression::Spawn`:** The `Spawn` handler wraps the call in a new task. The parallel function runs inside the spawned task, producing a `Future` that is already awaited within the spawn context. The outer spawn returns a `Future` wrapping the whole task.
- **Called directly (bare call, no await/spawn):** The implied-await logic kicks in. If `await_required` is not `required`, the interpreter immediately awaits the future. If `await_required` is `required`, the VM throws an error (the `await_required` audit).

The implied-await logic for bare calls to parallel functions is implemented in the `Expression::FunctionCall` handler (or in `call_function_inner` itself). When a call returns `Value::Future` and the call is not wrapped in `await`/`spawn`:

```rust
// In the FunctionCall handler (not Await, not Spawn):
let result = self.call_function_inner(task, yc, &callee_val, &arg_vals).await?;

if let Value::Future(ref future_ref) = result {
    // Check await_required
    let features = self.ledger.active_features();
    if is_await_required(&features) {
        // The await_required audit will catch this — return the Future
        // so the unawaited-future audit can report it
        Ok(result)
    } else {
        // Implied await at low assurance
        match future_ref.take() {
            Some(handle) => match handle.await {
                Ok(inner) => inner,
                Err(join_err) => Err(RuntimeError::system_error(
                    format!("implied await failed: {}", join_err), "E011")),
            },
            None => Err(RuntimeError::system_error(
                "future has already been awaited".to_string(), "E011")),
        }
    }
} else {
    Ok(result)
}
```

### Print/Println I/O Completion

For `print`/`println`, the `BuiltinConfig::output_sender` (crossbeam channel for ExternalPrinter) is captured in the `ParallelShim` closure. The execution path differs by mode:

**Non-interactive mode:** `execute_parallel` writes directly to stdout. The future resolves when the write completes.

**Interactive mode (output_sender active):**

1. The `ParallelShim` closure produces a `ParallelCall::Print` as usual
2. `execute_parallel` is extended: for `Print` calls, if an `output_sender` is available, send the message along with a `oneshot::Sender<()>` through the crossbeam channel
3. The shell receives the message, writes it, and sends acknowledgment back through the `oneshot`
4. The `spawn_blocking` task (or `spawn_local` bridge) waits for the acknowledgment before resolving
5. The future does not resolve until the output is actually written

| Builtin | Params | Args marshaled | Result |

### Test Impact

Because of implied await at low assurance, **existing tests do not need to change**. `println("hello")` continues to work — the VM internally creates a future, immediately awaits it, and returns the resolved value.

Tests under `@standard[cautious]` or any standard where `await_required` is `required` will need explicit `await`:

```ish
@standard[cautious] {
    await println("hello")    // required — no implied await
    let content = await read_file("path.txt")   // required
}
```

New tests should cover:
- Parallel builtins with implied await (low assurance) — output ordering preserved
- Parallel builtins with explicit await (high assurance)
- Parallel builtins with `spawn` — returns `Future`, output may be deferred
- `read_file` and `write_file` in parallel via `spawn` — concurrent I/O
- Parameter count errors on builtins (same E003 as regular functions)
- `await len([1,2])` → E012 (unyielding entry)
- `spawn len([1,2])` → E013 (unyielding entry)
- `await 42` → parse error (Incomplete node)
- `spawn 42` → parse error (Incomplete node)
