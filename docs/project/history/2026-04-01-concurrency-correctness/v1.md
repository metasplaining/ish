## Feature 1: FutureRef Identity Equality

### Issues to Watch Out For

- The `FutureRef` struct wraps `Rc<RefCell<Option<JoinHandle<...>>>>`. Cloning a `FutureRef` creates a new `Rc` pointing to the same `RefCell`. Two clones genuinely refer to the same future — pointer equality via `Rc::ptr_eq` is the correct identity check.
- After `take()`, the inner `Option` becomes `None`, but the `Rc` pointer identity is preserved. Two references to a consumed future should still be equal.
- The `id: u64` field added for the unawaited-future tracking system provides an alternative identity mechanism, but `Rc::ptr_eq` is more fundamental and doesn't require an extra field for equality purposes.
- Objects and Lists already use deep structural equality via `*a == *b` inside `GcCell::borrow()`. Futures are intentionally different — there is no meaningful structural comparison of `JoinHandle` internals.

### Critical Analysis

**Alternative A: `Rc::ptr_eq` on the inner `Rc<RefCell<...>>`** (recommended)

Two `Value::Future` values are equal if and only if their `FutureRef::inner` pointers are identical. This is the standard Rust idiom for identity-based equality of reference-counted values.

- Pros: Correct identity semantics. Zero overhead (pointer comparison). Matches how identity equality works in every mainstream language.
- Cons: None identified.

**Alternative B: Compare the `id: u64` field**

Two futures are equal if they have the same tracking ID.

- Pros: Simpler to understand. Works even if the `Rc` is somehow reconstructed (not possible in practice).
- Cons: The `id` field exists for the unawaited-future audit, not for equality — coupling these concerns is poor design. IDs are assigned sequentially, so there's a theoretical wraparound at `u64::MAX` (not a practical concern, but still an unnecessary dependency).

**Alternative C: Never equal (current behavior)**

- Pros: Conservative — prevents any accidental equality assumption.
- Cons: Incorrect. `let f = spawn work(); f == f` evaluates to `false`, which violates the basic expectation that a value equals itself (reflexivity). This breaks hash-map usage, set membership, and find operations.

### Proposed Implementation

Single line change in `proto/ish-vm/src/value.rs`:

```rust
// In impl PartialEq for Value:
(Value::Future(a), Value::Future(b)) => Rc::ptr_eq(&a.inner, &b.inner),
```

Update the comment from "Futures are never equal (identity-based, not value-based)" to "Futures use identity equality (same Rc pointer)".

### Decisions

**Decision 1:** Which equality mechanism for futures — `Rc::ptr_eq` (alternative A), `id` comparison (alternative B), or keep never-equal (alternative C)?
--> `Rc::ptr_eq` (alternative A)

## Feature 2: Await on Non-Future Is Always an Error

### Issues to Watch Out For

- Two existing acceptance tests (`spawn_await.sh`) assert pass-through behavior: `await 42` returns `42`, `await "hello"` returns `"hello"`. These tests must be updated to expect errors.
- The concurrency proposal (Decision 3) says "Default to await; low-assurance auto-upgrades" — this refers to **implicit await when calling async functions**, not to `await` accepting arbitrary value types. The implementation conflated these two concepts.
- Build-time detection of `await` on a statically-known non-future type requires type inference or type checking, which doesn't exist yet in the prototype. The high-assurance build-time error is a future concern — the runtime error is what matters now.
- The `await` keyword sits in `Expression::Await` in the AST. The inner expression can be any expression. At runtime, after evaluating the inner expression, the value is either `Value::Future` or something else.
- This error should use a new error code (E012 suggested), not E011, because E011 covers concurrency scheduling/audit issues, while this is a type mismatch at the await site.
- The error message should be clear: "cannot await a non-future value of type <actual_type>".

### Critical Analysis

**Alternative A: Always a runtime error, regardless of assurance level** (recommended for current prototype)

`await <non-future>` always produces a `TypeError` (E004) at runtime. No assurance-level distinction in the prototype since build-time type checking doesn't exist yet.

- Pros: Simple to implement. Catches bugs immediately. Consistent behavior regardless of configuration. Uses existing error code E004 (type mismatch).
- Cons: Doesn't demonstrate the assurance-level distinction the RFP envisions. But this is a prototype — build-time checking can be added when a type checker exists.

**Alternative B: Runtime error at all levels, with different error codes**

Low assurance: E004 (TypeError). High assurance: E004 but with an additional discrepancy flagged through the ledger audit system.

- Pros: Demonstrates assurance-level awareness even without build-time checking. The auditor could flag `await` on non-futures as a discrepancy at higher standards.
- Cons: Over-engineering. The await-on-non-future is always wrong — it doesn't need assurance-level nuance. It's not like `async_annotation` where the requirement varies by standard.

**Alternative C: Use a new error code E012 for await-type errors**

Introduce E012 specifically for "await type mismatch" errors.

- Pros: Clear provenance — E012 always means "await was applied to the wrong type."
- Cons: E004 (TypeError) already covers "operation applied to incompatible types." Await-on-non-future is a type mismatch. A new code isn't strictly necessary.

### Proposed Implementation

In `proto/ish-vm/src/interpreter.rs`, the `Expression::Await` handler currently has:

```rust
} else {
    // At low assurance: awaiting a non-future value returns it directly
    Ok(val)
}
```

Replace with:

```rust
} else {
    Err(RuntimeError::system_error(
        format!("cannot await a value of type '{}' — await requires a Future", val.type_name()),
        "E004",
    ))
}
```

Update two tests in `spawn_await.sh` from expecting pass-through to expecting an error.

### Decisions

**Decision 2:** Should await-on-non-future be a plain runtime error at all levels (alternative A), an assurance-aware error (alternative B), or use a new error code E012 (alternative C)?
--> await-on-non-future should be a plain runtime error at all levels using a new error code E012.  Also note that this is partially unimplementable at this time.  Awaiting an async function should succeed.  Awaiting a unyielding function should throw an E012 error.  Awaiting a function that is neither async or unyielding should not be possible, because every function should be categorized as one or the other at the time it is declared.  But that is not yet implemented, or even specified.  For now, we should allow functions with no Yielding entry to be awaited, to support code like `fn my_fn(){ println('I yield') }`.  We should, however, create an open question about this, so that we don't forget to fix it when function declaration behavior is improved.

**Decision 3:** Should the error use existing E004 (TypeError) or new E012?
--> new E012

## Feature 3: Parallel Builtins

### Issues to Watch Out For

- The current `BuiltinFn` signature is `Fn(&[Value]) -> Result<Value, RuntimeError>`. This is synchronous and runs on the Tokio `LocalSet` thread. Any blocking operation (file I/O, stdout write) blocks the entire task set.
- `print`/`println` already mitigate this in interactive mode by using `crossbeam::channel::Sender::send()` which is non-blocking (unbounded channel). But in non-interactive mode they use `println!()` which blocks on stdout.
- `read_file` and `write_file` use `std::fs::read_to_string` and `std::fs::write` — these are synchronous filesystem calls that block the thread.
- `tokio::task::spawn_blocking` moves work to a threadpool, but returns a `JoinHandle<T>` that must be `.await`ed. This means the calling code in the interpreter must be async at the call site.
- `Value` is `!Send` (it uses `Gc<GcCell<...>>` from the `gc` crate, which is single-threaded). This means we **cannot** pass `Value` arguments directly to `spawn_blocking`. Values must be converted to owned/Send types before crossing the thread boundary, and results must be converted back.
- For `print`/`println`, we only need `String` (already `Send`). For `read_file`, we pass a `String` path and get back a `String` content. For `write_file`, we pass `String` path + `String` content. All currently-blocking builtins happen to deal in `String` at the interface boundary.
- Concurrency proposal Decision 13 eliminated parallel functions that accept ish closures. Builtins don't accept closures, so this constraint is satisfied.
- The ledger spec defines `Parallel` as an entry type for functions that run on the thread pool. Parallel builtins should carry this entry implicitly.
- From the ish user's perspective, a parallel builtin call looks identical to a simple builtin call. The parallelism is transparent — no `await` is required from the caller for a builtin (regardless of assurance level). This matches Decision 3: at low assurance, async aspects are implicit.
--> This item is incorrect.  When the `await_required` feature is set to `required`, parallel builtins should require an `await` or a `spawn`, just like any other yielding function. 

### Critical Analysis

**Alternative A: Two-signature builtins — sync closures and async closures** (recommended)

Add a second builtin variant alongside the existing one:

```rust
pub enum BuiltinKind {
    Simple(Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>),
    Parallel(Rc<dyn Fn(&[Value]) -> Result<ParallelArgs, RuntimeError>>),
}
```

Where `ParallelArgs` is a `Send`-safe capture of the arguments, and the interpreter handles the `spawn_blocking` + `.await` + result conversion in `call_function_inner`.

- Pros: Clean separation. The interpreter knows which builtins might block. Simple builtins stay zero-overhead. Parallel builtins are transparent to the user.
- Cons: Requires a marshaling protocol for converting args to Send-safe types and results back.

**Alternative B: All I/O builtins return `Value::Future` explicitly**

Parallel builtins return a `Future` wrapping a `spawn_blocking` task. The user must `await` them (or they're auto-awaited at low assurance).

- Pros: Consistent with how user-defined async functions work. Makes the async nature visible at higher assurance levels.
- Cons: Significantly more complex. Changes the call semantics for fundamental operations like `println()`. At low assurance, every `println()` would need implicit await logic. At high assurance, users would write `await println("hello")` — awkward.

**Alternative C: Use `tokio::task::spawn_blocking` inline, await immediately**

Don't change the builtin signature. Instead, in `register_io`, use `tokio::runtime::Handle::current().block_on(spawn_blocking(...))` to block synchronously on the thread pool result.

- Pros: No signature changes. Minimal code change.
- Cons: `block_on` inside an async context panics in Tokio. This doesn't work with `LocalSet`. Would require a dedicated thread or complex workaround.

**Alternative D: Keep the crossbeam channel pattern, extend to file I/O**

Route `read_file`/`write_file` through channels to a dedicated I/O thread, similar to how `println` uses `crossbeam::channel::Sender`.

- Pros: Proven pattern (already works for println). No Send issues with Values — only strings cross the channel.
- Cons: Adding request/response channels for file I/O is more complex than println (println is fire-and-forget, file I/O needs a response). Essentially reinvents `spawn_blocking`.

### Proposed Implementation (Alternative A)

**Step 1: Define parallel builtin types**

In `proto/ish-vm/src/value.rs`:

```rust
/// Arguments marshaled into Send-safe form for a parallel builtin.
pub enum ParallelCall {
    /// print/println: formatted output string, whether to append newline
    Print { text: String, newline: bool },
    /// read_file: path
    ReadFile { path: String },
    /// write_file: path, content
    WriteFile { path: String, content: String },
}

/// Result from a parallel builtin, marshaled back into a Value.
pub enum ParallelResult {
    Null,
    Text(String),
}

impl ParallelResult {
    pub fn into_value(self) -> Value {
        match self {
            ParallelResult::Null => Value::Null,
            ParallelResult::Text(s) => Value::String(Rc::new(s)),
        }
    }
}
```

**Step 2: Add parallel variant to BuiltinFn**

```rust
pub enum BuiltinKind {
    Simple(Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>),
    Parallel(Rc<dyn Fn(&[Value]) -> Result<ParallelCall, RuntimeError>>),
}
```

**Step 3: Update builtin registration**

`print`, `println`, `read_file`, and `write_file` register as `Parallel` builtins. Their closures validate arguments and produce a `ParallelCall` instead of executing the I/O directly.

**Step 4: Update interpreter dispatch**

In `call_function_inner`, when a `Parallel` builtin is matched:

1. Call the closure to get `ParallelCall` (validates args, marshals to Send types)
2. Use `tokio::task::spawn_blocking(move || execute_parallel(call))` to run I/O on the threadpool
3. `.await` the join handle
4. Convert `ParallelResult` back to `Value`

For `print`/`println` specifically, in interactive mode, the existing `crossbeam::channel::Sender` pattern remains the better choice (fire-and-forget). The implementation can check whether an output sender is available and use it, falling back to `spawn_blocking` for non-interactive mode.

**Step 5: Update BuiltinConfig**

The `BuiltinConfig` already carries `output_sender`. No changes needed for print routing.

### Decisions

**Decision 4:** Which mechanism for parallel builtins — two-signature (A), user-visible Future return (B), inline block_on (C), or channel-based (D)?
--> two-signature

**Decision 5:** Which builtins become parallel? Recommended: `print`, `println`, `read_file`, `write_file`. Are there others?
--> No

**Decision 6:** Should parallel builtins be transparent to the user (auto-awaited internally by the interpreter), or should they return `Future<T>` that callers must handle?
--> They should return a future that callers must handle.  Note that builtins are not intended to be part of the language at all.  They will be completely replaced by standard library functions later, when there is a standard library.  Until that happens, their interface should conform to normal function interfaces as much as is possible.  Add an open question so that we remember to remove the builtins when the standard library becomes available.
