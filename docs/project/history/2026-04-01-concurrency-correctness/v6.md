## Decision Register

All decisions made during design, consolidated here as the authoritative reference.

| # | Decision | Outcome |
|---|----------|---------|
| 1 | FutureRef equality semantics | `Rc::ptr_eq` identity equality |
| 2 | Await-on-non-future behavior | E012 thrown in clear-cut cases: awaiting a call to a function with an explicit unyielding entry. **Pass-through only for the ambiguous case:** calling a function with no Yielding entry that returns a non-Future. |
| 3 | Error code for await-on-non-future | E012 — await type mismatch. Active in prototype for clear-cut cases. |
| 4 | Spawn-on-non-function behavior | E013 — spawning a call to a function with an explicit unyielding entry. |
| 5 | Builtin mechanism | Builtins become compiled functions (`FunctionImplementation::Compiled(Shim)`). No separate `BuiltinFn`/`Value::BuiltinFunction`. |
| 6 | Which builtins become parallel | `print`, `println`, `read_file`, `write_file` only |
| 7 | Parallel builtin return type | Return `Future<T>`. At low assurance, calling without `await`/`spawn` implies `await`. The only way to obtain a `Future` value is via `spawn`. |
| 8 | Parallel builtins and `await_required` | When `await_required` is `required`, VM throws if no explicit `await` or `spawn`. Caller never receives a bare `Future`. |
| 9 | Parallel builtin I/O completion | Future must not resolve until the I/O operation is actually complete. Shell communication requires an acknowledgment message back after output is written. |
| 10 | Builtin–function parity | Builtins are regular `IshFunction` values with compiled implementations. They carry parameter metadata (names, types), return type, and yielding classification. `Value::BuiltinFunction` is eliminated. Dispatch through `call_function_inner` is unified. |
| 11 | Grammar-level await/spawn restriction | `await` and `spawn` take a function call in the grammar, not an arbitrary expression. `await_op ~ call_expr` instead of `await_op ~ unary`. The AST `Await` and `Spawn` nodes contain callee + args. Invalid forms like `await 42` still parse (parser-matches-everything) but produce `Incomplete` nodes. |
| 12 | FunctionImplementation enum | Two variants only: `Interpreted(Statement)` and `Compiled(Shim)`. Shims are synchronous functions. The shim is responsible for marshalling parameters from `Value` to native types and marshalling results back to `Value`. For unyielding functions, the shim calls directly and returns a `Value`. For yielding or parallel functions, the shim internally spawns the work and returns `Value::Future`. |

## Feature 1: FutureRef Identity Equality

### Summary

The `PartialEq` implementation for `Value::Future` currently returns `false` for all comparisons. This violates reflexivity (`f == f` is `false`) and is incorrect. Futures use identity-based equality: two future values are equal if and only if they reference the same underlying `Rc`.

### Proposed Implementation

Single change in `proto/ish-vm/src/value.rs`, in `impl PartialEq for Value`:

```rust
// Futures use identity equality (same Rc pointer)
(Value::Future(a), Value::Future(b)) => Rc::ptr_eq(&a.inner, &b.inner),
```

This replaces the current line:

```rust
// Futures are never equal (identity-based, not value-based)
(Value::Future(_), Value::Future(_)) => false,
```

### Tests

- `let f = spawn work(); f == f` → `true`
- `let f1 = spawn work(); let f2 = spawn work(); f1 == f2` → `false`
- `let f = spawn work(); let g = f; f == g` → `true` (same Rc via clone)

## Feature 2: Grammar-Level Spawn/Await Restriction

### Summary

Currently, `await` and `spawn` are unary prefix operators in the grammar (`await_op ~ unary`), meaning they can be applied to any expression: `await 42`, `spawn "hello"`, etc. The AST `Await` and `Spawn` nodes wrap a generic `Box<Expression>`.

This is a grammar-level error. `await` and `spawn` should only accept function calls. The grammar is changed so these operators take a `call_expr` (function call expression) as their operand. The AST nodes are changed to contain callee + args instead of a generic `Expression`.

Invalid forms like `await 42` still parse — parser-matches-everything — but produce an `Incomplete` AST node rather than a valid `Await` node. This moves the structural validation from the interpreter to the parser.

### Grammar Change

Current grammar:

```pest
unary = {
    not_op ~ unary |
    neg_op ~ unary |
    await_op ~ unary |
    spawn_op ~ unary |
    postfix
}
```

New grammar:

```pest
unary = {
    not_op ~ unary |
    neg_op ~ unary |
    await_op ~ call_expr |
    spawn_op ~ call_expr |
    postfix
}

// A call_expr is a primary followed by call_args
call_expr = { primary ~ call_args }
```

When the parser encounters `await` followed by a non-call expression (e.g., `await 42`), the `call_expr` rule does not match. The parser-matches-everything philosophy means this still produces an AST node — an `Incomplete { kind: IncompleteKind::AwaitNonCall }` or similar — rather than failing.

### AST Change

Current:

```rust
Await {
    expr: Box<Expression>,
},
Spawn {
    expr: Box<Expression>,
},
```

New:

```rust
Await {
    callee: Box<Expression>,
    args: Vec<Expression>,
},
Spawn {
    callee: Box<Expression>,
    args: Vec<Expression>,
},
```

This mirrors the existing `FunctionCall` structure. The `callee` is the function expression and `args` are the call arguments. Since `await` and `spawn` now grammatically require a call, the callee/args split happens at parse time.

### Interpreter Change — Await

With the grammar guaranteeing a function call, `Expression::Await` can resolve the callee *before* calling it:

```rust
Expression::Await { callee, args } => {
    // 1. Audit async_annotation feature (existing)

    // 2. Resolve the callee
    let callee_val = self.eval_expression(task, yc, callee, env).await?;

    // 3. Check yielding classification before calling
    match &callee_val {
        Value::Function(f) if has_unyielding_entry(f) => {
            return Err(RuntimeError::system_error(
                format!("cannot await unyielding function '{}'",
                    f.name.as_deref().unwrap_or("<anonymous>")),
                "E012",
            ));
        }
        _ => {} // proceed
    }

    // 4. Evaluate arguments
    let arg_vals = eval_args(task, yc, args, env).await?;

    // 5. Call the function
    let val = self.call_function_inner(task, yc, &callee_val, &arg_vals).await?;

    // 6. If the result is a Future, await it
    if let Value::Future(ref future_ref) = val {
        match future_ref.take() {
            Some(handle) => match handle.await {
                Ok(result) => result,
                Err(join_err) => {
                    if join_err.is_cancelled() {
                        Err(RuntimeError::system_error(
                            "awaited task was cancelled".to_string(), "E011"))
                    } else {
                        Err(RuntimeError::system_error(
                            format!("awaited task panicked: {}", join_err), "E011"))
                    }
                }
            },
            None => Err(RuntimeError::system_error(
                "future has already been awaited".to_string(), "E011")),
        }
    } else {
        // Non-future result from a function without an unyielding entry.
        // This is the ambiguous case — pass through.
        Ok(val)
    }
}
```

### Interpreter Change — Spawn

```rust
Expression::Spawn { callee, args } => {
    // 1. Resolve the callee
    let callee_val = self.eval_expression(task, yc, callee, env).await?;

    // 2. Check yielding classification before spawning
    match &callee_val {
        Value::Function(f) if has_unyielding_entry(f) => {
            return Err(RuntimeError::system_error(
                format!("cannot spawn unyielding function '{}'",
                    f.name.as_deref().unwrap_or("<anonymous>")),
                "E013",
            ));
        }
        _ => {} // proceed
    }

    // 3. Evaluate arguments in the current context
    let arg_vals = eval_args(task, yc, args, env).await?;

    // 4. Spawn a new task that calls the function
    let mut spawned_vm = self.spawn_clone();
    let callee_clone = callee_val.clone();
    let handle = tokio::task::spawn_local(async move {
        let mut task_ctx = TaskContext::new();
        let mut yield_ctx = YieldContext::new();
        task_ctx.push_defer_frame();
        let result = spawned_vm.call_function_inner(
            &mut task_ctx, &mut yield_ctx, &callee_clone, &arg_vals,
        ).await;
        spawned_vm.pop_and_run_defers(&mut task_ctx, &mut yield_ctx).await;
        result
    });
    Ok(Value::Future(FutureRef::new(handle)))
}
```

### E012 Decision Table

Since `await` now grammatically requires a function call, the table simplifies:

| Callee classification | Result type | Outcome |
|----------------------|-------------|---------|
| Function with yielding entry | `Future` | Normal await |
| Function with unyielding entry | — | **E012** (before call) |
| Function with no Yielding entry | `Future` | Normal await |
| Function with no Yielding entry | non-`Future` | Pass-through (ambiguous) |

Cases like `await 42` or `await "hello"` no longer reach the interpreter — the parser produces an `Incomplete` node.

### Error Catalog

| Code | Structural Type | Summary | Production Sites |
|------|----------------|---------|-----------------|
| E011 | `ConcurrencyError` | Concurrency error — cancelled task, panicked task, assurance discrepancy, already-awaited future | `interpreter.rs` (await, spawn, audits) |
| E012 | `TypeError` | Await type mismatch — `await` applied to a call to an explicitly unyielding function | `interpreter.rs` (Expression::Await) |
| E013 | `ConcurrencyError` | Spawn type mismatch — `spawn` applied to a call to an explicitly unyielding function | `interpreter.rs` (Expression::Spawn) |

## Feature 3: Compiled Function Implementation

### Summary

The `IshFunction` struct currently has a `body: Statement` field that holds the interpreted AST body. This is replaced by an `implementation: FunctionImplementation` enum with two variants:

- **`Interpreted(Statement)`** — existing tree-walking execution via `exec_statement`
- **`Compiled(Shim)`** — a synchronous shim function that is responsible for marshalling parameters from `Value` to native types, executing the function (or spawning it), and marshalling results back to `Value`

All builtins become regular `IshFunction` values with `Compiled` implementations. `BuiltinFn` and `Value::BuiltinFunction` are eliminated. The `Value` enum loses the `BuiltinFunction` variant — builtins are just `Value::Function` with a `Compiled` implementation. To an outside observer, builtins are indistinguishable from regular functions.

### How Shims Work

A shim is a **synchronous** function. It receives `&[Value]` (the arguments, already arity-checked and type-audited by `call_function_inner`) and returns `Result<Value, RuntimeError>`. What the shim does internally depends on the function it wraps:

**Unyielding shim** — calls the underlying function directly, returns a plain `Value`:

```rust
// len(value) -> int
fn len_shim(args: &[Value]) -> Result<Value, RuntimeError> {
    match &args[0] {
        Value::String(s) => Ok(Value::Int(s.len() as i64)),
        Value::List(l) => Ok(Value::Int(l.borrow().len() as i64)),
        _ => Err(RuntimeError::system_error(
            format!("len: unsupported type '{}'", args[0].type_name()), "E004"
        )),
    }
}
```

**Yielding shim** — spawns the work on the `LocalSet` via `spawn_local`, returns `Value::Future`:

```rust
// A hypothetical yielding builtin
fn yielding_shim(args: &[Value]) -> Result<Value, RuntimeError> {
    let captured_arg = args[0].clone();
    let handle = tokio::task::spawn_local(async move {
        // ... do async work ...
        Ok(some_value)
    });
    Ok(Value::Future(FutureRef::new(handle)))
}
```

**Parallel shim** — marshals arguments into `Send`-safe form, spawns on `spawn_blocking` (or calls a function that returns a future), returns `Value::Future`:

```rust
// read_file(path: string) -> string
fn read_file_shim(args: &[Value]) -> Result<Value, RuntimeError> {
    let path = match &args[0] {
        Value::String(s) => (**s).clone(),
        _ => return Err(RuntimeError::system_error(
            "read_file: expected string path", "E004"
        )),
    };
    let blocking_handle = tokio::task::spawn_blocking(move || {
        std::fs::read_to_string(&path)
            .map_err(|e| format!("{}: {}", path, e))
    });
    // Bridge: spawn_local converts the native result back to Value
    let local_handle = tokio::task::spawn_local(async move {
        match blocking_handle.await {
            Ok(Ok(content)) => Ok(Value::String(Rc::new(content))),
            Ok(Err(msg)) => Err(RuntimeError::system_error(msg, "E008")),
            Err(join_err) => Err(RuntimeError::system_error(
                format!("parallel function panicked: {}", join_err), "E010"
            )),
        }
    });
    Ok(Value::Future(FutureRef::new(local_handle)))
}
```

The key insight: `spawn_local` and `spawn_blocking` are non-async — they schedule work and return a `JoinHandle` immediately. The shim is synchronous: it sets up the spawn chain, wraps the handle in `FutureRef`, and returns. The `spawn_local` bridge is where the native result (e.g., `String` from `read_to_string`) is marshalled back into `Value` (e.g., `Value::String(Rc::new(content))`). This bridge runs asynchronously when the future is eventually awaited, but the shim itself returns immediately.

### `FunctionImplementation` Type

```rust
/// The execution strategy for a function.
pub enum FunctionImplementation {
    /// Tree-walking interpreter — existing behavior for user-defined functions.
    Interpreted(Statement),

    /// Compiled function — a synchronous shim that marshals params/results
    /// and either executes directly or spawns work.
    Compiled(Shim),
}

/// A synchronous shim function.
/// Receives already-validated args. Returns a Value directly (unyielding)
/// or a Value::Future (yielding/parallel).
pub type Shim = Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>;
```

### `IshFunction` Change

```rust
// Current:
pub struct IshFunction {
    pub name: Option<String>,
    pub params: Vec<String>,
    pub param_types: Vec<Option<ish_ast::TypeAnnotation>>,
    pub return_type: Option<ish_ast::TypeAnnotation>,
    pub body: Statement,
    pub closure_env: Environment,
    pub is_async: bool,
}

// New:
pub struct IshFunction {
    pub name: Option<String>,
    pub params: Vec<String>,
    pub param_types: Vec<Option<ish_ast::TypeAnnotation>>,
    pub return_type: Option<ish_ast::TypeAnnotation>,
    pub implementation: FunctionImplementation,
    pub closure_env: Environment,
    pub is_async: bool,
    pub has_yielding_entry: Option<bool>,
}
```

The `has_yielding_entry` field distinguishes three cases:
- `None` — no Yielding entry (ambiguous; function has neither `async` nor `@[unyielding]`)
- `Some(true)` — yielding (parallel builtins, functions with `async` keyword)
- `Some(false)` — unyielding (simple builtins, functions with `@[unyielding]`)

### Dispatch in `call_function_inner`

With `Value::BuiltinFunction` eliminated, `call_function_inner` only handles `Value::Function`:

```rust
fn call_function_inner<'a>(...) -> Pin<Box<dyn Future<...>>> {
    Box::pin(async move {
        match func {
            Value::Function(f) => {
                // 1. Arity check (same for all implementations)
                if args.len() != f.params.len() {
                    return Err(RuntimeError::system_error(
                        format!("'{}' expects {} arguments, got {}",
                            f.name.as_deref().unwrap_or("<fn>"),
                            f.params.len(), args.len()),
                        "E003",
                    ));
                }

                // 2. Audit parameter types (same for all implementations)
                // ...existing param type audit logic...

                // 3. Dispatch by implementation
                match &f.implementation {
                    FunctionImplementation::Interpreted(body) => {
                        // Create call environment, bind parameters
                        let call_env = f.closure_env.child();
                        for (name, val) in f.params.iter().zip(args.iter()) {
                            call_env.define(name, val.clone());
                        }
                        // Existing logic: push defer frame, async_stack,
                        // yield budget, exec_statement, pop, audit return
                        // ...
                    }
                    FunctionImplementation::Compiled(shim) => {
                        // Call the shim synchronously — no await needed
                        shim(args)
                    }
                }
            }
            _ => Err(RuntimeError::system_error(
                format!("cannot call {}", func.type_name()), "E006"
            )),
        }
    })
}
```

For compiled functions, `call_function_inner` does arity checking and parameter type auditing, then calls the shim directly with `&[Value]`. The shim is synchronous — no `.await` needed. The shim returns either a plain `Value` (unyielding) or `Value::Future` (yielding/parallel). The caller handles the result normally.

### Ledger Builtins

Ledger builtins (`active_standard`, `feature_state`, `has_standard`, `has_entry_type`, `ledger_state`, `has_entry`) need access to the VM's ledger state. Since the shim signature is `Fn(&[Value]) -> Result<Value, RuntimeError>`, it cannot take `&mut IshVm`.

These builtins remain intercepted by name in `call_function_inner` as they are today, but they are still registered as `IshFunction` values with `Compiled` implementations (the shim is a stub that panics — it should never be called directly). The name-based interception occurs before the shim dispatch:

```rust
Value::Function(f) => {
    // ... arity check, param audit ...

    // Ledger builtins: intercepted by name (need &mut self)
    if let Some(name) = &f.name {
        match name.as_str() {
            "active_standard" => return self.builtin_active_standard(args),
            "feature_state" => return self.builtin_feature_state(args),
            // ... etc ...
            _ => {}
        }
    }

    // Normal dispatch
    match &f.implementation { ... }
}
```

This is a pragmatic exception. When the standard library replaces builtins, the ledger will be accessed through a different mechanism.

### Builtin Registration

All builtins are registered as `IshFunction` values via a helper. The `BuiltinConfig` (output channel) is captured in shim closures where needed.

**Unyielding builtins** — `Compiled` shim, `has_yielding_entry: Some(false)`:

```rust
env.define("len", new_compiled_function(
    "len",
    vec![("value", None)],                      // params
    Some(TypeAnnotation::Simple("int".into())),  // return type
    Rc::new(|args| {
        match &args[0] {
            Value::String(s) => Ok(Value::Int(s.len() as i64)),
            Value::List(l) => Ok(Value::Int(l.borrow().len() as i64)),
            _ => Err(RuntimeError::system_error(
                format!("len: unsupported type '{}'", args[0].type_name()), "E004"
            )),
        }
    }),
    false,  // has_yielding_entry: Some(false)
    global_env.clone(),
));
```

**Parallel builtins** — `Compiled` shim, `has_yielding_entry: Some(true)`:

```rust
// println captures the output_sender from BuiltinConfig
let sender = config.output_sender.clone();
env.define("println", new_compiled_function(
    "println",
    vec![("value", None)],
    Some(TypeAnnotation::Simple("null".into())),
    Rc::new(move |args| {
        let text = format_value(&args[0]);
        if let Some(ref tx) = sender {
            // Interactive mode: send through channel with ack
            let (ack_tx, ack_rx) = tokio::sync::oneshot::channel();
            let _ = tx.send(/* message with ack_tx */);
            let handle = tokio::task::spawn_local(async move {
                let _ = ack_rx.await;  // wait for shell to confirm output
                Ok(Value::Null)
            });
            Ok(Value::Future(FutureRef::new(handle)))
        } else {
            // Non-interactive: spawn_blocking for stdout
            let handle = tokio::task::spawn_blocking(move || {
                println!("{}", text);
                Ok(())
            });
            let local_handle = tokio::task::spawn_local(async move {
                match handle.await {
                    Ok(Ok(())) => Ok(Value::Null),
                    Ok(Err(_)) => Ok(Value::Null),
                    Err(e) => Err(RuntimeError::system_error(
                        format!("println panicked: {}", e), "E010"
                    )),
                }
            });
            Ok(Value::Future(FutureRef::new(local_handle)))
        }
    }),
    true,  // has_yielding_entry: Some(true)
    global_env.clone(),
));
```

### Implied Await for Parallel Functions

When `call_function_inner` returns a `Value::Future` from a compiled shim, the behavior depends on the call context:

- **Called via `Expression::Await`:** The `Await` handler (Feature 2) receives the `Future` and awaits it. The resolved value is returned.
- **Called via `Expression::Spawn`:** The `Spawn` handler wraps the entire call in a spawned task. The parallel function's shim runs inside the spawned task, producing a `Future` that is awaited within the spawn context. The outer spawn returns a `Future` wrapping the whole task.
- **Called directly (bare call, no await/spawn):** The implied-await logic kicks in. If `await_required` is not `required`, the interpreter immediately awaits the future. If `await_required` is `required`, the VM throws an error.

The implied-await logic for bare calls is implemented in the `Expression::FunctionCall` handler:

```rust
// In the FunctionCall handler (not Await, not Spawn):
let result = self.call_function_inner(task, yc, &callee_val, &arg_vals).await?;

if let Value::Future(ref future_ref) = result {
    let features = self.ledger.active_features();
    if is_await_required(&features) {
        // Return the Future — the unawaited-future audit will catch it
        Ok(result)
    } else {
        // Implied await at low assurance
        match future_ref.take() {
            Some(handle) => match handle.await {
                Ok(inner) => inner,
                Err(join_err) => Err(RuntimeError::system_error(
                    format!("implied await failed: {}", join_err), "E011")),
            },
            None => Err(RuntimeError::system_error(
                "future has already been awaited".to_string(), "E011")),
        }
    }
} else {
    Ok(result)
}
```

### Print/Println I/O Completion

For `print`/`println`, the future must not resolve until the I/O is actually complete (Decision 9). The shim handles this:

**Non-interactive mode:** The shim uses `spawn_blocking` to write to stdout. The `spawn_local` bridge waits for the blocking task to complete, then returns `Value::Null`. The future resolves when the write completes.

**Interactive mode (output_sender active):** The shim sends the message and a `oneshot::Sender<()>` through the crossbeam channel. The `spawn_local` task waits for the acknowledgment `oneshot::Receiver`. The shell receives the message, writes it, and sends acknowledgment back. The future resolves only after the shell confirms the output was written.

### Parallel Builtin Table

All parallel builtins become `Compiled` functions with `has_yielding_entry: Some(true)`:

| Builtin | Params | Return |
|---------|--------|--------|
| `print` | `(value: any)` | `null` (via `Future`) |
| `println` | `(value: any)` | `null` (via `Future`) |
| `read_file` | `(path: string)` | `string` (via `Future`) |
| `write_file` | `(path: string, content: string)` | `null` (via `Future`) |

### Test Impact

Because of implied await at low assurance, **existing tests do not need to change**. `println("hello")` continues to work — the VM internally creates a future, immediately awaits it, and returns the resolved value.

Tests under `@standard[cautious]` or any standard where `await_required` is `required` will need explicit `await`:

```ish
@standard[cautious] {
    await println("hello")    // required — no implied await
    let content = await read_file("path.txt")   // required
}
```

New tests should cover:
- Parallel builtins with implied await (low assurance) — output ordering preserved
- Parallel builtins with explicit await (high assurance)
- Parallel builtins with `spawn` — returns `Future`, output may be deferred
- `read_file` and `write_file` in parallel via `spawn` — concurrent I/O
- Parameter count errors on builtins (same E003 as regular functions)
- `await len([1,2])` → E012 (unyielding entry)
- `spawn len([1,2])` → E013 (unyielding entry)
- `await 42` → parse error (Incomplete node)
- `spawn 42` → parse error (Incomplete node)
