## Feature 1: FutureRef Identity Equality

### Summary

The `PartialEq` implementation for `Value::Future` currently returns `false` for all comparisons. This violates reflexivity (`f == f` is `false`) and is incorrect. Futures use identity-based equality: two future values are equal if and only if they reference the same underlying `Rc`.

### Proposed Implementation

Single change in `proto/ish-vm/src/value.rs`, in `impl PartialEq for Value`:

```rust
// Futures use identity equality (same Rc pointer)
(Value::Future(a), Value::Future(b)) => Rc::ptr_eq(&a.inner, &b.inner),
```

This replaces the current line:

```rust
// Futures are never equal (identity-based, not value-based)
(Value::Future(_), Value::Future(_)) => false,
```

### Tests

- `let f = spawn work(); f == f` â†’ `true`
- `let f1 = spawn work(); let f2 = spawn work(); f1 == f2` â†’ `false`
- `let f = spawn work(); let g = f; f == g` â†’ `true` (same Rc via clone)

## Feature 2: Await on Non-Future Is an Error (E012)

### Summary

`await` applied to a non-future value is always a runtime error, regardless of assurance level. This uses a new error code E012 ("await type mismatch"). The concurrency proposal's "implicit await" concept (Decision 3 of that proposal) refers to implicit awaiting of async function *calls*, not to `await` accepting arbitrary value types.
--> Conversely, spawning anything but a function (or builtin) is also an error (E013).  Spawning a function with an unyielding entry is also an E013 error.

### Nuance: Function Yielding Categorization

Ideally, every function would be categorized as either `yielding` or `unyielding` at declaration time. Under that model:

- Awaiting a yielding (async) function's result â†’ succeeds (it returns a `Future`)
- Awaiting an unyielding function's result â†’ E012 error (it returns a plain value)
- Awaiting a literal non-future value â†’ E012 error

However, the prototype does not yet categorize functions at declaration time. Functions declared without explicit `async` or `@[unyielding]` annotations have no Yielding entry. For now, the runtime check is simple: if the value produced by the `await` sub-expression is not `Value::Future`, throw E012.

This means code like `await my_sync_fn()` will correctly error because `my_sync_fn()` returns a plain value (not a `Future`). Code like `await my_async_fn()` succeeds because `spawn` wraps the result in a `Future` â€” but note that `my_async_fn()` without `spawn` would need to be implicitly spawned to produce a future, or the user writes `await spawn my_async_fn()`.

An open question is created to track the eventual requirement that every function be categorized as yielding or unyielding at declaration time.
--> No. No. No.  This is exactly why I called this out. Consider `fn my_fn() { println('I yield') }; await my_fn()` and `fn other_fn() { return 5}; await other_fn()`.  The first is valid ish.  The second is not, because it awaits an unyielding function.  The problem is that right now the interpreter can't tell the difference.  We have to choose between throwing an error for both, in which case will handle the first one incorrectly, and throwing an error for neither, in which case we will handle the second case incorrectly.  For now, we choose to NOT throw an error, since it would break correct programs.

### Proposed Implementation

In `proto/ish-vm/src/interpreter.rs`, replace the pass-through in `Expression::Await`:

```rust
// Current (incorrect):
} else {
    // At low assurance: awaiting a non-future value returns it directly
    Ok(val)
}

// Replacement:
} else {
    Err(RuntimeError::system_error(
        format!(
            "cannot await a value of type '{}' â€” await requires a Future",
            val.type_name()
        ),
        "E012",
    ))
}
```

### Error Catalog

Add E012 to `docs/errors/INDEX.md` and `docs/spec/errors.md`:

|------|----------------|---------|-----------------|
| E012 | `TypeError` | Await type mismatch â€” `await` applied to a non-future value | `interpreter.rs` (Expression::Await) |

Also add the missing E011:

|------|----------------|---------|-----------------|
| E011 | `ConcurrencyError` | Concurrency error â€” cancelled task, panicked task, assurance discrepancy, already-awaited future | `interpreter.rs` (await, spawn, audits) |

### Test Changes

Two existing tests in `spawn_await.sh` must change from expecting pass-through to expecting errors:

```bash
# Before:
output=$(run_ish 'let v = await 42; println(v)')
assert_output "await non-future passes through" "42" "$output"

output=$(run_ish 'let v = await "hello"; println(v)')
assert_output "await string passes through" "hello" "$output"

# After:
output=$(run_ish 'let v = await 42; println(v)')
assert_output_contains "await non-future is error" "E012" "$output"

output=$(run_ish 'let v = await "hello"; println(v)')
assert_output_contains "await non-future string is error" "E012" "$output"
```

### Open Question

Add to `docs/project/open-questions.md` under a new Concurrency section:

> - [ ] **Function yielding categorization at declaration.** Every function should be categorized as `yielding` or `unyielding` at declaration time. Currently, functions without explicit `async` or `@[unyielding]` annotations have no Yielding entry. This means `await non_async_fn()` produces E012 (correct) but via value-type checking rather than function-category checking. When function declaration behavior is improved, the `await` check should verify that the callee is a yielding function, not just that the result happens to be a `Future`. See concurrency-correctness proposal Decision 2.

## Feature 3: Parallel Builtins

### Summary

Builtins that perform blocking I/O (`print`, `println`, `read_file`, `write_file`) become **parallel builtins**. They use a two-signature architecture: simple builtins remain synchronous closures; parallel builtins marshal arguments into `Send`-safe form, execute on `tokio::task::spawn_blocking`, and return a `Future<T>`.

Parallel builtins return `Value::Future` to the caller. They behave exactly like any other yielding function:

- At low assurance (no `await_required` feature active): the caller must still `await` or `spawn` the result, because `await` on a non-future is always an error (Feature 2).
- When `await_required` is `required`: the caller must explicitly `await` or `spawn` the result, like any yielding function.

Builtins are temporary scaffolding. They will be replaced by standard library functions when the standard library becomes available. An open question is created to track this.

### Architecture

**Simple builtins** â€” pure computation, synchronous, zero overhead:

`len`, `type_of`, `is_type`, `to_string`, `to_int`, `to_float`, `char`, `str_concat`, `str_length`, `str_slice`, `str_contains`, `str_starts_with`, `str_replace`, `str_split`, `str_to_upper`, `str_to_lower`, `str_char_at`, `str_trim`, `list_push`, `list_pop`, `list_length`, `list_get`, `list_set`, `list_slice`, `list_join`, `obj_get`, `obj_set`, `obj_has`, `obj_keys`, `obj_values`, `obj_remove`, `is_error`, `error_message`, `error_code`

Ledger stubs (`active_standard`, `feature_state`, `has_standard`, `has_entry_type`, `ledger_state`, `has_entry`) remain intercepted by the interpreter and are simple.

**Parallel builtins** â€” I/O, run on threadpool, return `Future`:

| Builtin | Args marshaled | Result |
|---------|---------------|--------|
| `print` | `String` (formatted text) | `Null` |
| `println` | `String` (formatted text) | `Null` |
| `read_file` | `String` (path) | `String` (file content) |
| `write_file` | `String` (path), `String` (content) | `Null` |

### Proposed Implementation

**Step 1: Define parallel types in `value.rs`**

```rust
/// Send-safe arguments for a parallel builtin, ready for spawn_blocking.
pub enum ParallelCall {
    Print { text: String, newline: bool },
    ReadFile { path: String },
    WriteFile { path: String, content: String },
}

/// Send-safe result from a parallel builtin.
pub enum ParallelResult {
    Null,
    Text(String),
}
```

**Step 2: Add `Parallel` variant to `BuiltinFn`**

```rust
pub enum BuiltinKind {
    Simple(Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>),
    Parallel(Rc<dyn Fn(&[Value]) -> Result<ParallelCall, RuntimeError>>),
}
```

The `BuiltinFn` struct changes to hold a `BuiltinKind` instead of the current single closure. The `name` field is preserved.

**Step 3: Update registration in `builtins.rs`**

`print`, `println`, `read_file`, `write_file` register as `Parallel`. Their closures validate arguments and produce `ParallelCall` without executing I/O.

For `print`/`println`, the `BuiltinConfig::output_sender` (crossbeam channel for ExternalPrinter) is still captured. The execution function checks: if `output_sender` is `Some`, send through the channel (non-blocking, fire-and-forget); otherwise, perform the I/O in the `spawn_blocking` closure.

**Step 4: Update interpreter dispatch in `call_function_inner`**

When calling a `Parallel` builtin:

1. Invoke the closure to validate args and produce `ParallelCall`
2. For `Print` with an active `output_sender`: send directly through the channel and return `Value::Future` wrapping an immediately-resolved handle
--> The future of the parallel builtins should not resolve until their I/O operation is actually complete.  So, the communication with the shell will have to include the shell sending a message back after the message has been written.
3. Otherwise: `tokio::task::spawn_blocking(move || execute_parallel(call))`, wrap the `JoinHandle` in `FutureRef`, return `Value::Future`
4. The caller receives a `Future` and must `await` it (or the unawaited-future audit catches it if `future_drop` is active)

**Step 5: Execute function for `spawn_blocking`**

```rust
fn execute_parallel(call: ParallelCall) -> Result<Value, RuntimeError> {
    match call {
        ParallelCall::Print { text, newline } => {
            if newline { println!("{}", text); } else { print!("{}", text); }
            Ok(Value::Null)
        }
        ParallelCall::ReadFile { path } => {
            std::fs::read_to_string(&path)
                .map(|s| Value::String(Rc::new(s)))
                .map_err(|e| RuntimeError::system_error(
                    format!("{}: {}", path, e), "E008"
                ))
        }
        ParallelCall::WriteFile { path, content } => {
            std::fs::write(&path, &content)
                .map(|_| Value::Null)
                .map_err(|e| RuntimeError::system_error(
                    format!("{}: {}", path, e), "E008"
                ))
        }
    }
}
```

Note: `Value::String(Rc::new(s))` is created inside `spawn_blocking`, which runs on a different thread. `Rc` is `!Send`, so this won't compile. The `execute_parallel` function must return `ParallelResult` (which is `Send`), and the conversion to `Value` happens back on the `LocalSet` thread:

```rust
fn execute_parallel(call: ParallelCall) -> Result<ParallelResult, String> {
    match call {
        ParallelCall::Print { text, newline } => {
            if newline { println!("{}", text); } else { print!("{}", text); }
            Ok(ParallelResult::Null)
        }
        ParallelCall::ReadFile { path } => {
            std::fs::read_to_string(&path)
                .map(ParallelResult::Text)
                .map_err(|e| format!("{}: {}", path, e))
        }
        ParallelCall::WriteFile { path, content } => {
            std::fs::write(&path, &content)
                .map(|_| ParallelResult::Null)
                .map_err(|e| format!("{}: {}", path, e))
        }
    }
}
```

The interpreter wraps this in a `spawn_blocking` that converts back:

```rust
let handle = tokio::task::spawn_blocking(move || execute_parallel(call));
// Wrap in a spawn_local that awaits the blocking handle and converts to Value
let local_handle = tokio::task::spawn_local(async move {
    match handle.await {
        Ok(Ok(result)) => Ok(result.into_value()),
        Ok(Err(msg)) => Err(RuntimeError::system_error(msg, "E008")),
        Err(join_err) => Err(RuntimeError::system_error(
            format!("parallel builtin panicked: {}", join_err), "E010"
        )),
    }
});
Ok(Value::Future(FutureRef::new(local_handle)))
```

### Test Impact

All existing tests that call `println(...)` without `await` will now receive a `Future` instead of `Null`. This is a significant change:

- Every `println("hello")` becomes `await println("hello")` to actually wait for output
- Or at low assurance, the output still happens (the `spawn_blocking` task runs), but the return value is a `Future` not `Null`
--> No.  At low assurance, when a yielding function is called without either spawn or await, an await is implied. `println("whatever")` is equivalent to `await println("whatever")`. So, `println` returns a future, but the future is consumed by the `await`.
- Tests that check for specific output may need adjustment if output ordering changes due to async execution
--> That shouldn't happen, because of the implied `await`.

The implementation plan should address test migration carefully. A pragmatic approach: if no standard is active (low assurance), the interpreter could auto-await parallel builtins in the dispatch path, preserving backward compatibility. When `await_required` is active, the auto-await is suppressed and the caller must handle the `Future` explicitly.
--> When `await_required` is active, the VM will throw if there is no `await` or `spawn`.  The caller still doesn't get a `Future`.  The only way to get a ever get a `Future` is with a `spawn`.

Actually, per Decision 6, parallel builtins always return `Future`. There is no auto-await. All existing `println(...)` calls become `await println(...)`. This is a large but mechanical change across all test files.

### Open Questions

Add to `docs/project/open-questions.md` under the Concurrency section:

> - [ ] **Builtin replacement by standard library.** All builtins (`print`, `println`, `read_file`, `write_file`, and the pure computation builtins) are temporary scaffolding. They should be replaced by standard library functions when the standard library is available. Parallel builtins should become parallel stdlib shims. See concurrency-correctness proposal Decision 6.
