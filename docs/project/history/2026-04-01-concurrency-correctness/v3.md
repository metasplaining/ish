## Feature 1: FutureRef Identity Equality

### Summary

The `PartialEq` implementation for `Value::Future` currently returns `false` for all comparisons. This violates reflexivity (`f == f` is `false`) and is incorrect. Futures use identity-based equality: two future values are equal if and only if they reference the same underlying `Rc`.

### Proposed Implementation

Single change in `proto/ish-vm/src/value.rs`, in `impl PartialEq for Value`:

```rust
// Futures use identity equality (same Rc pointer)
(Value::Future(a), Value::Future(b)) => Rc::ptr_eq(&a.inner, &b.inner),
```

This replaces the current line:

```rust
// Futures are never equal (identity-based, not value-based)
(Value::Future(_), Value::Future(_)) => false,
```

### Tests

- `let f = spawn work(); f == f` → `true`
- `let f1 = spawn work(); let f2 = spawn work(); f1 == f2` → `false`
- `let f = spawn work(); let g = f; f == g` → `true` (same Rc via clone)

## Feature 2: Spawn Validation and Await Semantics

### Summary

Two changes to spawn/await behavior:

1. **Spawn validation (E013):** `spawn` applied to anything other than a function call (or builtin call) is a runtime error (E013). Spawning a function that has an explicit unyielding entry is also E013. This catches mistakes like `spawn 42` or `spawn my_pure_fn()` where `my_pure_fn` is known to be unyielding.

2. **Await on non-future: allowed for now.** The v2 proposal specified that `await` on a non-future throws E012. This is reversed. Consider:

```ish
fn my_fn() { println('I yield') }
await my_fn()
```

This is valid ish — `my_fn` calls a parallel builtin internally, making it a yielding function. But `my_fn` has no Yielding entry in the prototype, so `my_fn()` returns the result of the implied `await println(...)`, which is a plain value (not a `Future`). Throwing E012 on `await <non-future>` would incorrectly reject this valid program.

Conversely:

```ish
fn other_fn() { return 5 }
await other_fn()
```

This is incorrect — `other_fn` is unyielding, and awaiting it is meaningless. But the prototype cannot distinguish this from the first case.

We must choose between:
- Throwing E012 for both → breaks correct programs (first case)
- Allowing both → permits meaningless `await` (second case)

**We choose to allow both, since breaking correct programs is worse than permitting a no-op.** E012 is reserved for future use when function yielding categorization at declaration time is implemented.
--> No.  My previous decisions were not reversals.  They were clarifications.  I am trying to explain something nuanced, and you keep ignoring the nuance.  E012 should be thrown when awaiting anything other than a function or parallel builtin.  E012 should be thrown when awaiting a function with an unyielding entry.  Even in the prototype, a function can have an unyielding entry if it has an unyielding annotation.  The only ambiguous case is a function that has neither an `async` keyword or an unyielding annotation.  This explanation that is already in the proposal of how this ambiguous case should be handled is already correct.

### Proposed Implementation — Spawn Validation

In `proto/ish-vm/src/interpreter.rs`, in the `Expression::Spawn` handler, add validation:

```rust
// After evaluating the spawn sub-expression:
// 1. Check that the spawned expression is a function call
// 2. If the function has an explicit Yielding entry set to Unyielding, throw E013

Err(RuntimeError::system_error(
    format!("cannot spawn a non-function value of type '{}'", val.type_name()),
    "E013",
))

// For unyielding functions:
Err(RuntimeError::system_error(
    "cannot spawn an unyielding function",
    "E013",
))
```

### Proposed Implementation — Await Pass-Through

The current pass-through behavior in `Expression::Await` is preserved as-is:

```rust
} else {
    // Await on non-future: pass through the value.
    // This is intentional — the prototype cannot distinguish yielding
    // from unyielding functions at declaration time. See Decision 2.
    Ok(val)
}
```

The only change is updating the comment to document the intentional decision.


### Open Question

Add to `docs/project/open-questions.md` under a new Concurrency section:

> - [ ] **Function yielding categorization at declaration.** Every function should eventually be categorized as `yielding` or `unyielding` at declaration time. Currently, functions without explicit annotations have no Yielding entry. This means `await non_yielding_fn()` silently passes through the value instead of producing an error. When function declaration categorization is implemented, `await` should check the callee's category (not just whether the result is a `Future`), and E012 should be thrown for awaiting unyielding functions. See concurrency-correctness proposal Decisions 2–3.

## Feature 3: Parallel Builtins

### Summary

Builtins that perform blocking I/O (`print`, `println`, `read_file`, `write_file`) become **parallel builtins**. They use a two-signature architecture: simple builtins remain synchronous closures; parallel builtins marshal arguments into `Send`-safe form, execute on `tokio::task::spawn_blocking`, and return a `Future<T>`.

Parallel builtins return `Value::Future` internally, but **the caller never sees a bare `Future`** in normal usage:

- **Without `await_required`** (low assurance): calling a parallel builtin without `await` or `spawn` implies `await`. So `println("hello")` is equivalent to `await println("hello")`. The future is created, immediately awaited by the VM, and the resolved value is returned. Output ordering is preserved.
- **With `await_required` set to `required`**: the VM throws an error if a yielding function is called without explicit `await` or `spawn`. The caller must write `await println("hello")` or `spawn println("hello")`. In the `await` case, the future is created and awaited — the caller receives the resolved value. In the `spawn` case, the caller receives a `Future`.
- **The only way to obtain a `Future` value is via `spawn`.** In all other cases, the future is consumed by `await` (explicit or implied) and the caller receives the resolved value.

Builtins are temporary scaffolding. They will be replaced by standard library functions when the standard library becomes available. An open question is created to track this.
--> Expand the proposal to include full parity between builtins and regular functions.  Although the may have special handling internally, builtins should appear to an outside observer just like regular functions.  All of the builtins should have either an unyielding entry or both the parallel and yielding entries.  They should all have the same parameter checking and return type checking as regular functions.

### Architecture

**Simple builtins** — pure computation, synchronous, zero overhead:

`len`, `type_of`, `is_type`, `to_string`, `to_int`, `to_float`, `char`, `str_concat`, `str_length`, `str_slice`, `str_contains`, `str_starts_with`, `str_replace`, `str_split`, `str_to_upper`, `str_to_lower`, `str_char_at`, `str_trim`, `list_push`, `list_pop`, `list_length`, `list_get`, `list_set`, `list_slice`, `list_join`, `obj_get`, `obj_set`, `obj_has`, `obj_keys`, `obj_values`, `obj_remove`, `is_error`, `error_message`, `error_code`

Ledger stubs (`active_standard`, `feature_state`, `has_standard`, `has_entry_type`, `ledger_state`, `has_entry`) remain intercepted by the interpreter and are simple.

**Parallel builtins** — I/O, run on threadpool, return `Future`:

| Builtin | Args marshaled | Result |
|---------|---------------|--------|
| `print` | `String` (formatted text) | `Null` |
| `println` | `String` (formatted text) | `Null` |
| `read_file` | `String` (path) | `String` (file content) |
| `write_file` | `String` (path), `String` (content) | `Null` |

### Proposed Implementation

**Step 1: Define parallel types in `value.rs`**

```rust
/// Send-safe arguments for a parallel builtin, ready for spawn_blocking.
pub enum ParallelCall {
    Print { text: String, newline: bool },
    ReadFile { path: String },
    WriteFile { path: String, content: String },
}

/// Send-safe result from a parallel builtin.
pub enum ParallelResult {
    Null,
    Text(String),
}
```

**Step 2: Add `Parallel` variant to `BuiltinFn`**

```rust
pub enum BuiltinKind {
    Simple(Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>),
    Parallel(Rc<dyn Fn(&[Value]) -> Result<ParallelCall, RuntimeError>>),
}
```

The `BuiltinFn` struct changes to hold a `BuiltinKind` instead of the current single closure. The `name` field is preserved.

**Step 3: Update registration in `builtins.rs`**

`print`, `println`, `read_file`, `write_file` register as `Parallel`. Their closures validate arguments and produce `ParallelCall` without executing I/O.

For `print`/`println`, the `BuiltinConfig::output_sender` (crossbeam channel for ExternalPrinter) is still captured. The execution path uses the channel for interactive mode, but the future must not resolve until the I/O operation is actually complete. This means the shell must send an acknowledgment back after the message is written. A response channel (e.g., `oneshot::channel`) is included with each print message so the `spawn_blocking` task can wait for confirmation.

**Step 4: Update interpreter dispatch in `call_function_inner`**

When calling a `Parallel` builtin:

1. Invoke the closure to validate args and produce `ParallelCall`
2. `tokio::task::spawn_blocking(move || execute_parallel(call))`, wrap in `spawn_local` bridge, wrap in `FutureRef`, producing `Value::Future`
3. **Implied await at low assurance:** If no `await_required` feature is active (or it is set to a non-`required` level), the interpreter immediately awaits the future before returning the resolved value to the caller. This makes `println("hello")` behave synchronously from the caller's perspective while executing I/O off the main thread.
4. **Explicit await/spawn required at high assurance:** When `await_required` is `required`, the VM checks whether the call site has an explicit `await` or `spawn`. If neither is present, throw a runtime error. If `await` is present, await the future and return the resolved value. If `spawn` is present, return the `Future` to the caller.
5. The unawaited-future audit catches any `Future` values that escape without being awaited (when `future_drop` is active).

For `print`/`println` with an active `output_sender` (interactive shell mode):

1. Send the message and a `oneshot::Sender` through the crossbeam channel
2. The shell writes the output and sends the acknowledgment back through the `oneshot`
3. The `spawn_blocking` (or `spawn_local`) task waits for the acknowledgment before resolving the future
4. This ensures the future does not resolve until the output is actually written

**Step 5: Execute function for `spawn_blocking`**

```rust
fn execute_parallel(call: ParallelCall) -> Result<ParallelResult, String> {
    match call {
        ParallelCall::Print { text, newline } => {
            if newline { println!("{}", text); } else { print!("{}", text); }
            Ok(ParallelResult::Null)
        }
        ParallelCall::ReadFile { path } => {
            std::fs::read_to_string(&path)
                .map(ParallelResult::Text)
                .map_err(|e| format!("{}: {}", path, e))
        }
        ParallelCall::WriteFile { path, content } => {
            std::fs::write(&path, &content)
                .map(|_| ParallelResult::Null)
                .map_err(|e| format!("{}: {}", path, e))
        }
    }
}
```

The interpreter wraps this in a `spawn_blocking` → `spawn_local` bridge:

```rust
let handle = tokio::task::spawn_blocking(move || execute_parallel(call));
// Bridge back to LocalSet thread to create Value (which contains Rc, so !Send)
let local_handle = tokio::task::spawn_local(async move {
    match handle.await {
        Ok(Ok(result)) => Ok(result.into_value()),
        Ok(Err(msg)) => Err(RuntimeError::system_error(msg, "E008")),
        Err(join_err) => Err(RuntimeError::system_error(
            format!("parallel builtin panicked: {}", join_err), "E010"
        )),
    }
});
let future_val = Value::Future(FutureRef::new(local_handle));

// Implied await if no await_required or at low assurance:
// The interpreter checks the call context and, if no explicit await/spawn
// is present and await_required is not "required", immediately awaits
// the future and returns the resolved value.
```

### Test Impact

Because of implied await at low assurance, **existing tests do not need to change**. `println("hello")` continues to work — the VM internally creates a future, immediately awaits it, and returns the resolved value. Output ordering is preserved because the implied await blocks until the I/O completes.

Tests under `@standard[cautious]` or any standard where `await_required` is `required` will need explicit `await`:

```ish
@standard[cautious] {
    await println("hello")    // required — no implied await
    let content = await read_file("path.txt")   // required
}
```

New tests should cover:
- Parallel builtins with implied await (low assurance) — output ordering preserved
- Parallel builtins with explicit await (high assurance)
- Parallel builtins with `spawn` — returns `Future`, output may be deferred
- `read_file` and `write_file` in parallel via `spawn` — concurrent I/O

### Open Questions

Add to `docs/project/open-questions.md` under the Concurrency section:

> - [ ] **Builtin replacement by standard library.** All builtins (`print`, `println`, `read_file`, `write_file`, and the pure computation builtins) are temporary scaffolding. They should be replaced by standard library functions when the standard library is available. Parallel builtins should become parallel stdlib shims. See concurrency-correctness proposal Decision 7.
