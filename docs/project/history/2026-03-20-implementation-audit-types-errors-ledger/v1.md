## Questions and Answers

### Q: Does the spec clearly state that entry maintenance (narrowing, entry tracking) should always be performed regardless of active standard?

**Partially.** The assurance-ledger spec says:

> Type narrowing is not a separate pass — it is the natural consequence of the ledger maintaining entry sets through control flow.

This implies narrowing is unconditional. The spec also says:

> The VM always checks type compatibility; the standard determines *when* (runtime vs. build) and *how strictly* (optional vs. required annotations).

However, the spec conflates two distinct operations:

1. **Entry maintenance** — recording facts in the ledger (type entries, narrowing, actual-value tracking). This should always happen because the ledger exists to *record* facts, not to *enforce* them.
2. **Auditing** — checking entries for discrepancies (type mismatches, missing annotations). This is governed by the active standard's feature states.

The spec does not explicitly draw this distinction in a summary statement. The implementing agent apparently interpreted "when the types feature is active" as a condition for all type-related operations, including entry maintenance.

### Q: Are there other meaning changes introduced during the spec rewrite that were not requested by the proposal?

**Four changes were found:**

1. **Audit mode naming (intentional).** `optional`/`live`/`pre` → two-dimension model (`type_annotations` + `type_audit`). This was Decision 14 in the proposal. Correct change.

2. **Value entries (intentional).** Three new entry kinds (`actual_value`, `possible_values`, `allowed_values`) added to replace literal types. This was Decision 17 in the proposal. Correct change.

3. **Entry type section (intentional).** Built-in entry types table with error hierarchy, `Open`, `Closed`. This was Decisions 3 and 16 in the proposal. Correct change.

4. **No unintended meaning changes found** in the assurance-ledger spec beyond the requested updates. The core concepts (standards, entries, audits, discrepancies) retained their original meaning. Feature inheritance, standard scoping, annotation syntax, native syntax equivalence, and cross-scope interactions all preserved their original semantics.

The errors.md file was entirely new (no prior version to compare), so meaning regression is not applicable there except for the `defer` scoping issue which came from incorporating content from the error-handling proposal.

### Q: Did the spec rewrite lose the `defer` scoping decision?

**Yes.** The error-handling proposal (error-handling.md) originally deferred defer scoping as an open question. A separate proposal (defer-scoping.md) resolved it in favor of function scoping. The open-questions.md file records this resolution. However, when errors.md was written during the consistency rewrite, it included the line:

> `defer` is scoped to the enclosing block, not the function.

This contradicts the accepted defer-scoping proposal and the implementation (which correctly uses per-function defer stacks). The errors.md author likely copied the initial intuition from the error-handling proposal without checking the subsequent defer-scoping proposal.

## Issue 1: VM Gates Type Narrowing on `types` Feature

### Findings

In `proto/ish-vm/src/interpreter.rs` (lines 194–252), the `if` statement handler:

```rust
let features = self.ledger.active_features();
let types_active = features.contains_key("types");

if types_active {
    // ... narrowing logic, entry save/restore/merge ...
} else {
    // No type narrowing — simple if/else.
    if cond.is_truthy() {
        self.exec_statement(then_block, env)
    } else if let Some(eb) = else_block {
        self.exec_statement(eb, env)
    } else {
        Ok(ControlFlow::None)
    }
}
```

When no standard with a `types` feature is active (e.g., the `streamlined` standard), the entire narrowing mechanism — including entry snapshot, restore, and merge — is skipped. This means:

- No ledger entries are saved before the branch.
- No narrowing facts are applied.
- No entries are merged after the branch.
- If any other mechanism adds entries inside the branch, those entries leak into post-branch code because no save/restore was done.

### Issues to Watch Out For

- **Silent state corruption.** This is not a crash — it silently produces incorrect ledger state. Code after an if/else block may see entries from inside the branch.
- **Interaction with other features.** Even without the `types` feature, other features (e.g., `immutability`, `null_safety`) may create entries that need save/restore through branches.
- **Test invisibility.** Because the acceptance tests only assert printed output (not ledger state), this bug is invisible to the existing test suite.

### Critical Analysis

**The fix is straightforward:** Entry maintenance (save/restore/merge) must always happen in branching statements. The narrowing-specific logic (analyzing conditions for narrowing facts) should also always happen — it records observable facts about values that the ledger needs regardless of which standard is active.

**Alternative A: Always perform narrowing (recommended)**
- Pros: Correct by specification. Ledger state is always consistent. Simple — remove the conditional.
- Cons: Very slight performance overhead for code that doesn't use any standard. Negligible.

**Alternative B: Only manage entries (save/restore/merge) without narrowing analysis**
- Pros: Minimal change — just move the snapshot/restore/merge outside the conditional.
- Cons: Narrowing facts would still be lost when no `types` feature is active. Half a fix.

**Alternative C: Leave as-is but document the limitation**
- Pros: No code change.
- Cons: Contradicts the specification. Silent state corruption.

### Proposed Implementation

Remove the `types_active` check entirely from the `if` statement handler. The narrowing logic should always execute:

1. Analyze the condition for narrowing facts.
2. Save the entry snapshot.
3. Apply narrowing facts to the appropriate branch.
4. Execute the branch.
5. Restore entries and merge.

The condition analysis itself (`analyze_condition`) is a pure function that examines the AST — it has no dependency on feature states.

**Files affected:**
- `proto/ish-vm/src/interpreter.rs` — remove the `types_active` conditional, collapse the two branches into one.

### Decisions

**Decision:** Remove the `types` feature gate from narrowing in the `if` statement handler? (Recommended: yes.)
--> Yes.

## Issue 2: VM Gates Throw Audit on `types` Feature

### Findings

In `proto/ish-vm/src/interpreter.rs` (lines 333–344), the throw statement handler:

```rust
if self.ledger.active_features().contains_key("types") {
    if let Value::Object(ref obj_ref) = val {
        let map = obj_ref.borrow();
        let has_message = matches!(map.get("message"), Some(Value::String(_)));
        if !has_message {
            return Err(RuntimeError::system_error(
                "throw audit: thrown object must have a 'message: String' property",
                "E004",
            ));
        }
    }
}
```

The throw audit only runs when the `types` feature is active. Per spec:

> When a `throw` statement executes, the ledger audits the thrown value.

No condition on feature state. The throw audit ensures all values in the error propagation path carry proper error entries. This should always happen because:

1. Catch clauses depend on error entries being present.
2. The guarantee that `e.message` is always accessible in a catch block depends on the throw audit enforcing it.
3. The `@Error` auto-annotation is a core language behavior, not a standard-enforced quality check.

### Issues to Watch Out For

- **Catch reliability.** Without the throw audit, thrown values in low-assurance code might not have `message` properties, causing runtime errors in catch blocks.
- **Entry consistency.** Without auto-annotation, thrown objects won't carry `Error` entries, breaking `is_error()` checks.

### Critical Analysis

The throw audit serves a dual purpose:

1. **Entry maintenance** — auto-adding `Error`/`CodedError` entries to thrown objects. This is unconditional.
2. **Discrepancy reporting** — rejecting thrown objects without `message: String`. This could be seen as auditing.

However, the spec is clear that even the discrepancy case (throwing a non-qualifying object) produces a `SystemError` wrapping — it doesn't silently pass. The throw audit is part of the `throw` semantics, not a standard-controlled quality check.

**Alternative A: Always perform throw audit (recommended)**
- Pros: Matches spec. Ensures `catch(e) { e.message }` always works. Consistent entry tracking.
- Cons: None — this is a correctness fix, not a tradeoff.

**Alternative B: Split into "always auto-annotate" + "conditionally reject bad throws"**
- Pros: More nuanced. Auto-annotation always happens; the "bad throw → SystemError" only under stricter standards.
- Cons: The spec doesn't make this distinction. Adds complexity without clear benefit.

### Proposed Implementation

Remove the `active_features().contains_key("types")` check from the throw handler. The throw audit should always execute.

Additionally, the current implementation is incomplete — it only checks for `message` but doesn't auto-add entries or handle the full throw audit spec (auto-add `Error`, auto-add `CodedError` for objects with `code`). This should be addressed but may be a separate work item.

**Files affected:**
- `proto/ish-vm/src/interpreter.rs` — remove the feature gate from the `Statement::Throw` handler.

### Decisions

**Decision:** Remove the `types` feature gate from the throw audit? (Recommended: yes.)
--> Yes.

**Decision:** Should incomplete throw audit logic (missing auto-annotation) be addressed in this fix or deferred?
--> Addressed.

## Issue 3: Spec Clarity — Entry Maintenance vs. Auditing

### Findings

The assurance-ledger spec does not explicitly distinguish between two fundamentally different operations:

1. **Entry maintenance** — the act of recording, updating, narrowing, and merging entries. This is the ledger's bookkeeping function. It happens unconditionally because the ledger must always have an accurate picture of the program's state.

2. **Auditing** — the act of checking entries for discrepancies. This is governed by the active standard's feature states. The standard controls *what* is checked, *when*, and *how strictly*.

The spec says "The VM always checks type compatibility" and "Type narrowing is not a separate pass — it is the natural consequence of the ledger maintaining entry sets through control flow." These statements imply the distinction, but never state it as a principle. This ambiguity led the implementing agent to conflate "no `types` feature active" with "no type-related operations at all."

### Issues to Watch Out For

- **Future implementers.** Without a clear statement of principle, future agents or developers may make the same mistake.
- **Feature interactions.** If a new feature is added to the ledger, implementers need to know: "always maintain entries; only audit based on standards."

### Critical Analysis

The fix is a documentation improvement — adding a clear statement distinguishing entry maintenance from auditing.

### Proposed Implementation

Add a subsection to assurance-ledger.md under **Concepts** titled **Entry Maintenance vs. Auditing** that clearly states:

- Entry maintenance (recording, updating, narrowing, merging entries) is always performed. The ledger must maintain an accurate picture of the program's state at all times.
- Auditing (checking for discrepancies) is governed by the active standard's feature states.
- The VM's role is to notify the ledger of program events (variable declarations, assignments, branch points, function calls, throws). The ledger performs both maintenance and auditing. The VM does not decide what the ledger should or should not do.

**Files affected:**
- `docs/spec/assurance-ledger.md` — add "Entry Maintenance vs. Auditing" subsection.

### Decisions

**Decision:** Add an explicit "Entry Maintenance vs. Auditing" principle to the assurance-ledger spec? (Recommended: yes.)
--> Yes.

## Issue 4: Acceptance Test Observability — `ledger_state()` Builtin

### Findings

The type narrowing acceptance tests in `proto/ish-tests/type_narrowing/type_narrowing.sh` contain 8 test cases that all follow the same pattern: set up a typed variable, branch on a condition, print a constant string inside the branch, and assert the printed output.

Example:
```bash
output=$(run_ish '
standard typed_std [ types(optional, runtime) ]
@standard[typed_std]
let x: i32 | null = 42
if x != null { println("not null") } else { println("is null") }
')
assert_output "null exclusion true branch" "not null" "$output"
```

This tests that the `if` statement evaluates the condition correctly and picks the right branch — it does *not* test that the ledger performed narrowing. The test would pass identically whether or not narrowing happened.

### Issues to Watch Out For

- **False confidence.** All 8 narrowing tests pass with the current broken code.
- **Regression invisibility.** If narrowing is broken further, these tests won't catch it.

### Critical Analysis

**Alternative A: Add a `ledger_state()` builtin (recommended)**
A builtin function `ledger_state(variable_name)` that returns a string representation of the ledger entries for a named variable. Tests can then assert specific ledger state:

```bash
output=$(run_ish '
let x: i32 | null = 42
if x != null {
    println(ledger_state("x"))
}
')
assert_contains "narrowing entries" "ExcludeNull" "$output"
```

- Pros: Direct observability of ledger behavior. Tests verify the mechanism, not just the effect. Works with any standard or no standard.
- Cons: Adds a builtin that exists only for testing. The output format becomes a de facto contract.

**Alternative B: Add a `has_entry()` builtin for testing**
A boolean builtin: `has_entry("x", "ExcludeNull")` → `true`/`false`.
- Pros: Simpler output to assert. Less brittle than string format.
- Cons: Can only test one entry at a time. Less informative for debugging.

**Alternative C: Both `ledger_state()` and `has_entry()`**
`ledger_state()` for debugging and exploration, `has_entry()` for precise assertions.
- Pros: Best of both worlds.
- Cons: Two builtins instead of one.

### Proposed Implementation

Add `ledger_state` as a new ledger query stub in `builtins.rs` with VM interception in `interpreter.rs`. The builtin takes a variable name (string) and returns a string representation of all entries on that variable (e.g., `"Type(i32), ExcludeNull"`). The existing `has_entry_type` builtin could potentially serve the per-entry check use case.

Rewrite the type narrowing acceptance tests to assert ledger state. Each test should verify that the expected narrowing entries are present (or absent) at the relevant program point.

**Files affected:**
- `proto/ish-vm/src/builtins.rs` — add `ledger_state` stub.
- `proto/ish-vm/src/interpreter.rs` — add VM interception for `ledger_state`.
- `proto/ish-tests/type_narrowing/type_narrowing.sh` — rewrite tests.
- `proto/ish-tests/lib/test_lib.sh` — possibly add `assert_contains` helper if not present.

### Decisions

**Decision:** Which builtin(s) to add for ledger observability? (Recommended: `ledger_state()` for full visibility.)
--> Both.  Also, add TODO documentation for some deferred items: ledger_state should return complex object, not a string.  And then the test code should call a to_string function to convert it to a string.  But all this relies on a generic to_string mechanism that is out of scope for this proposal.

## Issue 5: Domain Error Subtypes Extend Wrong Parent

### Findings

In `proto/ish-vm/src/ledger/entry_type.rs` (lines 123–142), the built-in entry type registrations:

```rust
// Domain subtypes of SystemError
self.register(
    EntryType::new("TypeError")
        .with_parent("SystemError")
);
self.register(
    EntryType::new("ArgumentError")
        .with_parent("SystemError")
);
self.register(
    EntryType::new("FileError")
        .with_parent("SystemError")
);
```

Both specifications (`errors.md` and `assurance-ledger.md`) say domain subtypes extend `CodedError`, not `SystemError`:

| Domain subtype | Parent | Description |
|---------------|--------|-------------|
| `TypeError` | `CodedError` | Type mismatch or type system violation |
| `ArgumentError` | `CodedError` | Incorrect argument count or type |
| `FileError` | `CodedError` | File system operation failure |

### Issues to Watch Out For

- **`is_subtype` checking is wrong.** `entry_type_registry.is_subtype("TypeError", "CodedError")` currently returns `true` (because `SystemError` extends `CodedError`), but `is_subtype("TypeError", "SystemError")` also returns `true`, which is incorrect if `TypeError` should only inherit from `CodedError`.
- **Catch clause matching.** A `catch(e: SystemError)` would incorrectly match a `TypeError` thrown by user code if `TypeError` extends `SystemError`.

### Critical Analysis

This is clearly a bug in the implementation that doesn't match the spec. The spec hierarchy is:

```
Error
  └── CodedError
        ├── SystemError (interpreter-only)
        ├── TypeError
        ├── ArgumentError
        └── FileError
              ├── FileNotFoundError
              └── PermissionError
```

`SystemError` is a *sibling* of the domain subtypes under `CodedError`. Domain subtypes are not system errors — they can be thrown by user code. `SystemError` is specifically for interpreter-generated errors.

### Proposed Implementation

Change `.with_parent("SystemError")` to `.with_parent("CodedError")` for `TypeError`, `ArgumentError`, and `FileError` in `entry_type.rs`.

**Files affected:**
- `proto/ish-vm/src/ledger/entry_type.rs` — fix parent references.

### Decisions

**Decision:** Fix domain error subtypes to extend `CodedError` instead of `SystemError`? (Recommended: yes.)
--> Upon reflection, this is a little subtle.  The definitions should be something like:
```ish
type CodedError = Error & { code: String }
type FileNotFoundError = CodedError & { code = "EXXX" }
type FileError = FileNotFoundError | OtherFileError | ...
type SystemError = FileError | TypeError | ArgumentError | ...
```
So, FileNotFoundError is a SystemError, in the sense that `catch(e: SystemError)` will catch a FileNotFoundError.  But SystemError is not part of it's definition.  It is the other way around.

## Issue 6: Structural Error Hierarchy Model

### Findings

The RFP describes a structural error hierarchy where:

- **Error** is nominal + structural: requires `@Error` annotation + `message: String` property.
- **CodedError** is purely structural: an Error with a `code: String` property.
- **Leaf errors** are purely structural: a CodedError with a specific `code` value.
- **Domain unions** (e.g., `FileError`) are unions of related leaf errors.

The current implementation and spec use an inheritance-based model where each error type is a separate entry type (`CodedError extends Error`, `TypeError extends CodedError`, etc.).

### Issues to Watch Out For

- **Throw complexity.** Under the current model, `throw { message: "bad type", code: "E004" }` needs the throw audit to determine that this is a `TypeError` and add the appropriate entry. Under the structural model, the type system itself recognizes it as a `TypeError` because `code == "E004"`.
- **Catch clause implementation.** Under the structural model, `catch(e: TypeError)` matches any error object where `code == "E004"`. Under the inheritance model, it matches any error with a `TypeError` entry.
- **User-defined error types.** The structural model makes it trivial for users to create domain-specific errors — just throw with the right code.
- **Backwards compatibility.** Changing the model now (before v1) is cheap. The acceptance tests don't depend on the error hierarchy internals.

### Critical Analysis

**Alternative A: Structural model (as described in RFP)**

The structural model says error classification is determined by property values, not by separate entry types for each error category. The `@Error` annotation marks something as an error. The `code` property's value determines which *kind* of error it is. Domain groupings like `FileError` are union types over code values.

- Pros:
  - Simple throw: `throw { message: "not found", code: "E008" }` is automatically a `FileError` by its code value, without entry type lookup.
  - Fearless catch: `catch(e) { e.message }` always works because `@Error` guarantees `message`.
  - Easy user extension: define new error codes, they automatically participate in the type hierarchy.
  - Fewer entry types to register and maintain.
  - Matches the RFP's design intent.

- Cons:
  - The mapping from code to domain type must be maintained somewhere (a code→type table).
  - `catch(e: FileError)` requires knowing which codes constitute file errors — this is a type system concern, not an entry concern.
  - Less conventional — most languages use inheritance for error types.

**Alternative B: Inheritance model (current implementation, corrected)**

Keep the current entry-type inheritance tree but fix the parent chain (`TypeError extends CodedError`, not `SystemError`). The throw audit adds the appropriate entry types based on property analysis.

- Pros:
  - Conventional and familiar.
  - `catch(e: TypeError)` matches based on entry type, which is well-defined.
  - No code→type mapping table needed.
  - Already partially implemented.

- Cons:
  - The throw audit must do more work to classify errors and add the correct entries.
  - More entry types to maintain.
  - Doesn't match the RFP's design intent.

**Alternative C: Hybrid model**

`@Error` annotation is the only entry. Error classification is structural (by code value). Entry types exist for the purpose of catch matching. The catch clause type system knows that `TypeError` means `code == "E004"` or `code == "E006"`.

- Pros: Combines simplicity of throw with familiarity of catch syntax.
- Cons: Two parallel systems (entry types for catch, structural matching for classification).

### Proposed Implementation

This is a significant design question. The proposal recommends discussing the tradeoffs and recording a decision before implementing.

If the structural model is chosen:
- Remove domain error entry types (`TypeError`, `ArgumentError`, etc.) from the builtin registry.
- Add a code→category mapping to the type system or ledger.
- Update the throw audit to only add `@Error` (and `@CodedError` if `code` is present).
- Update catch clause matching to use structural checks.
- Update specs.

If the inheritance model is chosen (corrected):
- Fix the parent chain as described in Issue 5.
- Enhance the throw audit to detect the correct domain subtype based on the `code` value and add the appropriate entry.
- No structural changes needed.

### Decisions

**Decision:** Should the error hierarchy use the structural model (RFP intent) or the corrected inheritance model? (Needs discussion.)
--> Structural model.  But the proposed implementation is still confused about how the structural model works. Errors in the error hierarchy are NOT special types.  They are NOT defined in rust.  They are NOT pre-registered with the system or ledger.  The only special types are the @Error entry type, which IS predefined, and does get automatically checked and annotated by the `throw` mechanism.  The rest of the error hierarchy is defined in ish.  When we have projects, modules, and packages more completely working, it should be necessary to include a "use" statement in order to get the error hierarchy. The throw mechanics don't use the error hierarchy at all.  The catch mechanics don't need it.  But it is convenient to have pre-defined types to support catching specific kinds of errors. For now, just define the error hierarchy types in the tests themselves and in the spec.  Don't include them in the implementation at all until they can be implemented as ordinary standard library types.

## Issue 8: `audit_type_annotation` Separation of Concerns

### Findings

The `audit_type_annotation` method in `interpreter.rs` (line 1002) returns early when no `types` feature is active:

```rust
let types_feature = match features.get("types") {
    Some(fs) => fs,
    None => return Ok(()), // no types feature active → no checking
};
```

This method handles two concerns:
1. **Type compatibility checking** — when an annotation is present, check the value matches it.
2. **Annotation requirement** — when no annotation is present but the standard requires one, report a discrepancy.

The early return skips *both* concerns. This might be partially defensible for concern 2 (if no `types` feature is active, annotations shouldn't be *required*), but it's problematic for concern 1 — if an annotation is present, the value should still be checked for compatibility, even without a `types` feature.

### Critical Analysis

This is the most nuanced of the issues. There are valid arguments for gating:

- In `streamlined` mode (no `types` feature), developers expect zero type checking overhead.
- If someone writes `let x: i32 = "hello"` in streamlined mode, should it be an error?

Per the spec: "If an entry for the feature is present, it is still checked; if absent, no discrepancy is raised." This suggests that even in `optional` mode, annotations that are present should still be validated. But the `streamlined` standard doesn't even have a `types` feature — it's not `optional`, it's *absent*.

**Alternative A: Always check present annotations (recommended)**
If code has a type annotation, check it regardless of the active standard. This follows the principle that explicit annotations are always meaningful.
- Pros: Annotations mean something. Catches bugs even in low-assurance code.
- Cons: Slight behavior change for streamlined mode. Could surprise developers who use annotations as documentation without enforcement.

**Alternative B: Only check when `types` feature exists at any level**
Check type annotations when the `types` feature exists in the active standard, regardless of whether it's `optional` or `required`.
- Pros: Respects the streamlined standard's intent to do nothing.
- Cons: Annotations in streamlined mode are dead code — the ledger ignores them.

**Alternative C: Leave as-is but separate the ledger update from the check**
Always *record* the type annotation as an entry (entry maintenance). Only *audit* it when the `types` feature is active.
- Pros: Clean separation. Ledger state is always correct. Discrepancies reported only when appropriate.
- Cons: Requires refactoring to separate recording from checking.

### Proposed Implementation

Alternative C aligns best with the "entry maintenance vs. auditing" principle from Issue 3. Refactor `audit_type_annotation` into two operations:

1. `record_type_annotation(item, annotation)` — always records the type entry in the ledger.
2. `audit_type_annotation(item, value, annotation)` — checks compatibility only when the `types` feature is active.

This ensures the ledger always has accurate type information while respecting the standard's audit policy.

**Files affected:**
- `proto/ish-vm/src/interpreter.rs` — refactor `audit_type_annotation`.

### Decisions

**Decision:** How should `audit_type_annotation` handle the absence of a `types` feature? (Recommended: Alternative C — separate entry maintenance from auditing.)
--> `Alternative A: Always check present annotations ` is the correct behavior.  It has always been the intended behavior.  The fact that this isn't clear indicates that the spec needs improvement.  The statement "In `streamlined` mode (no `types` feature), developers expect zero type checking overhead" misinterprets how assurance levels work.  Low assurance ish does NOT reduce runtime overhead by skipping checks.  The reverse is true.  High assurance ish reduces runtime overhead by performing the checks at build time, determining that a runtime check is not necessary because it will never fail, and pruning the checking during optimization.  This high assurance behavior is not yet implemented, or even specified, but it was always intended.  Low assurance ish abstracts concerns away from the developer.  Those concerns don't magically go away; the developer just doesn't need to think about them because ish just substitutes a sub-optimal solution.  In the case of error checking, the suboptimal solution is throwing at runtime.  ** Low assurance ish does not disable error checking.  It just defers it to runtime. **  The assurance ledger spec needs to be improved to make this clear.

## Additional Findings from Audit

### Finding A: Acceptance Test for "No Types Feature" Masks the Bug

Test case "no types feature no crash" in `type_narrowing.sh`:

```bash
output=$(run_ish '
let x = 42
if x != null { println("ok") }
')
assert_output "no types feature no crash" "ok" "$output"
```

This test verifies that the code doesn't crash without the `types` feature. It passes precisely because the bug exists — the narrowing path is skipped entirely. Once the narrowing conditional is removed, this test will still pass (narrowing with no entries is a no-op), but it should be updated to also verify that ledger state is correct.

### Finding B: `assert_contains` Not Present in Test Library

The test library (`proto/ish-tests/lib/test_lib.sh`) likely needs an `assert_contains` helper for checking partial string matches in ledger state output. Existing tests use `assert_output` for exact match.

### Finding C: `undeclared_errors` Feature Interaction

The `streamlined` standard defaults `undeclared_errors` to `any` (documented in `errors.md`). However, the `streamlined` built-in standard in the code has an empty feature map. This means `undeclared_errors` is not `any` — it's *absent*. The distinction matters if code checks `features.get("undeclared_errors")` and gets `None`.
