## Questions and Answers

### Q: What is the current state of AGENTS.md, CLAUDE.md, and .github/copilot-instructions.md?

**AGENTS.md** (6001 bytes, ~108 lines) is the main agent instruction file. It contains: build/test commands, project structure table, key concepts, crate map, task playbooks, and conventions. It references `.github/copilot-instructions.md` in its `depends-on` frontmatter.

**CLAUDE.md** (1573 bytes, ~46 lines) contains: the proposal process description, authority order, implementation discipline, and resuming-implementation rules. This is exactly the content that Claude Code uses as project instructions — it is the file Claude Code reads on session start.

**.github/copilot-instructions.md** contains the same content as `CLAUDE.md` — identical proposal process, authority order, and implementation discipline. This is the file GitHub Copilot reads on session start.

**The problem:** These three files together constitute agent instructions, but they are fragmented. An agent reading only `AGENTS.md` misses the proposal process rules. An agent reading only `CLAUDE.md` misses the crate map and task playbooks. Neither vendor file knows about the other.

### Q: Where are skills currently located, and how do the two vendors access them?

Skills currently exist in two directories with identical content:
- `.claude/skills/` — Claude Code reads these
- `.github/skills/` — GitHub Copilot reads these

Both directories contain: `propose/`, `revise/`, `accept/`, `audit/`, `plan-implementation/`, `implement/`. Each contains a single `SKILL.md`. The files are currently identical duplicates (confirmed by diff). There is no symlink or reference between them — they are maintained in parallel and are a synchronization hazard.

### Q: Are additional skills or roles needed?

The current six skills cover: proposal creation, proposal revision, proposal acceptance, feature audit, implementation planning, and implementation execution. This leaves gaps:

1. **No path from broken implementation to fix.** After implementation, when something is wrong, there is no defined skill for handling it.
2. **No root-cause investigation.** "Something is broken" and "fix it" are conflated. In practice, these require different behaviors.
3. **No consistency verification.** The audit skill checks feature coherence, but does not verify post-implementation consistency.

The proposed Bug Fix, Troubleshoot, and Verify skills fill these gaps. An **Orchestrator** role (deciding which skill to invoke) is worth considering but could be handled by the human for now.

### Q: What terminology should be used?

"Skill" is the correct technical term for the SKILL.md artifacts: it is the vocabulary used by both Claude Code and GitHub Copilot for callable procedure files loaded at prompt time. Overloading this term with "agent" or "role" would create confusion.

For the conceptual pipeline stages (the *purposes* the skills serve), "role" is the clearest term: Propose, Revise, Accept, Plan, Implement, Bug Fix, Troubleshoot, Verify are roles. A skill implements a role.

Recommendation: use **skill** for the file artifact, **role** for the conceptual pipeline stage. Avoid "agent" as a synonym for either — reserve it for the autonomous AI process executing the skill.

### Q: Are any of these practices a bad idea?

**"Creativity is not allowed outside the proposal process"** is sound. It addresses a real failure mode: agents improvising solutions that contradict the spec, creating documentation drift. The risk is over-application: agents may refuse to make trivially obvious fixes (a typo in a comment, a formatting issue). A narrow escape hatch should be defined.

**Plan-per-sub-agent with phase directories** is ambitious but well-supported by current research on multi-agent LLM systems. The key risk is context file quality: if context files are incomplete, the implementing agent will hallucinate. The plan skill must be explicit about this risk and require scrutiny.

**~500-line AGENTS.md** is a reasonable target, but the "prune lines the agent already knows" guideline must be applied aggressively. An over-specified AGENTS.md is as bad as an under-specified one: it trains agents to ignore it.

### Q: What practices are missing?

1. **Regression testing** — after a bug fix, explicitly verify the fixed behavior and check for regressions. The Bug Fix skill should include this.
2. **Escalation hierarchy** — when an agent cannot proceed, a clear decision tree should specify what it does (request clarification, mark blocked, escalate to human).
3. **Debug artifact cleanup** — the Troubleshoot skill should include a cleanup step to remove debug instrumentation before handing off.
4. **Session continuity** — the agent bootstrap problem is partly about agents not knowing where they left off. The implementation plan `in-progress` status already addresses this partially; it should be made more prominent.

## Feature (A): AGENTS.md Consolidation and Vendor-Agnostic Layout

### Issues to Watch Out For

- **CLAUDE.md as symlink**: Claude Code reads `CLAUDE.md` as project instructions. If `CLAUDE.md` is a symlink to `AGENTS.md`, the entire ~500-line `AGENTS.md` becomes Claude Code's project instructions. This is likely fine — it is the intent — but the content of `AGENTS.md` must remain appropriate for use as project instructions (no secrets, no build-time artifacts).
- **YAML frontmatter conflict**: Both CLAUDE.md and AGENTS.md currently have YAML frontmatter. The merged file needs a single frontmatter block. Claude Code may or may not render frontmatter — test this.
- **GitHub Copilot symlink support**: GitHub Copilot's `.github/copilot-instructions.md` following symlinks depends on the OS and how Copilot reads the file. On Linux/macOS this will work; on Windows it may not. Document this as a known limitation.
- **Skill symlinks**: If `.github/skills/` is a symlink to a vendor-agnostic directory, GitHub Copilot must be able to follow it. This should be verified.
- **500-line target**: Merging CLAUDE.md (~46 lines) + AGENTS.md (~108 lines) gives ~154 lines. Expanding to 500 lines means adding ~346 lines of new content. This expansion must be purposeful — not padding.

### Critical Analysis

**Option 1 (RFP): CLAUDE.md symlinks to AGENTS.md**
- Pros: Single source of truth. Changes to agent instructions automatically propagate to both vendors.
- Cons: AGENTS.md must serve dual purpose (project instructions + agent guide). Claude Code's project instructions display in a context that may not expect full agent docs. If AGENTS.md grows to 500 lines, it all becomes "project instructions."

**Option 2: AGENTS.md symlinks to CLAUDE.md**
- Pros: CLAUDE.md stays lean (project instructions only); AGENTS.md points to it and adds its own content.
- Cons: This inverts the relationship. AGENTS.md becomes the follower, not the leader. Doesn't solve the "agents reading only AGENTS.md miss proposal rules" problem.

**Option 3: AGENTS.md is canonical; CLAUDE.md and .github/copilot-instructions.md are both symlinks**
- This is Option 1 restated. AGENTS.md is the canonical file.
- The distinction from current CLAUDE.md: currently CLAUDE.md contains only "meta-process" rules (proposal process, authority order). Under Option 3, AGENTS.md contains everything.

**Option 4: Keep files separate but include CLAUDE.md content in AGENTS.md explicitly**
- AGENTS.md includes the proposal process and authority order inline (not via symlink).
- CLAUDE.md and AGENTS.md remain separate files that happen to share some content.
- Cons: Content can drift again.

**Recommendation**: Option 3 (AGENTS.md canonical, others are symlinks). This eliminates drift. AGENTS.md is already described in the project as the bootstrap file. Claude Code supports project instructions from any file name if configured; the current `CLAUDE.md` convention just means Claude Code defaults to reading that file, and a symlink satisfies it.

**For skills:**

**Option 1: `docs/skills/` is canonical; `.claude/skills/` and `.github/skills/` are symlinks**
- Pros: Vendor-agnostic. Adding a new skill is a single operation.
- Cons: `docs/` is documentation; skills are executable instructions, not documentation. Slightly odd placement.

**Option 2: `.claude/skills/` is canonical; `.github/skills/` symlinks to it**
- Pros: Claude Code is the primary tool. This keeps skills close to their primary consumer.
- Cons: Claude Code-specific directory becomes canonical. Violates vendor-agnostic intent.

**Option 3: New top-level `skills/` directory**
- Pros: Neutral location, clearly vendor-agnostic.
- Cons: Adds a top-level directory. May require updating both vendors' skill path configuration.

**Recommendation**: `docs/skills/` is cleanest because this project already treats `docs/` as the home for all authoritative project artifacts. Skills are a form of agent documentation. Document in `AGENTS.md` that `docs/skills/` is canonical.

### Proposed Implementation

1. Create `docs/skills/` with subdirectories for each skill, copying current content.
2. Replace `.claude/skills/` with a symlink: `.claude/skills/ → ../docs/skills/` (or individual per-skill symlinks if Claude Code does not follow directory symlinks).
3. Replace `.github/skills/` with a symlink: `.github/skills/ → ../../docs/skills/`.
4. Verify both vendors can discover the skills after symlinking.
5. Merge CLAUDE.md and AGENTS.md content into a new AGENTS.md (~500 lines, following §1.3 rules).
6. Replace CLAUDE.md with a symlink to AGENTS.md.
7. Replace `.github/copilot-instructions.md` with a symlink to AGENTS.md.

### Decisions

**Decision:** Should AGENTS.md be canonical with CLAUDE.md and .github/copilot-instructions.md as symlinks (Option 3), or keep files separate with shared content (Option 4)?

## Feature (A): AGENTS.md Content Improvements

### Issues to Watch Out For

- **"Agent-friendly style" is not yet defined.** The RFP references it but the project has no style guide for writing agent instructions. This needs to be defined before the AGENTS.md rewrite.
- **500 lines is a hard target.** If new skills add task playbook entries and the tech stack section grows, 500 lines may be exceeded. The maintenance skill should enforce the limit.
- **"Would the agent make a mistake without this?"** applies differently at different session points. Some content is needed on first read (orientation); other content is only needed for specific tasks. The split between AGENTS.md and per-task files must be deliberate.

### Critical Analysis

The current AGENTS.md is 108 lines and covers the essentials well. Expanding to 500 lines is a significant increase. The risk is that agents skip or skim long files. Research suggests agents perform better with shorter, denser context files that use structured formats (tables, checklists) over prose.

A counter-argument: the 500-line target includes content currently in CLAUDE.md (~46 lines) plus new content for skills, queries, never-touch list, and style guidelines. The actual expansion may be closer to 250–350 lines of content, with the rest being structure and formatting.

The "agent-friendly style guidelines" concept comes from the AI agent infrastructure literature. Key principles:
- Use imperative mood ("Read X" not "You should read X")
- Lead with commands, not explanations
- Prefer tables and checklists over prose
- Every instruction must be actionable
- No ambiguous pronouns
- File paths must be explicit
- Commands must be copy-pasteable

### Proposed Implementation

1. Define agent-friendly style guidelines as a new section in `AGENTS.md` itself (meta: the file documents how it is written).
2. Add a "Never Touch" section listing: secrets files (`.env`, credentials), `proto/target/` build artifacts, `Cargo.lock` (unless specifically asked), production config files, `.github/workflows/` unless explicitly part of a plan.
3. Expand tech stack section with hyper-specific detail: Rust edition, key crate versions, Tokio runtime configuration, pest PEG version.
4. Add Rust style guideline: "Write idiomatic Rust. Prefer standard library idioms. Use `?` for error propagation. Avoid `unwrap()` in non-test code."
5. Add an Architecture Index section pointing to spec docs, architecture docs, and user guide by feature area.
6. Update task playbooks to include all new skills (Bug Fix, Troubleshoot, Verify) once they are defined.
7. Add per-task query guides: "To find rules for your task, look up your task type in the playbooks table. To find product requirements, read the relevant spec docs. To find architecture, read the relevant architecture docs."

### Decisions

**Decision:** Should agent-friendly style guidelines be documented in AGENTS.md itself, in CONTRIBUTING.md, or in a separate `docs/ai-guide/style.md`?

## Feature (A): AGENTS.md Maintenance Skill

### Issues to Watch Out For

- **When to trigger**: The maintenance skill should be run after any change that adds new skills, renames files, changes the crate structure, or adds new doc sections. If it is run too often, it wastes context. If too rarely, AGENTS.md drifts.
- **"Would the agent make a mistake without this?"** is a judgment call. Different agents may answer differently. The maintenance skill should apply the criterion conservatively: when in doubt, keep content.
- **Size enforcement**: The skill must track line count and flag if the 500-line target is exceeded.

### Critical Analysis

A maintenance skill is a good idea. The risk is that a poorly-implemented maintenance skill removes content that seems redundant but is in fact needed for specific failure modes. The skill must be conservative: it should propose deletions rather than execute them unilaterally.

**Alternative**: Instead of a skill, use a recurring audit that checks AGENTS.md for stale references. This is less intrusive but also less effective at removing outdated content.

**Recommendation**: Create the maintenance skill. It should propose changes to the human rather than auto-applying them.

### Proposed Implementation

New skill: `docs/skills/update-agents/SKILL.md`

The skill procedure:
1. Read current AGENTS.md.
2. Read the complete project structure (docs index, skills list, crate map).
3. For each line in AGENTS.md, evaluate "Would an agent make a mistake without this?"
4. Check all file references for broken links.
5. Check that all current skills appear in the task playbooks table.
6. Check that the crate map is current.
7. Count lines; flag if over 500.
8. Produce a proposed new AGENTS.md and a diff summary for human review.
9. Do not save until the human approves.

### Decisions

**Decision:** Should the maintenance skill auto-apply changes (with confirmation) or produce a diff for human review?

## Feature (B): Revise Skill Improvements

### Issues to Watch Out For

- **Detecting "silent" parts**: An agent may not know what it doesn't know. "Silent" areas are specification gaps — the agent must infer these from the implementation context, not just the proposal text.
- **Splitting proposals**: Splitting at the wrong boundary can create dependencies between proposals that are harder to implement than one large proposal.
- **Version history**: Splitting a proposal creates new history directories, but the parent RFP spawned both. History must link the split proposals back to the parent.

### Critical Analysis

The current Revise skill is focused on punch list processing. The proposed additions — detecting gaps, splitting proposals — are qualitatively different: they require the agent to reason about completeness, not just incorporate feedback.

**Detecting gaps**: Research on specification completeness suggests checking for: missing error cases, missing concurrency behavior, missing backward-compatibility analysis, and missing test coverage statements. These can be turned into a checklist that Revise applies systematically.

**Splitting proposals**: The proposed criterion (independent pieces, clear implementation order, best-understood piece first) is sound. The risk is that the agent splits too aggressively, creating unnecessary overhead. A heuristic: split only when a proposal has more than 3 independent features, or when any single feature has more than 10 implementation steps.

### Proposed Implementation

Add to Revise skill:

**Gap detection (after step 5, before saving):**
1. For each feature in the proposal, check:
   - Are error cases specified?
   - Is concurrency behavior specified (if relevant)?
   - Are backward-compatibility impacts addressed?
   - Is there a testability statement?
2. For any unchecked item, add an open question: `**Open Question:** [specific gap]` in the relevant section.

**Proposal splitting (after gap detection):**
1. Count independent features (features with no intra-proposal dependencies).
2. If the count exceeds a threshold (proposed: 4+ features, or 15+ total implementation steps), evaluate splitting.
3. Identify dependency order. Group features into two or more proposals, with the most independent and best-understood group first.
4. Present the proposed split to the human as a decision: `**Decision:** Split into Proposal A (features X, Y) and Proposal B (features Z)? -->`.
5. If the human accepts the split, create new proposal files and update the history directory.

### Decisions

**Decision:** What threshold should trigger a split recommendation: feature count, implementation step count, or a subjective assessment?

## Feature (B): Plan Skill Improvements

### Issues to Watch Out For

- **Phase-file directory structure**: The current plan skill produces a single `docs/project/plans/<name>.md`. The new structure produces `docs/project/plans/<name>/phase-1.md`, `phase-2.md`, etc. plus context files. This is a breaking change for the implement skill, which must learn to read the new structure.
- **Context file quality**: If context files are incomplete or wrong, the implementing agent will hallucinate. The plan skill must extract context from authoritative sources (specs, architecture docs), not summarize from memory.
- **"Send back for revision" mechanism**: When the plan skill identifies gaps, it needs a structured way to report them. Currently there is no such mechanism.

### Critical Analysis

The single-file plan worked for the current project because all implementation was done by one agent in one session. As features grow larger, multi-phase, multi-agent plans are necessary. The proposed directory structure is the right direction.

The key insight from multi-agent research: context files must be complete and self-contained. An implementing agent that must "figure out" missing context will make things up. Context files are expensive to write but cheap compared to debugging hallucinated implementations.

**Alternative**: Instead of phase directories, use a single plan file with embedded context sections. Simpler, but loses the ability to hand off individual phases to sub-agents.

**Recommendation**: Adopt phase directories. The added complexity is worth it for large implementations. For small implementations (1–3 steps), a single plan file is still acceptable.

### Proposed Implementation

Update Plan skill procedure:
1. After reading the proposal, count the estimated implementation steps.
2. If ≤ 5 steps: produce a single `docs/project/plans/<name>.md` (current behavior).
3. If > 5 steps: produce `docs/project/plans/<name>/` directory containing:
   - `overview.md` — summary, requirements, dependency graph of phases
   - `context/` subdirectory — one file per spec/architecture area referenced
   - `phase-1.md`, `phase-2.md`, ... — one per implementation phase
4. Each phase file format:
   ```
   # Phase N: <name>
   ## Context Files
   - [context/types.md](context/types.md)
   ## Requirements
   - <testable statement>
   ## Tasks
   - [ ] 1. <task> — <file>
   ## Verification
   - Run: <command>
   - Check: <what to look for>
   ```
5. Before saving, scrutinize each phase: does it have all detail needed? If not, produce a gap list and return to proposal for revision.

Add a **"Send Back" mechanism** (see Feature B: Backchannels).

### Decisions

**Decision:** At what step count should the plan skill switch from single-file to directory structure? (Proposed: 5 steps)

**Decision:** Should context files be extracted verbatim from source docs, or should they be summaries? (Verbatim is safer; summaries risk distortion)

## Feature (B): Implement Skill Guard Rails

### Issues to Watch Out For

- **Defining "unclear"**: An agent may perceive ambiguity where none exists, and request clarification unnecessarily. The guard rail should specify: request clarification only when the ambiguity affects correctness, not style.
- **"Impossible" detection**: An agent may incorrectly conclude a task is impossible due to a misunderstanding. Before declaring impossibility, the agent should re-read the relevant spec and try one alternative approach.

### Critical Analysis

The current implement skill says "complete all TODO items." The proposed addition — "request clarification when unclear" — is the right guard rail. The risk is over-triggering: agents that request clarification for every uncertainty will frustrate users.

**Recommendation**: Specify a two-step protocol: (1) re-read the relevant spec and proposal section; (2) if still unclear after re-reading, request clarification. This reduces noise while preserving the guard rail.

### Proposed Implementation

Add to Implement skill, step 4:
> Before executing each TODO item, verify that the item is unambiguous. If a task is unclear:
> 1. Re-read the relevant specification section and proposal section.
> 2. If still unclear after re-reading, do not attempt the task. Write a clarification request (see Backchannels) and stop.
> If a task appears impossible:
> 1. Verify the task requirements against the proposal.
> 2. Try one alternative interpretation.
> 3. If still impossible, write a clarification request and stop.

### Decisions

**Decision:** Should the implement skill stop entirely on the first unclear item, or continue with subsequent items and batch all clarification requests?

## Feature (B): Bug Fix Skill (New)

### Issues to Watch Out For

- **Test semantics**: The prohibition on changing test meaning is critical but subtle. A test that checks output format may need to be updated when output format changes — but only if the spec changed. The bug-fix agent must determine whether the test or the code is authoritative.
- **Documentation updates**: "All documentation" is too broad. The bug-fix skill must specify which documentation must always be updated (history, affected spec sections) vs. which may be optional.
- **Re-running tests**: After a bug fix, tests may reveal new failures unrelated to the fix. The bug-fix agent must distinguish: expected failures (known pre-existing issues) vs. new regressions introduced by the fix.

### Critical Analysis

A Bug Fix skill is clearly needed. The key design tension is between completeness (update everything) and focus (just fix the bug). The RFP correctly identifies that "fix the bug" is insufficient — documentation and history must be updated. But an overly broad update mandate will slow bug fixes and introduce risk.

**Recommendation**: The Bug Fix skill should update: history, the implementation plan (mark the fix), and only the spec/architecture docs that are directly incorrect. It should not re-audit the entire codebase.

### Proposed Implementation

New skill: `docs/skills/bug-fix/SKILL.md`

Procedure:
1. Read the bug report (human provides description of what is wrong and where).
2. Read the implementation plan for the affected feature.
3. Read the relevant spec and architecture docs.
4. Identify the root cause (if not provided — if uncertain, use Troubleshoot first).
5. Fix the code.
6. Run acceptance tests. For each failing test:
   - If the test failure is unrelated to the fix: document it; do not fix.
   - If the test failure is due to the fix and the test expected wrong behavior: stop; report to human.
   - If the test failure is due to the fix and the test expected correct behavior: fix the test (format/reference only, never the assertion logic).
7. Fix the unit tests (same rules as acceptance tests).
8. Update affected spec/architecture docs (only if they are incorrect, not as a general re-audit).
9. Add a history entry documenting the bug and fix.
10. Mark the fix in the implementation plan.
11. Run acceptance tests again to confirm all pass.
12. Report completion.

### Decisions

**Decision:** Should Bug Fix always be preceded by Troubleshoot, or should Bug Fix include its own root-cause analysis for simple bugs?

## Feature (B): Troubleshoot Skill (New)

### Issues to Watch Out For

- **Debug artifact cleanup**: Debug logs and temporary behavior changes must be removed before handoff to Bug Fix. The troubleshoot skill must track all changes made during investigation.
- **Specification vs. implementation**: Determining whether a bug is a spec problem or an implementation problem is the hardest part. The troubleshoot agent must not make this determination by changing the spec — it reads the spec and determines if it is complete, unambiguous, and achievable.
- **Scope creep**: A troubleshooter finding one bug may find others. It should report all findings but fix none.

### Critical Analysis

The proposed Troubleshoot skill cleanly separates "understand what's wrong" from "fix it." This is correct: debugging and fixing are different cognitive tasks. The skill ensures the root cause is documented before any fix is attempted.

The key guard rail — "do not change the specification" — is important. Troubleshooters often feel pressure to fix specs as they go. This skill should resist that pressure and escalate to the proposal process.

### Proposed Implementation

New skill: `docs/skills/troubleshoot/SKILL.md`

Procedure:
1. Read the problem report.
2. Read the implementation plan and proposal for the affected feature.
3. Read the relevant spec and architecture docs.
4. Form a hypothesis about the root cause.
5. Gather evidence:
   - Add debug logging if needed (document all changes in a `debug-log.md` scratch file).
   - Run tests to confirm/refute the hypothesis.
   - Do not change any behavior beyond adding debug instrumentation.
6. Determine root cause.
7. Classify:
   - **Specification incomplete/ambiguous/unachievable**: Write a clarification request. Remove debug instrumentation. Stop.
   - **Implementation error**: Document root cause in `docs/project/clarifications/<date>-<topic>.md`. Remove debug instrumentation. Hand off to Bug Fix.
8. Remove all debug instrumentation.
9. Report findings.

### Decisions

**Decision:** Should debug instrumentation go in a scratch file that is deleted after troubleshooting, or should it be a documented temporary change committed to a branch?

## Feature (B): Verify Skill (New)

### Issues to Watch Out For

- **Scope**: "Verify everything" is impractical. The verify skill must scope its check to the feature or phase just implemented.
- **Incomplete spec**: If the spec is incomplete, verification against it will always succeed (nothing contradicts an empty spec). The verify skill should flag incomplete specs as a finding.
- **vs. Audit skill**: The existing Audit skill checks feature coherence. The Verify skill checks post-implementation consistency. They overlap. The relationship must be defined.

### Critical Analysis

The Audit and Verify skills serve overlapping purposes. The distinction proposed in the RFP is:
- **Audit**: checks all project artifacts for a feature, at any time
- **Verify**: checks consistency after a specific implementation event (phase completion, bug fix)

This distinction is useful: Audit is broad and periodic; Verify is narrow and triggered. Verify hands off to Troubleshoot; Audit produces a report.

**Alternative**: Merge Verify into Audit by adding a "triggered mode" that scopes the audit to a specific change. Simpler, but loses the direct Troubleshoot handoff.

**Recommendation**: Keep them separate. Verify is narrower, faster, and has a defined output action (hand to Troubleshoot). Audit is broader and produces a report for human review.

### Proposed Implementation

New skill: `docs/skills/verify/SKILL.md`

Procedure:
1. Read the implementation plan phase or bug fix being verified.
2. For each changed file, identify the authoritative source (spec, proposal, test).
3. Check consistency:
   - Code ↔ tests: does the code implement what the tests assert?
   - Tests ↔ spec: do the tests cover what the spec requires?
   - Spec ↔ proposal: does the spec reflect the accepted proposal decisions?
4. For each inconsistency found:
   - Document: what was found, in which files, and which is likely authoritative.
5. If inconsistencies are found: hand off to Troubleshoot with the documented findings.
6. If no inconsistencies: report "Verified: no inconsistencies found."

### Decisions

**Decision:** Should Verify be a standalone skill or an invocable phase of the Implement skill (automatically triggered at each checkpoint)?

## Feature (B): Backchannel Communication

### Issues to Watch Out For

- **File proliferation**: If every agent writes clarification request files, the `docs/project/clarifications/` directory may fill up. A cleanup protocol is needed.
- **Human discoverability**: The human must know when a clarification request has been written. The agent should report the file path prominently at the end of its output.
- **Structured format**: Clarification requests must be machine-readable enough for the next agent to parse them, and human-readable enough for the human to respond.

### Critical Analysis

Current backchannels are entirely in the conversation context. When an agent hits a problem, it reports in-conversation and waits. This works for interactive sessions but fails for async/sub-agent scenarios where the conversation context may not be visible.

Three proposed mechanisms:

**Mechanism 1: Clarification Request Files**
- Path: `docs/project/clarifications/<date>-<topic>.md`
- Contains: what the agent was doing, what is unclear, specific questions, recommended resolution paths
- Pros: Durable, discoverable, structured
- Cons: Requires human to check a directory; file accumulation

**Mechanism 2: Blocked Status in Plan**
- Implement agents mark a plan item `BLOCKED: <reason>` when they cannot proceed
- Pros: Integrated with existing plan tracking
- Cons: Only works for implement/bug-fix agents; doesn't work for troubleshoot/verify

**Mechanism 3: Agent Output Status Field**
- Each skill's output includes a status: `completed | needs-clarification | escalate-to-bug-fix | escalate-to-troubleshoot`
- Pros: Composable; orchestrators can act on status
- Cons: Requires orchestrator logic; more complex

**Recommendation**: All three, in combination. Clarification request files are the primary mechanism (durable, human-readable). Blocked status in plan is an additional signal for implement agents. Output status field is added to skill output format for future orchestration support.

### Proposed Implementation

1. Create `docs/project/clarifications/` directory with an `INDEX.md`.
2. Define clarification request file format:
   ```markdown
   # Clarification Request: <topic>
   *Created by: <skill name> on <date>*
   ## Context
   <what was being done>
   ## Blocked On
   <specific unclear/impossible item>
   ## Questions
   1. <question>
   ## Recommended Resolution
   <proposal: send back to revise / new RFP / direct answer>
   ```
3. Update all skills to write clarification requests when blocked.
4. Add "output status" section to each skill's output format.
5. Add `BLOCKED: <reason>` convention to plan file task items.
6. Document cleanup protocol: clarification requests are deleted after resolution and a history note is added.

### Decisions

**Decision:** Should clarification requests go in `docs/project/clarifications/` or in the relevant proposal/plan directory?
