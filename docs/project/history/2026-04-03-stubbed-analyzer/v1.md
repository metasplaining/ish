## Questions and Answers

### Q: How does the current PENDING_INTERP_CALL mechanism work, and why is it a defect?

The mechanism uses a thread-local `RefCell<Option<InterpCall>>` to bridge the synchronous `Shim` type signature (`Fn(&[Value]) -> Result<Value, RuntimeError>`) with the async interpreter. When an interpreted function's shim is called, it stores the function body and captured environment in the thread-local, returns `Value::Null` as a placeholder, and then `call_function_inner` picks up the pending call and executes the body asynchronously.

The defect is that this only works when shims are called from `call_function_inner`. If a compiled function calls an interpreted function's shim directly (which is the whole point of cross-boundary calls), the shim stores the pending call but nobody retrieves it. The `apply` builtin was implemented as a special-case intercept in `call_function_inner` (line 1425) specifically to work around this limitation.

### Q: The RFP asks whether we need yielding and unyielding shim variants. Do we?

Yes. The `Shim` type is `Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>` — a synchronous function. For yielding functions, the shim needs to return a `Value::Future` that wraps the async body execution. For unyielding functions, the shim can execute the body synchronously and return the result directly. Both return `Result<Value, RuntimeError>`, so the `Shim` type signature doesn't change. What changes is what the shim *does internally*:

- **Unyielding shim**: Calls `exec_statement_unyielding` synchronously, returns result.
- **Yielding shim**: Spawns a Tokio `spawn_local` task that calls `exec_statement_yielding`, returns `Value::Future`.

This difference is behavioral — the `Shim` type itself remains the same. No type-level variant is needed.

### Q: What about `apply` returning a `Value::Future` when passed a yielding function?

If `apply` is a normal compiled function whose shim calls the passed-in function's shim, and the passed-in function is yielding, then the yielding function's shim will return `Value::Future`. `apply` just returns that `Value::Future` as its own result. This is exactly the behavior we want — `apply(yielding_fn, args)` returns a `Future` that can be `await`ed.

## Feature 1: Stubbed Code Analyzer

### Issues to Watch Out For

1. **Nested functions.** A function body may contain inner function declarations or lambdas. The analyzer should not recurse into them — an inner function's yielding is independent of its parent's.
2. **Conditional yielding.** A function might have `await` in one branch of an `if` but not another. The stub analyzer just needs to find *any* yielding node in the body — it doesn't need to reason about reachability.
3. **`CommandSubstitution`.** The `$()` syntax executes a shell command via `tokio::process::Command`, which is inherently async. This should be treated as yielding.
4. **Annotations.** The `@[unyielding]` annotation on a function body already forces unyielding. The analyzer needs to check for this too, or defer to the existing mechanism.
5. **`is_async` flag.** Functions declared with `async fn` have `is_async: true` in the AST. This should immediately classify as yielding without walking the body.

### Critical Analysis

**Alternative A: AST-walking analyzer in ish-vm** — Put `analyzer.rs` in `ish-vm`. The analyzer walks the AST and returns yielding/unyielding classification. It has access to both `ish-ast` types and the VM context.

*Pros:* Simple to implement. Close to where it's consumed (the interpreter). Can access VM state if needed later.
*Cons:* Couples analysis to the VM crate. Harder to reuse from `ish-codegen`.

**Alternative B: AST-walking analyzer in ish-parser** — Put `analyzer.rs` in `ish-parser`, next to the AST construction. Pure AST analysis, no runtime dependency.

*Pros:* Available to all downstream crates. Clean separation of concerns.
*Cons:* `ish-parser` is about parsing, not semantic analysis. Adds a responsibility that doesn't belong there.

**Alternative C: New `ish-analysis` crate** — Create a dedicated crate for code analysis. Depends on `ish-ast` and `ish-core`. Both `ish-vm` and `ish-codegen` depend on it.

*Pros:* Clean separation. Clearly extensible for future analysis passes.
*Cons:* Another crate to maintain. Possibly premature for a stub.

**Alternative D: AST-walking analyzer in ish-ast** — Put the analyzer in `ish-ast` since it only operates on AST types.

*Pros:* Zero new crate. Analysis is inherently about the AST.
*Cons:* `ish-ast` is about types and structure, not analysis. Mixes concerns.

**Recommendation:** Alternative A for now. The analyzer is consumed by the interpreter during function declaration. When `ish-codegen` needs it, extracting to a shared crate is straightforward. The stub is small enough that moving it later is cheap.

### Proposed Implementation

Create `proto/ish-vm/src/analyzer.rs`:

```rust
use ish_ast::{Statement, Expression};

pub enum YieldingClassification {
    Yielding,
    Unyielding,
}

/// Walk a function body and determine whether the function is yielding or unyielding.
/// This is a stub implementation — it only checks for the presence of yielding
/// AST nodes. It does not handle automatic yielding or complex inference.
pub fn classify_function(body: &Statement, is_async: bool) -> YieldingClassification {
    if is_async {
        return YieldingClassification::Yielding;
    }
    if contains_yielding_node(body) {
        YieldingClassification::Yielding
    } else {
        YieldingClassification::Unyielding
    }
}
```

The `contains_yielding_node` function walks the AST recursively, returning `true` if it finds:
- `Expression::Await`
- `Expression::Spawn`
- `Statement::Yield`
- `Expression::CommandSubstitution`

--> There is another case: Function calls with no explicit await or spawn.  In this case, the interpeter will automatically await if the called function is yielding.  So, the analyzer should mark the calling function as yielding if the called function is yielding.  This has two deeper implications: 1. The analyzer needs to live in the VM, or at least have access to it,  because it needs the access to the defined functions in order to do its job. 2. Functions must be defined in order, and call cycles are not supported.  This is not good enough, and we will need to fix it later. For now, just throw an error when a function is declared which calls an undefined function.

It does **not** recurse into `Statement::FunctionDecl` or `Expression::Lambda` bodies — those are independent functions with their own classifications.

## Feature 2: Remove the Ambiguous Case

### Issues to Watch Out For

1. **Specification changes.** The concurrency spec (docs/spec/concurrency.md) documents the three-state model: `Some(true)`, `Some(false)`, `None`. The `None` case and its pass-through behavior must be removed from the spec.
2. **Open questions.** The open question on "Function yielding categorization at declaration" (docs/project/open-questions.md) is resolved by this change — it should be marked done.
3. **Test breakage.** Any tests that rely on the ambiguous pass-through behavior (calling unclassified functions with `await` and getting non-`Future` values passed through) will break.
4. **Lambda classification.** Lambdas currently always get `has_yielding_entry: None`. They need to go through the analyzer too.
5. **`has_yielding_entry` type change.** With no ambiguous case, `Option<bool>` can become a simple `bool`. However, this is a pervasive change across crates. Consider keeping `Option<bool>` but asserting `Some` everywhere, or changing the type.

### Critical Analysis

**Alternative A: Change `has_yielding_entry` from `Option<bool>` to `bool`** — Direct. Every function is `true` (yielding) or `false` (unyielding).

*Pros:* Type-safe — impossible to represent the ambiguous state. Clear intent.
*Cons:* Pervasive change across `ish-runtime`, `ish-vm`, `ish-codegen`, `ish-stdlib`. All call sites of `new_compiled_function` change.

**Alternative B: Keep `Option<bool>` but treat `None` as `Unyielding`** — Backward compatible. Assert `is_some()` in debug builds but fall back to unyielding.

*Pros:* Smaller diff. Easier migration for compiled function registrations.
*Cons:* Confusing — the type says "optional" but `None` is no longer valid.

**Alternative C: Introduce an enum** — `pub enum YieldingEntry { Yielding, Unyielding }`.

*Pros:* Self-documenting. Impossible to misuse. Named variants clearer than `true`/`false`.
*Cons:* New type that propagates across crates. Slightly more verbose.

**Recommendation:** Alternative C. The `YieldingClassification` enum from the analyzer aligns with this. Use the same enum everywhere, or re-export it from `ish-runtime` with a shorter name. This also serves the analyzer's return type.

### Proposed Implementation

1. Define `YieldingClassification` in `ish-core` (shared by all crates):
   ```rust
   pub enum YieldingClassification { Yielding, Unyielding }
   ```
2. Change `IshFunction.has_yielding_entry` from `Option<bool>` to `YieldingClassification`.
3. Update `new_compiled_function` to take `YieldingClassification`.
4. Update all builtin registrations to use `Yielding` or `Unyielding`.
5. Remove `None` handling from `call_function_inner`, `await` handling, and `spawn` handling.
6. Update the concurrency spec to remove the three-state model.
7. Close the open question on function yielding categorization.

### Decisions

**Decision 6:** What happens to `has_yielding_entry: None`?
- A) Change to `bool` (no `None` allowed)
- B) Keep `Option<bool>`, treat `None` as unyielding
- C) Introduce `YieldingClassification` enum (recommended)
--> Keep `Option<bool>`, treat `None` as unyielding.  This data structure is going to need a more significant refactoring.  There's no point in doing half the job now.

## Feature 3: Remove PENDING_INTERP_CALL and Fix Shim Architecture

### Issues to Watch Out For

1. **Shim type is synchronous.** `Shim = Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>`. Yielding shims need to spawn a `tokio::task::spawn_local` to return a `Value::Future`. But `spawn_local` requires a Tokio `LocalSet` runtime context. The shim is called from within the interpreter which already runs on the `LocalSet`, so this is safe.
2. **Shim captures.** Yielding shims need to capture: the function body (`Statement`), the declaration-time environment (`Environment`), parameter names, and a reference to the VM (`Rc<RefCell<IshVm>>`). This is a lot of captured state but it's all `Clone` or `Rc`-shared.
3. **TaskContext and YieldContext.** Currently passed to `exec_statement` as mutable references. For yielding shims that spawn a new task, a new `TaskContext` and `YieldContext` must be created inside the spawned task. For unyielding shims, these must be available in the calling context.
4. **Unyielding shim execution context.** Unyielding shims need `TaskContext` and `YieldContext` to call `exec_statement_unyielding`. But `TaskContext` contains the defer stack, async stack, etc. A new lightweight `TaskContext` could be created for unyielding execution.
5. **Return value handling.** Currently `call_function_inner` handles `ControlFlow::Return`, `ControlFlow::ExprValue`, etc. This logic must move into the shim (or into a helper called by the shim).
6. **Parameter type auditing and return type auditing.** Currently done in `call_function_inner`. Must move into the shims or remain in a wrapper around the shim call.
7. **Defer stack.** Currently `call_function_inner` manages `push_defer_frame` / `pop_and_run_defers`. For yielding shims, the spawned task manages its own defer stack. For unyielding shims, defer management happens in the calling context.

### Critical Analysis

**Alternative A: Shim does everything** — The shim captures everything it needs, creates its own `TaskContext`, and either executes synchronously (unyielding) or spawns a task (yielding). `call_function_inner` becomes a thin wrapper that only calls the shim and returns its result.

*Pros:* Clean. PENDING_INTERP_CALL eliminated entirely. `call_function_inner` becomes trivial. Compiled functions calling interpreted functions "just works" because the shim is self-contained.
*Cons:* Shim closures capture a lot of state. Arity checking, parameter auditing, and return type auditing would need to either move into the shim or be duplicated.

**Alternative B: Shim returns a "thunk"** — The shim returns a special `Value::Thunk` that `call_function_inner` executes. Similar to PENDING_INTERP_CALL but type-safe and carried in the return value instead of a thread-local.

*Pros:* Avoids thread-local hack. Type-safe.
*Cons:* Doesn't fix the core problem — compiled functions still can't call interpreted functions without going through `call_function_inner`.

**Alternative C: Make Shim async** — Change `Shim` to `Rc<dyn Fn(&[Value]) -> Pin<Box<dyn Future<Output = Result<Value, RuntimeError>>>>>`. 

*Pros:* Yielding shims naturally async. No thread-local needed.
*Cons:* **All** shims become async, including simple compiled functions. Compiled code that calls shims must be async. This is a fundamental architecture change that ripples into `ish-runtime` and `ish-codegen`. Unyielding compiled functions would need to wrap their synchronous result in `async { Ok(value) }`.

**Recommendation:** Alternative A. The shim should be self-contained. Parameter arity checking can remain in `call_function_inner` (it just checks counts before calling the shim). Type auditing remains in `call_function_inner` as a wrapper. The shim handles body execution, defer management, and ControlFlow → Value conversion.

### Proposed Implementation

**Unyielding interpreted function shim:**
```rust
let shim: Shim = Rc::new(move |args: &[Value]| {
    let call_env = captured_env.child();
    for (param, arg) in captured_params.iter().zip(args.iter()) {
        call_env.define(param.clone(), arg.clone());
    }
    // Execute body synchronously
    let mut task = TaskContext::new();
    let mut yc = YieldContext::new_unyielding();
    let result = Interpreter::exec_statement_unyielding(
        &captured_vm, &mut task, &mut yc, &captured_body, &call_env
    )?;
    // Convert ControlFlow to Value
    match result {
        ControlFlow::Return(v) | ControlFlow::ExprValue(v) => Ok(v),
        ControlFlow::None => Ok(Value::Null),
        ControlFlow::Throw(v) => Err(RuntimeError::thrown(v)),
    }
});
```

**Yielding interpreted function shim:**
```rust
let shim: Shim = Rc::new(move |args: &[Value]| {
    let call_env = captured_env.child();
    for (param, arg) in captured_params.iter().zip(args.iter()) {
        call_env.define(param.clone(), arg.clone());
    }
    let body = captured_body.clone();
    let vm = captured_vm.clone();
    let env = call_env.clone();
    // Spawn async task, return Future
    let future = FutureRef::new(async move {
        let mut task = TaskContext::new();
        let mut yc = YieldContext::new_yielding();
        task.push_defer_frame();
        let result = Interpreter::exec_statement_yielding(
            &vm, &mut task, &mut yc, &body, &env
        ).await;
        Interpreter::pop_and_run_defers_static(&vm, &mut task, &mut yc).await;
        match result? {
            ControlFlow::Return(v) | ControlFlow::ExprValue(v) => Ok(v),
            ControlFlow::None => Ok(Value::Null),
            ControlFlow::Throw(v) => Err(RuntimeError::thrown(v)),
        }
    });
    Ok(Value::Future(future))
});
```

**Simplified `call_function_inner`:**
```rust
fn call_function_inner(func: &Value, args: &[Value]) -> Result<Value, RuntimeError> {
    match func {
        Value::Function(f) => {
            // Arity check
            // Parameter type audit
            // Call the shim — it handles everything
            let result = (f.shim)(args)?;
            // Return type audit
            Ok(result)
        }
        _ => Err(not_callable_error()),
    }
}
```

Note: `call_function_inner` can potentially become synchronous since the shim handles all async work. However, the `await` expression handler still needs an async path to unwrap `Value::Future` results. The `await` handling should remain in `eval_expression` — it calls `call_function_inner` (sync), gets a `Value::Future`, then `.await`s the future inline.

### Decisions

- A) Shim is self-contained — captures VM, body, env, executes directly (recommended)
- B) Shim returns a `Value::Thunk` that `call_function_inner` executes
- C) Make `Shim` type async
--> Shim is self-contained — captures VM, body, env, executes directly

- B) No — all shims are async, unyielding ones just don't yield
--> Yes — yielding shims spawn a task and return `Value::Future`; unyielding shims execute synchronously and return the result directly

## Feature 4: Yielding/Unyielding Execution Context

### Issues to Watch Out For

1. **The `YieldContext` already tracks `unyielding_depth`.** This suppresses yield budget checks. The question is whether we need a broader mechanism.
2. **Unyielding functions cannot contain `await`, `spawn`, or `yield`.** The analyzer ensures this at declaration time. But the unyielding exec functions should also enforce this as a runtime invariant.
3. **`exec_statement_unyielding` is synchronous.** It cannot call `.await`. If it encounters an `Await` or `Spawn` node, it must return an error. This is a compile-time guarantee from the analyzer, but defense-in-depth says enforce it at runtime too.

### Critical Analysis

**Alternative A: Use `YieldContext` with a boolean flag** — Add a `is_yielding: bool` field to `YieldContext`. Yielding functions set it to `true`, unyielding to `false`. The same `exec_statement` function checks the flag.

*Pros:* Minimal change. Already familiar pattern.
*Cons:* One function doing two things. Can't eliminate the `Pin<Box<dyn Future>>` overhead for unyielding code since the function must still be async.

**Alternative B: Separate functions, no shared context** — `exec_statement_yielding` and `exec_statement_unyielding` are completely separate functions. They share no state tracking. Each knows its context from being called.

*Pros:* Clear separation. Unyielding version is truly synchronous. No runtime flag checking needed.
*Cons:* Must extract common logic into reusable helpers to avoid duplication.

**Alternative C: Generic parameter / trait-based dispatch** — Define a trait `ExecutionContext` with `Yielding` and `Unyielding` implementations. The exec functions are generic over the context.

*Pros:* Elegant. Zero-cost abstraction via monomorphization.
*Cons:* Complex. Trait objects don't monomorphize. Hard to make work when the yielding version needs `async` but the unyielding version doesn't.

**Recommendation:** Alternative B. The whole point is to have the unyielding path be truly synchronous — no `Pin<Box<dyn Future>>`, no `.await`, no Tokio runtime requirement. A shared context flag would defeat this purpose.

### Proposed Implementation

The execution context is implicit in which function is called:
- `exec_statement_yielding` / `eval_expression_yielding` — async, returns `Pin<Box<dyn Future<...>>>`, calls `yc.check_yield_budget().await`
- `exec_statement_unyielding` / `eval_expression_unyielding` — sync, returns `Result<ControlFlow, RuntimeError>` / `Result<Value, RuntimeError>`, no yield checks

The `YieldContext` is still passed to yielding functions for budget management. Unyielding functions receive a `TaskContext` only (for defer stack management).

### Decisions

**Decision 2:** What mechanism tracks yielding/unyielding execution context?
- A) Boolean flag on `YieldContext`
- B) Separate function pairs — context is implicit in which function you call (recommended)
- C) Generic trait-based dispatch
--> Separate function pairs — context is implicit in which function you call

## Feature 5: Split eval_expression and exec_statement

### Issues to Watch Out For

1. **Code duplication.** The match arms for `exec_statement` span ~600 lines. The match arms for `eval_expression` span ~300 lines. Without careful extraction, the yielding/unyielding variants would double this.
2. **Reusable helpers.** Most match arm logic is the same between yielding and unyielding — variable lookup, assignment, object construction, arithmetic, etc. The only differences are: (a) recursive calls use the yielding or unyielding variant, (b) yield budget checks, (c) function call handling.
3. **Method visibility.** Reusable helpers must be accessible to both variants. If they're methods on `Interpreter`, they need appropriate visibility.
4. **`call_function_inner` changes.** If shims are self-contained (Feature 3), `call_function_inner` becomes synchronous. Both yielding and unyielding `eval_expression` variants can call it the same way — the difference is whether the result (`Value::Future`) is awaited inline (for implied await).
5. **ControlFlow propagation.** Both variants use `ControlFlow`. The type doesn't change, just the return wrapper (`Pin<Box<dyn Future<Output = Result<ControlFlow, RuntimeError>>>>` vs `Result<ControlFlow, RuntimeError>`).

### Critical Analysis

**Alternative A: Extract match arm bodies into free functions** — Each match arm body becomes a standalone function that takes the relevant fields and returns the result. Both yielding and unyielding variants call the same extraction functions.

```rust
fn exec_variable_decl(vm, task, name, value, type_ann, env) -> Result<ControlFlow, RuntimeError>
fn exec_assignment(vm, task, target, value, env) -> Result<ControlFlow, RuntimeError>
// etc.
```

*Pros:* Maximum code reuse. Each match arm body exists once.
*Cons:* Many small functions with complex signatures. Some arms need `await` (e.g., anything that evaluates a sub-expression), which means the extraction function must itself be async for the yielding variant. This doesn't help — you still need two versions.

**Alternative B: Macro-based duplication** — Use a macro to generate both yielding and unyielding variants from a single template, inserting `.await` and yield checks where needed.

*Pros:* Single source of truth. Changes to match arms automatically apply to both variants.
*Cons:* Complex macro. Hard to debug. Rust macros don't handle `async`/non-async duality well.

**Alternative C: Extract pure computation, keep structure in variants** — Extract the parts that are *purely synchronous* (no sub-expression evaluation) into helpers. The match arm structure stays in each variant, calling extractable helpers where possible but duplicating the small amount of "glue" that differs.

*Pros:* Practical. The bulk of the logic that differs is recursive calls (`eval_expression` calls itself). The actual per-arm computation (variable lookup, arithmetic, object construction) is often a few lines.
*Cons:* Some duplication between variants for arms that evaluate sub-expressions.

**Recommendation:** Alternative C. Most match arms in `exec_statement` and `eval_expression` follow the pattern: evaluate sub-expressions recursively, then do a small amount of synchronous work. The synchronous work is extractable. The recursive calls are the only part that differs (yielding vs unyielding). Trying to eliminate *all* duplication leads to macros or complex abstractions that are harder to maintain than a modest amount of parallel structure.

### Proposed Implementation

**Step 1:** Extract synchronous helpers for each match arm where the arm body does significant work beyond recursive evaluation:

```rust
// Example: binary operation logic (after both operands are evaluated)
fn apply_binary_op(op: &BinaryOp, left: &Value, right: &Value) -> Result<Value, RuntimeError> { ... }

// Example: variable declaration (after the initializer is evaluated)
fn define_variable(env: &Environment, name: &str, value: Value, type_ann: &Option<TypeAnnotation>) -> Result<ControlFlow, RuntimeError> { ... }
```

**Step 2:** Create the four variants:

```rust
// Yielding (async) - essentially the current code with renamed recursive calls
fn exec_statement_yielding<'a>(...) -> Pin<Box<dyn Future<Output = Result<ControlFlow, RuntimeError>> + 'a>> {
    Box::pin(async move {
        match stmt {
            Statement::VariableDecl { name, value, .. } => {
                let val = Self::eval_expression_yielding(vm, task, yc, value, env).await?;
                define_variable(env, name, val, type_ann)
            }
            // ...
        }
    })
}

// Unyielding (sync) - same structure, no async, no yield checks
fn exec_statement_unyielding(...) -> Result<ControlFlow, RuntimeError> {
    match stmt {
        Statement::VariableDecl { name, value, .. } => {
            let val = Self::eval_expression_unyielding(vm, task, value, env)?;
            define_variable(env, name, val, type_ann)
        }
        // ...
    }
}
```

**Step 3:** Unyielding variants return errors for async-only constructs:

```rust
// In eval_expression_unyielding:
Expression::Await { .. } => Err(RuntimeError::system_error(
    "await not allowed in unyielding context",
    ErrorCode::AwaitUnyielding,
)),
Expression::Spawn { .. } => Err(RuntimeError::system_error(
    "spawn not allowed in unyielding context",
    ErrorCode::SpawnUnyielding,
)),
// In exec_statement_unyielding:
Statement::Yield => Err(RuntimeError::system_error(
    "yield not allowed in unyielding context",
    ErrorCode::YieldUnyielding,
)),
```

### Decisions

**Decision 3:** How are reusable match-arm functions organized?
- A) Free functions extracted from each arm (maximum deduplication but many functions)
- B) Macro-based generation of both variants
- C) Extract pure computation into helpers, keep parallel structure for match arms (recommended)
--> Extract pure computation into helpers, keep parallel structure for match arms

## Feature 6: Fix Function Declaration and apply

### Issues to Watch Out For

1. **Capturing the VM reference.** Yielding shims need `Rc<RefCell<IshVm>>` to call `exec_statement_yielding`. The VM reference must be capturable at declaration time. Currently `IshVm` is accessed via `Rc<RefCell<IshVm>>`, which is `Clone`, so this works.
2. **`apply` as a normal function.** With self-contained shims, `apply` just calls the passed-in function's shim: `(func.shim)(args)`. If the function is yielding, the shim returns `Value::Future`. If unyielding, it returns the result directly. `apply` doesn't need to know or care.
3. **`apply` yielding classification.** `apply` itself is unyielding — it doesn't `await` anything. It calls a shim synchronously. The result may be a `Value::Future`, but `apply` doesn't unwrap it.
4. **Existing cross-boundary tests.** The tests in `proto/ish-tests/functions/cross_boundary.sh` currently test `apply(fn, [args])` with unyielding lambdas. New tests should include yielding functions.
5. **Ledger builtin intercepts.** The special-case intercepts in `call_function_inner` for `active_standard`, `feature_state`, etc. need review. These are compiled builtins with shims — they should work through normal dispatch. The special-case intercepts may also be a workaround that can be eliminated.
6. **`Interpreter::` static method requirement.** `exec_statement_yielding` is currently a method on `Interpreter`. The shim closure can't easily call instance methods. These should be static methods (they already are — they receive `vm` as a parameter). The challenge is naming: `Interpreter::exec_statement_yielding(...)` requires the function to be `pub` or at least `pub(crate)`.

### Critical Analysis

**For `apply`:**

`apply` becomes trivially simple with self-contained shims:

```rust
fn apply_shim(args: &[Value]) -> Result<Value, RuntimeError> {
    let func = &args[0];
    let arg_list = match &args[1] {
        Value::List(list) => list.borrow().iter().cloned().collect::<Vec<_>>(),
        _ => return Err(type_mismatch_error()),
    };
    match func {
        Value::Function(f) => (f.shim)(&arg_list),
        _ => Err(not_callable_error()),
    }
}
```

No special-casing in `call_function_inner`. No need for `vm`, `task`, or `yc`. Just call the shim.

**For `call_function_inner`:**

With shims handling body execution:
1. Arity checking stays in `call_function_inner`.
2. Parameter type auditing stays.
3. Shim is called — returns either a direct value or `Value::Future`.
4. Return type auditing stays.
5. No PENDING_INTERP_CALL check.
6. No special-case builtin intercepts (they're all proper shims now).

`call_function_inner` can become **synchronous**:

```rust
fn call_function_inner(
    vm: &Rc<RefCell<IshVm>>,
    func: &Value,
    args: &[Value],
) -> Result<Value, RuntimeError> {
    match func {
        Value::Function(f) => {
            // Arity check
            // Param type audit
            let result = (f.shim)(args)?;
            // Return type audit
            Ok(result)
        }
        _ => Err(not_callable_error()),
    }
}
```

The async handling moves to `eval_expression_yielding`'s `FunctionCall` arm, which calls `call_function_inner`, gets the result, and handles implied await if needed.

### Proposed Implementation

1. **Update `apply` registration** in `builtins.rs` to be a normal compiled function with a simple shim.
2. **Remove the `apply` intercept** from `call_function_inner`.
3. **Remove all PENDING_INTERP_CALL** code: the struct, the thread-local, the stores, the retrieval.
4. **Update function declaration** (`Statement::FunctionDecl`) to create self-contained shims based on analyzer classification.
5. **Update lambda expression** (`Expression::Lambda`) to create self-contained shims.
6. **Make `call_function_inner` synchronous.**
7. **Move implied-await logic** to `eval_expression_yielding`'s function call handling.
8. **Remove ledger builtin intercepts** from `call_function_inner` if they work through normal shim dispatch.

### Decisions

- A) Simple shim that calls the passed-in function's shim directly (recommended)
- B) `apply` remains a special case but without PENDING_INTERP_CALL
