## Questions and Answers

### Q: What are the best practices and standard terminology for using agents for requirements and design work?

The field of agent-assisted software engineering is emerging rapidly. Several frameworks and patterns have converged around common practices:

#### Standard Terminology

The software engineering community has settled on terminology that distinguishes between **specification artifacts** (what to build) and **planning artifacts** (how to build it). The most widely used terms:

| Term | Meaning | Analogues |
|------|---------|-----------|
| **Request for Proposal (RFP)** | A cleaned-up, structured request describing what is wanted. Written by or on behalf of the human. | Brief, Requirements Document, User Story |
| **Design Proposal** | A response to an RFP that analyzes the request, proposes alternatives, and recommends a design. May go through multiple iterations. | Design Document, Technical Design, RFC |
| **Implementation Plan** | A concrete, ordered list of changes to make, derived from the accepted design. Optimized for execution, not for understanding the design rationale. | Work Breakdown, Sprint Plan, Task List |
| **Decision Record** | A record of a specific decision, its context, alternatives considered, and rationale. | ADR (Architecture Decision Record) |
| **Design History** | A narrative account of how the design evolved through proposal/response/decision exchanges. | Design Log, Decision Journal |

#### Industry Patterns for Agent-Driven Design

**1. Specification → Plan → Execute (SPE) Pattern**

Used by Devin, Cursor Composer, and similar coding agents. The agent receives a specification, creates an internal plan (often invisible to the user), then executes. The weakness is that the plan is opaque — the user cannot inspect or correct it before execution begins.

**2. Iterative Refinement Pattern**

Used by Anthropic's recommended workflows and GitHub Copilot's agent mode. The process is conversational: human provides requirements → agent proposes → human refines → agent updates → repeat. Each iteration produces a complete document. The weakness is that the "current state" is spread across a conversation thread, making it hard to extract the final agreed design.

**3. Document-Driven Development (DDD)**

Popularized by Amazon's "working backwards" approach and adapted for AI agents by several teams. All requirements and designs exist as documents, not conversation messages. The agent reads and writes documents; the human reviews and edits documents. Versions are tracked. The weakness is overhead for small changes.

**4. Dual-Track Development**

Common in agile organizations and increasingly adapted for agent workflows. Two parallel tracks run: **Discovery** (understanding what to build) and **Delivery** (building it). Discovery produces specifications; Delivery consumes them. The key insight is that these tracks have different rhythms and artifacts.

**5. Authoritative Document Pattern**

Emerging best practice specifically for agent workflows. One document is designated the "source of truth" for implementation. All iterative discussion may happen in other documents or conversations, but the authoritative document is always kept current. The agent implementing the feature reads only this document. This prevents the "stale context" problem where an agent reads an outdated intermediate version.

#### Best Practices for Agent Context Management

1. **Single source of truth.** The implementing agent should read exactly one document to understand what to build. That document must be self-contained and current.

2. **Authority ordering.** When multiple artifacts describe the same feature, there must be an explicit precedence order. The agent must update the most authoritative artifact first, so that if it reads any artifact during implementation, it gets the current truth.

3. **Explicit state tracking.** Features should have visible status (not-started, in-progress, completed) in high-authority documents like the roadmap, so that agents do not infer incorrect status from stale text.

4. **Checkpoint-based execution.** Long implementation tasks should be broken into checkpoints where the agent verifies its work against the specification before continuing. This prevents drift.

5. **Batch communication.** For complex design work, batch exchanges (long document in, long document out) outperform conversational turn-taking because they allow the human to address all concerns simultaneously and the agent to produce a coherent, internally consistent response.

## Feature 1: Revised Proposal Lifecycle

The current process is: prompt → RFP → proposal (with decisions inline) → implementation. The RFP observed several failure modes with this process. This feature redesigns the lifecycle.

### Issues to Watch Out For

1. **Migration burden.** Existing proposals follow the current format. Changing the process should not require retroactively reformatting accepted proposals.

2. **Overhead for small changes.** A three-phase process (RFP → design proposal → implementation plan) could be excessive for a simple bug fix or minor feature. The process needs a lightweight path.

3. **Duplication risk.** If the design proposal and implementation plan are separate documents, information will be duplicated and can drift. The boundary between "design" and "plan" must be crisp.

4. **History capture vs. implementation clarity.** The RFP identifies a tension: history captures the evolution of ideas, but evolving documents confuse implementing agents. These are fundamentally different audiences (historians vs. executors) and should be served by different documents.

### Critical Analysis

#### Alternative A: Three-Document Lifecycle (RFP → Design Proposal → Implementation Plan)

Separate the current "proposal" into two documents:
- **Design Proposal**: Analysis, alternatives, pros/cons, recommendations, decisions. Organized for understanding *why*.  
- **Implementation Plan**: Consolidated final state, ordered TODO list, file-by-file changes. Organized for understanding *what to do*.

The design proposal is iterative (multiple versions as human and agent converge). The implementation plan is generated once from the accepted design proposal.

**Pros:**
- Clean separation of audiences (design reviewers vs. implementing agents)
- Implementation plan is always current — no stale historical text
- Design proposal preserves the full deliberation for future reference
- Matches the Authoritative Document Pattern from industry
- The TODO list is part of the implementation plan, not the design proposal, so it reflects only accepted decisions

**Cons:**
- Three document types instead of two — more files to manage
- Generating the implementation plan is an additional step
- Risk that the implementation plan diverges from the design proposal if the design is amended post-acceptance

#### Alternative B: Single Evolving Document with Sections

Keep the current single-proposal format but add a "Final State" section at the top that is always kept current as decisions are made. Below it, the historical analysis sections remain but are clearly marked as "Design History — do not use for implementation."

**Pros:**
- Fewer files — everything in one place
- No new document type to learn or manage
- The "Final State" section serves as the implementation plan

**Cons:**
- A single document serving two audiences is the root cause of the current problems
- Long proposals become unwieldy with both sections
- Agents may still read the historical sections and get confused
- The "do not use for implementation" marker is a convention that agents may not respect

#### Alternative C: Two-Document Lifecycle with In-Place Consolidation

Keep RFP → Proposal, but when the proposal is accepted, it is *rewritten* into an implementation plan. The original proposal (with all the deliberation) is moved to history. The implementation plan replaces it at the same path.

**Pros:**
- Only two document types (RFP and proposal/plan)
- Clean: once accepted, the document is purely an implementation plan
- History is preserved in the history directory where it belongs

**Cons:**
- Destructive: the original proposal analysis is no longer in its original location
- The same path means different things at different times (proposal vs. plan)
- Moving files creates link breakage

#### Alternative D: Current Process with Stricter Agent Instructions

Keep the current process but add stronger agent instructions (via copilot-instructions.md and skills) that require the agent to:
- Always re-read the full proposal before starting implementation
- Verify each step against the proposal
- Complete all checklist items before reporting success

**Pros:**
- No process change — just better instructions
- Zero migration cost
- Simpler

**Cons:**
- Does not address the structural problem — proposals are still organized for deliberation, not execution
- "Try harder" is not a reliable fix for systematic failure modes
- Does not address the contradictory-information problem in follow-on proposals

#### Recommendation

**Alternative A (Three-Document Lifecycle)** addresses all four observed failure modes:
- Contradictory information → the implementation plan is generated fresh from accepted decisions, not accumulated through iterations
- Poor implementation readability → the implementation plan is optimized for execution
- Specification drift → the implementation plan is the single authoritative source
- Incomplete task execution → the implementation plan includes a comprehensive TODO list

The overhead concern is addressed by making the lightweight path explicit: for changes that require no design deliberation, the RFP and design proposal can be skipped, and an implementation plan can be created directly.

### Proposed Implementation

#### New Artifact Types

| Artifact | Location | Purpose | Audience |
|----------|----------|---------|----------|
| RFP | `docs/project/rfp/` | Cleaned-up request from human | Design reviewers |
| Design Proposal | `docs/project/proposals/` | Analysis, alternatives, decisions | Design reviewers |

```

    ↓ (clean up, format)

    ↓ (analyze, propose alternatives)

    ↓ (human punch list)
Design Proposal v2
    ↓ (repeat until accepted)
Design Proposal vN [accepted]
    ↓ (consolidate, generate TODO)
Implementation Plan
    ↓ (execute)
Code + Docs + Tests + History
```

#### Design Proposal Iteration Rules

During iterative refinement:
1. Each new version of the design proposal is a **complete replacement**, not a diff or addendum.
2. The proposal must be internally consistent — no references to "the previous version said X."
3. Decisions are recorded inline with `-->` notation (as currently done).
4. When a decision contradicts an earlier analysis section, the analysis section is updated to reflect the decision, not left as-is with the decision appended.

## Feature 2: Authority-Ordered Execution

The RFP identifies a specific failure mode: the agent read the roadmap during implementation and found a feature marked as "not yet implemented" when it had in fact been implemented. This caused the agent to proceed as if the feature was out of scope.

### Issues to Watch Out For

1. **Circular updates.** If the roadmap must be updated before code changes, but the roadmap entry says "completed" before the code exists, there's a brief window of inconsistency. The solution is a three-state status: not-started → in-progress → completed. "In progress" means "the agent is currently working on this."

2. **Authority order varies by project.** "Docs before code" is appropriate for a design-heavy project like ish, but many projects follow "code first, docs after." The authority order should be configurable per project, not hardcoded.

3. **Partial completion.** If the agent completes steps 1–7 of 13 and then the session ends, the project is in an inconsistent state. The TODO list needs to be a durable artifact that survives session boundaries.

4. **Agent compliance.** The authority order is only useful if the agent follows it. This requires explicit instructions in AGENTS.md, copilot-instructions.md, and/or skills.

### Critical Analysis

The authority-ordering concept is sound and well-supported by the "single source of truth" principle. The key insight is: **if an agent might read an artifact during implementation, that artifact must be updated before any work that depends on it.**

The specific order proposed in the RFP (docs > assurance tests > code) is a reasonable default. The full order should be:

1. **Glossary** — New terms must exist before any document references them.
2. **Roadmap** — Feature status set to "in progress" so no agent infers it's out of scope.
3. **Specification** — The normative definition of behavior.
4. **Architecture** — How the spec is realized in the codebase.
5. **User guide / AI guide** — How to use the feature.
6. **Agent documentation** — Skills, instructions, AGENTS.md updates.
7. **Acceptance tests** — Tests that define "done" for the feature.
8. **Code** — Implementation.
9. **Unit tests** — Tests for internal correctness.
10. **Roadmap** — Feature status set to "completed."
11. **History** — Narrative of the work.
12. **Index files** — Updated last (they reference everything above).

There is no real alternative to explore here — the principle is well-established. The decision is about the specific order and how strictly to enforce it.

### Proposed Implementation

1. Add an "Authority Order" section to the implementation plan template (shown in Feature 1).
2. Add a glossary entry for "authority order."
3. Add the default authority order to AGENTS.md.
4. The `implement` skill (see Feature 5) enforces execution in authority order.

### Decisions

**Decision:** Accept the 12-step default authority order above, or modify it?
--> Accept

**Decision:** Should authority order be enforced by tooling (a script that checks timestamps/git history) or by convention (agent instructions only)?
--> Agent instructions only

## Feature 3: Improved Terminology

The RFP requests clearer terminology to distinguish design-phase artifacts from implementation-phase artifacts.

### Issues to Watch Out For

1. **Glossary consistency.** All new terms must be added to GLOSSARY.md and used consistently. Existing documents referencing the old implicit terminology need not be retroactively updated, but new documents must use the new terms.

2. **Overloaded term: "proposal."** Currently, "proposal" means both "the document containing analysis and alternatives" and "the thing that tells the agent what to implement." Separating these meanings is the core of this feature.

### Critical Analysis

#### Proposed Terminology

| Current Term | New Term | Definition |
|--------------|----------|------------|
| (prompt file) | **Prompt** | Raw input from the human, before any cleanup. Not a project artifact. |
| RFP | **RFP (Request for Proposal)** | Cleaned-up prompt, formatted as a proper request. Input to the design process. |
| Proposal | **Design Proposal** | Analysis of the RFP with alternatives, pros/cons, recommendations, and decisions. Output of the design process. May go through multiple iterations (v1, v2, …). |
| (new) | **Implementation Plan** | Consolidated, authoritative document derived from the accepted design proposal. Contains the TODO list and file-by-file changes. Input to the implementation process. |
| (new) | **Design History** | Narrative of the deliberation process: what was proposed, what alternatives were considered, what was decided, and by whom. Output of the design process, stored in `docs/project/history/`. |
| (new) | **Punch List** | A list of corrections, additions, or changes that the human wants made to a design proposal before accepting it. Input to the next iteration. |

#### Process Terminology

| Term | Definition |
|------|------------|
| **Design phase** | The iterative process of creating and refining a design proposal from an RFP. Ends when the human accepts the design proposal. |
| **Planning phase** | The process of generating an implementation plan from an accepted design proposal. Typically one step, not iterative. |
| **Implementation phase** | The process of executing an implementation plan. Ends when all TODO items are complete. |
| **Authority order** | The defined sequence in which project artifacts must be updated during implementation, from most authoritative (glossary, roadmap) to least authoritative (index files). |
| **Feature coherence audit** | A cross-check of all project artifacts related to a single feature, verifying consistency. |

No real alternative exists — the question is whether these specific terms are clear enough. The key improvement over the current state is the explicit split between "design proposal" (for deliberation) and "implementation plan" (for execution).

### Proposed Implementation

1. Add all new terms to GLOSSARY.md.
2. Update AGENTS.md to reference the new terminology.
3. Update the skill definitions (Features 5–7) to use the new terms.

### Decisions

**Decision:** Accept the proposed terminology, or suggest modifications?
--> Accept

## Feature 4: Batch-Oriented Conversation Protocol

The RFP specifies that the design process should remain batch-oriented: the human submits a detailed document, the agent produces a detailed response, the human responds with a punch list, and so on.

### Issues to Watch Out For

1. **Skill invocation model.** Currently, the `/propose` skill is invoked once and produces a single output. Iterative refinement happens by invoking `/propose` again with a follow-on prompt (as in the shell-build → shell-build-ii → shell-build-iii chain). This creates the chain-of-documents problem.

2. **Punch list processing.** The agent must process every item in a punch list and produce a complete replacement document, not a diff. This is a convention that must be documented and enforced by agent instructions.

3. **Version tracking.** If the design proposal goes through v1, v2, v3, only the latest version should exist in `docs/project/proposals/`. Prior versions are captured in the design history, not as separate files.

### Critical Analysis

#### Alternative A: In-Place Replacement (punch list → updated proposal)

Each iteration replaces the design proposal file in-place. The punch list and the agent's responses are captured in the design history file.

**Pros:**
- Only one proposal file exists at any time — always current
- No chain-of-documents problem
- The history file captures the deliberation narrative

**Cons:**
- Git diff shows changes, but the intermediate states are not easily browseable as standalone documents
- The history file must be written carefully to capture what changed and why

#### Alternative B: Versioned Files (proposal-v1.md, proposal-v2.md, …)

Each iteration creates a new numbered file. The latest version is the authoritative one.

**Pros:**
- All versions are directly browseable
- Easy to compare versions

**Cons:**
- File proliferation
- Ambiguity about which version is current
- This is the chain-of-documents approach that caused the original problems

#### Alternative C: Single File with Append-Only History Section

The design proposal has a "History" section at the bottom where each iteration appends a dated entry describing what changed. The body of the proposal is always the current state.

**Pros:**
- Single file, always current
- History is inline — no need to cross-reference
- Simple to implement

**Cons:**
- Documents become very long
- The history section may confuse implementing agents (same problem as before, just in a different section)

#### Recommendation

**Alternative A (In-Place Replacement)** is the cleanest approach. The design proposal reflects only the current accepted state. The design history captures the narrative of how it got there. This mirrors the separation in Feature 1: design proposals are for the current design, design history is for the evolution of the design.

The key convention: when the human sends a punch list, the agent must:
1. Read the current design proposal
2. Read the punch list
3. Produce a complete replacement design proposal incorporating all punch list items
4. Append a section to the design history describing the iteration

### Proposed Implementation

1. Document the batch protocol in the updated `/propose` skill.
2. Add a `/revise` skill that takes a punch list and updates an existing design proposal.
3. Document the convention that design proposals are always complete replacements.

### Decisions

**Decision:** Which conversation alternative to adopt? (A) In-place replacement, (B) Versioned files, (C) Append-only history section?
--> In-place replacement.  Note that history should be verbose.  History should contain every version of every proposal, and the associated punch list.  It should also contain a narrative explaining how the proposal evolved between versions.  This has the effect that decisions-not-taken and questions-already-answered can be removed from subsequent versions of the proposal, because the information is still available in historical versions of the proposal.

**Decision:** Should the `/revise` skill be a separate skill, or a mode of the `/propose` skill (e.g., `/propose --revise`)?
--> A separate skill

## Feature 5: Updated Agent Documentation and Skills

The RFP identifies that agent documentation needs updating so agents consistently follow the new process. This includes copilot instructions, new skills, and updates to AGENTS.md.

### Issues to Watch Out For

1. **Instruction hierarchy.** VS Code Copilot has multiple instruction sources: `.github/copilot-instructions.md` (always loaded), `.instructions.md` files (pattern-scoped), skill files, and AGENTS.md. These need to be consistent and not contradictory.

2. **Skill granularity.** Too many skills creates confusion about which to invoke. Too few means each skill is too complex. The skills should map to natural user intents.

3. **Agent compliance enforcement.** Instructions are only effective if agents actually follow them. Critical instructions should be reinforced by skill procedures, not just stated once in a general instructions file.

### Critical Analysis

#### Proposed Skills

| Skill | Trigger | Purpose |
|-------|---------|---------|
| `/propose` (updated) | "propose", "proposal", "design" | Create a design proposal from an RFP. Handles initial creation. |
| `/revise` (new) | "revise", "punch list", "update proposal" | Update a design proposal based on a punch list. Produces a complete replacement. |
| `/plan` (new) | "plan", "implementation plan", "ready to implement" | Generate an implementation plan from an accepted design proposal. |
| `/implement` (new) | "implement", "build", "execute plan" | Execute an implementation plan, following authority order, with checkpoint verification. |
| `/audit` (new) | "audit", "coherence check", "consistency check" | Run a feature coherence audit across all project artifacts. |

#### Copilot Instructions File

A `.github/copilot-instructions.md` file would provide always-loaded context about the proposal process, authority order, and key conventions. This is appropriate because these conventions apply to all work in the repository, not just specific file patterns.

#### AGENTS.md Updates

AGENTS.md should be updated to reference the new terminology, lifecycle, and skills. The Task Playbooks table should include entries for the new skills.

### Proposed Implementation

#### `.github/copilot-instructions.md`

```markdown
## Proposal Process

This project uses a structured proposal process for all non-trivial changes:

1. **RFP** → 2. **Design Proposal** (iterative) → 3. **Implementation Plan** → 4. **Implementation**

See GLOSSARY.md for definitions of these terms.

## Authority Order

When implementing changes, update project artifacts in this order:

1. GLOSSARY.md (new terms)
2. Roadmap (status → "in progress")
3. Specification docs
4. Architecture docs
5. User guide / AI guide
6. Agent documentation (AGENTS.md, skills)
7. Acceptance tests
8. Code
9. Unit tests
10. Roadmap (status → "completed")
11. History file
12. Index files

Always update more authoritative artifacts before less authoritative ones.
If you read an artifact during implementation and it seems to contradict the
implementation plan, the implementation plan takes precedence.

## Implementation Discipline

- The implementation plan is the single source of truth during implementation.
- Complete all TODO items in the implementation plan before reporting success.
- At each checkpoint, verify your work against the implementation plan.
- Do not inject behavior that contradicts the implementation plan, even if it
  seems like an improvement. Propose changes in a follow-up, not during implementation.
```

#### Skill: `/revise`

```
1. Read the punch list (provided by user).
2. Read the current design proposal (at the path specified in the punch list,
   or ask if not specified).
3. For each item in the punch list, identify the affected sections of the
   design proposal.
4. Produce a complete replacement design proposal that incorporates all
   punch list items. The replacement must be internally consistent — no
   references to "the previous version" or "as changed by this punch list."
5. Save the replacement design proposal at the same path (overwriting the
   previous version).
6. Append an iteration summary to the design history file, describing what
   the punch list requested and how the proposal was updated.
```

#### Skill: `/plan`

```
1. Read the accepted design proposal.
2. Extract all accepted decisions.
3. For each feature, compile the requirements (testable statements of behavior).
4. Determine the authority order (use the default unless the proposal specifies otherwise).
5. Generate the ordered TODO list:
   a. For each artifact type in authority order, list the files that need to change
      and what changes are needed.
   b. Include checkpoint items at natural boundaries (e.g., after docs, after tests,
      after code).
6. Include a Reference section with any historical information needed during
   implementation (original file paths, old terminology, etc.).
7. Save to docs/project/plans/<name>.md.
```

#### Skill: `/implement`

```
1. Read the implementation plan.
2. Verify all prerequisite artifacts exist (design proposal is accepted,
   dependencies are met).
3. Execute TODO items in order:
   a. Before each item, mark it as in-progress in the plan.
   b. After each item, verify the change against the requirements.
   c. Mark the item as completed in the plan.
   d. At checkpoint items, pause and verify all preceding work against
      the implementation plan.
4. After all items are complete, run acceptance tests.
5. Report completion.
```

### Decisions

**Decision:** Accept the five-skill decomposition (propose, revise, plan, implement, audit), or combine/split differently?
--> Modify. The skills as listed do not account for two things.  First, the punch list might be delivered in the form of a separate document, but it is more likely to be delivered in the form of decisions made in-line in the document.  Second, the final version of the proposal is likely to still contain decision points and a punch list, although the punch list may be of a trivial "accept everything" nature.  The revise skill should be modified to explicitly allow for in-line punch lists.  The "accept" skill should be added.  The purpose of the "accept" skill is to revise the proposal one last time so that all decisions are recorded, the content of the proposal is consistent with all the decisions made, and no new decisions are requested.  The accept skill should not result in any surprises.  Also, the plan skill will often be called with a proposal that has not actually been accepted yet. In this case, the agent should first run the accept skill against the proposal before proceeding. Note that the human has the option of sending it back after the plan step.  After reviewing the accepted proposal and the implementation plan, the human may revise the plan again.  In this case, removing the old implementation plan should be part of the revise skill.

**Decision:** Should the copilot-instructions.md contain the full authority order list, or just a reference to where it is documented?
--> The full authority list

**Decision:** Should the `/implement` skill exist, or should implementation remain a manual agent interaction guided by instructions?
--> An implement skill should exist.

## Feature 6: Feature Coherence Audit

The RFP requests a skill that verifies consistency of a single feature across all project artifacts.

### Issues to Watch Out For

1. **Feature identification.** How does the agent know which artifacts relate to a given feature? There's no explicit feature-to-file mapping. The agent must discover this through search, metadata, and cross-references.

2. **Consistency definition.** "Consistent" means different things in different contexts. A spec saying "the language supports X" and a roadmap saying "X is completed" are consistent. A spec saying "X behaves as Y" and a test asserting Y are consistent. The audit must understand these different kinds of consistency.

3. **False positives.** At the specification level, documents may describe planned behavior that is not yet implemented. This is not an inconsistency — it's expected. The audit needs to distinguish between "not yet implemented" (expected) and "described differently in two places" (a problem).

4. **Scope of audit.** A full audit of a large feature could be extremely expensive (reading dozens of files). The audit should be structured with levels of depth.

### Critical Analysis

#### Alternative A: Checklist-Based Audit

The audit follows a fixed checklist of artifact types and, for each one, verifies that the feature is described consistently.

**Checklist:**
1. **Glossary**: Does the feature use terms consistently?
2. **Roadmap**: Is the feature status accurate?
3. **Specification**: Is the feature behavior fully specified?
4. **Architecture**: Does the architecture doc describe the implementation?
5. **User guide**: Does the user guide explain how to use the feature?
6. **AI guide**: Does the AI guide cover the feature?
7. **Acceptance tests**: Do tests exist for the feature? Do they match the spec?
8. **Code**: Does the implementation match the spec and tests?
9. **Unit tests**: Do unit tests cover internal behavior?
10. **AGENTS.md / Skills**: Is the feature reflected in agent documentation?

**Pros:**
- Systematic — no artifact type is missed
- Easy to implement as a skill
- Produces a clear report with pass/fail per artifact type

**Cons:**
- Expensive — reads many files for every audit
- Some checks are subjective ("does X match Y?")
- The checklist may not cover all artifact types for all features

#### Alternative B: Dependency-Driven Audit

Use the `depends-on` frontmatter to build a dependency graph. Audit only the files in the dependency chain for the feature's primary document. Report inconsistencies.

**Pros:**
- Scoped — only audits relevant files
- Uses existing infrastructure (depends-on metadata)

**Cons:**
- `depends-on` may be incomplete or stale
- Doesn't discover files that *should* reference the feature but don't

#### Alternative C: Hybrid Audit (Checklist + Dependency)

Start with the checklist to identify which artifact types should contain information about the feature. For each artifact type, use search to find relevant files. Then cross-check for consistency.

**Pros:**
- Combines the thoroughness of the checklist with the precision of dependency tracking
- Can discover missing coverage (an artifact type that should have the feature but doesn't)
- Adapts to different features (some features don't touch all artifact types)

**Cons:**
- Most complex to implement
- Search-based discovery may produce false positives

#### Recommendation

**Alternative C (Hybrid Audit)** is the most robust approach. The checklist ensures nothing is missed; the search ensures all relevant files are found, even if `depends-on` is incomplete.

#### Audit Skill Procedure

```
1. Identify the feature to audit (from user input or implementation plan).
2. For each artifact type in the checklist:
   a. Search for files related to the feature (keyword search, depends-on traversal).
   b. Read relevant sections.
   c. Extract claims about the feature (behavior, status, syntax, etc.).
3. Cross-check claims across artifact types:
   a. Spec vs. acceptance tests: do tests cover all specified behaviors?
   b. Spec vs. code: does the implementation match the spec?
   c. Roadmap vs. actual: is the status accurate?
   d. User guide vs. spec: does the guide accurately describe the feature?
   e. AI guide vs. spec: same check.
4. Report inconsistencies and missing coverage.
5. Save the audit report to a session file or a designated location.
```

## Feature 7: Design Proposal Iteration Quality

The RFP identifies that follow-on proposals sometimes contain contradictory information because the agent did not scan for elements that contradicted new decisions.

### Issues to Watch Out For

1. **This is primarily a process problem, not a tooling problem.** The agent has contradictory text because it appends new text without revising old text. The fix is the convention that each iteration produces a complete replacement (Feature 4).

2. **Verification step.** Even with complete replacements, the agent might introduce new contradictions. A self-consistency check should be part of the `/revise` skill.

### Critical Analysis

The complete-replacement convention from Feature 4 is the primary fix. Two additional safeguards are worth considering:

#### Self-Consistency Check

After generating a replacement design proposal, the agent scans the entire document for internal contradictions. This is a natural-language reasoning task, not a syntactic check.

**Implementation:** Add a step to the `/revise` skill: "After generating the replacement, review the entire document for internal contradictions. If any are found, resolve them before saving."

#### Decision Register

Maintain a list of all accepted decisions in a dedicated section of the design proposal. Before generating any analysis text, the agent must check the decision register and ensure no analysis contradicts a recorded decision.

**Implementation:** The design proposal template includes a "Decision Register" section at the top. The `/revise` skill updates this section first, then updates the body to be consistent with it.

### Proposed Implementation

Both safeguards are lightweight and should be adopted:

1. The design proposal template includes a "Decision Register" section.
2. The `/propose` skill populates the decision register as decisions are made.
3. The `/revise` skill:
   a. Updates the decision register first.
   b. Rewrites the body to be consistent with the decision register.
   c. Scans the full document for contradictions.
   d. Resolves any contradictions before saving.

### Decisions

**Decision:** Adopt the Decision Register approach, the self-consistency check, or both?
--> Both

## Feature 8: Implementation Plan TODO Durability

The RFP notes that agents lose track of tasks across session boundaries.

### Issues to Watch Out For

1. **The implementation plan file is already durable.** It lives on disk. The agent can read it at the start of a new session.

2. **The issue is that agents don't read it.** The fix is in agent instructions and skills, not in a new mechanism.

3. **TODO state tracking.** The implementation plan's TODO list needs a convention for marking items as not-started, in-progress, or completed. Markdown checkboxes (`- [ ]`, `- [x]`) are a natural fit.

### Critical Analysis

The implementation plan template (Feature 1) already includes a TODO section with checkboxes. The `/implement` skill (Feature 5) already specifies that the agent marks items as in-progress and completed. The remaining need is:

1. **Session start behavior.** When an agent starts a new session and is asked to continue implementation, it should read the implementation plan and resume from the first uncompleted item.

2. **Copilot instructions.** The `.github/copilot-instructions.md` should include: "If an implementation plan exists with uncompleted items, resume from the first uncompleted item."

### Proposed Implementation

Add to `.github/copilot-instructions.md`:

```markdown
## Resuming Implementation

If you are asked to continue implementing a feature and an implementation plan
exists in docs/project/plans/, read it and resume from the first uncompleted
TODO item. Do not start over.
```

### Decisions

**Decision:** Accept this approach, or propose a different mechanism for cross-session continuity?
--> Accept
