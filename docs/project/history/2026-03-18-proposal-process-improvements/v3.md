## Research Summary

Research into agent-assisted software engineering identified five industry patterns and five best practices that inform this design. The full research is preserved in Version 1 (see [design history](../history/2026-03-18-proposal-process-improvements/v1.md)). The key findings that shaped the design:

- **Authoritative Document Pattern:** One document is the source of truth for implementation. All iterative discussion happens elsewhere. This prevents stale-context problems.
- **Single source of truth:** The implementing agent reads exactly one document. That document must be self-contained and current.
- **Authority ordering:** When multiple artifacts describe the same feature, update the most authoritative first, so any artifact read during implementation reflects the current truth.
- **Batch communication:** For complex design work, long-document exchanges outperform conversational turn-taking.
- **Checkpoint-based execution:** Long tasks should include verification checkpoints to prevent drift.

## Feature 1: Three-Document Proposal Lifecycle

The current process (prompt → RFP → proposal → implementation) is replaced by a three-document lifecycle that separates design deliberation from implementation instructions.

### Design

The lifecycle uses three artifact types, each serving a distinct audience:

| Artifact | Location | Purpose | Audience |
|----------|----------|---------|----------|
| RFP | `docs/project/rfp/` | Cleaned-up request from human | Design reviewers |
| Design Proposal | `docs/project/proposals/` | Analysis, alternatives, decisions | Design reviewers |
| Implementation Plan | `docs/project/plans/` | Consolidated TODO, file changes | Implementing agent |
| Design History | `docs/project/history/<slug>/` | Narrative of deliberation | Future reference |

### Lifecycle Flow

```
prompt
    ↓ (agent cleans up and formats)
RFP
    ↓ (agent decides: needs design deliberation?)
    ├── yes → Design Proposal v1
    │           ↓ (human punch list / inline decisions)
    │         Design Proposal v2
    │           ↓ (repeat until accepted)
    │         Design Proposal vN [accepted]
    │           ↓ (accept skill finalizes)
    │         Accepted Design Proposal
    │           ↓ (plan skill generates TODO)
    │         Implementation Plan
    └── no → Implementation Plan (directly from RFP)
                ↓ (implement skill executes)
              Code + Docs + Tests + History
```

The process always begins with a prompt, which the agent formats as an RFP. The agent then decides whether the RFP needs further elaboration through a design proposal, or whether it can proceed directly to an implementation plan.

### Design Proposal Iteration Rules

During iterative refinement:

1. Each new version is a **complete replacement**, not a diff or addendum.
2. The proposal must be internally consistent — no references to "the previous version said X."
3. Decisions are recorded inline with `-->` notation.
4. When a decision contradicts an earlier analysis section, the analysis section is updated to reflect the decision.
5. Decisions-not-taken and questions-already-answered may be removed from subsequent versions, because the information is preserved in the design history.

### Design History Convention

Each proposal's design history is stored in its own directory under `docs/project/history/`. The directory is named with the date and proposal slug (e.g., `2026-03-18-proposal-process-improvements/`). Within the directory:

| File | Contents |
|------|----------|
| `summary.md` | Narrative of how the proposal evolved between versions: what each punch list requested, what changed, what was decided. |
| `v1.md` | Full text of Version 1. |
| `v2.md` | Full text of Version 2. |
| ... | One file per version. |
| `accepted.md` | Full text of the accepted version (final). |

The `docs/project/history/` directory itself contains an `INDEX.md` listing all proposal history directories.

This structure preserves full traceability while keeping the current proposal clean and preventing any single file from becoming too large.

## Feature 2: Authority-Ordered Execution

When an agent implements a feature, it must update project artifacts in a defined sequence so that any artifact read during implementation reflects the current truth. This prevents the failure mode where an agent reads the roadmap, finds a feature marked "not yet implemented," and proceeds as if it is out of scope.

### Design

The **authority order** is the defined sequence in which artifacts are updated:

1. **Glossary** — New terms must exist before any document references them.
2. **Roadmap** — Feature status set to "in progress" so no agent infers it's out of scope.
3. **Specification** — The normative definition of behavior.
4. **Architecture** — How the spec is realized in the codebase.
5. **User guide / AI guide** — How to use the feature.
6. **Agent documentation** — Skills, instructions, AGENTS.md updates.
7. **Acceptance tests** — Tests that define "done" for the feature.
8. **Code** — Implementation.
9. **Unit tests** — Tests for internal correctness.
10. **Roadmap** — Feature status set to "completed."
11. **History** — Narrative of the work.
12. **Index files** — Updated last (they reference everything above).

The roadmap uses three-state tracking: **not-started → in-progress → completed**. Setting "in progress" before code changes prevents agents from drawing incorrect conclusions about feature scope.

Authority order is enforced by agent instructions only (in copilot-instructions.md, AGENTS.md, and the implement skill), not by tooling.

## Feature 3: Terminology

The following terms are added to the project glossary. All new documents must use these terms consistently.

### Artifact Terminology

| Term | Definition |
|------|------------|
| **Prompt** | Raw input from the human, before any cleanup. Not a project artifact. |
| **RFP (Request for Proposal)** | Cleaned-up prompt, formatted as a proper request. Input to the design process. |
| **Design Proposal** | Analysis of the RFP with alternatives, pros/cons, recommendations, and decisions. Output of the design process. May go through multiple iterations. |
| **Implementation Plan** | Consolidated, authoritative document derived from the accepted design proposal. Contains the TODO list and file-by-file changes. Input to the implementation process. |
| **Design History** | Narrative of the deliberation process: what was proposed, what alternatives were considered, what was decided, and by whom. Stored as a directory per proposal under `docs/project/history/`, with a summary file and separate files for each version. |
| **Punch List** | A list of corrections, additions, or changes that the human wants made to a design proposal before accepting it. May be delivered as a separate document or as inline decisions in the proposal itself. |
| **Decision Register** | A consolidated list of all accepted decisions in a design proposal. Maintained at the top of the proposal as the authoritative reference. |

### Process Terminology

| Term | Definition |
|------|------------|
| **Design phase** | The iterative process of creating and refining a design proposal from an RFP. Ends when the human accepts the design proposal. |
| **Planning phase** | The process of generating an implementation plan from an accepted design proposal. Typically one step, not iterative. |
| **Implementation phase** | The process of executing an implementation plan. Ends when all TODO items are complete. |
| **Authority order** | The defined sequence in which project artifacts must be updated during implementation, from most authoritative (glossary, roadmap) to least authoritative (index files). |
| **Feature coherence audit** | A cross-check of all project artifacts related to a single feature, verifying consistency. |

## Feature 4: Batch-Oriented Conversation Protocol

The design process uses in-place replacement for proposal iterations. Only one version of the design proposal exists on disk at any time — always the current version. Prior versions are preserved in the proposal's design history directory.

### Protocol

When the human sends a punch list (whether as a separate document or as inline decisions in the proposal):

1. The agent reads the current design proposal.
2. The agent reads the punch list or inline decisions.
3. The agent saves the current version to the design history directory (as a new version file).
4. The agent produces a **complete replacement** design proposal incorporating all punch list items.
5. The replacement must be internally consistent — no references to previous versions.
6. The agent appends a narrative to the summary file in the design history directory explaining how the proposal evolved.

### Punch List Formats

The punch list may take two forms:

- **Separate document:** A file or message listing corrections, additions, or changes.
- **Inline decisions:** The human edits the proposal directly, adding `-->` decisions to decision points and/or annotating sections with corrections.

Both formats are handled by the `/revise` skill.

## Feature 5: Agent Skills

Six skills support the proposal lifecycle. Each maps to a natural user intent.

### Skill Overview

| Skill | Trigger | Purpose |
|-------|---------|---------|
| `/propose` (updated) | "propose", "proposal", "design" | Create a design proposal from an RFP. Initial creation only. |
| `/revise` (new) | "revise", "punch list", "update proposal" | Update a design proposal based on a punch list or inline decisions. Produces a complete replacement. If an implementation plan exists from a prior iteration, removes it. |
| `/accept` (new) | "accept", "finalize proposal" | Finalize a design proposal: record all decisions, ensure internal consistency, remove open decision points. No surprises. |
| `/plan` (new) | "plan", "implementation plan", "ready to implement" | Generate an implementation plan from a design proposal. Runs accept first if the proposal has not been accepted. |
| `/implement` (new) | "implement", "build", "execute plan" | Execute an implementation plan following authority order with checkpoint verification. |
| `/audit` (new) | "audit", "coherence check", "consistency check" | Run a feature coherence audit across all project artifacts. |

### Copilot Instructions File

A `.github/copilot-instructions.md` file provides always-loaded context:

```markdown
## Proposal Process

This project uses a structured proposal process for all non-trivial changes:

1. **RFP** → 2. **Design Proposal** (iterative) → 3. **Implementation Plan** → 4. **Implementation**

See GLOSSARY.md for definitions of these terms.

## Authority Order

When implementing changes, update project artifacts in this order:

1. GLOSSARY.md (new terms)
2. Roadmap (status → "in progress")
3. Specification docs
4. Architecture docs
5. User guide / AI guide
6. Agent documentation (AGENTS.md, skills)
7. Acceptance tests
8. Code
9. Unit tests
10. Roadmap (status → "completed")
11. History
12. Index files

Always update more authoritative artifacts before less authoritative ones.
If you read an artifact during implementation and it seems to contradict the
implementation plan, the implementation plan takes precedence.

## Implementation Discipline

- The implementation plan is the single source of truth during implementation.
- Complete all TODO items in the implementation plan before reporting success.
- At each checkpoint, verify your work against the implementation plan.
- Do not inject behavior that contradicts the implementation plan, even if it
  seems like an improvement. Propose changes in a follow-up, not during
  implementation.

## Resuming Implementation

If you are asked to continue implementing a feature and an implementation plan
exists in docs/project/plans/, read it and resume from the first uncompleted
TODO item. Do not start over.
```

### Skill: `/propose` (updated)

The existing propose skill is updated to:
- Include a Decision Register section in the output template.
- Populate the decision register as decisions are made during initial creation.
- Reference the new terminology (design proposal, not just proposal).

### Skill: `/revise`

```
1. Read the current design proposal.
2. Identify the punch list:
   a. If the user provides a separate punch list document, read it.
   b. If the user indicates that decisions have been made inline in the
      proposal (via --> notation or annotations), treat the inline
      edits as the punch list.
3. Save the current proposal version to the design history directory (as a new version file).
4. Update the decision register first — add or update all decisions from
   the punch list.
5. Rewrite the body of the proposal to be consistent with the updated
   decision register. Produce a complete replacement; do not reference
   previous versions.
6. Scan the entire document for internal contradictions. Resolve any
   contradictions before saving.
7. Save the replacement at the same path, overwriting the previous version.
8. Append a narrative to the summary file in the design history directory
   explaining what the punch list requested and how the proposal changed.
9. If an implementation plan exists from a prior iteration (in
   docs/project/plans/), delete it — the design has changed and the
   plan is stale.
```

### Skill: `/accept`

```
1. Read the current design proposal.
2. Verify that all decision points have been resolved (every --> has
   a recorded outcome).
3. If any decision points remain unresolved, present them to the user
   and ask for resolution before proceeding.
4. Save the current proposal version to the design history directory (as a new version file).
5. Update the decision register to reflect all decisions as final.
6. Rewrite the body of the proposal:
   a. Remove alternatives that were not chosen.
   b. Remove decision prompts (the --> lines).
   c. Present the chosen design as settled fact.
   d. Ensure internal consistency throughout.
7. Scan for contradictions and resolve them.
8. Set the proposal status to "accepted" in the YAML frontmatter.
9. Save the accepted proposal at the same path.
10. Append a narrative to the summary file in the design history directory
    marking acceptance and summarizing the final state.
```

The accept skill must not produce surprises. It records decisions that have already been made and ensures the document reflects them. It does not make new design choices.

### Skill: `/plan`

```
1. Read the design proposal.
2. If the proposal status is not "accepted", run the /accept skill first.
   (The human may send the proposal back after reviewing the accepted
   version and the plan.)
3. Extract all accepted decisions from the decision register.
4. For each feature, compile the requirements (testable statements of
   behavior).
5. Determine the authority order (use the 12-step default unless the
   proposal specifies otherwise).
6. Generate the ordered TODO list:
   a. For each artifact type in authority order, list the files that
      need to change and what changes are needed.
   b. Include checkpoint items at natural boundaries.
   c. Include feature coherence audit steps at appropriate points.
7. Include a Reference section with any information needed during
   implementation that would otherwise require reading historical
   documents.
8. Save to docs/project/plans/<name>.md.
```

### Skill: `/implement`

```
1. Read the implementation plan.
2. Verify the design proposal is accepted and the plan status is "ready."
3. Set the plan status to "in-progress."
4. Execute TODO items in order:
   a. Before each item, mark it as in-progress in the plan.
   b. After each item, verify the change against the requirements.
   c. Mark the item as completed in the plan.
   d. At checkpoint items, pause and verify all preceding work against
      the implementation plan.
5. After all items are complete, run acceptance tests.
6. Set the plan status to "completed."
7. Report completion.
```

### Skill: `/audit`

Uses a hybrid approach combining a fixed checklist with dependency traversal and search-based discovery.

```
1. Identify the feature to audit (from user input or implementation plan).
2. For each artifact type in the checklist:
   a. Search for files related to the feature (keyword search, depends-on
      traversal).
   b. Read relevant sections.
   c. Extract claims about the feature (behavior, status, syntax, etc.).
3. Cross-check claims across artifact types:
   a. Spec vs. acceptance tests: do tests cover all specified behaviors?
   b. Spec vs. code: does the implementation match the spec?
   c. Roadmap vs. actual: is the status accurate?
   d. User guide vs. spec: does the guide accurately describe the feature?
   e. AI guide vs. spec: same check.
   f. Glossary: are all terms used consistently?
   g. AGENTS.md / skills: is the feature reflected in agent documentation?
4. Distinguish between "not yet implemented" (expected for planned features)
   and "described differently in two places" (an inconsistency).
5. Report inconsistencies and missing coverage.
6. Save the audit report to a file that the human can review and delete.
```

Audit report format:

```markdown
# Feature Coherence Audit: <Feature Name>

*Audited on <date>.*

## Summary
- Artifacts checked: <N>
- Consistent: <N>
- Inconsistent: <N>
- Missing coverage: <N>

## Glossary
**Status:** ✅ Consistent / ⚠️ Inconsistent / ❌ Missing
<details>

## Roadmap
**Status:** ✅ / ⚠️ / ❌
<details>

...
```

## Feature 6: Design Proposal Quality Safeguards

Two safeguards prevent contradictions in design proposals.

### Decision Register

Every design proposal includes a Decision Register section at the top — a consolidated table of all accepted decisions. This is the authoritative record of what has been decided.

- The `/propose` skill creates the decision register when the proposal is first generated.
- The `/revise` skill updates the decision register **first**, then rewrites the body to be consistent with it.
- The `/accept` skill finalizes the decision register and removes all open decision prompts.

### Self-Consistency Check

After generating or revising a design proposal, the agent scans the entire document for internal contradictions. If any analysis text contradicts a decision in the decision register, the analysis is updated or removed. This is a mandatory step in both the `/revise` and `/accept` skills.

## Feature 7: Cross-Session Continuity

The implementation plan file on disk is the durable artifact for tracking progress across session boundaries. The TODO list uses markdown checkboxes (`- [ ]`, `- [x]`) for state tracking.

The `.github/copilot-instructions.md` includes instructions to resume from the first uncompleted TODO item when asked to continue implementing a feature. The `/implement` skill marks items as in-progress and completed as it executes.
