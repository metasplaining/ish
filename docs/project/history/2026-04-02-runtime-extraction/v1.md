## Questions and Answers

### Q: Are there any other types beyond `Value`, `Shim`, and error codes that would be useful to compiled functions in ish packages?

Yes. Tracing through the dependency chain from `Shim` reveals several types that must move together and a few additional candidates:

**Must move (transitive closure of `Shim`):**

| Type | Current location | Why it must move |
|------|-----------------|------------------|
| `Value` | `ish-vm::value` | `Shim` signature: `Fn(&[Value]) -> Result<Value, RuntimeError>` |
| `RuntimeError` | `ish-vm::error` | `Shim` return type |
| `ObjectRef` | `ish-vm::value` | `Value::Object(ObjectRef)` — compiled code creates/reads objects |
| `ListRef` | `ish-vm::value` | `Value::List(ListRef)` — compiled code creates/reads lists |
| `FunctionRef` / `IshFunction` | `ish-vm::value` | `Value::Function(FunctionRef)` — compiled code may receive/return functions |
| `FunctionImplementation` | `ish-vm::value` | Field of `IshFunction` |
| `FutureRef` | `ish-vm::value` | `Value::Future(FutureRef)` — yielding compiled functions return futures |
| `Environment` | `ish-vm::environment` | Field of `IshFunction` (closure environment) |

**Strong candidates:**

| Type | Current location | Rationale |
|------|-----------------|-----------|
| `new_compiled_function()` | `ish-vm::value` | The factory used by builtins, codegen, and stdlib to create compiled functions. Every compiled package needs this. |
| `new_object()`, `new_list()`, `empty_object()` | `ish-vm::value` | Convenience constructors that compiled code will use. |
| Error code constants (E001–E013) | scattered in `ish-vm` | Compiled packages need to throw structured errors with the same codes. Currently string literals with no registry. |

**Not candidates for ish-runtime:**

| Type | Why not |
|------|---------|
| `IshVm` (interpreter) | Consumers of the runtime are compiled packages, not interpreters. The interpreter depends on the runtime, not the other way around. |
| `LedgerState` | Compile-time and interpreter concern, not a package runtime need. |
| AST types (`Statement`, `TypeAnnotation`) | These are in `ish-ast`. `IshFunction` references them, but `ish-ast` is already a separate crate and can remain a dependency of `ish-runtime`. |
| `ControlFlow` enum | Interpreter-internal dispatch mechanism. |

## Feature 1: Move Value (and its dependency closure) to ish-runtime

### Issues to Watch Out For

1. **GC dependency propagation.** `Value` uses `Gc<GcCell<...>>` for objects, lists, and functions. Moving it to `ish-runtime` means `ish-runtime` must depend on the `gc` crate. This is a significant change — `ish-runtime` was designed as a minimal, GC-free FFI boundary type. Adding GC turns it into a substantially larger dependency.

2. **Tokio dependency propagation.** `FutureRef` wraps a `tokio::task::JoinHandle`. Moving `Value` means `ish-runtime` must also depend on `tokio`. This further inflates the crate.

3. **ish-ast dependency propagation.** `IshFunction` references `ish_ast::Statement` and `ish_ast::TypeAnnotation`. Moving `IshFunction` to `ish-runtime` means `ish-runtime` depends on `ish-ast`.

4. **Re-export churn.** Every file in `ish-vm` that currently uses `use crate::value::Value` (and there are many — interpreter, builtins, reflection, environment) must switch to `use ish_runtime::Value`. This is a large but mechanical refactor.

5. **Circular dependency risk.** `Environment` references `Value` and `Value::Function` contains `Environment`. Both must move together. If `Environment` stays in `ish-vm`, we get a circular dependency.

6. **IshValue retirement.** The current `IshValue` in `ish-runtime` was a placeholder for the FFI boundary. Once the full `Value` moves in, `IshValue` becomes redundant or needs a clear separate purpose (e.g., a `repr(C)` FFI projection). The codegen crate currently uses raw `i64` over FFI anyway, so `IshValue` is not actively used in the compilation pipeline.

### Critical Analysis

**Alternative A: Move the full transitive closure (Value, RuntimeError, Environment, IshFunction, FutureRef) to ish-runtime**

Pros:
- Clean separation: `ish-runtime` contains all types needed to write a compiled ish package.
- `ish-vm` becomes purely the interpreter, depending on `ish-runtime` for its value types.
- Compiled packages depend only on `ish-runtime` (and `ish-ast` transitively), never on `ish-vm`.
- Matches the stated goal: "the Rust compiler needs the shim definition" in a standalone crate.
- Enables future package ecosystem where third-party packages don't pull in the interpreter.

Cons:
- `ish-runtime` grows from ~60 lines to several hundred. It is no longer "minimal."
- `ish-runtime` gains `gc`, `tokio`, `crossbeam`, and `ish-ast` as dependencies.
- Large mechanical refactor across `ish-vm` (every `use crate::value` → `use ish_runtime`).
- `Environment` is tightly coupled with the interpreter (e.g., `define`, `get`, `set` for variable lookup) — it's debatable whether it's a "runtime" type or an "interpreter" type.

**Alternative B: Create a new intermediate crate (e.g., `ish-core` or `ish-types`) for the shared types**

Pros:
- Preserves `ish-runtime` as a minimal FFI crate.
- New crate clearly communicates "these are shared types, not the full runtime."
- Could later split into `ish-runtime` (FFI boundary) and `ish-core` (shared types).

Cons:
- Adds another crate to maintain.
- "Runtime" is the standard name for this layer in language implementations (V8, CPython, JVM rt.jar).
- The original RFP explicitly calls for moving types to `ish-runtime`, not a new crate.
- `IshValue` in the current `ish-runtime` would still need reconciliation.

**Alternative C: Move Value and RuntimeError to ish-runtime, but keep Environment and IshFunction in ish-vm**

Pros:
- Smaller change surface: `ish-runtime` grows moderately.
- `Environment` (which is very interpreter-flavored) stays in the interpreter.

Cons:
- `Value::Function(FunctionRef)` contains `IshFunction` which contains `Environment`. You cannot split them without breaking the type. You'd have to make `Value::Function` opaque or use a trait object, adding complexity.
- Shim's signature depends on `Value` and `RuntimeError` but not `IshFunction`, so compiled packages could theoretically avoid constructing `IshFunction` values — but they'd still receive them as arguments.
- Partial extraction is inherently unstable; you'd likely need to complete the move anyway.

**Recommendation:** Alternative A is the cleanest path despite the dependency growth. The name "runtime" accurately describes this layer, and language runtimes are expected to contain GC, value types, and error handling. The growth is a natural evolution, not bloat.

### Proposed Implementation

1. **Add dependencies** to `ish-runtime/Cargo.toml`: `gc`, `tokio` (with required features), `crossbeam`, `ish-ast`.

2. **Move source files** from `ish-vm/src/` to `ish-runtime/src/`:
   - `value.rs` → `ish-runtime/src/value.rs`
   - `error.rs` → `ish-runtime/src/error.rs`
   - `environment.rs` → `ish-runtime/src/environment.rs`

3. **Update `ish-runtime/src/lib.rs`** to declare modules and re-export key types.

4. **Decide fate of `IshValue`**: retire it (remove) or rename it (e.g., `FfiValue`) and keep it alongside `Value` for the raw FFI projection. If retired, update codegen template.

5. **Update `ish-vm`**:
   - Remove moved files.
   - Add `ish-runtime` as a dependency (it currently does not depend on `ish-runtime`).
   - Replace all `use crate::value::*` with `use ish_runtime::*` (or equivalent).
   - Replace all `use crate::error::RuntimeError` with `use ish_runtime::RuntimeError`.
   - Replace all `use crate::environment::Environment` with `use ish_runtime::Environment`.

6. **Update downstream crates** (`ish-codegen`, `ish-stdlib`, `ish-shell`) to import types from `ish-runtime` instead of `ish-vm` where applicable.

7. **Update tests** — unit tests in the moved files travel with them to `ish-runtime`. Integration tests in `ish-vm` continue to work via the re-imports.

### Decisions

**Decision:** Move the full transitive closure (Alternative A), create an intermediate crate (Alternative B), or partial extraction (Alternative C)?
--> We first need to break the dependency that `Value` has on `Environment`, `Statement`, and `IshValue`.  Then we can move the full transitive closure of Value.  The thing to note here is that the difficulties are originating from Value::Function.  How does the VM pass a closure or interpreted function to a compiled function?  How does the compiled function call it?  We need to answer these questions as part of runtime extraction.  The answer is that the `implementation` of `IshFunction` should be redefined to always be a shim.  Compiled functions can both define and call shims.  Whenever a closure or interpreted function is declared, the VM needs to instantiate a shim that carries the environment, body, and VM, and so has everything it needs to execute the function in an interpreted mode when it is called.  We should add a builtin that takes a function as an argument so that we can test nesting interpreted -> compiled -> interpreted. (Actually lets test it a few layers deeper than that, just to flush out any 'it works one layer deep but not recursively' bugs.)

## Feature 2: Move Shim to ish-runtime

### Issues to Watch Out For

1. **Shim depends on Value and RuntimeError.** This is why the RFP says Value must also move. If Decision 1 approves moving Value, Shim moves naturally alongside it.

2. **`new_compiled_function()` must also move.** This factory constructs `IshFunction` values with `Compiled(Shim)` implementation. It's used by `ish-vm` builtins, `ish-codegen`, and `ish-stdlib`. If `IshFunction` moves, this function should too.

3. **Shim captures `Environment` closures.** Some shims (particularly in builtins) capture state from the interpreter. After the move, these closures still compile because `Shim` is just a trait object — but the captured types must be accessible at the construction site. This is not a problem if `Environment` also moves.

### Critical Analysis

There is no real alternative here. If `Value` moves (Feature 1), `Shim` must move with it — they are defined in the same file and `Shim` is parameterized over `Value`. The only decision is whether `Shim` should be a type alias (current) or a trait.

**Current design (type alias):**
```rust
pub type Shim = Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>;
```

Pros:
- Simple, zero boilerplate.
- Easy to construct: any closure works.

Cons:
- No metadata (name, arity) attached to the shim itself.
- Not `Send`/`Sync` (due to `Rc`).

For now, the type alias is sufficient. Evolving to a trait or struct can happen in a future proposal if package catalogs need richer metadata.

### Proposed Implementation

Shim moves to `ish-runtime/src/value.rs` alongside `Value` as part of Feature 1. No separate implementation needed.

### Decisions

**Decision:** Keep `Shim` as a type alias, or promote to a trait/struct?
-->  I would prefer to keep the type alias.  I'm not sure whether the previous decision impacts our ability to do that.

## Feature 3: Move Error Codes to ish-runtime

### Issues to Watch Out For

1. **No error code registry exists today.** Error codes (E001–E013) are currently string literals scattered across `ish-vm/src/interpreter.rs`, `ish-vm/src/builtins.rs`, and `ish-vm/src/environment.rs`. There's no shared constant or enum.

2. **`RuntimeError::system_error()` is the main constructor.** It takes a message string and a code string, and constructs a `Value::Object` with `message` and `code` fields. This ties the error code system to `Value`.

3. **Compiled packages will need to throw structured errors.** Without a shared registry, every compiled package must hardcode the same string literals. This is fragile.

### Critical Analysis

**Alternative A: Move RuntimeError as-is, add an error code enum**

Create an enum (e.g., `ErrorCode`) in `ish-runtime` with variants for each code (E001–E013), with a `as_str()` method returning the string code. Update `RuntimeError::system_error()` to accept `ErrorCode` instead of `&str`.

Pros:
- Compiled packages get a type-safe error code API.
- Prevents typos in error code strings.
- Central registry for documentation and tooling.

Cons:
- Breaking change to all error construction sites in `ish-vm`.
- Enum is closed — adding new codes requires updating `ish-runtime`.

**Alternative B: Move RuntimeError as-is, defer error code registry**

Just move `RuntimeError` with its current string-based API. Compiled packages use the same string literals.

Pros:
- Smallest change.
- Error code registry can be added later.

Cons:
- No compile-time safety for error codes.
- Duplicated string literals across crates.

**Recommendation:** Alternative B for this proposal. The error code registry is valuable but orthogonal to the extraction. It deserves its own proposal where the full error catalog can be designed.

### Proposed Implementation

`RuntimeError` moves to `ish-runtime/src/error.rs` as part of Feature 1. Error code constants remain string literals. A follow-up proposal should address the error code registry.

### Decisions

**Decision:** Add an error code enum now (Alternative A), or defer (Alternative B)?
--> Add an error code enum now

## Feature 4: Retire or Repurpose IshValue

### Issues to Watch Out For

1. **`IshValue` is currently unused in the compilation pipeline.** The codegen crate generates `extern "C" fn(i64) -> i64` functions — it doesn't use `IshValue` at all. The type exists but serves no active purpose beyond a single test.

2. **`IshValue` was designed as a `repr(C)`-safe FFI type.** `Value` (with GC pointers) cannot cross a C FFI boundary safely. If the codegen pipeline later wants to pass rich values over FFI, a projection type like `IshValue` would be needed.

3. **Name collision.** Having both `IshValue` and `Value` in the same crate would be confusing.

### Critical Analysis

**Alternative A: Remove `IshValue` entirely**

Pros:
- Eliminates confusion.
- No code uses it in production.

Cons:
- Loses the FFI-safe value concept. Would need to reinvent later.

**Alternative B: Keep `IshValue`, rename to `FfiValue`**

Pros:
- Preserves the FFI-safe projection concept.
- Clear naming: `Value` for the GC runtime, `FfiValue` for C-ABI boundaries.

Cons:
- Still unused code until codegen evolves.

**Alternative C: Keep `IshValue` as-is, alongside `Value`**

Pros:
- Least change.

Cons:
- Confusing naming in the same crate.

**Recommendation:** Alternative A (remove for now). The FFI projection concept can be reintroduced when codegen actually needs it. YAGNI.

### Proposed Implementation

1. Remove `IshValue` enum and its tests from `ish-runtime/src/lib.rs`.
2. Update `ish-codegen/src/template.rs` if it references `IshValue` (currently it doesn't in the template code).

### Decisions

**Decision:** Remove `IshValue` (Alternative A), rename to `FfiValue` (Alternative B), or keep as-is (Alternative C)?
--> Remove `IshValue`. YAGNI.
