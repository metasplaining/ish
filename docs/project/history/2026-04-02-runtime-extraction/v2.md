## Questions and Answers

### Q: Are there any other types beyond `Value`, `Shim`, and error codes that would be useful to compiled functions in ish packages?

Yes. The revised analysis (after Decision 1) identifies these types:

**Must move (transitive closure of `Value` after shim-only refactor):**

| Type | Current location | Why it must move |
|------|-----------------|------------------|
| `Value` | `ish-vm::value` | `Shim` signature: `Fn(&[Value]) -> Result<Value, RuntimeError>` |
| `RuntimeError` | `ish-vm::error` | `Shim` return type |
| `Shim` | `ish-vm::value` | Compiled packages define and call shims |
| `ObjectRef` | `ish-vm::value` | `Value::Object(ObjectRef)` — compiled code creates/reads objects |
| `ListRef` | `ish-vm::value` | `Value::List(ListRef)` — compiled code creates/reads lists |
| `IshFunction` / `FunctionRef` | `ish-vm::value` | `Value::Function(FunctionRef)` — but now shim-only, no `Environment` or `Statement` dependency |
| `FutureRef` | `ish-vm::value` | `Value::Future(FutureRef)` — yielding compiled functions return futures |

**Also moves:**

| Type | Current location | Rationale |
|------|-----------------|-----------|
| `ErrorCode` enum | new | Type-safe error code registry for compiled packages |
| `new_compiled_function()` | `ish-vm::value` | Factory for creating shim-based functions. The only function constructor after the refactor. |
| `new_object()`, `new_list()`, `empty_object()` | `ish-vm::value` | Convenience constructors for compiled code. |

**No longer needs to move (changed from v1):**

| Type | Why not |
|------|---------|
| `Environment` | The shim-only architecture breaks the `IshFunction` → `Environment` link. `Environment` is interpreter-internal. The VM captures it inside the shim closure instead. |
| `FunctionImplementation` enum | Eliminated entirely. All functions use shims. |
| AST types (`Statement`) | No longer referenced by `IshFunction`. |

**Not candidates for ish-runtime (unchanged from v1):**

| Type | Why not |
|------|---------|
| `IshVm` (interpreter) | Consumers of the runtime are compiled packages, not interpreters. |
| `LedgerState` | Compile-time and interpreter concern. |
| `ControlFlow` enum | Interpreter-internal dispatch mechanism. |

### Q: How does the VM pass a closure or interpreted function to a compiled function? How does the compiled function call it?

The answer is that all functions are shims. When the VM encounters a function declaration (closure or top-level), it wraps the function body in a shim that captures:

1. The `Environment` (closure scope)
2. The `Statement` (function body)
3. A reference to the VM (or its `exec_statement` method)

The resulting shim has the standard `Fn(&[Value]) -> Result<Value, RuntimeError>` signature. Compiled functions receive and call this shim through the normal `Value::Function` dispatch — they don't need to know whether the function was originally interpreted or compiled.

This is the key insight that unblocks the extraction: `IshFunction` no longer stores an `Environment` or `Statement`. It stores a `Shim`. The VM creates the shim at function declaration time, capturing everything needed for execution inside the closure.

## Feature 1: Shim-Only Function Architecture

This feature is a prerequisite for the extraction. It refactors `IshFunction` to remove the `Interpreted` variant, breaking the dependency chain that would otherwise force `Environment`, `Statement`, and `ish-ast` into `ish-runtime`.

### Issues to Watch Out For

1. **VM self-reference in shim closures.** The shim for an interpreted function needs to call back into the VM's `exec_statement` or `call_function`. This requires the shim to capture a reference to the VM. Since `IshVm` uses `&mut self`, this creates a borrowing conflict — you can't create a closure that captures `&mut self` and also be called from within `self`. Solutions include using `Rc<RefCell<IshVm>>` or passing the VM as a parameter to a trampoline.

2. **Closure environment capture timing.** Currently, `new_function()` stores the `Environment` directly in `IshFunction`. With shim wrapping, the environment must be captured inside the shim closure at declaration time. The timing is the same, but the mechanism changes from struct field to closure capture.

3. **Async functions.** Currently `IshFunction.is_async` controls whether the interpreter spawns a task. With shim-only architecture, async interpreted functions need their shim to handle the spawning. The `is_async` field remains on `IshFunction` as metadata for the caller.

4. **Performance.** Adding an `Rc<dyn Fn>` indirection for every interpreted function call adds one virtual dispatch. This is negligible compared to tree-walking interpretation overhead.

5. **`new_function()` removal.** The current `new_function()` creates an `Interpreted` variant. After this change, the VM's function-declaration handling replaces `new_function()` with shim construction. `new_compiled_function()` becomes the sole factory function (possibly renamed to just `new_function()`).

6. **Yielding classification.** `IshFunction.has_yielding_entry` remains — it's metadata about the function, not an implementation detail. Shims don't change how yielding is classified.

### Critical Analysis

The shim-only architecture is not an alternative — it's the decision from the review. The question is how to handle the VM self-reference problem.

**Option 1: `Rc<RefCell<IshVm>>` — VM behind shared interior mutability**

The VM is wrapped in `Rc<RefCell<IshVm>>`. Shims capture `Rc<RefCell<IshVm>>` and call `borrow_mut()` when invoked.

Pros:
- Simple ownership model.
- Shims are self-contained — they carry everything needed.

Cons:
- Runtime borrow checking. A shim calling back into the VM while the VM is already borrowed panics.
- This is a real risk: the VM calls a compiled function (borrowing self), which calls an interpreted-function-shim (tries to borrow self again) → panic.

**Option 2: Trampoline — shim receives VM as a parameter**

Change the shim signature to include an executor callback or pass the VM at call time. The shim signals "I need interpretation" and the VM handles it.

Pros:
- No double-borrow risk.

Cons:
- Changes the `Shim` type signature, which affects all existing shims (builtins).
- Adds complexity to the calling convention.

**Option 3: Split VM state — separate execution engine from mutable state**

Factor the VM into an immutable "engine" (methods) and mutable "state" (environment stack, defer stack, ledger). Shims capture the engine; state is passed as a parameter.

Pros:
- Clean separation of concerns.
- No borrow conflicts.

Cons:
- Significant refactor of the interpreter.
- Over-engineering for the prototype stage.

**Option 4: Shim captures only data, VM wraps at call site**

The interpreted-function shim stores the `Environment` and `Statement` as opaque data (not calling back into the VM directly). When the VM dispatches a function call and the function is a shim, the VM itself handles interpretation — the shim is just a container.

This is essentially what we have today, with the `Interpreted` variant. The difference is that we want compiled code to be able to call the function too. So the shim must be callable without the VM.

**Option 5: Shim captures `exec_fn` callback at declaration time**

At function declaration, the VM creates a closure that captures a callback (not the VM itself) for executing statements. This callback is a `Rc<dyn Fn(Statement, Environment) -> Result<Value, RuntimeError>>` that the VM provides. The VM constructs this callback once and reuses it.

Pros:
- No direct VM reference in shims.
- Callback can be an `Rc` clone — cheap to share.
- Compiled code can call the shim without knowing about the VM.

Cons:
- The callback still needs access to VM state (defer stack, ledger). It must close over them or receive them.
- Essentially still `Rc<RefCell<...>>` under the hood, just narrower.

**Recommendation:** Option 1 (`Rc<RefCell<IshVm>>`) is the pragmatic choice for the prototype. The double-borrow risk is real but manageable: the VM drops its `borrow_mut()` before calling into shims, and the shim re-borrows. This works because tree-walking interpretation is synchronous — there's never a true overlap if the borrow is dropped before the call. The interpreter's `call_function` method needs restructuring to release the borrow before invoking the shim, but this is a contained change.

### Proposed Implementation

1. **Wrap `IshVm` in `Rc<RefCell<IshVm>>`** at the shell/entry level.

2. **Remove `FunctionImplementation` enum.** Replace with a single `Shim` field on `IshFunction`.

3. **Remove `closure_env` field from `IshFunction`.** The environment is captured inside the shim.

4. **Modify the VM's function-declaration handling:**
   - Where the VM currently calls `new_function(name, params, ..., body, env, is_async)`, it instead constructs a shim closure that captures `body`, `env`, and `Rc<RefCell<IshVm>>`.
   - The shim closure, when called, borrows the VM, creates a child scope from the captured `env`, binds parameters, and calls `exec_statement` on the body.

5. **Remove `new_function()`.** Rename `new_compiled_function()` to reflect it's now the only constructor. Or keep both names for ergonomics.

6. **Update `call_function` in the interpreter:** Currently dispatches on `FunctionImplementation::Interpreted` vs `Compiled`. After this change, all functions are shims — just invoke the shim directly. The interpreter still handles arity checking, parameter type auditing, and return type auditing around the shim call.

7. **Verify async function handling.** Ensure async interpreted functions' shims correctly spawn tasks.

### Decisions

**Decision:** How to handle VM self-reference in interpreted-function shims?
--> The current approach of passing a mutable IshVM everywhere is fragile.  We should globally change it to a `Rc<RefCell<IshVm>>`.  Then everywhere that we actually need to mutate the VM, we borrow it mutably, mutate, and then immediately release the borrow.  This approach makes it much easier to verify that we never double borrow.

*Deferred to implementation plan. Option 1 (`Rc<RefCell<IshVm>>`) is the recommended starting point, with the understanding that the borrow must be released before shim invocation.*

## Feature 2: Move Value and Shim to ish-runtime

With the shim-only architecture from Feature 1 complete, `Value` no longer depends on `Environment` or `Statement`. The transitive closure is now clean:

- `Value` → `ObjectRef` (gc), `ListRef` (gc), `FunctionRef` → `IshFunction` → `Shim` → `Value` + `RuntimeError`, `FutureRef` → tokio

No `ish-ast` dependency. No `Environment` dependency.

### Issues to Watch Out For

1. **GC dependency.** `ish-runtime` must add the `gc` crate. `Value`, `ObjectRef`, `ListRef`, and `FunctionRef` all use `Gc<>`.

2. **Tokio dependency.** `FutureRef` wraps `tokio::task::JoinHandle`. `ish-runtime` must add `tokio`.

3. **Re-export churn.** All `use crate::value::*` and `use crate::error::*` in `ish-vm` must change to `use ish_runtime::*`. Mechanical but large.

4. **TypeAnnotation references.** `IshFunction` currently has `param_types: Vec<Option<ish_ast::TypeAnnotation>>` and `return_type: Option<ish_ast::TypeAnnotation>`. These are metadata fields (not needed for execution), but they are part of the struct. This creates an `ish-ast` dependency unless we make these fields generic, use `String` type names, or move the annotation types. The simplest approach: keep `ish-ast` as a dependency of `ish-runtime` for now — `ish-ast` is small and has no heavy dependencies (only `serde`). Alternatively, represent type annotations as strings in `IshFunction` and defer structured types to a future pass.

5. **`Shim` type alias stays as a type alias.** The shim-only refactor doesn't change the `Shim` type — it's still `Rc<dyn Fn(&[Value]) -> Result<Value, RuntimeError>>`. Decision 2 confirms this.

### Critical Analysis

After the shim-only refactor, the extraction is straightforward. The only remaining question is whether `ish-runtime` should depend on `ish-ast` for the `TypeAnnotation` references in `IshFunction`.

**Option A: Allow `ish-runtime` → `ish-ast` dependency**

Pros:
- Simplest: just move the file.
- `ish-ast` is small (only `serde` dependency), so it doesn't meaningfully inflate `ish-runtime`.
- Type annotations on functions are useful metadata for compiled packages (introspection).

Cons:
- Slightly wider dependency surface.
- Couples the runtime to the parser's type representation.

**Option B: Replace TypeAnnotation with String in IshFunction**

Pros:
- No `ish-ast` dependency in `ish-runtime`.
- Simpler struct.

Cons:
- Loses structured type information.
- Would need to serialize `TypeAnnotation` to string and back, losing fidelity.
- Every existing callsite that passes `TypeAnnotation` values must serialize.

**Recommendation:** Option A. The `ish-ast` dependency is lightweight and the type annotations are genuinely useful. The alternative adds complexity for minimal gain.
--> Extract `TypeAnnotation` from `ish-ast` and move it to a new crate `ish-core`.  `ish-ast` depends on `ish-core` and so does `ish-runtime`.

### Proposed Implementation

1. **Add dependencies** to `ish-runtime/Cargo.toml`: `gc` (with derive), `tokio` (workspace features), `ish-ast`.

2. **Move source files:**
   - `ish-vm/src/value.rs` → `ish-runtime/src/value.rs` (minus `FunctionImplementation` enum, minus `new_function()`, minus `Environment` references)
   - `ish-vm/src/error.rs` → `ish-runtime/src/error.rs`

3. **Update `ish-runtime/src/lib.rs`:** declare `value` and `error` modules, re-export key types (`Value`, `Shim`, `RuntimeError`, `IshFunction`, `FunctionRef`, `ObjectRef`, `ListRef`, `FutureRef`, `new_compiled_function`, `new_object`, `new_list`, `empty_object`).

4. **Remove `IshValue`** from `ish-runtime/src/lib.rs`.

5. **Update `ish-vm`:**
   - Add `ish-runtime` as a dependency.
   - Delete `ish-vm/src/value.rs` and `ish-vm/src/error.rs`.
   - Replace all `use crate::value::*` with `use ish_runtime::*`.
   - Replace all `use crate::error::RuntimeError` with `use ish_runtime::RuntimeError`.
   - Re-export from `ish-vm/src/lib.rs` for backward compatibility if desired.

6. **Update downstream crates** (`ish-codegen`, `ish-stdlib`, `ish-shell`): import shared types from `ish-runtime`. They may still depend on `ish-vm` for the interpreter but not for types.

### Decisions

No new decisions required. This feature follows from Decisions 1, 2, 4, 5, and 8.

## Feature 3: Error Code Enum

### Issues to Watch Out For

1. **Existing codes are string literals.** The 13 error codes (E001–E013) are scattered across the interpreter, builtins, and environment modules with no central registry.

2. **`RuntimeError::system_error()` constructs `Value::Object`.** This method creates a `Value::Object` with `message` and `code` properties. After the extraction, both `RuntimeError` and `Value` are in `ish-runtime`, so this method can stay intact.

3. **Extensibility.** The enum is closed. New codes require updating `ish-runtime`. This is acceptable — error codes are a core language concern, not something third-party packages should define ad hoc.

4. **Breaking change.** All error creation sites must change from string literals to `ErrorCode` variants.

### Proposed Implementation

1. **Define `ErrorCode` enum** in `ish-runtime/src/error.rs`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ErrorCode {
    /// E001: Unhandled throw / system error
    UnhandledThrow,
    /// E002: Division by zero / modulo by zero
    DivisionByZero,
    /// E003: Argument count mismatch
    ArgumentCountMismatch,
    /// E004: Type mismatch / undefined operation
    TypeMismatch,
    /// E005: Undefined variable
    UndefinedVariable,
    /// E006: Cannot call non-function
    NotCallable,
    /// E007: Index out of bounds
    IndexOutOfBounds,
    /// E008: I/O error
    IoError,
    /// E009: Null unwrap via `?` operator
    NullUnwrap,
    /// E010: External command / shell error
    ShellError,
    /// E011: Async / concurrency error
    AsyncError,
    /// E012: Cannot await unyielding function
    AwaitUnyielding,
    /// E013: Cannot spawn unyielding function
    SpawnUnyielding,
}

impl ErrorCode {
    pub fn as_str(&self) -> &'static str {
        match self {
            ErrorCode::UnhandledThrow => "E001",
            ErrorCode::DivisionByZero => "E002",
            // ... etc
        }
    }
}

impl fmt::Display for ErrorCode {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.as_str())
    }
}
```

2. **Update `RuntimeError::system_error()`** to accept `ErrorCode` instead of `&str`:

```rust
pub fn system_error(message: impl Into<String>, code: ErrorCode) -> RuntimeError { ... }
```

3. **Update all call sites** in the interpreter, builtins, and environment to use `ErrorCode` variants.

### Decisions

No new decisions required. Decision 7 settled this.

## Feature 5: Cross-Boundary Function Call Testing

The shim-only architecture must be validated with deep nesting tests that exercise the interpreted→compiled→interpreted boundary multiple times.

### Issues to Watch Out For

1. **Borrow conflicts.** The most likely failure mode is a double-borrow panic when a compiled shim calls an interpreted-function shim that borrows the VM while the VM is already borrowed. Testing must specifically exercise this.

2. **Stack depth.** Deep nesting should verify that the Rust call stack doesn't overflow. The shim indirection adds frames.

3. **Closure correctness.** Interpreted functions closed over local variables must correctly resolve those variables when called back from compiled code.

### Proposed Implementation

1. **Add a builtin function** (e.g., `apply` or `call_with`) that takes a function and arguments, then calls the function. This is implemented as a compiled shim, so calling it with an interpreted function produces the interpreted→compiled→interpreted path.

2. **Write acceptance tests** that chain calls through multiple layers:
   ```
   // interpreted function
   let f = fn(x) { x + 1 }
   
   // compiled builtin calls interpreted function
   let result = apply(f, [10])  // → 11
   
   // interpreted function that calls compiled builtin that calls interpreted function
   let g = fn(x) { apply(f, [x]) }
   let result2 = apply(g, [20])  // → 21 (4 boundary crossings)
   
   // deeper nesting
   let h = fn(x) { apply(g, [x]) }
   let result3 = apply(h, [30])  // → 31 (6 boundary crossings)
   ```

3. **Test closures across boundaries:**
   ```
   let base = 100
   let closed = fn(x) { x + base }
   let result = apply(closed, [5])  // → 105
   ```

4. **Test error propagation across boundaries:**
   ```
   let throws = fn() { throw "boom" }
   // compiled builtin calls interpreted function that throws
   // error should propagate back through the compiled layer
   ```
--> Skip this test for now.  Error propagation across the the compiled layer is a behavior of the compiled function, not the interpreter.

### Decisions

No new decisions required. Decision 10 settled this.

## Implementation Order

The features have dependencies:

1. **Feature 4: Remove IshValue** — independent, do first to simplify the crate.
2. **Feature 1: Shim-only architecture** — refactor `IshFunction`, remove `FunctionImplementation::Interpreted`, modify interpreter.
3. **Feature 5: Cross-boundary testing** — validate Feature 1 works correctly.
4. **Feature 3: Error code enum** — can be done in parallel with Feature 2, but doing it first means `RuntimeError` moves in its final form.
5. **Feature 2: Move Value and Shim** — the actual extraction, depends on Features 1, 3, and 4 being complete.
