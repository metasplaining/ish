## Questions and Answers

### Q: What is the conflict with Copilot's builtin plan feature?

VS Code Copilot has a builtin "plan" mode/feature. When a skill is named `plan`, invoking `/plan` in the Copilot chat may trigger the builtin plan feature instead of (or in addition to) loading the project's custom `/plan` skill. Renaming the skill to a unique name avoids this ambiguity and ensures the custom skill is reliably invoked.

### Issues to Watch Out For

1. **Many cross-references.** The `/plan` skill is referenced in multiple active documents: `AGENTS.md`, `docs/ai-guide/orientation.md`, `docs/project/proposals/proposal-process-improvements.md`, `docs/project/plans/proposal-process-improvements.md`, and `audit-proposal-process-improvements.md`. 2. **History files contain `/plan` references.** The history directory `docs/project/history/2026-03-18-proposal-process-improvements/` contains `v1.md`, `v2.md`, `v3.md`, and `summary.md`, all of which reference `/plan`. History files are records of what happened at a point in time â€” modifying them would falsify the historical record.

3. **Skill YAML frontmatter.** The `name:` field in the SKILL.md frontmatter and the `description:` field both reference "plan." The trigger words in the description include "plan" â€” these need careful updating to avoid re-triggering the Copilot builtin while still being discoverable.

4. **Directory rename.** The skill lives at `.github/skills/plan/`. The directory must be renamed, not just the file contents. This is a filesystem operation that must happen atomically with the content updates.

5. **Trigger word "plan" in the description.** The current description includes trigger words: `"plan", "implementation plan", "ready to implement"`. The bare word "plan" as a trigger may still conflict. The trigger words should be updated to avoid the bare "plan" trigger.

### Critical Analysis

The request is straightforward â€” a naming collision with a builtin feature must be resolved. The main design question is what the new name should be.

#### Alternative 1: `/plan_implementation` (as proposed in the RFP)

**Pros:**
- Clear about what the skill does â€” it plans an implementation.
- Distinct from Copilot's builtin `plan` feature.
- Follows the existing naming pattern of descriptive skill names.

**Cons:**
- Longer to type than `/plan`.
- Uses underscore, which is inconsistent with the existing skill names (all single-word: `propose`, `revise`, `accept`, `implement`, `audit`).

#### Alternative 2: `/plan-implementation`

**Pros:**
- Same clarity as Alternative 1.
- Hyphen is more natural in URL-like contexts.

**Cons:**
- Still longer than the original.
- Hyphens in skill names may or may not be supported by Copilot's skill resolution (underscore is safer for identifiers).

#### Alternative 3: `/planimpl`

**Pros:**
- Shorter, faster to type.
- Clearly distinct from `plan`.

**Cons:**
- Abbreviated and less readable.
- Not immediately obvious what it does.

#### Alternative 4: `/implementation_plan`

**Pros:**
- Very clear about what is produced.
- Distinct from builtin `plan`.

**Cons:**
- Word order feels less natural as a command ("create an implementation plan" vs. "plan an implementation").
- Long.
