## Questions and Answers

### Q: How many duration entries can a block have, and are they mutually exclusive?

The RFP lists five entries: `simple`, `complex`, `yielding`, `unyielding`, `blocking`. These are not all in the same dimension. Analysis:

- **Duration dimension:** `simple` vs. `complex` — these are mutually exclusive. A block either completes quickly or it doesn't.
- **Yielding dimension:** `yielding` vs. `unyielding` — these are mutually exclusive. A block either yields the CPU or it doesn't.
- **Blocking dimension:** `blocking` vs. (implicitly) non-blocking — a block either can cause the whole thread to wait or it can't.

A block would carry one entry per dimension. For example, a block could be `simple + unyielding + non-blocking` (a quick computation), or `complex + yielding + non-blocking` (a long async operation). The combination `yielding + blocking` is always a discrepancy per the RFP, which makes sense: if a function yields cooperatively, it should not also block the thread.

So a block can have up to three entries (one per dimension), and entries within the same dimension are mutually exclusive, but entries across dimensions are not.

### Q: What is the correct terminology, consistent with other languages that support async/await?

Standard terminology from JavaScript, Python, Rust, C#, Kotlin, and Swift:

| ish concept (from RFP) | Standard term | Notes |
|------------------------|---------------|-------|
| yielding function | async function | A function that can suspend execution and resume later |
| await | await | Suspend the caller until the async operation completes |
| accept a promise | (no await) / spawn | Start the operation without waiting for its result |
| promise | future / promise | A handle representing an eventual result |
| yield (the act) | suspend / yield | The point at which an async function gives up control |
| blocking | blocking | Universally understood |

Rust uses `Future` and `.await`; JavaScript uses `Promise` and `await`; Python uses `Coroutine`/`Awaitable` and `await`; Kotlin uses `suspend` functions and coroutines. The choice of "promise" vs. "future" is mostly aesthetic. "Promise" is more common in web ecosystems; "future" is more common in systems programming.

### Q: What does "quickly" mean for the `simple` entry?

The RFP suggests "less than a millisecond or fewer than 100,000 CPU cycles." This is a reasonable heuristic but difficult to enforce precisely since execution time varies by hardware. Most languages with similar concepts (like Go's goroutine scheduling heuristics) do not expose this as a user-facing guarantee. The primary value of `simple` vs. `complex` is as a signal to the scheduler and as documentation, not as a hard contract.

## Feature 1: Cooperative Multitasking (Async/Await)

### Issues to Watch Out For

1. **Colored function problem.** Async functions create a "function color" split — sync code cannot easily call async code and vice versa. This is a well-known pain point in JavaScript, Python, and Rust. ish's assurance ledger could mitigate this by making the distinction gradual rather than binary, but the fundamental tension remains.

2. **Entry inference complexity.** Automatically inferring `simple`/`complex`, `yielding`/`unyielding`, and `blocking`/`non-blocking` for every block is a significant analysis burden. The inference must be conservative (if any called function is `complex`, the caller is `complex`), which could make most code `complex` by default.

3. **Guaranteed yield and loop configuration.** The RFP's example of configuring nested loops to yield every N iterations is powerful but complex. The developer must reason about iteration counts across multiple loop levels. Getting this wrong silently degrades cooperative scheduling.

4. **Cancellation.** The RFP does not mention cancellation of in-flight async operations. This is a critical concern — what happens when a promise is dropped or a scope exits while an async operation is pending?

5. **Error propagation across async boundaries.** When an awaited function throws, does the error propagate through the caller's try/catch? When a promise's function throws, how is the error surfaced? This must integrate with ish's existing throw/try/catch/finally model.

6. **Interaction with `defer` and `with` blocks.** If a function yields in the middle of a `with` block, the resource remains held across the suspension. This could cause resource starvation.

7. **Backwards compatibility with interpreted mode.** The prototype's tree-walking interpreter is single-threaded. Cooperative multitasking requires either: (a) a true async runtime (like Tokio), (b) coroutine-style transformation of the AST, or (c) green threads. This is a significant implementation concern.

### Critical Analysis

#### Alternative A: Standard Async/Await (Recommended)

Adopt the standard `async`/`await` model with ish-specific assurance ledger integration. Async functions are declared with `async fn`, called with `await`, and return futures/promises when called without `await`.

**Pros:**
- Familiar to developers from JavaScript, Python, Rust, Kotlin, C#
- Well-understood semantics with extensive literature and tooling experience
- Integrates naturally with ish's entry system — `async` implies `yielding`
- Error propagation is well-defined: `await` propagates errors, futures capture errors

**Cons:**
- Function coloring problem persists (mitigated by ledger, not eliminated)
- Requires an async runtime, adding complexity to the prototype

#### Alternative B: Goroutine-Style Cooperative Concurrency

All functions are potentially concurrent. The runtime handles scheduling transparently, inserting yield points automatically at function calls and loop iterations.

**Pros:**
- No function coloring — any function can be concurrent
- Simpler mental model for developers
- Matches ish's philosophy of deferring complexity to the runtime

**Cons:**
- Harder to reason about when suspension happens
- Implicit yielding conflicts with the assurance ledger's explicitness philosophy
- Difficult to implement in Rust without significant runtime infrastructure (Go's runtime is 50K+ lines)
- The `simple`/`complex` entry distinction becomes less meaningful if everything can yield

#### Alternative C: Structured Concurrency (Kotlin-Style)

Async operations exist only within explicitly-scoped concurrency regions. A `concurrent` block creates a scope; `launch` and `async` start tasks within it; the block does not exit until all tasks complete.

**Pros:**
- Solves cancellation: scope exit cancels all child tasks
- Solves resource leaking: resources tied to the scope's lifetime
- Structured — aligns with ish's emphasis on correctness
- Integrates well with the ledger — the `concurrent` block carries the entries

**Cons:**
- Less flexible than unrestricted async/await
- Still has function coloring (functions that launch tasks vs. those that don't)
- Less familiar to developers (though Kotlin and Java's Project Loom are making it more common)

#### Alternative D: Hybrid — Async/Await with Structured Concurrency Primitives

Use standard `async`/`await` as the core mechanism, but provide structured concurrency primitives (`concurrent` blocks, scoped task groups) as the recommended way to manage task lifetimes. The ledger enforces structured concurrency at higher assurance levels but allows unstructured async at lower levels.

**Pros:**
- Familiar async/await for low-assurance code
- Structured concurrency enforced for high-assurance code
- The assurance continuum works naturally — more structure at higher levels
- Cancellation and resource management are clean at high assurance

**Cons:**
- Two mental models (simple async for beginners, structured for experts)
- More to specify and implement

### Proposed Implementation (for Alternative D — Hybrid)

#### Duration Entries as Ledger Entries

Reframe the RFP's five entries as three orthogonal ledger entry types:

```ish
entry type Duration {
    applies_to: [function, block],
    values: [simple, complex],
}

entry type Yielding {
    applies_to: [function, block],
    values: [yielding, unyielding],
}

entry type Blocking {
    applies_to: [function, block],
    values: [blocking, non_blocking],
}
```

These entries are inferred from function bodies during analysis:
- A function calling any `complex` function is itself `complex`.
- A function calling any `yielding` function (with `await`) is itself `yielding`.
- A function calling any `blocking` function is itself `blocking`.

#### Async Functions and Await

```ish
// Declare an async function
async fn fetch_data(url: String) -> Response {
    let raw = await http.get(url)
    return parse(raw)
}

// Await — caller yields until result is ready
let response = await fetch_data("https://example.com")

// Without await — get a future
let future = fetch_data("https://example.com")
// ... do other work ...
let response = await future
```

#### Structured Concurrency

```ish
// Task group — all tasks must complete before the block exits
concurrent {
    let a = async fetch_data("/api/users")
    let b = async fetch_data("/api/posts")
    // Both fetched in parallel
}
// a and b are available here; both have completed

// Parallel iteration
concurrent for item in large_list {
    process(item)
}
```

## Feature 2: Guaranteed Yield Mechanism

### Issues to Watch Out For

1. **Yield frequency tuning.** The RFP describes manually choosing which loop level to yield at and how often. This requires understanding iteration counts, which may not be known at build time.

2. **Performance overhead.** Each yield point has overhead (save/restore coroutine state). Yielding too frequently degrades throughput. Yielding too infrequently degrades latency for other tasks.

3. **Nested loop analysis.** The RFP's example of six nested loops with 100 items each shows the complexity — the right yield point depends on the product of iteration counts across levels.

4. **Dynamic iteration counts.** When loop bounds are runtime values, the compiler cannot determine the ideal yield frequency. The developer must either annotate or accept automatic heuristics.

### Critical Analysis

#### Alternative A: Explicit Yield Configuration (RFP Approach)

The developer annotates loops with `yield every N` to control yield frequency.

**Pros:**
- Developer has full control
- Predictable behavior
- Integrates with the ledger — the annotation is an entry that can be audited

**Cons:**
- Requires understanding iteration counts and performance characteristics
- Brittle — changing inner loop bodies changes the ideal yield frequency
- Tedious for deeply nested structures

#### Alternative B: Compiler-Inserted Yield Points

The compiler inserts yield points at loop back-edges and function call sites, using heuristics to avoid excessive overhead.

**Pros:**
- No developer burden
- Adapts to code changes automatically
- Similar to Go's preemptive scheduling approach

**Cons:**
- Less predictable — the developer cannot reason about when yields happen
- Heuristics may be wrong for unusual workloads
- Conflicts with the assurance ledger's explicitness philosophy at high assurance

#### Alternative C: Time-Based Yielding (Recommended)

The runtime tracks CPU time consumed since the last yield. When a threshold is exceeded, the next yield-eligible point (loop iteration boundary, function call) causes a yield. The threshold is configurable.

**Pros:**
- Adapts to hardware speed automatically ("quickly" is measured, not guessed)
- No need to count iterations
- Developers can tune the threshold without understanding loop structure
- Compatible with both explicit annotation and automatic heuristics

**Cons:**
- Requires runtime time tracking (overhead, though small)
- Less deterministic than explicit `yield every N`
- Harder to test — behavior depends on execution speed

#### Alternative D: Hybrid — Time-Based with Override

Time-based yielding by default, with `yield every N` available as an explicit override for performance-critical inner loops.

**Pros:**
- Sensible default for most code
- Manual control available when needed
- Aligns with the assurance continuum — override is a higher-assurance annotation

**Cons:**
- Two mechanisms to understand
- Override might mask timing issues

### Proposed Implementation (for Alternative D — Hybrid)

The runtime maintains a yield budget (configurable time quantum, e.g., 1ms by default). At each yield-eligible point (loop back-edge, function call), the runtime checks whether the budget is exhausted. If so, it yields.

Developers can override with explicit annotation:

```ish
// Time-based (default) — runtime handles it
for item in items {
    process(item)
}

// Explicit override
for item in items yield every 500 {
    process(item)
}

// Suppress yielding (requires unyielding entry, may cause discrepancy)
@[unyielding]
for item in items {
    process(item)
}
```

### Decisions

**Decision 5:** Yield mechanism — explicit only (A), compiler-inserted (B), time-based (C), or hybrid (D)?
-->  Hybrid.  Low assurance ish has time based yielding at documented yield points.  High assurance ish allows fine grained control of both the yield points and the yield criteria (both time based and statement count based.)

## Feature 3: Parallel Multitasking

### Issues to Watch Out For

1. **Rust's `Send`/`Sync` requirements.** Any data sent to another thread must implement `Send`; any data shared between threads must implement `Sync`. This is a fundamental constraint from the Rust compilation target. ish must either expose these constraints to the developer or ensure them automatically.

2. **Garbage collector thread safety.** The prototype uses a GC-managed heap. A tracing garbage collector shared across threads requires synchronization (stop-the-world pauses or concurrent GC). This is a major implementation concern.

3. **Data races.** Parallel code can introduce data races if mutable data is shared without synchronization. ish's assurance ledger can detect some of these statically (e.g., two threads accessing the same mutable variable), but runtime races are harder.

4. **Deadlocks.** Any synchronization mechanism (mutexes, channels) can deadlock. Static deadlock detection is undecidable in general.

5. **Interaction with cooperative multitasking.** Can an async function run on a separate thread? Can a thread-spawned function use `await`? The interaction between these two models must be well-defined.

6. **Complexity budget.** The RFP emphasizes simplicity and correctness over completeness. This is wise — concurrent programming is a rich area where more features do not always help.

### Critical Analysis

#### Alternative A: Thread Spawn with Message Passing

Expose `spawn` to run a function on a new OS thread. Communication between threads uses channels (typed, bounded message queues). No shared mutable state.

**Pros:**
- Simple mental model — threads communicate by sending messages
- No data races by construction (no shared mutable state)
- Channels are well-understood (Go, Rust, Erlang)
- Easy to map to Rust's `std::sync::mpsc` or crossbeam channels
- Satisfies `Send` automatically — values are moved into messages

**Cons:**
- Message passing has overhead (serialization, allocation)
- Some patterns (shared counters, caches) are awkward with message passing alone
- Cannot share immutable data without copying

#### Alternative B: Shared-State Concurrency with Locks

Allow threads to share data through synchronized access (mutexes, read-write locks). The assurance ledger tracks which variables are shared and ensures proper locking.

**Pros:**
- Supports all concurrency patterns
- More efficient than message passing for read-heavy workloads
- Familiar to systems programmers

**Cons:**
- Deadlocks, priority inversion, lock contention
- Harder to reason about correctness
- Requires tracking shared-state entries in the ledger
- `Sync` requirement is harder to enforce automatically

#### Alternative C: Parallel Iterators Only

No general thread spawning. Instead, provide parallel versions of collection operations (`parallel_map`, `parallel_filter`, `parallel_for_each`). This is the approach that Rayon takes in Rust.

**Pros:**
- Extremely simple — no thread management, no synchronization
- Correct by construction (no shared mutable state within the iteration body)
- Covers the most common use case identified in the RFP (parallelizing list processing)
- Maps directly to Rayon in Rust

**Cons:**
- Does not cover queue-based producer/consumer patterns
- Does not cover background tasks or server-style concurrency
- Too limited for a general-purpose language

#### Alternative D: Structured Parallelism with Scoped Threads (Recommended)

Parallel tasks exist only within `parallel` blocks. Within a `parallel` block, the developer can launch tasks and iterate in parallel. The block does not exit until all tasks complete. Scoped threads allow safe borrowing of parent-scope data (Rust's `std::thread::scope`).

Communication between parallel tasks uses channels. Shared immutable data is accessed directly (no copying needed). Shared mutable data requires explicit `@[shared]` annotation and uses interior mutability patterns.

**Pros:**
- Covers the common use cases: parallel iteration, producer/consumer, fork/join
- Structured — tasks cannot outlive their scope, preventing dangling references
- Scoped threads satisfy Rust's borrowing rules
- Immutable sharing is free (no copies)
- Mutable sharing is explicit and auditable via the ledger
- The assurance continuum works: `streamlined` allows `parallel for`; `rigorous` also checks for shared-state safety

**Cons:**
- Scoped threads cannot outlive their parent scope (no background tasks that survive function return)
- More complex than parallel iterators alone
- Interior mutability patterns add cognitive overhead

#### Alternative E: Actor Model

Each concurrent entity is an actor with its own state and mailbox. Actors communicate exclusively through asynchronous messages.

**Pros:**
- Strong isolation — no shared state
- Scales well (Erlang, Akka)
- Fault tolerance through supervision trees

**Cons:**
- Significant runtime infrastructure required
- Unfamiliar to many developers
- Overhead for fine-grained concurrency
- Poor fit for parallel iteration patterns
- Hard to map to Rust's concurrency primitives directly

### Proposed Implementation (for Alternative D — Structured Parallelism)

#### Parallel Blocks and Iteration

```ish
// Parallel iteration — most common case
parallel for item in large_list {
    process(item)
}

// Fork-join — run tasks in parallel, collect results
parallel {
    let a = task fetch_data("/api/users")
    let b = task fetch_data("/api/posts")
}
// a and b are available here

// Producer-consumer with channels
let (sender, receiver) = channel<Message>()
parallel {
    task produce(sender)
    task consume(receiver)
}
```

#### Data Sharing Rules

| Data kind | Rule |
|-----------|------|
| Immutable local | Shared freely across parallel tasks (borrowed) |
| Mutable local | Cannot be accessed from parallel tasks without `@[shared]` |
| `@[shared]` mutable | Accessed through atomic operations or locks; the ledger audits correct usage |
| Channel | Moved into the receiving task; sender/receiver are not shared |

#### Assurance Features

| Feature | `streamlined` | `cautious` | `rigorous` |
|---------|--------------|------------|------------|
| `parallel_annotation` | optional | optional | required |
| `shared_state_audit` | runtime | runtime | build |
| `channel_type_required` | optional | required | required |

#### Ledger Entry Types

```ish

    applies_to: [variable, property],
    implies: [@[mutable]],
    conflicts: [],
}

    applies_to: [variable, property, function],
    // Functions with ThreadSafe can be called from parallel tasks

```

### Decisions

**Decision 6:** Parallel multitasking model — thread spawn with message passing (A), shared-state with locks (B), parallel iterators only (C), structured parallelism (D), or actor model (E)?
--> ish concurrency and parallelism is implemented on top of Tokio.  Concurrency is implemented using `LocalSet` and `spawn_local`, so memory sharing between tasks is not an issue.  Parallelism is implemented using normal Tokio tasks and `spawn`.  Parallel functions have a `parallel` entry.  But there is no way to define parallel functions in ish.  The standard library comes with some parallel functions.  All I/O in the standard library is implemented using Tokio's async I/O capabilities.  Additional parallel functions can be created by writing a library in rust that contains a Tokio task and a shim.  Making sure the task is 'Send' becomes a problem for the implementer or the library, writing in rust.  The is runtime includes support for parallel shims.  A parallel shim accepts parameters like a regular shim, and returns a `Future`.  Tokio supports calling normal tasks, and awaiting normal futures from within a `LocalSet` task, so there is no problem with yielding is functions calling parallel ish functions.  Calling yielding functions from parallel functions could get tricky, but the problem is bypassed, because there is no way to create a parallel function. Blocking functions go away.  There is no way to define one in ish, and if one is needed in the standard library, it is defined as parallel, not blocking. Examine and elaborate on this model.  Check it carefully to make sure it is workable, and report any concerns.

## Feature 4: Blocking Detection and Audit

### Issues to Watch Out For

1. **Defining "blocking."** What operations are blocking? File I/O, network I/O, mutex acquisition, sleep, subprocess wait — these are clear. But what about CPU-intensive computation without yields?

2. **Third-party code.** If ish calls external functions (FFI, shell commands), it cannot know whether they block. Conservative assumption: all external calls are blocking.

3. **False positives.** A function that acquires a mutex and releases it quickly is technically blocking but practically instant. Overly aggressive blocking detection will annoy developers.

### Critical Analysis

#### Alternative A: Annotation-Based (Recommended)

Functions are annotated as `blocking` or `non_blocking`. The ledger infers blocking status from call graphs. FFI and shell commands are assumed blocking unless annotated otherwise.

**Pros:**
- Fits the existing entry/ledger model
- Composable — blocking propagates through call chains
- Clear and auditable

**Cons:**
- Requires annotation of standard library and built-in functions
- FFI boundary requires manual annotation

#### Alternative B: Runtime Detection

The runtime monitors thread state and flags operations that block for too long.

**Pros:**
- No annotation burden
- Catches actual blocking, not theoretical

**Cons:**
- Reactive, not preventive
- Cannot catch blocking at build time
- Overhead of monitoring

### Decisions

**Decision 9:** Blocking detection model — annotation-based (A), runtime detection (B), or both?
--> No blocking.  See decision 6.
