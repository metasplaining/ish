## Questions and Answers

### Q: How many duration entries can a block have, and are they mutually exclusive?

**Decided.** There are three orthogonal dimensions (Decision 1). A block carries one entry per dimension:

- **Duration:** `simple` (completes quickly) vs. `complex` (may take a long time)
- **Yielding:** `yielding` (suspends execution cooperatively) vs. `unyielding` (never suspends)
- **Blocking:** eliminated from the ish language (Decision 9); only relevant at the Rust shim boundary

Since blocking is eliminated, ish code effectively has two dimensions: Duration and Yielding. A block can be `simple + unyielding` (quick computation), `complex + unyielding` (long computation without yields — a discrepancy under guaranteed yield), `simple + yielding` (quick but suspends, e.g., a fast async I/O), or `complex + yielding` (long async operation with yield points).

Entries within the same dimension are mutually exclusive. Entries across dimensions are independent.

### Q: What is the correct terminology, consistent with other languages that support async/await?

**Decided.** ish adopts standard keywords (Decision 2):

| Concept | ish keyword/type | Notes |
|---------|-----------------|-------|
| Async function declaration | `async fn` | Declares a function that can yield |
| Suspend until complete | `await` | Suspends caller until the awaited future resolves |
| Start without waiting | `spawn` | Starts an async operation, returns a `Future` |
| Future handle | `Future<T>` | A value representing an eventual result of type `T` |
| Cooperative suspension | `yield` | The point at which a function gives up control |

**Open question:** Should the ledger entry names match the keywords (`yielding`, `async`) or use distinct vocabulary? For example, the `Yielding` entry type could be named `Async` instead, but this might cause confusion between "this function is declared async" and "this block contains yield points." Recommendation: keep `Yielding` and `Duration` as entry type names since they describe properties of code, while `async`/`await` are developer-facing syntax. This parallels how `Mutable` is an entry type while `mut` is the keyword.

### Q: What does "quickly" mean for the `simple` entry?

The RFP suggests "less than a millisecond or fewer than 100,000 CPU cycles." This serves primarily as a signal to the runtime's yield-point insertion logic and as documentation. In the Tokio model (Decision 6), `simple` tells the runtime that no yield points are needed within this block. `complex` tells the runtime that yield points should be inserted (either automatically via time-based budgets, or explicitly via developer annotation).

## Feature 1: Cooperative Multitasking

### Design (Decisions 1–4, 6)

ish uses standard `async`/`await` semantics, implemented on top of a Tokio runtime with `LocalSet` for cooperative scheduling.

#### Runtime Architecture

All ish user code runs inside a Tokio `LocalSet` on the main thread. Tasks are spawned with `spawn_local`, which does not require the `Send` trait. This means:

- GC-managed heap objects can be shared freely between concurrent tasks.
- No synchronization overhead for memory access between ish tasks.
- The single-threaded cooperative model avoids data races by construction.
- The existing tree-walking interpreter can be adapted to yield at `.await` points without fundamental architectural changes.

#### Low-Assurance Behavior (Decision 3)

At low assurance (`streamlined` standard), concurrency is transparent:

- All code is treated as `simple + unyielding` by default.
- When an ish function calls an async standard library function (e.g., file I/O), the call implicitly awaits — the developer does not need to write `await`.
- The runtime automatically upgrades the calling function to `yielding` at execution time.
- The `async`, `await`, `spawn`, and `yield` keywords are available but not required.

This means a developer at low assurance can write:

```ish
let data = file.read("input.txt")
let result = process(data)
file.write("output.txt", result)
```

The `file.read` and `file.write` calls are async under the hood (Tokio I/O), but the developer sees synchronous-looking code. The runtime awaits each call automatically.

#### High-Assurance Behavior

At higher assurance levels (`cautious`, `rigorous`), the developer must be explicit:

```ish
// Explicit async function declaration
async fn fetch_and_process(path: String) -> Result<String> {
    let data = await file.read(path)
    let result = process(data)
    return result
}

// Caller must explicitly await or spawn
let result = await fetch_and_process("input.txt")

// Or spawn to get a future
let future = spawn fetch_and_process("input.txt")
// ... do other work ...
let result = await future
```

#### Assurance Features

| Feature | `streamlined` | `cautious` | `rigorous` |
|---------|--------------|------------|------------|
| `async_annotation` | optional (auto-inferred) | required | required |
| `await_required` | optional (default: await) | required | required |
| `guaranteed_yield` | disabled | enabled | enabled |

#### Cancellation

When a `Future` value is dropped (goes out of scope without being awaited), the underlying `spawn_local` task must be cancelled. Tokio supports this via `JoinHandle::abort()`. Cancellation semantics:

- A cancelled task stops at its next `.await` point.
- `defer` blocks in the cancelled task still execute (cleanup is guaranteed).
- `with` block resources in the cancelled task are still closed.
- Awaiting a cancelled future returns a cancellation error, catchable via `try`/`catch`.

**Open question:** Should dropping a `Future` without awaiting it be a discrepancy at higher assurance levels? This would catch resource leaks from forgotten futures.
--> Yes.  There should be an assurance feature for throwing an error when a future is dropped.

#### Error Propagation

Errors propagate naturally through `await`:

- If an awaited function throws, the error propagates through the caller's `try`/`catch` chain, identical to synchronous error propagation.
- If a spawned function throws, the error is captured in the `Future`. When the future is awaited, the error is re-thrown at the await site.
- If a `Future` is dropped without being awaited and the task threw an error, the error is logged (not silently swallowed).

#### Concurrent For Loop (Open Question from Decision 4)

The `concurrent` block was rejected because variables declared inside the block would be out of scope after the block exits, making it difficult to return values without surprising semantics.

The `concurrent for` loop remains an open question. It would spawn each iteration as a separate `spawn_local` task and wait for all to complete:

```ish
concurrent for item in items {
    process(item)
}
```

The practical question is whether this offers advantages over a library function:

```ish
concurrent_for_each(items, (item) => {
    process(item)
})
```

**Analysis:**

| Aspect | Syntax (`concurrent for`) | Library function |
|--------|--------------------------|-----------------|
| **Scope access** | Natural — loop body sees enclosing scope | Closure captures are equivalent since all code runs on one `LocalSet` thread; no `Send` issues |
| **Error propagation** | `throw` propagates naturally from loop body | Must propagate errors from the callback; doable but slightly more verbose |
| **Yield point insertion** | Compiler knows the loop body and can insert yield points | Compiler can still analyze the callback body |
| **Type inference** | Inferred from the iterable | Parameter types may need explicit annotation |
| **Implementation** | Requires parser and AST changes | No language changes needed |

Given the Tokio `LocalSet` model, the differences are modest. The library function approach avoids language changes and can be provided immediately as a standard library function. Syntax sugar could be added later if the pattern proves common enough.

**Decision:** `concurrent for` — provide as a library function or add syntax support?
--> Library function.  Error propagation is an important consideration.  A concurrent for loop needs to have predictable error propagation semantics.  In practice, that means it needs to run all of the concurrent tasks to completion, and return a list of errors.  This would be a suprising behavior for a `concurrent for` but not for a standard library function.

### Duration and Yielding Entries as Ledger Entry Types

The two remaining dimensions (blocking is eliminated) are modeled as ledger entry types:

```ish
entry type Duration {
    applies_to: [function, block],
    values: [simple, complex],
}

entry type Yielding {
    applies_to: [function, block],
    values: [yielding, unyielding],
}
```

Inference rules:
- A function calling any `complex` function is itself `complex`.
- A function containing an `await`, `spawn`, or `yield` expression is `yielding`.
--> Calling `spawn` does not make an expression `yielding`.
- A function calling any `yielding` function (with `await`) is itself `yielding`.
- A function that spawns a `yielding` function (with `spawn`) is not necessarily `yielding` — the spawn returns immediately.

## Feature 2: Guaranteed Yield Mechanism (Decision 5)

### Design

The yield mechanism is a hybrid: time-based yielding by default, with explicit overrides available at higher assurance levels.

#### Time-Based Yielding (Low Assurance)

The Tokio runtime budget system provides a natural foundation. Tokio already tracks "coop budget" — a counter that decrements on each poll, forcing a yield when exhausted. ish extends this concept to the language level:

- The runtime maintains a yield budget (configurable time quantum, default ~1ms).
- At each **yield-eligible point** — loop back-edges and function call sites — the runtime checks whether the budget is exhausted.
- If exhausted, the runtime inserts a `tokio::task::yield_now().await`, returning control to the Tokio scheduler.
- The budget is reset after each yield.

The **documented yield points** are:
1. Every loop back-edge (`for`, `while`)
2. Every function call (before the call)
3. Every explicit `yield` statement

At low assurance, only the time-based mechanism is active. The developer does not need to think about yielding.

#### Explicit Yield Control (High Assurance)

At higher assurance levels, the developer can control yield behavior precisely:

```ish
// Time-based — runtime handles it (default at all levels)
for item in items {
    process(item)
}

// Statement-count-based — yield every N iterations
for item in items yield every 500 {
    process(item)
}

// Time-based with custom threshold
@[yield_budget(500us)]
for item in items {
    process(item)
}

// Suppress yielding (marks the block as unyielding)
@[unyielding]
for item in items {
    process(item)
}
```

#### Guaranteed Yield as a Discrepancy

When the `guaranteed_yield` feature is enabled (`cautious` and `rigorous` standards), a `complex + unyielding` block is a discrepancy. The developer must either:
- Accept automatic time-based yielding (the block becomes `complex + yielding`)
- Add explicit yield annotations
- Mark the block `@[unyielding]` with justification (an escape hatch that may require review)

#### Assurance Feature

| Feature | `streamlined` | `cautious` | `rigorous` |
|---------|--------------|------------|------------|
| `guaranteed_yield` | disabled (time-based active but no discrepancies) | enabled (complex + unyielding = discrepancy) | enabled |
| `yield_control` | time-based only | time-based + statement-count | time-based + statement-count |

## Feature 3: Parallelism via Tokio (Decision 6)

### Design

Parallelism in ish is not a language-level feature — it is a runtime capability exposed through the standard library and the shim system. ish developers consume parallel functions; they do not write them.

#### Architecture

```
┌─────────────────────────────────────────────────┐
│ Tokio Runtime (multi-threaded)                   │
│                                                  │
│  ┌──────────────────────────────────┐            │
│  │ LocalSet (main thread)           │            │
│  │                                  │            │
│  │  ┌────────┐ ┌────────┐          │            │
│  │  │ Task 1 │ │ Task 2 │  ...     │ ← ish code │
│  │  │(spawn_ │ │(spawn_ │          │   (!Send)  │
│  │  │ local) │ │ local) │          │            │
│  │  └────────┘ └────────┘          │            │
│  └──────────────────────────────────┘            │
│                                                  │
│  ┌──────────┐ ┌──────────┐                       │
│  │ Parallel │ │ Parallel │  ...     ← Rust libs  │
│  │ Task A   │ │ Task B   │            (Send)     │
│  │ (spawn)  │ │ (spawn)  │                       │
│  └──────────┘ └──────────┘                       │
└─────────────────────────────────────────────────┘
```

- **ish tasks** (cooperative): Run on the `LocalSet`. Share the GC heap. Scheduled cooperatively via `spawn_local`. No `Send` requirement.
- **Parallel tasks** (true parallelism): Run on Tokio's thread pool via `spawn`. Implemented in Rust. Require `Send`. Cannot access the GC heap directly.

#### Parallel Shims

The existing shim system is extended with a parallel variant. A **parallel shim** has the same interface as a regular shim (accepts symbol name, value object, parent shim) but returns a `Future` instead of a value:

```
Regular shim:  (symbol, value, parent_shim) -> Value
Parallel shim: (symbol, value, parent_shim) -> Future<Value>
```

When ish code calls a parallel function:
1. The runtime marshals ish values into the shim's parameter format.
2. The parallel shim spawns a Tokio task (via `tokio::spawn`) and returns a `Future`.
3. The ish code awaits (or spawns) the future from within the `LocalSet`.
4. When the parallel task completes, its result is marshaled back into an ish value.

#### Standard Library Parallel Functions

The standard library provides parallel functions for common use cases:

| Function | Purpose |
|----------|---------|
| `parallel_map(list, fn)` | Apply a function to each element in parallel, return results |
| `parallel_filter(list, fn)` | Filter a list in parallel |
| `parallel_for_each(list, fn)` | Execute a function for each element in parallel (no return) |
| `parallel_reduce(list, fn, init)` | Reduce a list in parallel |

All I/O in the standard library is async via Tokio:

| Domain | Implementation |
|--------|---------------|
| File I/O | `tokio::fs` |
| Network I/O | `tokio::net` |
| Process execution | `tokio::process` |
| Timers/sleep | `tokio::time` |

#### From ish Code

```ish
// Parallel map — process a large list across multiple CPU cores
let results = await parallel_map(large_list, (item) => {
    expensive_computation(item)
})

// Async file I/O — transparently async via Tokio
let data = file.read("input.txt")    // implicitly awaited at low assurance
let data = await file.read("input.txt")  // explicit at high assurance
```

#### Why No User-Defined Parallel Functions

Allowing ish developers to write parallel functions would require:
1. Ensuring all captured values are `Send` — this leaks Rust's ownership model into ish.
2. Thread-safe GC — the GC would need synchronization for cross-thread access.
3. Data race prevention — shared mutable state across threads.

By restricting parallelism to Rust-implemented shims, these problems become the Rust library author's responsibility. The ish developer sees only async functions with `Future` return types — the parallelism is invisible.

#### Ledger Entry

Functions from parallel shims carry a `parallel` entry in the ledger:

```ish
entry type Parallel {
    applies_to: [function],
    implies: [@[yielding]],
    // A parallel function is always yielding from the caller's perspective
}
```

The `parallel` entry is informational — it tells the developer and the analyzer that the function executes on a separate thread. It implies `yielding` because calling a parallel function always involves an async boundary (the future).

### Concerns and Open Questions

The following concerns were identified during analysis of the Tokio-based model:

#### 1. Callbacks from Parallel Tasks into ish Code

If a Rust-implemented parallel function needs to invoke an ish callback (e.g., `parallel_map` needs to call the user-provided function for each element), the callback must run on the `LocalSet` thread because it accesses `!Send` GC data.

**Approaches:**
- **Marshal via channel.** The parallel task sends callback requests to the `LocalSet` thread via a channel and awaits responses. This serializes callback execution, limiting parallelism.
- **Copy parameters.** The parallel task receives copied (marshaled) values, processes them in pure Rust, and returns results. The ish callback is never invoked from the parallel thread — instead, the function passed to `parallel_map` must be a Rust function, not an ish closure.
- **Hybrid.** Some parallel functions accept Rust functions only (true parallelism). Others accept ish closures but execute them on the `LocalSet` with work distributed cooperatively (not true parallelism, just concurrent iteration).

This is a fundamental tension: if the per-element function is ish code, it must run on the `LocalSet` thread and cannot be truly parallel. The standard library `parallel_map` etc. would need to either:
- Accept only built-in / Rust-implemented transformation functions (limiting but truly parallel), or
- Accept ish closures but distribute them cooperatively across `LocalSet` tasks, not truly parallel (concurrent but single-threaded).

**Decision:** How do parallel collection functions interact with ish closures? True parallelism (Rust-only callbacks) vs. cooperative distribution (ish closures on `LocalSet`)?
--> They don't.  Strike all of the parallel standard library functions that take closures from the list.  Any parallel function that accepts closures just pulls all of the parallelism complexity back into the language.

#### 2. Parameter Marshaling Overhead

Values passed to parallel shims must be copied/serialized across the thread boundary. For large data structures (a list of 10 million elements), this copy could be expensive. Possible mitigations:
- Use `Arc` for immutable data sharing (the Rust shim wraps the data in `Arc` before spawning).
- Parallel iterator functions could use shared immutable references via scoped threads.
- Accept the overhead and document it as a tradeoff.

**Decision:** Accept marshaling overhead, or require the shim system to support zero-copy sharing of immutable data?
--> This probably needs to be addressed on a per-function basis by the standard library implementation.  The ish VM and compiler should support a mechanism to transfer read only byte buffers between threads without copying.  At the least, this means making sure the library implementer can wrap a byte buffer in a `Value` without copying the buffer.  We may need to discriminate between mutable and immutable types in the `Value` enumeration in order to appease the rust compiler.  Most I/O functions will accept and return either byte buffers or streams wrapped on top of byte buffers, and so can be handled this way.  If a parallel function does accept a complex ish type, the function will most likely be implemented with `copy in`/`copy out` semantics in the shim.  But that is a problem for the function implementer.  ish will need to support a stream type, that supports sending bytes between threads efficiently.  We need to consider streams carefully.  They will be how most data is transferred between the main thread and I/O threads.  Presumably there are some Tokio-provided stream implementations that we can use.  But we will need to adapt them to ish.  In addition, we should support streams of complex objects.  Propose some alternative stream implementations, with pros and cons.

#### 3. Shell Command Execution

The current shell executes external commands via `std::process::Command`. In the Tokio model, this must migrate to `tokio::process::Command` to avoid blocking the `LocalSet` thread. This is a significant change to the shell implementation.

At low assurance, shell commands should still look synchronous:
```ish
let output = $(ls -la)  // implicitly awaited
```

This requires the shell command execution path to become async internally.

#### 4. Interpreter Architecture Impact

The tree-walking interpreter must become async-aware. Each evaluation step that might yield (function calls, loop iterations) must be an `.await` point. This likely means the interpreter's `eval` function becomes `async fn eval(...)`.

This is a significant refactor of the prototype. However, since the prototype is a design vehicle (not a production compiler), the change validates that the Tokio model is implementable.

#### 5. Entry Terminology (Decision 2 — Open Question)

Should the `Yielding` entry type be renamed to `Async`? Comparison:

| Entry name | Pros | Cons |
|------------|------|------|
| `Yielding` | Describes the mechanism (cooperative yield); distinct from keyword | Unfamiliar; "yielding function" is not standard terminology |
| `Async` | Matches the keyword; familiar | Conflates declaration (`async fn`) with property (contains yield points); a non-`async` function that calls an awaited function is also yielding |

Recommendation: Keep `Yielding` as the entry type name. A function can be `yielding` without being declared `async` (at low assurance, the runtime infers it). The entry describes the runtime behavior, not the syntactic declaration.

**Decision:** Entry terminology — keep `Yielding`/`Duration` or rename to `Async`/`Complexity`?
--> `Yielding`/`Complexity`

## Feature 4: Blocking (Eliminated — Decision 9)

The `blocking` dimension from the original RFP is eliminated from the ish language. The rationale (from Decision 6):

- There is no way to define a blocking function in ish. All ish functions run on the `LocalSet` and use Tokio async I/O.
- Standard library functions that perform I/O use Tokio's async implementations (`tokio::fs`, `tokio::net`, `tokio::process`).
- If a Rust library needs synchronous I/O, it wraps it in a parallel shim (which runs on the Tokio thread pool). From ish's perspective, this is a parallel function, not a blocking one.
- The `Blocking` entry type is not needed. The `Parallel` entry type covers the relevant case.

The only residual concern is FFI: if ish eventually supports calling arbitrary C/Rust functions, those functions might block. At that point, a `blocking` entry could be reintroduced, or all FFI calls could be required to go through parallel shims.
