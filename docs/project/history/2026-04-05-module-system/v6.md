# Proposal A: Module Core — v6 Snapshot

*Snapshot of [module-system-core.md](../../proposals/module-system-core.md) taken before revision on 2026-04-05. This version contained two inline decisions awaiting incorporation.*

## Inline Decisions Pending Incorporation

**D16 — cross-tree project directive:**
> Rename and redefine the `project` directive. A file that is part of a project gets its project file by iterating parent directories until it finds a project.json file. This is the only way to be part of a project. The `project` directive exists to allow scripts that are not part of a project to handle the logistical concerns that a project.json addresses. So, the solution is to modify the directive so that it excludes access to project internals, and to rename it so that it does not imply project membership. Maybe `bootstrap` would be a good name. Propose some alternatives.

**New open questions — script/module distinction and deployment models:**
> Let's carefully consider the distinction between files that contain only declarations and script files that are executable, and in conjunction, the distinction between files that are part of a project and files that are standalone. There is interaction. Packages contain only declarations. Because the file path variant of `use` is only allowed within projects, the only way to access declarations outside the file is to either belong to their project or depend on their project. This almost kicks script files out of projects. Scripts included in projects are not available to consumers of those projects through `use`. But we need to consider the deployment models of projects. I see three: 1. Libraries, 2. Scripts/executables, 3. Bundles of scripts/executables. We have not defined a way for an ish project to export one or more executables. Also, some scripts are meant for public consumption, and some are meant for internal consumption (build scripts, etc.). Also, because of the interpreted/compiled nature of ish, any interpreted script could be compiled into a functionally equivalent executable. I think we have two decisions to make: 1. How do we distinguish executable scripts from declarative modules? 2. How do ish projects expose scripts/compiled executables as public?
>
> Alternative for question 1: by convention, but not strictly required, ish scripts have a shebang and lack a .ish extension, while modules lack a shebang and contain a .ish extension.
>
> Alternative for question 2: projects have a scripts directory and a bin directory, where the scripts directory contains scripts meant to be exported by the project, and the bin directory contains compiled versions of those same scripts, with the same names.
