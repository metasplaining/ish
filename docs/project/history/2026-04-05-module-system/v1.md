## Questions and Answers

### Q: What order should modules, projects, packages, lexical scopes, and the use directive be implemented?

The natural dependency order is:

1. **Lexical scopes** — foundation for everything else. Modules need scope boundaries, the use directive needs a target scope, and packages are collections of scoped modules.
2. **Modules** — file-to-module mapping, module paths, basic visibility rules. Depends on scopes being well-defined.
3. **Use directive** — importing names from one module into another. Depends on modules existing.
4. **Projects** — project manifest, metadata, local dependency declarations. Depends on modules and use being functional so there's something to organize.
5. **Packages** — versioning, distribution, remote dependencies. The most complex layer; depends on all the above.

This order mirrors how OCaml, Rust, and Go evolved — all started with in-process modules before adding package management.

### Q: What other problems should we be aware of?

Research identified several additional problems not listed in the RFP:

12. **Cyclic dependencies** — Modules that mutually depend on each other. OCaml forbids cross-file cycles entirely. Rust allows cycles within a crate but not across crates. Go forbids package-level cycles. Haskell supports mutual recursion within a module but requires `.hs-boot` files across modules. This needs explicit design.

13. **Build reproducibility** — Ensuring that builds produce the same result everywhere. Go uses `go.sum` with a checksum database. Rust uses `Cargo.lock`. Haskell had "Cabal hell" for years before adopting Nix-style local builds.

14. **Module identity vs. module path** — When the same source can be reached via different paths (symlinks, re-exports, vendoring), what is the module's identity? Go addresses this with canonical module paths. Rust addresses it with crate names.

15. **Conditional compilation / platform-specific code** — Go uses build tags and filename conventions. Rust uses `cfg` attributes and features. This intersects with ish's execution configurations.

16. **Incremental compilation** — Module boundaries are natural compilation units. The module design should support incremental compilation from the start.

17. **Re-exports and facade modules** — The ability to create a module that re-exports contents from other modules (Rust's `pub use`, Haskell's reexported-modules, Scheme's import/export). Important for API stability when refactoring internals.

18. **Security and trust** — Who can publish packages? How are they authenticated? Go uses a transparency log (sum.golang.org). This intersects with ish's assurance ledger.

## Feature: Physical Downloading and Caching (Problem 1)

### Issues to Watch Out For

- Network availability: builds must work offline after initial fetch.
- Cache invalidation: when does a cached module become stale?
- Disk space management: caches grow without bound.
- Security: downloaded code must be verified against tampering.
- Proxy/mirror support for restricted environments.

### Critical Analysis

**Alternative A: Centralized registry (Rust/Haskell model)**
A single registry (like crates.io or Hackage) is the primary source of packages.
- *Pros:* Simple discovery, single source of truth for names, easy search and documentation hosting.
- *Cons:* Single point of failure, governance burden, namespace squatting, requires standing up infrastructure.

**Alternative B: Decentralized VCS-based (Go model)**
Module paths are URLs. The tool resolves them to VCS repositories or module proxies.
- *Pros:* No central authority needed, works with any hosting provider, leverages existing VCS infrastructure, optional caching proxies.
- *Cons:* No single search/discovery mechanism, module paths are long, hosting provider lock-in in module paths, need a mechanism for vanity paths.

**Alternative C: Hybrid with proxy protocol (Go's GOPROXY)**
Decentralized by default with a proxy protocol that allows caching, mirroring, and corporate control.
- *Pros:* Best of both worlds. Works directly with VCS, but proxies add reliability, caching, censorship/policy control. Go's proxy protocol is well-proven.
- *Cons:* More complex specification.

**Recommendation:** Alternative C (Go's GOPROXY model). This aligns with the existing spec's three-phase distribution strategy (git → OCI/ORAS → dedicated registry). The proxy protocol should be simple enough to implement as a static file server.

### Proposed Implementation

- Define a module proxy protocol (HTTP-based, similar to Go's `$base/$module/@v/$version.{info,mod,zip}`).
- Use a content-addressable cache directory (~/.ish/cache/) for downloaded modules.
- Store cryptographic hashes of all downloaded modules for verification.
- Support a `ISH_PROXY` environment variable (like GOPROXY) with fallback chains.
- Support `ISH_PRIVATE` for modules that should bypass proxy/checksum servers.
- Support vendoring (copying dependencies into a `vendor/` directory) for hermetic builds.

### Decisions

**Decision 11:** Should ish use a centralized registry, decentralized VCS-based resolution, or a proxy protocol?
--> a proxy protocol

## Feature: Versioning (Problem 2)

### Issues to Watch Out For

- Pre-1.0 versions need different compatibility expectations.
- Version resolution must be deterministic across machines.
- Yanking/retracting broken versions needs support.
- Pre-release and build metadata affect ordering.

### Critical Analysis

**Alternative A: Semantic Versioning with lockfile (Rust/Haskell model)**
Declare version ranges; a solver picks compatible versions; results are locked in a lockfile.
- *Pros:* Familiar, allows automatic minor/patch upgrades, lockfile ensures reproducibility.
- *Cons:* SAT solving is NP-hard (though practical), lockfile merge conflicts, solver behavior can be surprising.

**Alternative B: Minimal Version Selection (Go model)**
Always use the *minimum* version that satisfies all requirements. No lockfile needed because the algorithm is deterministic from go.mod files alone.
- *Pros:* Deterministic without lockfile, simple algorithm, predictable behavior, no solver surprises, builds get newer dependencies only when explicitly requested.
- *Cons:* Must explicitly upgrade to get security patches, major versions require path changes (Go's /v2 suffix), less automatic than range-based approaches.

**Alternative C: Curated snapshot sets (Haskell Stackage model)**
Provide curated sets of package versions known to work together.
- *Pros:* Eliminates version conflicts entirely within a snapshot, great for beginners.
- *Cons:* Requires active curation, packages outside the snapshot are harder to use, heavy infrastructure requirement.

**Recommendation:** Alternative B (MVS). Go's experience shows MVS is simpler, more predictable, and avoids the class of problems that plagued Haskell ("Cabal hell") and can complicate Cargo resolution. The lockfile-free determinism is particularly valuable. Semver should still be the version format, but selection should use MVS.

### Proposed Implementation

- Require semantic versioning (major.minor.patch) for all packages.
- Use Minimal Version Selection: always select the minimum version satisfying all constraints.
- Major version changes create distinct module paths (like Go's /v2 suffix or a similar mechanism).
- Support `retract` directives for yanking broken versions.
- No lockfile needed; the project manifest is the source of truth.

### Decisions

**Decision 2:** Should ish use MVS (no lockfile, deterministic from manifests), SAT solving with lockfile (Cargo-style), or curated snapshots (Stackage-style)?
--> Should ish use MVS (no lockfile, deterministic from manifests)

## Feature: Diamond Dependencies / Multiple Versions (Problem 3)

### Issues to Watch Out For

- Type identity: if two parts of the code use different versions of a type, are they compatible?
- Binary size: multiple versions of the same package bloat the binary.
- Semantic confusion: which "version" of a function is being called?
- This is the single hardest problem in module system design.

### Critical Analysis

**Alternative A: Forbid multiple versions (Go model for same major)**
Within a major version, only one version exists in the dependency graph (the minimum that satisfies all constraints). Different major versions are treated as different packages entirely.
- *Pros:* No type confusion, simpler mental model, smaller binaries.
- *Cons:* Can prevent valid dependency graphs if two dependencies need incompatible versions of a transitive dependency.

**Alternative B: Allow multiple versions (Cargo model)**
The resolver can select multiple versions of a package if semver ranges are incompatible. Each dependant gets the version it asked for.
- *Pros:* More permissive, dependencies rarely block each other.
- *Cons:* Type confusion (a `Foo` from v1.2 is not the same as `Foo` from v1.3 if both are in the graph), larger binaries, surprising runtime behavior.

**Alternative C: Major version suffixes (Go model)**
Different major versions have different import paths (e.g., `example.com/foo/v2`). Within a major version, MVS ensures exactly one version.
- *Pros:* Clean separation of incompatible versions, MVS eliminates conflicts within a major version, explicit in code which major version is used.
- *Cons:* API authors must manage path changes at major bumps, import paths get longer.

**Recommendation:** Alternative C (Go's major version suffix model). Combined with MVS, this eliminates the diamond dependency problem for compatible versions while allowing coexistence of truly incompatible (different major) versions. This is the cleanest solution observed across all five languages.

### Proposed Implementation

- Packages at major version 2+ must include the major version in the module path.
- Within a major version, MVS selects exactly one version.
- Different major versions are treated as independent packages and can coexist.

### Decisions

**Decision 3:** Should ish forbid multiple versions of the same major, allow them freely (Cargo), or use major version path suffixes (Go)?
--> major version path suffixes

## Feature: Encapsulation (Problem 4)

### Issues to Watch Out For

- Too restrictive: makes code hard to test or extend.
- Too permissive: makes code hard to reason about.
- Need to support ish's assurance levels — higher assurance may require stricter encapsulation.
- Testing often needs access to internals (friend/test visibility).

### Critical Analysis

The five languages take very different approaches:

**Alternative A: File-based interfaces (OCaml model)**
Each module (file) has an explicit interface file (.mli) that lists public declarations. Anything not in the interface is private. Supports abstract types (type name visible, representation hidden) and read-only types (pattern matching allowed, construction restricted).
- *Pros:* Very explicit, interface is separate from implementation, supports abstract types naturally, easy to audit what's public.
- *Cons:* Maintaining parallel .mli files is boilerplate, changes require updating two files.

**Alternative B: Annotation-based visibility (Rust model)**
Items are private by default. `pub` makes them public. `pub(crate)` restricts to the crate. `pub(super)` restricts to the parent module.
- *Pros:* Visibility is co-located with code, fine-grained control, private by default is safe.
- *Cons:* No abstract types (though Rust achieves this via newtype pattern), visibility is scattered through code rather than centralized.

**Alternative C: Package-level visibility (Go model)**
All names starting with uppercase are public; lowercase names are package-private. No finer granularity. The `internal` directory convention restricts access to within a module subtree.
- *Pros:* Extremely simple, no annotations needed, the convention is easy to teach.
- *Cons:* Only two levels (public/private), no module-level or subpackage-level privacy, uppercase convention is unusual.

**Alternative D: Explicit export lists (Scheme/Haskell model)**
Modules declare exactly what they export. Haskell's `module M (foo, bar) where ...` and Scheme's `(export foo bar)`.
- *Pros:* Exports are centralized and easy to find, everything not exported is private, refactoring internals doesn't affect the interface.
- *Cons:* Export lists must be maintained manually, easy to forget to export new public items.

**Alternative E: Hybrid (existing ish spec)**
The current ish spec defines Rust-like annotation-based visibility with five levels: `pub(self)`, `pub(super)`, `pub(in path)`, `pub(project)`, `pub(global)`.
- *Pros:* Fine granularity, familiar to Rust users, distinguishes project from global visibility.
- *Cons:* Five levels may be more than needed; the distinction between `pub(project)` and `pub(global)` may not be meaningful until packages exist.

**Recommendation:** Keep the existing ish spec's visibility model (Alternative E) but simplify the initial implementation to three levels: private (default), `pub` (project-visible), and `pub(global)` (visible to dependents). Add `pub(super)` and `pub(in path)` later if needed. The Scheme/Haskell export-list style could be added as an optional interface file mechanism (like OCaml's .mli) for high-assurance modules.

### Proposed Implementation

- Private by default (like Rust). Items without `pub` are visible only within their module.
- `pub` makes items visible within the project.
- `pub(global)` makes items visible to external dependents.
- Consider supporting optional interface files for high-assurance modules (leveraging ish's assurance continuum).

### Decisions

**Decision 4:** Should ish use annotation-based visibility (Rust-style), export lists (Scheme/Haskell-style), interface files (OCaml-style), or the existing five-level spec?
--> 3 layer annotation based visibility + interface file that is generated and overwritten at low assurance but must be manually maintained at high assurance.  Three layers are pub, priv, pkg.  pkg is the default.

## Feature: Forward References (Problem 5)

### Issues to Watch Out For

- Unbounded forward references make compilation ordering impossible.
- Cyclic type definitions across modules are common and need support.
- Cyclic value/function definitions across modules are rare but arise in callback patterns.

### Critical Analysis

**Alternative A: No cross-module cycles (OCaml/Go model)**
OCaml forbids circular dependencies between compilation units entirely. Go forbids package-level import cycles. Mutual recursion is only within a single file/package.
- *Pros:* Simple compilation model, clear dependency DAG, easier to understand code.
- *Cons:* Sometimes requires restructuring code to break cycles, may need a "common types" module pattern.

**Alternative B: Explicit mutual recursion groups (Haskell `.hs-boot` model)**
Haskell allows cross-module recursion via `.hs-boot` files that declare the types/signatures needed to break the cycle. The compiler uses these to compile modules in a cycle.
- *Pros:* Supports genuine cyclic dependencies when needed.
- *Cons:* Complex, error-prone, `.hs-boot` files are universally considered a pain point.

**Alternative C: `rec` module groups (OCaml's `-rectypes` / proposed recursive modules)**
Designate a group of modules that should be compiled together as a recursive unit.
- *Pros:* Explicit about which modules form a cycle, compiler knows the full cycle.
- *Cons:* Not widely used, complex implementation, group must be compiled as a unit.

**Alternative D: Within-module recursion only, with trait/interface-based decoupling**
Forbid cross-module cycles. Provide interfaces/traits to decouple modules that would otherwise need cycles.
- *Pros:* Clean architecture, encourages good design patterns, simple compilation.
- *Cons:* Requires upfront design effort to identify interfaces.

**Recommendation:** Alternative A (no cross-module cycles), augmented with Alternative D (interfaces for decoupling). This is the approach Go and OCaml take, and it works well in practice. Within a single module file, forward references and mutual recursion should be fully supported. Cross-module cycles should be a compile error with a clear diagnostic suggesting how to restructure.

### Proposed Implementation

- Within a module (file), all definitions are visible to each other (mutual recursion supported).
- Cross-module dependencies must form a DAG. Cycles are a compile error.
- Provide clear error messages showing the cycle and suggesting extraction of shared types into a common module.

### Decisions

**Decision 5:** Should ish forbid cross-module cycles entirely (Go/OCaml), allow them with boot files (Haskell), or support explicit recursive module groups?
--> The compilation unit is a source file. Cyclical dependencies between source files are disallowed.  Cyclical dependencies within source files are allowed.  When declaring from the shell, there is a syntax that allows declaration of a whole module atomically, allowing cyclical dependencies within the module.

## Feature: Project Metadata and Dependencies (Problem 6)

### Issues to Watch Out For

- The manifest format sets the tone for the entire developer experience.
- Too much required metadata discourages casual use.
- Too little makes packages hard to discover and manage.
- ish's multiple execution configurations (thin shell, fat shell, compiler, executable) need representation.

### Critical Analysis

**Alternative A: Minimal declarative file (Go `go.mod` model)**
A small, focused file with just the module path, language version, and dependencies.
- *Pros:* Minimal friction, easy to read and edit manually, machine-readable, tooling can manage it.
- *Cons:* Separate metadata (author, license, description) needs another location.

```
module example.com/myproject
ish 0.1.0
require (
    example.com/lib v1.2.3
    example.com/util v0.4.0
)
```

**Alternative B: Rich declarative file (Cargo.toml / Cabal model)**
A comprehensive manifest with metadata, dependencies, build configuration, and feature flags.
- *Pros:* Single source of truth, supports publishing, build configuration is centralized.
- *Cons:* Verbose for simple projects, TOML/YAML learning curve.

**Alternative C: Code-as-configuration (some Scheme implementations)**
Dependencies and metadata are expressed in the language itself.
- *Pros:* No separate format to learn, can use language features for conditional logic.
- *Cons:* Hard to parse without running the code, security risks from arbitrary code in manifests, tooling can't statically analyze dependencies.

**Alternative D: ish manifest in ish syntax**
Write the manifest as an ish file, but restricted to a safe, statically-analyzable subset.
- *Pros:* Developer uses one syntax everywhere, natural for ish's shell-like nature, no new format to learn.
- *Cons:* Must define and enforce the "safe subset," parsing complexity.

**Recommendation:** Alternative A (minimal declarative file), similar to Go's go.mod. The manifest should be called `project.ish` or `ish.mod` — a simple, custom format that's easy for both humans and tools to read. Start minimal (module path, ish version, dependencies) and add metadata fields as needed for package publishing.

### Proposed Implementation

- Create a manifest file format (e.g., `ish.mod` or `project.ish`).
- Required fields: module path, ish version requirement.
- Optional fields: require (dependencies), replace (local overrides), exclude, retract.
- Metadata fields for publishing: description, license, authors, repository.
- ish's execution configurations should be declarable in the manifest or in the module itself.

### Decisions

**Decision 6:** Should the project manifest be a minimal declarative file (Go-style), a rich TOML file (Cargo-style), ish code, or a restricted ish subset?
--> Minimal declarative file.  JSON formatted.  Supports inheritance -- Can specify ordered list of parent projects, whose manifests are merged to provide defaults for optional parameters. Tooling for materializing file with commented source for tracing.

## Feature: Developer Workspace Flexibility (Problem 7)

### Issues to Watch Out For

- Replace/override directives must not leak into published packages.
- Workspace mode must be compatible with CI/CD (where workspaces may not exist).
- Need to handle the case where a developer's local fork diverges from upstream.

### Critical Analysis

**Alternative A: Replace directive in manifest (Go model)**
The `replace` directive in `go.mod` redirects a dependency to a local path or alternative module.
- *Pros:* Simple, explicit, in the manifest, can replace all versions or specific versions.
- *Cons:* Must be removed before publishing (Go enforces this), tempting to leave replace directives permanently.

**Alternative B: Workspace file (Go `go.work` / Cargo workspace)**
A separate workspace file lists local modules that should be used together. Not committed to VCS (Go) or committed as a workspace root (Cargo).
- *Pros:* Separates workspace concerns from package concerns, multiple developers can have different workspace layouts.
- *Cons:* Another file to manage, workspace-specific behavior can hide bugs.

**Alternative C: Path + version dependencies (Cargo model)**
Dependencies can specify both a path (for local development) and a version (for publishing). Cargo uses the path when found, falls back to the registry version.
- *Pros:* Smooth transition between local and published dependencies, one declaration serves both purposes.
- *Cons:* Path can silently take precedence when you don't expect it.

**Alternative D: Combined approach**
Support `replace` in the manifest for ad-hoc overrides, plus a workspace file for multi-module development. Replace directives in non-root modules are ignored (only the main module's replace directives take effect).
- *Pros:* Handles both use cases cleanly.
- *Cons:* Two mechanisms to learn.

**Recommendation:** Alternative D (combined approach). Go's experience shows both are needed: `replace` for quick ad-hoc overrides, workspaces for sustained multi-module development. The workspace file (`ish.work`) should not be committed to VCS. Replace directives should only take effect in the main module (not in dependencies).

### Proposed Implementation

- Support `replace` directive in the manifest: `replace example.com/foo => ../local-foo`.
- Support a workspace file (`ish.work`) listing local modules to develop together.
- Replace directives in dependency manifests are ignored.
- Workspace file is gitignored by default.
- Support ISH_PRIVATE for configuring which modules bypass proxies.

### Decisions

**Decision 7:** Should ish use replace directives, workspace files, path+version dependencies, or a combination?
--> Workspace files.  We may revisit later to add replace directives as well.

## Feature: Shell/Programming Mixed Nature (Problem 8)

### Issues to Watch Out For

- Shell scripts are often standalone — importing a full module system for a one-liner is too heavy.
- Shell scripts may want to use libraries without a project manifest.
- The module system must not make the shell experience worse.
- ish's four execution configurations (thin shell, fat shell, compiler, executable) have different needs.

### Critical Analysis

**Alternative A: Implicit module for scripts**
Single-file scripts are implicitly a module with no exports. They can import dependencies but don't need a manifest. Dependencies are specified inline (similar to Go's tool directive or Rust's cargo-script RFC).
- *Pros:* Zero friction for scripts, gradual adoption of the module system.
- *Cons:* No version pinning for scripts, reproducibility concerns.

**Alternative B: Script header with dependencies**
Scripts can declare dependencies in a header comment or directive:
```
#!/usr/bin/env ish
//require example.com/lib v1.2.3
```
- *Pros:* Self-contained scripts with pinned dependencies, familiar pattern (similar to Python's PEP 723).
- *Cons:* Comments-as-code is fragile, parsing complexity.

**Alternative C: Two modes — script mode and project mode**
In script mode (no manifest found), the file is executed directly with ad-hoc dependency resolution. In project mode (manifest present), full module resolution applies.
- *Pros:* Clean separation, scripts stay simple, projects get full power.
- *Cons:* Behavior changes based on presence of a file, which can surprise users.

**Recommendation:** Alternative C (two modes) with elements of Alternative A (inline requires for scripts). When no manifest is found, ish operates in script mode where files are standalone modules. A `use` directive at the top of a script can still import from the standard library or from globally-installed packages. For project mode, the full module system applies.

### Proposed Implementation

- **Script mode:** No manifest required. The file is a standalone module. `use` can import from stdlib and globally-installed packages. No versioned external dependencies without a manifest.
- **Project mode:** Manifest present. Full module resolution, versioning, and dependency management.
- The shell REPL operates in script mode by default, but can be pointed at a project.
- Thin shell execution uses script mode. Fat shell, compiler, and executable use project mode.

### Decisions

**Decision 8:** Should ish have distinct script/project modes, or require a manifest for all code?
--> A project directive loads a project file.  A file without a project directive loads a default project file.  The search path for the default starts with the directory hierarchy, supporting directory tree based projects. The ish install package comes with a default project file at the last point of the search path, which includes only packages distributed with the ish install package.

## Feature: Simplify Dependency Management (Problem 9)

### Issues to Watch Out For

- Dependency management should require minimal manual intervention.
- Tooling should handle the common operations automatically.
- Error messages for version conflicts must be clear and actionable.

### Critical Analysis

This is more an emergent property of the other decisions than a standalone feature. The key simplification strategies from the researched languages:

- **Go's MVS** eliminates the need for a lockfile and makes dependency resolution deterministic and understandable.
- **Go's `go mod tidy`** automatically adds missing and removes unused dependencies.
- **Cargo's `cargo add`** provides a CLI for adding dependencies rather than manual file editing.
- **Go's module proxy** makes downloads fast and reliable without requiring developer configuration.
- **Cargo's workspace inheritance** (`[workspace.dependencies]`) lets workspaces declare dependency versions once.

### Proposed Implementation

- Provide an `ish mod tidy` command that syncs the manifest with actual imports.
- Provide `ish mod add/remove` commands for CLI-based dependency management.
- MVS eliminates lockfile maintenance.
- The proxy protocol ensures reliable, fast downloads.
- Clear error messages for resolution failures, showing the conflict chain.

### Decisions

No standalone decisions — this feature is a consequence of the other choices.

## Feature: Inherited Dependency Management (Problem 10)

### Issues to Watch Out For

- Workspace-level dependency declarations must compose cleanly with module-level ones.
- Overly magical inheritance makes debugging dependency issues harder.
- Organizations want to standardize dependency versions across many projects.

### Critical Analysis

**Alternative A: Workspace dependency table (Cargo model)**
A workspace root declares shared dependency versions. Member projects reference these with `workspace = true` rather than specifying versions.
- *Pros:* Single source of truth for versions across a workspace, reduces duplication, easy to update.
- *Cons:* Only works within a workspace, not across independent projects.

**Alternative B: Bill of Materials / Platform (Maven BOM model)**
A published package declares a set of recommended dependency versions. Projects import the BOM to inherit versions.
- *Pros:* Works across independent projects, organizations can publish a BOM.
- *Cons:* More complex, another concept to learn, requires publishing the BOM itself.

**Alternative C: Go module graph (Go model)**
No explicit inheritance. Each module declares its own dependencies. MVS resolves the graph deterministically. Go workspaces provide workspace-level coherence for local development.
- *Pros:* Simple, no magic, each module is self-contained.
- *Cons:* Duplication across modules in the same workspace, no organizational standardization.

**Recommendation:** Start with Alternative C (Go model — no inheritance, each module is self-contained). Add Alternative A (workspace dependency table) when workspace support is added. Consider Alternative B (BOMs) as a future post-prototype feature.

### Proposed Implementation

- Initially, each project declares its own dependencies.
- When workspace support is added, allow a workspace-level dependency table.
- Projects in a workspace can reference workspace dependencies instead of specifying versions directly.

### Decisions

**Decision 9:** Should ish support workspace dependency inheritance from the start, or add it later?
--> This was covered by my previous decisions.  Projects can specify parent projects (which is distinct from specifying dependencies.)  Organizations can define projects with no content, specifically to act as BOMs.  Project file also configure assurance levels.  This allows organizations to define cross project assurance levels.  Re-using the package mechanism ensures the same stability/repeatability as a regular package dependency.

## Feature: Traceability (Problem 11)

### Issues to Watch Out For

- Traceability must work across the assurance continuum.
- Build provenance should be embeddable in compiled artifacts.
- Supply chain attacks are a growing concern.

### Critical Analysis

**Alternative A: Checksum verification (Go model)**
A checksum file (`go.sum`) records expected hashes of all dependencies. A global checksum database provides transparency.
- *Pros:* Tamper detection, supply chain security, transparency log prevents equivocation.
- *Cons:* Privacy concerns (all module fetches visible to the checksum database).

**Alternative B: Embedded build info (Go `go version -m` model)**
Compiled binaries embed the module versions used to build them.
- *Pros:* Audit any binary to see its dependencies, essential for compliance.
- *Cons:* Increases binary size slightly.

**Alternative C: Dependency graph commands**
Tools to query and visualize the dependency graph (`go mod graph`, `go mod why`, `cargo tree`).
- *Pros:* Developers can understand and audit dependencies.
- *Cons:* Only available at development time, not in deployed artifacts.

**Recommendation:** All three. Checksum verification (A) is essential for security. Embedded build info (B) ties into ish's assurance ledger — high-assurance code should carry full provenance. Dependency graph tools (C) are essential for developer experience.

### Proposed Implementation

- Maintain a checksum file alongside the manifest.
- Embed dependency versions and hashes into compiled artifacts.
- Provide `ish mod graph`, `ish mod why`, and `ish mod verify` commands.
- Integrate with the assurance ledger: dependencies at a given assurance level must have dependencies at the same or higher level.

### Decisions

No standalone decisions — all three approaches are recommended.

## Feature: Import Syntax / Use Directive (Problem 4 + cross-cutting)

### Issues to Watch Out For

- Import syntax is used thousands of times per project — it must be ergonomic.
- Ambiguous imports are a source of bugs.
- Import filtering (importing only specific names) reduces namespace pollution.

### Critical Analysis

**Alternative A: Path-based imports (Go model)**
```go
import "example.com/foo/bar"
```
The full module path is the import specifier. Access via the last path component: `bar.Func()`.
- *Pros:* Unambiguous, self-documenting (you see where code comes from), no central name registry.
- *Cons:* Verbose, import paths can be long.

**Alternative B: Name-based imports with filtering (Scheme R6RS model)**
```scheme
(import (only (rnrs lists) filter map)
        (prefix (rnrs io simple) io:))
```
Import specific names, with renaming, prefixing, and filtering.
- *Pros:* Maximum control over what enters the namespace, avoids conflicts, very precise.
- *Cons:* Verbose import sections, must look up available exports.

**Alternative C: Hierarchical imports with aliases (Rust/Python model)**
```rust
use std::collections::HashMap;
use crate::module::Type as MyType;
```
Tree-structured imports with aliasing via `as`.
- *Pros:* Familiar to many developers, tree imports (`use foo::{bar, baz}`) are concise, aliasing resolves conflicts.
- *Cons:* Ambiguity if module names collide without aliases.

**Alternative D: Simple imports with qualified access (OCaml model)**
```ocaml
open Printf     (* brings all names into scope *)
Printf.printf   (* qualified access without open *)
```
Modules are always accessible by qualified name. `open` optionally brings names into scope.
- *Pros:* No import needed for qualified access, `open` is optional convenience.
- *Cons:* `open` brings everything in (no selective imports in basic form).

**Recommendation:** A hybrid of Alternatives A and C. The `use` directive should support:
- `use example.com/foo/bar` — import the module, access as `bar.name`
- `use example.com/foo/bar as b` — import with alias
- `use example.com/foo/bar { Type, func }` — selective import
- `use example.com/foo/bar { Type as T }` — selective import with rename

This combines Go's path-based clarity with Rust's selective imports and aliasing.

### Proposed Implementation

- Module paths are URL-like for external packages, relative for within-project.
- `use` brings a module into scope; qualified access is the default.
- Selective imports (`use mod { a, b }`) bring specific names into the local scope.
- `as` provides aliasing at both the module and item level.
- Glob imports (`use mod { * }`) are available but discouraged (and possibly restricted at higher assurance levels).

### Decisions

**Decision 10:** Should ish use Go-style path imports, Rust-style hierarchical imports, Scheme-style filtered imports, or a hybrid?
--> Hierarchical imports with aliases (Rust/Python model)

## Feature: Module-to-File Mapping (Problem 4 + cross-cutting)

### Issues to Watch Out For

- Strict 1:1 mapping (Go/OCaml) is simple but can lead to many small files.
- Submodule support (Rust) adds flexibility but complexity.
- The mapping must be predictable — given a module path, a developer should know which file to open.

### Critical Analysis

**Alternative A: Strict file = module (Go/OCaml model)**
Each file is a module. The file path determines the module path.
- *Pros:* Simple, predictable, no ambiguity about where a module lives.
- *Cons:* Can't split a large module across files, can't have submodules within a file.

**Alternative B: File = module with inline submodules (Rust model)**
Each file is a module. Modules can contain submodules, declared inline or in subdirectories.
- *Pros:* Flexible, hierarchical modules mirror directory structure, inline submodules for small helper modules.
- *Cons:* Multiple ways to define the same structure (mod.rs vs directory), can be confusing.

**Alternative C: Package = directory, module = file (Go's package model)**
A directory is a package containing files that contribute to the same namespace. Each file in a directory is part of the same module.
- *Pros:* Splits large modules across files naturally, directory structure is the organizing principle.
- *Cons:* No file-level privacy within a package, all files in a directory must agree on the package name.

**Recommendation:** Alternative A (strict file = module), matching the existing ish spec. This is the simplest model and aligns with OCaml. If needed, the Rust-style submodule mechanism can be added later. The directory structure mirrors the module hierarchy.

### Proposed Implementation

- Each `.ish` file defines one module.
- The module's path is derived from its filesystem path relative to the project root.
- Directories create module hierarchy (e.g., `src/net/http.ish` → module path `net/http`).
- No submodules within files in the initial implementation.

### Decisions

**Decision 12:** Should ish use strict file=module (Go/OCaml), file+submodules (Rust), or directory=package (Go packages)?
--> Strict file = module

## Feature: Package Distribution (Problem 1 + cross-cutting)

### Issues to Watch Out For

- Distribution format affects build speed (source vs. precompiled).
- ish has unique needs due to compiled execution (AST → Rust → .so).
- The existing spec already outlines a three-phase strategy (git → OCI/ORAS → registry).

### Critical Analysis

The existing spec's three-phase distribution strategy is well-considered:

1. **Phase 1 (git):** Packages are VCS repositories. Simple, leverages existing infrastructure.
2. **Phase 2 (OCI/ORAS):** Packages are OCI artifacts. Efficient binary distribution, leverages container registry infrastructure.
3. **Phase 3 (dedicated registry):** Purpose-built registry with search, discovery, documentation hosting.

This aligns well with Go's evolution (VCS → module proxy) and Rust's approach (source distribution via registry). The key question is what the package format should contain.

**Package encoding options** (from existing spec):
- Annotated AST (works for interpreter, analyzable)
- Static object code (.a/.o — fast startup but platform-specific)
- Dynamic object code (.so/.dylib — supports dynamic linking but platform-specific)
- Source code (most portable, slowest to use)

**Recommendation:** Start with source distribution in Phase 1 (git). The package format should include source code and the manifest. Precompiled artifacts can be added in later phases for performance. This matches Go's approach (source distribution via proxy, compilation happens locally).

### Proposed Implementation

- Phase 1: Packages are git repositories. The tool clones/fetches and caches.
- Package consists of: source files + manifest + checksum.
- Support zip archives as the distribution unit (like Go's module zip format).
- Platform-specific compiled artifacts as an optimization in later phases.

### Decisions

No standalone decisions — follows the existing spec's phased approach, starting with source distribution.
