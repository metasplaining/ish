## Questions and Answers

### Q: What order should modules, projects, packages, lexical scopes, and the use directive be implemented?

**Decision 1 is still pending.** The Q&A recommends the following order, which remains the working assumption:

1. **Lexical scopes** — foundation. Modules need scope boundaries; the `use` directive needs a target scope.
2. **Modules** — file-to-module mapping, module paths, visibility rules.
3. **Use directive** — importing names from one module into another.
4. **Projects** — project manifest, project directive, hierarchy lookup.
5. **Packages** — versioning, distribution, remote dependencies.

### Q: What other problems should we be aware of?

Research identified six problems not in the original RFP:

- **Cyclic dependencies** — cross-file cycles are disallowed (see D5); within-file mutual recursion is allowed.
- **Build reproducibility** — MVS ensures deterministic resolution; checksums ensure tamper detection (see D11, Traceability feature).
- **Module identity vs. path** — strict file=module (D12) and major-version path suffixes (D3) together give each module a unique, stable path.
- **Conditional compilation** — deferred; intersects with execution configurations.
- **Incremental compilation** — module boundaries are natural units; the design supports this without prescribing it.
- **Re-exports and facade modules** — the `pub` visibility level combined with the `use` directive enables re-export patterns; explicit re-export syntax is deferred.
- **Security and trust** — proxy protocol (D11) provides the distribution layer; assurance-ledger integration is addressed under Traceability.

## Feature: Visibility / Encapsulation

### Issues to Watch Out For

- Too restrictive: makes code hard to test.
- Too permissive: makes code hard to reason about.
- Assurance levels should interact with the visibility model.
- Testing often needs access to internals.

### Design

Three visibility levels:

| Keyword | Scope | Notes |
|---------|-------|-------|
| `priv` | Current module (file) only | Module-private; not visible to other files in the same project |
| `pkg` | Entire project | **Default.** Visible anywhere within the project |
| `pub` | External dependents | Required for items exported from the project |

The default visibility is `pkg` — items are project-visible without annotation. This follows the principle that code within a project is trusted; explicit `priv` is needed only when an item is an implementation detail that should not be used by other modules.

**Interface file:** Each module may have an associated interface file. At low assurance, the interface file is generated automatically from the visibility annotations and may be regenerated at any time by tooling. At high assurance, the interface file is manually maintained and serves as the authoritative declaration of the module's public contract; the compiler checks that the implementation satisfies the interface.

**Open Question:** What is the naming convention for interface files? (e.g., same stem with a different extension like `.ishi`, or a separate `interfaces/` directory?)
--> `.ishi`

**Open Question:** When a manually maintained interface file declares a symbol that does not exist in the implementation, what error is produced? Is the check performed at compile time for the module or lazily at the point of use?
--> The check is performed at compile time.  We will have separate error codes for symbol not in interface, symbol not in implementation, symbol doesn't match.  Maybe more error codes.  Error codes should be granular.

### Acceptance Tests

- A `priv` item in module A is inaccessible from module B in the same project (compile error).
- A `pkg` item in module A is accessible from module B in the same project.
- A `pub` item in module A is accessible from an external dependent project.
- A `pub` item becomes inaccessible when only `pkg` is declared (external dependent gets error).

## Feature: Forward References / Mutual Recursion

### Issues to Watch Out For

- Cyclic type definitions across modules require careful design.
- Cyclic function definitions across modules are rare but arise in callback patterns.
- The shell REPL needs a way to define mutually recursive functions interactively.

### Design

The **compilation unit** is the source file. Within a file, all definitions are mutually visible — forward references and mutual recursion are supported without restriction. The programmer does not need to order declarations within a file.

Cross-file dependency cycles are a **compile error**. The dependency graph across files must be a DAG. Error messages must show the full cycle and suggest extracting shared types into a common module.

**Shell atomic module declaration:** When working in the shell REPL, there is a syntax for declaring an entire module atomically in a single interaction. This allows the programmer to define a group of mutually recursive functions that form a module without committing each definition separately.

**Open Question:** What is the concrete syntax for the shell's atomic module declaration? How does it interact with the line-by-line input model of the REPL?
--> A mod directive with a open curly brace.  Everything until the end of the block is considered part of the same module.  This decision revises earlier decisions.  A file with no module directives is assigned to a module using the relative path of the project file.  It contains a single module.  A file with module directives can contain multiple modules whose path's bear no resemblance to the directory structure.  Forward and cyclical dependencies within a module are supported, but not between modules.  There is an assurance feature to treat module directives as a discrepancy.  Module directives are intended to be used for standalone shell script only.  That is not enforced in low assurance mode, but no effort is put into supporting projects that have multiple files and some of the files have module directives.

### Acceptance Tests

- Two functions in the same file that call each other compile and run correctly.
- Two files that import each other produce a cycle compile error with a diagnostic naming both files.
- A type defined in file A, used in file B, where file B is also imported by file A, produces a cycle error.

## Feature: Module-to-File Mapping

### Issues to Watch Out For

- The mapping must be predictable: given a module path, a developer should know which file to open.
- Directory structure should mirror module hierarchy.

### Design

Each `.ish` file defines exactly one module. The module's path is derived from its filesystem path relative to the project root. Directories create module hierarchy.

Example: `src/net/http.ish` → module path `net/http` (with a `src/` root configured in the manifest).

No submodules within files in the initial implementation.

### Acceptance Tests

- Importing `net/http` resolves to `src/net/http.ish`.
- A module path with no corresponding file produces a "module not found" error naming the expected path.

## Feature: Import Syntax / Use Directive

### Issues to Watch Out For

- Import syntax is used thousands of times per project — ergonomics matter.
- Ambiguous imports cause bugs.
- Selective imports reduce namespace pollution.

### Design

The `use` directive is hierarchical with aliasing (Rust/Python model):

```ish
use example.com/foo/bar          -- import; access as bar.Name
use example.com/foo/bar as b     -- import with alias; access as b.Name
use example.com/foo/bar { Type, func }      -- selective import into local scope
use example.com/foo/bar { Type as T }       -- selective import with rename
```
--> Example syntax incorrectly uses full path.  It is a leftover from a previous iteration of the proposal.

Module paths are URL-like for external packages and relative for within-project:

```ish
use net/http                     -- within-project import
use ./util                       -- relative import (same package subtree)
```

Glob imports (`use mod { * }`) are available but discouraged and may be restricted at higher assurance levels.

Qualified access without an explicit `use` is supported: `net/http.Get(...)` resolves the module by path without bringing it into scope.

### Acceptance Tests

- `use foo/bar` resolves `bar.Func` to the function in `foo/bar.ish`.
- `use foo/bar as b` resolves `b.Func`.
- `use foo/bar { Func }` brings `Func` into local scope.
- `use foo/bar { Func as F }` brings `F` into local scope.
- Importing a name that is not `pub` or `pkg`-visible from another project produces an access error.

## Feature: Project Directive and Hierarchy Lookup

### Issues to Watch Out For

- Shell scripts are often standalone; loading a full project manifest for a one-liner is too heavy.
- The module system must not make the shell experience worse.
- Every file must resolve to exactly one project.

### Design

Every ish file is associated with a project. The resolution proceeds as follows:

1. **Explicit directive:** If the file contains a `project` directive at the top, it names the project file to load. The path may be a filesystem path or a package path (resolved via the module system).

2. **Directory hierarchy search:** If no directive is present, ish searches up the directory hierarchy for a project file (e.g., `project.json`), starting from the directory containing the source file. The first file found wins.

3. **Installation default:** If no project file is found in the hierarchy, ish uses the default project file bundled with the ish installation. This project includes only the packages distributed with ish itself (the standard library).

This model means a single-file script in a user's home directory gets the installation default automatically, while a file inside a project directory gets that project's manifest. A file can opt into a different project explicitly via the `project` directive.

**Open Question:** When a `project` directive references a package path (not a filesystem path), that package must be fetched before the file can be processed. This creates a bootstrapping dependency: the module system is needed to load the project file that configures the module system. What is the resolution strategy for this case? (Possible answer: the `project` directive may only reference filesystem paths or packages already present in the installation default project.)
--> The `ISH_PROXY` environment variable defines how the package system resolves URL paths.  A project with URL based parents is able to download the project files of those parents without bootstrapping anything else.  The project file does not actually configure the module system.  It primarily configures dependencies, assurance levels, and publishing metadata.
--> A `project` directive may contain a file path to a JSON file, a URL path to a JSON file, or an inline JSON object.

### Acceptance Tests

- A file with no `project` directive in an isolated directory uses the installation default project.
- A file inside a directory containing `project.json` uses that project's manifest.
- A file with an explicit `project` directive uses the specified manifest regardless of directory contents.
- The directory hierarchy search stops at the filesystem root without error.

## Feature: Project Manifest Format

### Issues to Watch Out For

- The manifest format sets the tone for the entire developer experience.
- Too much required metadata discourages casual use.
- ish's multiple execution configurations need representation.
- Inheritance cycles in parent project chains must be detected.

### Design

The project manifest is a **JSON file** (`project.json`) with minimal required fields and optional extension.

**Required fields:**

```json
{
  "module": "example.com/myproject",
  "ish": "0.1.0"
}
```

**Optional fields:**

```json
{
  "module": "example.com/myproject",
  "ish": "0.1.0",
  "parents": [
    "example.com/org-defaults v1.0.0",
    "example.com/platform-bom v2.3.0"
  ],
  "dependencies": {
    "example.com/lib": "v1.2.3",
    "example.com/util": "v0.4.0"
  },
  "retract": ["v0.3.0", "v0.3.1"],
  "description": "...",
  "license": "Apache-2.0",
  "authors": ["..."],
  "repository": "https://..."
}
```
--> Everything in `project.json` is optional.
--> The `module` field should be `name`.

**Inheritance:** The `parents` field is an ordered list of project packages. Parent manifests are fetched and merged to provide defaults for optional fields not specified in the current manifest. Earlier entries in the list take precedence over later entries. The current manifest always takes final precedence over inherited values. This allows organizations to publish a project package that acts as a Bill of Materials — declaring dependency versions and assurance configuration — which child projects inherit automatically.
--> Since all fields are optional, I should clarify that some specific fields can't be imported.  Right now `name` is the only field like that can't be imported.

**Materialization tooling:** `ish mod materialize` generates a fully resolved, annotated version of the manifest showing the source of each inherited field. This is not stored; it is generated on demand for debugging and auditing.

**Open Question:** If project A declares parent B and project B declares parent A (or a longer cycle), how is this detected? Since parent projects are fetched via the module system, cycle detection requires tracking the fetch chain. What is the error message and where in the process is it reported?
--> Let's rename `parents` to `defaults` and limit it to a single value.

### Acceptance Tests

- A manifest with only `module` and `ish` fields is valid.
- A manifest with a `parents` list merges dependency versions from the parent correctly.
- A field specified in the current manifest overrides the same field in a parent.
- An unresolvable `module` path in `dependencies` produces a "module not found" error.

## Feature: Versioning (MVS)

### Issues to Watch Out For

- Pre-1.0 versions need different compatibility expectations.
- Version resolution must be deterministic across machines.
- Yanking/retracting broken versions needs support.

### Design

All packages use **semantic versioning** (major.minor.patch). Version resolution uses **Minimal Version Selection (MVS)**: among all versions that satisfy the constraints in the transitive dependency graph, always select the minimum. This makes resolution deterministic without a lockfile — the manifest files alone determine the build.

Major version changes require a new module path (e.g., `example.com/foo/v2`). Within a major version, exactly one version is selected. MVS prevents diamond dependency conflicts within a major version.

`retract` directives in the manifest allow package authors to yank broken versions. Retracted versions are skipped during resolution.

No lockfile is produced. The manifests are the source of truth.

### Acceptance Tests

- Two dependencies requiring `v1.2.0` and `v1.3.0` of a common package resolve to `v1.3.0`.
- A retracted version is not selected even if it would otherwise satisfy constraints.
- Resolution is identical across two machines given the same manifest files.

## Feature: Diamond Dependencies / Multiple Versions

### Issues to Watch Out For

- Type identity: two different versions of the same type are not compatible.
- Multiple versions bloat compiled artifacts.

### Design

Within a major version, MVS guarantees exactly one version is selected (see Versioning feature). Different major versions are treated as **distinct packages** with distinct paths. A project may depend on `example.com/foo v1.5.0` and `example.com/foo/v2 v2.1.0` simultaneously — they are different modules with different paths.

Package authors must update the module path on major version bumps. This is an explicit cost; the benefit is that code is unambiguous about which major version it uses.

### Acceptance Tests

- A project depending on both `foo v1.x` and `foo/v2 v2.x` compiles without conflict.
- Two transitive dependencies requiring different minor versions of the same major resolve to the higher version (MVS).

## Feature: Developer Workspace Flexibility

### Issues to Watch Out For

- Workspace overrides must not leak into published packages.
- Workspace mode must be compatible with CI/CD.

### Design

A workspace file (`ish.work`) lists local modules to develop together. When present, the workspace file causes ish to use local filesystem paths for the listed modules instead of fetching them from the proxy.

```json
{
  "use": [
    "./mylib",
    "./myapp"
  ]
}
```

The workspace file is added to `.gitignore` by default. Workspace overrides do not affect the published manifests of the participating modules.

Replace directives (ad-hoc single-module overrides in the manifest) are deferred to a future revision.

### Acceptance Tests

- With a workspace file pointing `example.com/mylib` to `../mylib`, `import example.com/mylib` resolves to the local directory.
- Removing the workspace file reverts to proxy resolution.

## Feature: Dependency Management (Emergent)

This feature is an emergent property of MVS, the proxy protocol, and the manifest format. No independent design decisions are required.

Tooling:

- `ish mod tidy` — adds missing and removes unused dependencies from the manifest.
- `ish mod add <module>[@version]` — adds a dependency.
- `ish mod remove <module>` — removes a dependency.
- `ish mod graph` — prints the dependency graph.
- `ish mod why <module>` — explains why a module is in the dependency graph.
- `ish mod verify` — verifies checksums of downloaded modules.
- `ish mod materialize` — generates annotated manifest with inherited field sources.

## Feature: Dependency Inheritance / Parent Projects

### Issues to Watch Out For

- Parent project inheritance is distinct from package dependencies.
- Assurance level configuration must compose correctly across parents.

### Design

The `parents` field in the manifest (see Project Manifest Format) declares **parent projects**, which are regular packages distributed through the same proxy system. Parents are not code dependencies — they contribute manifest configuration, dependency version defaults, and assurance level settings to the child project.

Use cases:
- **Organization BOM:** An organization publishes a package (containing only a `project.json`) that declares approved versions of common dependencies. All project manifests in the organization list this as a parent.
- **Assurance configuration:** A parent project declares the assurance level requirements for the organization. All child projects inherit these requirements.
- **Toolchain defaults:** A parent project configures the default ish version and execution configuration.

Inheritance uses the ordered-merge semantics described in the manifest format feature.

**Open Question:** If parent projects specify conflicting assurance levels, which takes precedence? The ordered list implies the first parent wins, but this should be stated explicitly in the spec. Does the child project's own assurance setting always win over all parents, even if it is lower?
--> Multiple inheritance is removed.  The projects own values take precedence over direct dependency, which takes precedence over transitive dependencies.

### Acceptance Tests

- A project with a parent that declares `dep/foo v2.0.0` defaults to that version if the child does not specify `dep/foo`.
- A project that explicitly declares `dep/foo v1.5.0` overrides the parent's declaration.

## Feature: Physical Downloading and Caching

### Issues to Watch Out For

- Builds must work offline after initial fetch.
- Downloaded code must be verified against tampering.

### Design

ish uses a **proxy protocol** for package distribution (similar to Go's GOPROXY). Module paths are URL-like; the tool resolves them through a configurable proxy chain.

- **Cache directory:** `~/.ish/cache/` stores downloaded modules as content-addressed archives.
- **Checksum verification:** All downloaded modules are verified against a checksum file. The checksum file is stored alongside the manifest.
- **Proxy configuration:** The `ISH_PROXY` environment variable configures the proxy chain (e.g., `https://proxy.ish.example.com,direct`). `direct` means VCS resolution.
- **Private modules:** `ISH_PRIVATE` configures module path prefixes that bypass proxy and checksum servers.
- **Vendoring:** `ish mod vendor` copies all dependencies into a `vendor/` directory for hermetic builds.

### Acceptance Tests

- A module downloaded once is served from cache on subsequent builds (no network access).
- A tampered module (mismatched checksum) produces a "checksum mismatch" error.
- With `ISH_PROXY=off`, resolution uses VCS directly.

## Feature: Traceability

### Issues to Watch Out For

- Traceability must work across the assurance continuum.
- Build provenance must be embeddable in compiled artifacts.

### Design

All three traceability mechanisms are adopted:

1. **Checksum file** — a file alongside the manifest records cryptographic hashes of all dependencies. Used for tamper detection and reproducibility verification.

2. **Embedded build info** — compiled artifacts embed the module path, version, and dependency hashes used to build them. This is required for high-assurance execution; it enables auditing any deployed artifact.

3. **Dependency graph tools** — `ish mod graph`, `ish mod why`, and `ish mod verify` are provided (see Dependency Management).

**Assurance integration:** Dependencies at a given assurance level must themselves be at the same or higher assurance level. The analyzer enforces this. A high-assurance module cannot take a dependency on a low-assurance package without explicit annotation lowering the assurance boundary.

## Feature: Package Distribution

### Issues to Watch Out For

- Distribution format affects build speed (source vs. precompiled).
- ish has unique needs due to compiled execution (AST → Rust → `.so`).

### Design

Distribution follows a three-phase evolution:

1. **Phase 1 (git):** Packages are git repositories. The tool fetches source and manifest. Compilation happens locally.
2. **Phase 2 (OCI/ORAS):** Packages are OCI artifacts served via container registry infrastructure. Supports precompiled artifacts for faster builds.
3. **Phase 3 (dedicated registry):** Purpose-built registry with search, documentation hosting, and discovery.

Phase 1 is the target for prototype implementation. Package format: source files + `project.json` + checksum file, distributed as a zip archive (similar to Go's module zip format).

--> This is complicated.  We will need to fully specify it later.  But for now, there are some key constraints to understand:
--> 1. Some ish packages are compiled from ish source. Some ish packages are compiled from Rust.  Therefore, ish source format is not available for every package.
--> 2. Some of the most import standard library packages (I/O) are Rust based.  They MUST be supported.
--> 3. In order for the interpreter to support Rust based package,  it must be available as a shared library.
--> 4. In order for the compiler to support a Rust based package, it must be available in a .a format.
--> 5. In order for the interpreter to support an ish based package before the compiler becomes available, it must be available in either source or AST format.
--> These constraints combine to give us the minimum viable packaging system:
--> - Interpreter loads ish projects from source.
--> - Interpreter loads shared library packages.
--> - Rust based standard library builds into shared library packages.
--> The AST package format is a trivial addition to the minimum viable packaging system, and will probably be implemented right after it.

## Split Evaluation

**Decision: Split into Proposal A (Module Core) and Proposal B (Package Management)?**
--> Yes

This proposal contains 14 independent implementation steps. The natural split is:

**Proposal A — Module Core** (can be implemented with no network dependencies):
1. Visibility / Encapsulation
2. Forward References / Mutual Recursion
3. Module-to-File Mapping
4. Import Syntax / Use Directive
5. Project Directive & Hierarchy Lookup (filesystem search + installation default only)

**Proposal B — Package Management** (builds on Proposal A):
6. Project Manifest Format (JSON + inheritance)
7. Versioning (MVS)
8. Diamond Dependencies (major version path suffixes)
9. Physical Downloading & Caching (proxy protocol)
10. Developer Workspace Flexibility
11. Dependency Management Tools (`tidy`, `add`, `remove`, `graph`, `why`, `verify`, `materialize`)
12. Dependency Inheritance / Parent Projects
13. Traceability (checksums, embedded build info)
14. Package Distribution (source via proxy, Phase 1)

The split resolves **Decision 1** (implementation order): Proposal A first, then Proposal B.
