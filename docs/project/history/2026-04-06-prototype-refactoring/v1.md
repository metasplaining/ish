## Scope Note

The accepted [stubbed-analyzer proposal](stubbed-analyzer.md) (Decision 3) established that
the yielding/unyielding split should use "parallel structure for match arms" while extracting
"pure computation into helpers." H1 below is consistent with that decision — it proposes only
to extract helpers for arms whose bodies are *already identical* between the two variants, not
to collapse the parallel structure.

## Feature H1: Extract Shared Helpers from Yielding/Unyielding Interpreter Pair

### Issues to Watch Out For

- The parallel yielding/unyielding structure is a deliberate design decision (stubbed-analyzer
  Decision 2 and 3). Any helper must be purely synchronous — it cannot call back into either
  interpreter variant.
- Extracted helpers that take `&mut Environment` or `&Value` must be careful about borrow
  lifetimes. The interpreter uses `Rc<RefCell<IshVm>>` pervasively; a helper must not attempt
  to hold a borrow across a call boundary.
- The `Statement::Assignment` arms differ subtly between yielding and unyielding: they both
  perform the same structural checks but call different evaluation functions (`eval_expression_yielding`
  vs. `eval_expression_unyielding`). A shared helper can encapsulate only the post-evaluation
  side-effect logic (e.g., writing to `obj_ref`, bounds checking), not the evaluation itself.
- Test coverage: all four functions (`exec_statement_*`, `eval_expression_*`) are covered by
  existing unit and acceptance tests. Extract, run tests, verify no regressions.

### Critical Analysis

The four functions (`exec_statement_yielding` 667 lines, `exec_statement_unyielding` 498 lines,
`eval_expression_yielding` 437 lines, `eval_expression_unyielding` 277 lines) total ~1,880
lines. A bug fixed in one variant is routinely missed in the other — this has already happened
at least once with the `AssignTarget::Index` bounds check.

**Alternative A — Extract shared helpers (recommended)**

Extract `fn` helpers for match-arm bodies that are identical between the two variants.
Concrete candidates:

1. `fn eval_literal(lit: &Literal) -> Value` — the six-arm `Literal` match at the top of both
   `eval_expression_*` functions is character-for-character identical. A helper eliminates one
   copy.
2. `fn apply_index_write(obj: &Value, idx: &Value, val: Value) -> Result<(), RuntimeError>` —
   the `AssignTarget::Index` arm in both `exec_statement_*` functions (lists and objects) is
   identical; the only thing that differs is how `obj` and `idx` were evaluated. A shared helper
   receives the post-evaluation values and performs the write.
3. `fn apply_property_write(obj: &Value, property: &str, val: Value) -> Result<(), RuntimeError>` —
   same argument as (2) for `AssignTarget::Property`.

Helpers (1)–(3) together reduce the duplicated line count by roughly 80 lines.

*Pros:* Directly eliminates the identified bug-replication risk. Small, localized changes.
Consistent with stubbed-analyzer Decision 3. Tests pass unchanged.

*Cons:* Doesn't solve the larger duplication problem — ~1,800 lines of parallel structure
remain. A future reader still sees two giant match blocks.

**Alternative B — Macro-based dispatch**

Define a single `exec_statement!` macro that expands to either the async or sync variant by
substituting `await?` vs. `?`. This has been considered before (see concurrency design) and
rejected because macro-heavy interpreter code becomes very hard to debug. Not recommended for
a prototype.

**Alternative C — Status quo**

Leave the four functions unchanged. The parallel structure is documented and the test suite
catches divergence. Cost: continued manual synchronization burden.

*Pros:* Zero risk of introducing regressions. Zero scope.
*Cons:* The identified bug-replication risk remains.

**Recommendation:** Alternative A, targeting the three helper extractions listed above.
Additional arms can be extracted in follow-up passes.

### Proposed Implementation

All changes in `proto/ish-vm/src/interpreter.rs`.

1. Add three private `fn` helpers in the `impl IshVm` block (or as free functions in the
   same file, below the interpreter impl):
   ```rust
   fn eval_literal(lit: &Literal) -> Value { ... }
   fn apply_property_write(obj: &Value, property: &str, val: Value) -> Result<(), RuntimeError> { ... }
   fn apply_index_write(obj: &Value, idx: &Value, val: Value) -> Result<(), RuntimeError> { ... }
   ```
2. In `eval_expression_yielding` and `eval_expression_unyielding`, replace the `Expression::Literal`
   arm bodies with `Ok(Self::eval_literal(lit))`.
3. In `exec_statement_yielding` and `exec_statement_unyielding`, replace the
   `AssignTarget::Property` and `AssignTarget::Index` bodies with calls to
   `Self::apply_property_write(...)` and `Self::apply_index_write(...)`.
4. Run `cargo test --workspace` to confirm no regressions.

No new files. No changes to any other crate.

### Decisions

**Decision:** Should shared helpers (eval_literal, apply_property_write, apply_index_write) be
extracted from the interpreter pair, or leave the parallel structure as-is?
--> shared helpers (eval_literal, apply_property_write, apply_index_write)

**Decision:** Are there additional arms worth extracting in the same pass (e.g., the
`Expression::UnaryOp` arm, which is also identical between the two variants)?
--> Research further and propose candidates.

## Feature H2: Replace Panicking `unwrap()` Calls in ast_builder.rs

### Issues to Watch Out For

- The `ast_builder.rs` uses `inner.next().unwrap()` on pest `Pairs` iterators in ~80 of the
  102 sites. These are grammar-structural guarantees — the rule matched means the child token
  must exist. If the grammar changes, these become silent panics.
- Two sites use `.parse::<i64>().unwrap()` and `.parse::<f64>().unwrap()` (lines 1564, 1568).
  These are not grammar-structural — an integer literal that overflows `i64` would panic here.
  This is a real (if unlikely) user-visible failure mode.
- `lines.last().unwrap()` (line 1445) in `strip_triple_quote_indentation` is called on a
  non-empty `Vec` that was just constructed — safe in context, but brittle if the surrounding
  logic changes.
- `value.unwrap()` (line 120) is used where `value` is populated by a `for` loop over inner
  pairs. If the grammar rule changes to make the value optional, this becomes a panic.
- Converting all 102 sites to `Result` propagation is a large change and will require updating
  all call sites to handle the new error type.

### Critical Analysis

**Alternative A — Introduce `expect_next` helper + fix numeric parse sites**

Add one helper:
```rust
fn expect_next<'i>(pairs: &mut Pairs<'i, Rule>, ctx: &str) -> Result<Pair<'i, Rule>, ParseError> {
    pairs.next().ok_or_else(|| ParseError {
        message: format!("internal: expected token in '{}'", ctx),
        span: None,
    })
}
```
Replace all `inner.next().unwrap()` with `expect_next(&mut inner, "<rule>")`.
Replace `.parse::<i64>().unwrap()` with `.parse::<i64>().map_err(|_| ParseError { ... })`.

This converts all 102 panic sites to structured `ParseError` results.

*Pros:* Eliminates all panic sites from the parser's production path. Grammar evolution is
safe — a wrong grammar produces a `ParseError`, not a panic. Consistent with the Rust
convention for library code.

*Cons:* Large change (~102 sites). Every call site must propagate `?`. Some of the iterator
calls are inside closures passed to `.map()`, requiring those closures to return `Result`.
High mechanical cost for what is mostly a prototype-stage concern.

**Alternative B — Fix only the non-grammar-structural panics**

Target only the sites where the panic is *not* guaranteed by the grammar:
- `lines.last().unwrap()` (line 1445)
- `.parse::<i64>().unwrap()` (line 1564) — overflow risk
- `.parse::<f64>().unwrap()` (line 1568) — overflow risk
- `value.unwrap()` (line 120) — `Option` from a loop that depends on the grammar

Leave the ~95 `inner.next().unwrap()` sites as grammar-structural panics.

*Pros:* Small, targeted change (~5–8 sites). Eliminates the real runtime risks while avoiding
the large mechanical refactor. Appropriate for prototype stage.

*Cons:* Grammar-structural panics remain. If the grammar is refactored, these will surface as
panics during development (not in user hands), which is acceptable.

**Alternative C — Status quo**

*Pros:* Zero risk.
*Cons:* Numeric overflow panics remain as user-visible failure modes.

**Recommendation:** Alternative B for this proposal. Alternative A is appropriate to revisit
when the parser is considered production-ready.

### Proposed Implementation

All changes in `proto/ish-parser/src/ast_builder.rs`.

1. Line 1564: `n: i64 = inner.as_str().parse().unwrap()` →
   ```rust
   n: i64 = inner.as_str().parse().map_err(|_| ParseError {
       message: format!("integer literal '{}' out of range", inner.as_str()),
       span: Some(inner.as_span()),
   })?
   ```
2. Line 1568: Same pattern for `f64`.
3. Line 1445: `lines.last().unwrap()` → `lines.last().unwrap_or(&String::new())` (no-op
   for non-empty input; removes the unwrap without changing behavior).
4. Line 120: `value: value.unwrap()` — the grammar guarantees a value in a `VariableDecl`, but
   add `.ok_or_else(|| ParseError { message: "variable declaration missing value", ... })?`
   and propagate via `?`.

No new files.

### Decisions

**Decision:** Should this proposal address only the non-grammar-structural unwraps (Alt B), or
do the full 102-site conversion to `expect_next` (Alt A)?
--> address only the non-grammar-structural unwraps

## Feature M1: Reduce `register_ast_builtins` Boilerplate in reflection.rs

### Issues to Watch Out For

- `ast_literal` has a special case: it inspects `args[0]` to determine `literal_type` and adds
  an extra `literal_type` field. Any helper must accommodate builtins with custom body logic
  beyond simple field mapping.
- `new_compiled_function` is in `ish-runtime` and is the correct low-level primitive. The
  helper proposed here would sit in `ish-vm/src/reflection.rs` as a local `fn`.

### Critical Analysis

**Alternative A — Local helper function (recommended)**

```rust
fn simple_ast_builtin(name: &'static str, arity: usize, fields: &'static [&'static str]) -> Value {
    new_compiled_function(name, vec![], vec![], None, move |args| {
        if args.len() != arity {
            return Err(RuntimeError::system_error(
                format!("{} expects {} argument(s)", name, arity),
                ErrorCode::TypeMismatch,
            ));
        }
        let mut map = HashMap::new();
        map.insert("kind".to_string(), str_val(name.trim_start_matches("ast_")));
        for (field, arg) in fields.iter().zip(args.iter()) {
            map.insert(field.to_string(), arg.clone());
        }
        Ok(Value::Object(Gc::new(GcCell::new(map))))
    }, Some(false))
}
```

Builtins with custom logic (like `ast_literal`) keep their current inline form. The ~16
structurally simple builtins use `simple_ast_builtin`. Estimated reduction: 380 → ~120 lines.

*Pros:* Reduces boilerplate substantially. Makes adding a new AST node type a one-liner.
*Cons:* The `kind` field derivation (`trim_start_matches("ast_")`) is a naming convention
dependency — the builtin name must always be `ast_<kind>`. This is currently true but is an
implicit contract.

**Alternative B — Status quo**

*Pros:* Zero risk of introducing a naming-convention dependency.
*Cons:* Adding a new AST node type requires copy-pasting a ~15-line block.

**Recommendation:** Alternative A for the structurally simple builtins; keep `ast_literal`
and any other builtins with custom logic as inline closures.

### Proposed Implementation

In `proto/ish-vm/src/reflection.rs`:

1. Add `fn simple_ast_builtin(...)` before `register_ast_builtins`.
2. Replace the ~16 structurally simple builtin registrations with calls to `simple_ast_builtin`.
3. Leave `ast_literal` (and any others with non-trivial bodies) as-is.
4. Run `cargo test --workspace`.

No new files.

### Decisions

**Decision:** Should `simple_ast_builtin` be introduced, or leave `register_ast_builtins`
as-is?
--> `simple_ast_builtin`

**Decision:** If introduced, is the naming-convention dependency (`ast_<kind>`) acceptable?
If not, should the `kind` string be passed explicitly as a separate argument?
--> The naming convention is acceptable.

## Feature M2: Reduce Arity-Check Boilerplate in builtins.rs

### Issues to Watch Out For

- `new_builtin` would hard-code `vec![], vec![], None` for params/param_types/return_type.
  If a future builtin needs typed parameters (for analyzer integration), the wrapper would
  need updating. For now, no builtins use typed parameters.
- The `arity` helper is trivial but removes a repeated pattern. It must return `Result<(), RuntimeError>`
  so it can be used with `?`.

### Critical Analysis

**Alternative A — `arity` helper + `new_builtin` wrapper (recommended)**

```rust
fn arity(name: &str, args: &[Value], n: usize) -> Result<(), RuntimeError> {
    if args.len() != n {
        return Err(RuntimeError::system_error(
            format!("{} expects {} argument(s)", name, n),
            ErrorCode::ArgumentCountMismatch,
        ));
    }
    Ok(())
}

fn new_builtin(
    name: &'static str,
    f: impl Fn(&[Value]) -> Result<Value, RuntimeError> + 'static,
) -> Value {
    new_compiled_function(name, vec![], vec![], None, f, Some(false))
}
```

*Pros:* Removes the most repetitive boilerplate (~30 arity-check blocks). Makes individual
builtin closures shorter and easier to read.

*Cons:* `new_builtin` always sets `has_yielding_entry: Some(false)`. If a builtin needs
`Some(true)` or `None`, the wrapper doesn't fit. Currently no builtin needs this.

**Alternative B — Status quo**

*Pros:* Zero risk.
*Cons:* Every new builtin requires 3 extra lines for the arity check.

**Recommendation:** Alternative A.

### Proposed Implementation

In `proto/ish-vm/src/builtins.rs`:

1. Add `fn arity(...)` and `fn new_builtin(...)` at the top of the file.
2. Convert all `register_strings`, `register_lists`, `register_objects`, `register_conversion`,
   and `register_io` builtins to use `arity(name, args, n)?` and `new_builtin(name, |args| ...)`.
3. Run `cargo test --workspace`.

No new files.

### Decisions

**Decision:** Should `arity` and `new_builtin` helpers be added to builtins.rs?
--> Yes

## Feature M3: Extract Brace-Scanning Helper in `interpolate_shell_quoted`

### Issues to Watch Out For

- `interpolate_shell_quoted` is a pure function operating on `chars: Vec<char>`. Extracting
  a helper is safe — no shared state, no async, no `Rc<RefCell>`.
- The bare `{var}` syntax (no `$` prefix) is distinct from Bash conventions. Its interaction
  with `${var}` must remain unchanged.

### Critical Analysis

**Alternative A — Extract `scan_to_close_brace` helper**

```rust
fn scan_to_close_brace(chars: &[char], start: usize) -> Option<(String, usize)> {
    let mut j = start;
    while j < chars.len() && chars[j] != '}' {
        j += 1;
    }
    if j < chars.len() {
        Some((chars[start..j].iter().collect(), j + 1))
    } else {
        None
    }
}
```

Call from both the `${var}` branch (start = i+2) and the `{var}` branch (start = i+1).
Also remove the `.unwrap()` on `name.chars().next()` (line 3247) by using `name.starts_with(...)`.

*Pros:* Removes duplicate brace-scanning loops (~15 lines). Eliminates a minor panic site.
*Cons:* Small gain for a 63-line function; may not be worth the churn.

**Alternative B — Status quo**

*Pros:* Zero risk. The function is short enough to be readable as-is.
*Cons:* The inline `.unwrap()` at line 3247 remains.

**Recommendation:** Alternative A, but only because it also removes the `.unwrap()` panic
site. The deduplication benefit alone would not justify the change.

### Proposed Implementation

In `proto/ish-vm/src/interpreter.rs`:

1. Add `fn scan_to_close_brace(chars: &[char], start: usize) -> Option<(String, usize)>`
   as a free function near `interpolate_shell_quoted`.
2. Replace both brace-scanning inline loops with calls to `scan_to_close_brace`.
3. Replace `name.chars().next().unwrap().is_ascii_alphabetic()` with
   `name.starts_with(|c: char| c.is_ascii_alphabetic() || c == '_')`.

No new files.

### Decisions

**Decision:** Should `scan_to_close_brace` be extracted, or leave `interpolate_shell_quoted`
as-is?
--> extract `scan_to_close_brace`

## Sequencing

If all features are approved, the recommended implementation order is:

1. **L1** — Trivial constant; no risk.
2. **M3** — Isolated free function; touches only `interpolate_shell_quoted`.
3. **M2** — Builtin helpers; all changes within `builtins.rs`.
4. **M1** — AST builtin helper; all changes within `reflection.rs`.
5. **H2** — Parser unwrap fixes; changes within `ast_builder.rs`.
6. **H1** — Interpreter helpers; highest-complexity change, do last.

Each step is independently testable with `cargo test --workspace`.
